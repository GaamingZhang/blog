# Kubernetes Deployment滚动更新时502/504错误排查与解决

## 问题现象

在Kubernetes Deployment滚动更新过程中，客户端偶尔会遇到502 Bad Gateway或504 Gateway Timeout错误：

```
502 Bad Gateway
504 Gateway Timeout
```

这些错误通常发生在：
- 新版本发布时
- Pod扩缩容时
- 节点维护时

## 问题原因分析

### 502错误的根本原因

502错误表示网关（Ingress或Service）无法从上游服务（Pod）获得有效响应。在滚动更新场景下，主要原因是：

**请求被路由到已终止的Pod**

```
时间线:
T1: Pod-A 正在处理请求（需要5秒）
T2: 滚动更新开始，Pod-A收到SIGTERM信号
T3: Pod-A立即停止接收新请求，但仍在处理旧请求
T4: Ingress/Service仍将请求路由到Pod-A
T5: 请求失败，返回502
```

### 504错误的根本原因

504错误表示网关等待上游服务响应超时。主要原因：

**新Pod未就绪就接收流量**

```
时间线:
T1: 新Pod-B启动
T2: 容器进程启动，但应用还在初始化
T3: Service将Pod-B加入Endpoints
T4: 请求被路由到Pod-B
T5: 应用未准备好，无法响应
T6: 超时，返回504
```

## 完整的请求流程

理解问题需要先理解完整的请求流程：

```
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│ Client  │────→│ Ingress │────→│ Service │────→│   Pod   │
└─────────┘     └─────────┘     └─────────┘     └─────────┘
                    │               │               │
                    ↓               ↓               ↓
              Ingress Controller   kube-proxy    应用容器
              (nginx/traefik)    (iptables/IPVS)
```

**关键组件**：
1. **Ingress Controller**：外部流量入口，维护后端Pod列表
2. **Service**：通过Endpoints维护健康的Pod列表
3. **kube-proxy**：将Service规则同步到节点
4. **Pod**：实际处理请求的应用容器

## 问题场景详解

### 场景1：Pod终止时的竞态条件

```
┌─────────────────────────────────────────────────────────────┐
│                     Pod终止流程                              │
│                                                              │
│  API Server标记Pod为Terminating                              │
│         │                                                    │
│         ├──────────────────────┐                            │
│         ↓                      ↓                            │
│  Kubelet侧                 Endpoints Controller侧            │
│         │                      │                            │
│  执行preStop Hook          从Endpoints移除Pod               │
│         │                      │                            │
│  发送SIGTERM                 更新iptables规则               │
│         │                      │                            │
│  等待容器退出               kube-proxy同步规则              │
│         │                      │                            │
│         │                      │                            │
│         └──────────────────────┘                            │
│                    ↓                                         │
│            【竞态条件】                                       │
│                                                              │
│  问题：iptables规则更新可能慢于Pod停止                       │
│  结果：请求仍被路由到已停止的Pod                              │
└─────────────────────────────────────────────────────────────┘
```

### 场景2：新Pod启动时的就绪问题

```
┌─────────────────────────────────────────────────────────────┐
│                     Pod启动流程                              │
│                                                              │
│  Pod创建                                                     │
│         │                                                    │
│  容器启动                                                    │
│         │                                                    │
│  应用初始化（可能很慢）                                       │
│         │                                                    │
│  ReadinessProbe开始检查                                      │
│         │                                                    │
│  【问题窗口】                                                 │
│  如果没有正确配置ReadinessProbe，                             │
│  Pod可能在应用未就绪时就被加入Endpoints                        │
│         │                                                    │
│  ReadinessProbe成功                                          │
│         │                                                    │
│  Pod加入Endpoints                                            │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 场景3：Ingress Controller的延迟

```
┌─────────────────────────────────────────────────────────────┐
│              Ingress Controller更新流程                      │
│                                                              │
│  Endpoints更新                                               │
│         │                                                    │
│  Ingress Controller监听到变化                                │
│         │                                                    │
│  更新内部后端列表                                             │
│         │                                                    │
│  【延迟窗口】                                                 │
│  Nginx需要reload配置                                         │
│  这个过程可能需要几秒                                         │
│         │                                                    │
│  新配置生效                                                   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 解决方案

### 方案1：配置preStop Hook

在Pod终止前等待一段时间，让Endpoints更新传播：

```yaml
lifecycle:
  preStop:
    exec:
      command:
      - /bin/sh
      - -c
      - "sleep 10"
```

**原理**：
```
Pod收到删除请求
    ↓
执行preStop Hook（sleep 10）
    ↓
【这10秒内】
    - Endpoints Controller移除Pod
    - kube-proxy更新iptables
    - Ingress Controller更新后端列表
    ↓
发送SIGTERM给容器
    ↓
容器优雅关闭
```

### 方案2：配置ReadinessProbe

确保只有真正就绪的Pod才接收流量：

```yaml
readinessProbe:
  httpGet:
    path: /health/ready
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 5
  failureThreshold: 3
  successThreshold: 1
```

**关键点**：
- `/health/ready` 应该检查应用是否真正准备好
- 包括数据库连接、缓存预热等
- 不要只检查进程是否存在

### 方案3：配置minReadySeconds

给新Pod一个稳定期：

```yaml
spec:
  minReadySeconds: 30
```

**作用**：Pod就绪后等待30秒，才被认为"可用"，才会继续滚动更新。

### 方案4：优化滚动更新策略

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1
    maxUnavailable: 0
```

**说明**：
- `maxUnavailable: 0`：确保始终有足够的Pod可用
- `maxSurge: 1`：允许临时多一个Pod，加快更新

### 方案5：配置terminationGracePeriodSeconds

```yaml
spec:
  terminationGracePeriodSeconds: 60
```

给Pod足够的时间优雅关闭。

### 完整配置示例

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  minReadySeconds: 30
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  template:
    spec:
      terminationGracePeriodSeconds: 60
      containers:
      - name: nginx
        image: nginx:1.24
        ports:
        - containerPort: 8080
        readinessProbe:
          httpGet:
            path: /health/ready
            port: 8080
          initialDelaySeconds: 10
          periodSeconds: 5
          failureThreshold: 3
        livenessProbe:
          httpGet:
            path: /health/live
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 10
        lifecycle:
          preStop:
            exec:
              command:
              - /bin/sh
              - -c
              - "sleep 15 && nginx -s quit"
```

## 应用层面的优化

### 优雅关闭实现

应用需要正确处理SIGTERM信号：

**Go示例**：

```go
package main

import (
    "context"
    "net/http"
    "os"
    "os/signal"
    "syscall"
    "time"
)

func main() {
    server := &http.Server{Addr: ":8080"}
    
    http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        w.Write([]byte("Hello"))
    })
    
    go server.ListenAndServe()
    
    quit := make(chan os.Signal, 1)
    signal.Notify(quit, syscall.SIGTERM, syscall.SIGINT)
    <-quit
    
    ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
    defer cancel()
    
    server.Shutdown(ctx)
}
```

**Java Spring Boot示例**：

```yaml
server:
  shutdown: graceful
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s
```

**Node.js示例**：

```javascript
const http = require('http');
const server = http.createServer((req, res) => {
    res.end('Hello');
});

server.listen(8080);

process.on('SIGTERM', () => {
    server.close(() => {
        process.exit(0);
    });
});
```

### 健康检查端点实现

```yaml
readinessProbe:
  httpGet:
    path: /health/ready
    port: 8080

livenessProbe:
  httpGet:
    path: /health/live
    port: 8080
```

**健康检查逻辑**：

```go
func readinessHandler(w http.ResponseWriter, r *http.Request) {
    if !isReady {
        w.WriteHeader(http.StatusServiceUnavailable)
        return
    }
    w.WriteHeader(http.StatusOK)
}

func livenessHandler(w http.ResponseWriter, r *http.Request) {
    w.WriteHeader(http.StatusOK)
}
```

## Ingress Controller优化

### Nginx Ingress优化

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: nginx-ingress-controller
  namespace: ingress-nginx
data:
  proxy-connect-timeout: "10"
  proxy-read-timeout: "60"
  proxy-send-timeout: "60"
  proxy-next-upstream: "error timeout http_502 http_503 http_504"
  proxy-next-upstream-tries: "3"
```

**关键配置**：
- `proxy-next-upstream`：失败时尝试下一个后端
- `proxy-next-upstream-tries`：最大重试次数

### Traefik配置

```yaml
apiVersion: traefik.containo.us/v1alpha1
kind: ServersTransport

    responseHeaderTimeout: 60s

    interval: 10s
```

## 排查方法

### 1. 检查Pod状态

```bash
kubectl get pods -w
kubectl describe pod <pod-name>
kubectl logs <pod-name> --previous
```

### 2. 检查Endpoints

```bash
kubectl get endpoints <service-name>
kubectl describe endpoints <service-name>
```

### 3. 检查事件

```bash
kubectl get events --sort-by='.lastTimestamp'
```

### 4. 检查Ingress日志

```bash
kubectl logs -n ingress-nginx <ingress-controller-pod>
```

### 5. 模拟请求

```bash
for i in {1..100}; do
  curl -s -o /dev/null -w "%{http_code}\n" http://my-service/
done
```

## 最佳实践总结

### 检查清单

| 配置项 | 推荐值 | 说明 |
|--------|--------|------|
| readinessProbe | 必须配置 | 确保Pod真正就绪 |
| livenessProbe | 必须配置 | 检测应用是否存活 |
| preStop | sleep 10-15s | 等待Endpoints更新 |
| minReadySeconds | 30s | Pod稳定期 |
| terminationGracePeriodSeconds | 60s | 优雅关闭超时 |
| maxUnavailable | 0 | 零停机更新 |
| maxSurge | 1 | 临时额外Pod |

### 排查流程

```
1. 确认问题发生时间
    ↓
2. 检查Pod状态变化
    kubectl get pods -w
    ↓
3. 检查Endpoints变化
    kubectl get endpoints -w
    ↓
4. 检查Ingress日志
    kubectl logs <ingress-controller>
    ↓
5. 检查应用日志
    kubectl logs <pod-name>
    ↓
6. 分析配置是否正确
    - readinessProbe
    - preStop
    - minReadySeconds
```
