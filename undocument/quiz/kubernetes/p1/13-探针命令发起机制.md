# Kubernetes探针命令的发起机制

## 探针由谁发起？

在Kubernetes中，探针（Liveness、Readiness、Startup Probe）的健康检查命令是由**kubelet**发起的。

```
┌─────────────────────────────────────────────────────────────┐
│                        节点架构                              │
│                                                              │
│  ┌─────────────────────────────────────────────────────┐    │
│  │                   kubelet                            │    │
│  │                                                      │    │
│  │  ┌─────────────────────────────────────────────┐    │    │
│  │  │           Probe Manager                     │    │    │
│  │  │                                             │    │    │
│  │  │  - 存活探针管理器                            │    │    │
│  │  │  - 就绪探针管理器                            │    │    │
│  │  │  - 启动探针管理器                            │    │    │
│  │  └─────────────────────────────────────────────┘    │    │
│  │                        │                             │    │
│  │                        ↓                             │    │
│  │              执行探针检查命令                         │    │
│  └─────────────────────────────────────────────────────┘    │
│                            │                                 │
│                            ↓                                 │
│  ┌─────────────────────────────────────────────────────┐    │
│  │              容器运行时 (CRI)                         │    │
│  │              containerd / docker                     │    │
│  └─────────────────────────────────────────────────────┘    │
│                            │                                 │
│                            ↓                                 │
│  ┌─────────────────────────────────────────────────────┐    │
│  │                   容器                               │    │
│  │              应用进程                                │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## kubelet的探针管理器

kubelet内部有专门管理探针的组件：

### ProbeManager结构

```go
type ProbeManager interface {
    AddPod(pod *v1.Pod)
    RemovePod(pod *v1.Pod)
    CleanupPods(desiredPods map[types.UID]sets.Empty)
}

type probeManager struct {
    livenessManager  *livenessManager
    readinessManager *readinessManager
    startupManager   *startupManager
    prober           *prober
}
```

### 探针工作流程

```
1. Pod创建
    ↓
2. kubelet监听到Pod事件
    ↓
3. ProbeManager.AddPod(pod)
    ↓
4. 为每个容器创建探针Worker
    ↓
5. Worker定期执行探针检查
    ↓
6. 根据检查结果更新Pod状态
```

## 探针的执行方式

### HTTP探针

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
    scheme: HTTP
    httpHeaders:
    - name: Custom-Header
      value: value
```

**执行过程**：

```
kubelet
    ↓
构造HTTP请求
    ↓
通过容器网络发送请求到容器
    ↓
容器返回响应
    ↓
kubelet判断响应状态码
```

**关键点**：
- 请求从kubelet进程发出
- 目标是容器的IP:端口
- 使用容器网络命名空间

### TCP探针

```yaml
livenessProbe:
  tcpSocket:
    port: 8080
```

**执行过程**：

```
kubelet
    ↓
尝试建立TCP连接
    ↓
连接成功 → 探针成功
连接失败 → 探针失败
```

### Exec探针

```yaml
livenessProbe:
  exec:
    command:
    - /bin/sh
    - -c
    - "test -f /tmp/healthy"
```

**执行过程**：

```
kubelet
    ↓
通过CRI接口调用Exec
    ↓
containerd/docker在容器内执行命令
    ↓
返回命令退出码
    ↓
退出码0 → 探针成功
非0退出码 → 探针失败
```

### gRPC探针

```yaml
livenessProbe:
  grpc:
    port: 50051
    service: liveness
```

**执行过程**：

```
kubelet
    ↓
发送gRPC健康检查请求
    ↓
容器内gRPC服务响应
    ↓
根据响应判断健康状态
```

## 探针执行的技术细节

### HTTP探针的实现

kubelet使用Go的HTTP客户端发送请求：

```go
func (pb *prober) probeHTTP(probe *v1.HTTPGetAction) (probe.Result, error) {
    url := formatURL(probe.Host, probe.Port, probe.Path)
    
    client := &http.Client{
        Timeout: time.Duration(probeTimeoutSeconds) * time.Second,
    }
    
    resp, err := client.Get(url.String())
    if err != nil {
        return probe.Failure, err
    }
    
    if resp.StatusCode >= 200 && resp.StatusCode < 400 {
        return probe.Success, nil
    }
    
    return probe.Failure, nil
}
```

### Exec探针的实现

kubelet通过CRI接口执行命令：

```go
func (pb *prober) probeExec(probe *v1.ExecAction) (probe.Result, error) {
    // 通过CRI执行命令
    result, err := pb.runner.RunInContainer(
        containerID,
        probe.Command,
        time.Duration(probeTimeoutSeconds)*time.Second,
    )
    
    if err != nil {
        return probe.Unknown, err
    }
    
    if result.ExitCode == 0 {
        return probe.Success, nil
    }
    
    return probe.Failure, nil
}
```

### 网络层面

HTTP和TCP探针需要通过网络访问容器：

```
┌─────────────────────────────────────────────────────────────┐
│                        节点网络                              │
│                                                              │
│  kubelet (主机网络命名空间)                                   │
│       │                                                      │
│       │ HTTP/TCP请求                                         │
│       ↓                                                      │
│  ┌─────────────────────────────────────────────────────┐    │
│  │              容器网络命名空间                         │    │
│  │                                                      │    │
│  │    eth0 (veth pair)                                  │    │
│  │         │                                            │    │
│  │         ↓                                            │    │
│  │    容器进程监听端口                                   │    │
│  │                                                      │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**关键点**：
- kubelet运行在主机网络命名空间
- 需要通过容器IP或localhost访问容器
- 如果容器使用hostNetwork，则直接访问localhost

## 探针的执行时机

### 定时执行

```go
func (w *worker) run() {
    probeTickerPeriod := time.Duration(w.spec.PeriodSeconds) * time.Second
    
    probeTicker := time.NewTicker(probeTickerPeriod)
    defer probeTicker.Stop()
    
    for {
        select {
        case <-probeTicker.C:
            w.doProbe()
        case <-w.stopCh:
            return
        }
    }
}
```

### 初始延迟

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 30  # 容器启动后等待30秒
```

kubelet在容器启动后等待`initialDelaySeconds`才开始第一次探针检查。

### 启动探针优先

如果配置了启动探针，存活探针和就绪探针会等待启动探针成功后才开始：

```yaml
startupProbe:
  httpGet:
    path: /healthz
    port: 8080
  failureThreshold: 30
  periodSeconds: 10

livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  # 只有startupProbe成功后才开始检查
```

## 探针失败的处理

### 存活探针失败

```
存活探针失败
    ↓
达到failureThreshold
    ↓
kubelet杀死容器
    ↓
根据restartPolicy决定是否重启
```

### 就绪探针失败

```
就绪探针失败
    ↓
Pod的Ready条件变为False
    ↓
从Service的Endpoints中移除
    ↓
停止接收流量
```

### 启动探针失败

```
启动探针失败
    ↓
达到failureThreshold
    ↓
kubelet杀死容器
    ↓
根据restartPolicy决定是否重启
```

## 探针的并发执行

每个容器可能有多个探针，kubelet会并发执行：

```go
type worker struct {
    // 每个探针有独立的worker
    probeType     probeType
    spec          *v1.Probe
    containerID   string
    // ...
}

// 每个探针独立运行
for _, container := range pod.Spec.Containers {
    if container.LivenessProbe != nil {
        w := newWorker(liveness, container.LivenessProbe, ...)
        go w.run()
    }
    if container.ReadinessProbe != nil {
        w := newWorker(readiness, container.ReadinessProbe, ...)
        go w.run()
    }
    if container.StartupProbe != nil {
        w := newWorker(startup, container.StartupProbe, ...)
        go w.run()
    }
}
```

## 探针的资源消耗

### kubelet的资源使用

探针执行会消耗kubelet的资源：

- **HTTP探针**：网络IO、内存（响应体）
- **TCP探针**：网络IO
- **Exec探针**：进程创建、CPU

### 容器的资源使用

Exec探针会在容器内创建进程：

```yaml
livenessProbe:
  exec:
    command:
    - /bin/sh
    - -c
    - "curl -s http://localhost:8080/health"
```

每次探针检查都会创建新进程，可能影响容器性能。

## 常见问题

### Q1: 探针超时怎么处理？

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  timeoutSeconds: 1  # 超时时间
```

如果探针在`timeoutSeconds`内没有响应，视为失败。

### Q2: 探针检查频率如何控制？

```yaml
livenessProbe:
  periodSeconds: 10  # 检查间隔
```

`periodSeconds`控制检查频率，不建议设置太小。

### Q3: 为什么Exec探针比HTTP探针慢？

Exec探针需要：
1. 通过CRI接口
2. 创建新进程
3. 等待进程退出
4. 获取退出码

而HTTP探针只是发送网络请求。

### Q4: 探针可以访问其他服务吗？

可以，但不建议。探针应该只检查容器本身的状态，不应该检查外部依赖。

## 最佳实践

### 1. 探针端点要轻量

```yaml
livenessProbe:
  httpGet:
    path: /healthz    # 轻量级健康检查
    port: 8080
```

健康检查端点应该快速响应，不要执行复杂逻辑。

### 2. 合理设置超时时间

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  timeoutSeconds: 5    # 根据应用响应时间设置
```

### 3. 使用启动探针处理慢启动

```yaml
startupProbe:
  httpGet:
    path: /healthz
    port: 8080
  failureThreshold: 30
  periodSeconds: 10
```

### 4. 避免Exec探针

```yaml
livenessProbe:
  httpGet:        # 推荐
    path: /healthz
    port: 8080

livenessProbe:   # 不推荐
  exec:
    command:
    - /bin/sh
    - -c
    - "curl http://localhost:8080/health"
```
