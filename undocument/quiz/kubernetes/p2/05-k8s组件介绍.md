# Kubernetes核心组件深度解析：从架构到实践

## 引言：Kubernetes架构全景图

在云原生时代，Kubernetes（K8s）已成为容器编排的事实标准。但要真正掌握K8s，必须深入理解其核心组件的工作原理和协作机制。想象一下，K8s就像一个精密运转的数据中心操作系统，各个组件如同操作系统中的进程管理器、调度器、文件系统等，各司其职又紧密协作。

K8s采用Master-Worker架构，整体架构可以分为三个层次：

```
┌─────────────────────────────────────────────────────────────┐
│                        Master Node                          │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐     │
│  │ API Server   │  │ Scheduler    │  │ Controller   │     │
│  │  (网关)      │  │  (调度器)     │  │ Manager      │     │
│  └──────────────┘  └──────────────┘  └──────────────┘     │
│         │                  │                  │             │
│         └──────────────────┼──────────────────┘             │
│                            │                                │
│                    ┌───────▼────────┐                       │
│                    │     etcd       │                       │
│                    │  (数据存储)     │                       │
│                    └────────────────┘                       │
└─────────────────────────────────────────────────────────────┘
                             │
                    ┌────────▼────────┐
                    │   API Server    │
                    │   (REST API)    │
                    └────────┬────────┘
                             │
        ┌────────────────────┼────────────────────┐
        │                    │                    │
┌───────▼────────┐  ┌────────▼────────┐  ┌──────▼──────┐
│  Worker Node 1 │  │  Worker Node 2  │  │Worker Node N│
│ ┌────────────┐ │  │ ┌────────────┐  │ │             │
│ │  kubelet   │ │  │ │  kubelet   │  │ │             │
│ ├────────────┤ │  │ ├────────────┤  │ │             │
│ │ kube-proxy │ │  │ │ kube-proxy │  │ │             │
│ ├────────────┤ │  │ ├────────────┤  │ │             │
│ │ Container  │ │  │ │ Container  │  │ │             │
│ │ Runtime    │ │  │ │ Runtime    │  │ │             │
│ └────────────┘ │  │ └────────────┘  │ │             │
└────────────────┘  └─────────────────┘ └─────────────┘
```

这个架构图展示了K8s的核心设计理念：**声明式API + 控制器模式**。用户通过API Server声明期望状态，各个控制器不断调谐，使实际状态向期望状态收敛。

## 一、Master节点组件详解

Master节点是K8s集群的大脑，负责集群的统一调度和管理。让我们深入剖析每个组件的工作原理。

### 1.1 API Server - 集群的统一入口

#### 核心作用

API Server是K8s控制平面的前端，是集群内部各个组件通信的枢纽，也是外部用户管理集群的唯一入口。它扮演着"网关"的角色，所有的REST请求都必须经过API Server。

#### 工作原理深度解析

**1. 请求处理流水线**

API Server的请求处理流程是一个精心设计的流水线架构：

```
Request → Authentication → Authorization → Admission Control → 
Validation → ETCD Storage → Response
```

- **认证（Authentication）**：验证请求者身份。支持多种认证方式：
  - X509客户端证书
  - Service Account Token
  - OpenID Connect Token
  - Webhook Token认证
  
- **授权（Authorization）**：验证请求者是否有权限执行该操作。支持：
  - RBAC（Role-Based Access Control）
  - ABAC（Attribute-Based Access Control）
  - Node授权器
  - Webhook授权器

- **准入控制（Admission Control）**：在对象持久化前进行拦截和修改。常见的准入控制器：
  - `NamespaceLifecycle`：防止在已删除的namespace中创建对象
  - `LimitRanger`：限制资源使用
  - `ServiceAccount`：自动为Pod添加service account
  - `DefaultStorageClass`：设置默认存储类
  - `ResourceQuota`：限制namespace资源配额

**2. 数据存储机制**

API Server是唯一能直接访问etcd的组件。这种设计保证了数据一致性：

- 所有对象以JSON格式存储在etcd中
- 使用乐观并发控制（Optimistic Concurrency Control）
- 每个对象都有resourceVersion字段，用于版本控制和冲突检测
- 支持Watch机制，客户端可以监听对象变化

**3. API聚合层**

API Server支持API聚合，允许扩展API：

```yaml
# 自定义API示例
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: crontabs.stable.example.com
spec:
  group: stable.example.com
  versions:
    - name: v1
      served: true
      storage: true
  scope: Namespaced
  names:
    plural: crontabs
    singular: crontab
    kind: CronTab
```

#### 关键配置参数

```bash
--etcd-servers=https://127.0.0.1:2379  # etcd地址
--service-cluster-ip-range=10.0.0.0/24  # Service VIP范围
--enable-admission-plugins=NodeRestriction,PodSecurityPolicy
--audit-log-path=/var/log/kubernetes/audit.log  # 审计日志
--authorization-mode=RBAC,Node  # 授权模式
--max-requests-inflight=400  # 最大并发请求数
--request-timeout=60s  # 请求超时时间
```

#### 常见问题与排查

**问题1：API Server启动失败**

排查步骤：
1. 检查etcd连接状态：`etcdctl endpoint health`
2. 检查证书配置：确保证书路径正确且未过期
3. 查看日志：`journalctl -u kube-apiserver -f`

**问题2：请求被拒绝（403 Forbidden）**

常见原因：
- RBAC权限配置不足
- ServiceAccount未正确绑定角色
- 准入控制器拦截

解决方案：
```bash
# 检查用户权限
kubectl auth can-i list pods --as=system:anonymous

# 查看RBAC配置
kubectl get rolebindings,clusterrolebindings -A
```

**问题3：性能瓶颈**

优化建议：
- 启用API Server缓存
- 调整`--max-requests-inflight`参数
- 使用API聚合减少单个API Server压力
- 启用Watch Cache

### 1.2 etcd - 集群的数据大脑

#### 核心作用

etcd是一个分布式键值存储系统，存储了K8s集群的所有状态数据。它是K8s的"数据库"，存储了集群的完整状态信息，包括Pod、Service、ConfigMap等所有对象。

#### 工作原理深度解析

**1. Raft一致性协议**

etcd使用Raft协议保证分布式一致性，其核心机制：

```
┌─────────────────────────────────────────┐
│         Raft Consensus Protocol         │
├─────────────────────────────────────────┤
│                                         │
│  Leader Election (领导者选举)            │
│  ├─ 所有写操作必须经过Leader            │
│  ├─ 心跳机制维持领导地位                 │
│  └─ 超时触发重新选举                     │
│                                         │
│  Log Replication (日志复制)              │
│  ├─ Leader接收写请求                     │
│  ├─ 日志条目复制到Followers             │
│  └─ 多数节点确认后提交                   │
│                                         │
│  Safety (安全性保证)                     │
│  ├─ 已提交的日志不会丢失                 │
│  ├─ 所有节点最终一致                     │
│  └─ Leader包含所有已提交日志             │
│                                         │
└─────────────────────────────────────────┘
```

**2. 数据存储结构**

etcd采用B+树存储数据，支持范围查询和前缀查询：

```
/registry/pods/default/nginx-pod
/registry/services/default/kubernetes
/registry/configmaps/default/app-config
/registry/secrets/default/db-password
```

每个对象的存储路径遵循：`/registry/{resource}/{namespace}/{name}`

**3. Watch机制**

Watch是etcd的核心特性，允许客户端监听数据变化：

- 基于revision的增量通知
- 支持历史事件回放
- 客户端断线重连后可从上次revision继续

**4. Lease机制**

Lease用于实现TTL（Time To Live）功能：

```bash
# 创建一个10秒的租约
etcdctl lease grant 10
# lease 3269759358566980745 granted with TTL(10s)

# 将key绑定到租约
etcdctl put --lease=3269759358566980745 /temp/key value
```

#### 关键配置参数

```bash
# 集群配置
--initial-cluster etcd1=https://10.0.0.1:2380,etcd2=https://10.0.0.2:2380
--initial-cluster-state new
--initial-cluster-token etcd-cluster-1

# 性能调优
--heartbeat-interval=100  # 心跳间隔(毫秒)
--election-timeout=1000   # 选举超时(毫秒)
--snapshot-count=10000    # 快照阈值
--quota-backend-bytes=8589934592  # 存储配额(8GB)

# 数据压缩
--auto-compaction-retention=1h  # 自动压缩保留时间
```

#### 常见问题与排查

**问题1：etcd集群数据不一致**

症状：API Server返回错误，集群状态异常

排查：
```bash
# 检查集群健康状态
etcdctl endpoint health --cluster

# 检查数据一致性
etcdctl endpoint status --cluster -w table

# 查看Leader信息
etcdctl member list -w table
```

**问题2：etcd性能下降**

常见原因：
- 数据量过大，超过8GB
- 磁盘I/O性能不足
- 网络延迟高

解决方案：
```bash
# 定期压缩历史版本
etcdctl compact $(etcdctl endpoint status --write-out="json" | grep -o '"revision":[0-9]*' | head -1 | grep -o '[0-9]*')

# 清理碎片
etcdctl defrag

# 监控指标
etcdctl check perf
```

**问题3：etcd备份与恢复**

备份：
```bash
# 定期备份
etcdctl snapshot save /backup/etcd-snapshot-$(date +%Y%m%d).db

# 验证备份
etcdctl snapshot status /backup/etcd-snapshot.db -w table
```

恢复：
```bash
# 停止所有etcd实例
# 恢复数据
etcdctl snapshot restore /backup/etcd-snapshot.db \
  --data-dir=/var/lib/etcd-restore \
  --initial-cluster etcd1=https://10.0.0.1:2380 \
  --initial-cluster-token etcd-cluster-1 \
  --initial-advertise-peer-urls https://10.0.0.1:2380
```

### 1.3 Controller Manager - 集群的自动化大脑

#### 核心作用

Controller Manager是集群的"自动化运维中心"，运行着多种控制器，每个控制器负责一种特定的资源类型。控制器通过不断调谐（Reconcile），确保集群的实际状态与期望状态一致。

#### 工作原理深度解析

**1. 控制器模式**

所有控制器都遵循统一的控制循环模式：

```
┌─────────────────────────────────────────────┐
│          Controller Loop Pattern            │
├─────────────────────────────────────────────┤
│                                             │
│  1. List/Watch 监听资源变化                  │
│     ↓                                       │
│  2. 将事件加入WorkQueue                      │
│     ↓                                       │
│  3. Worker从队列取出事件                     │
│     ↓                                       │
│  4. 执行Reconcile逻辑                        │
│     ├─ 获取期望状态                          │
│     ├─ 获取实际状态                          │
│     ├─ 计算差异                              │
│     └─ 执行调谐操作                          │
│     ↓                                       │
│  5. 更新资源状态                             │
│     ↓                                       │
│  6. 返回步骤1                                │
│                                             │
└─────────────────────────────────────────────┘
```

**2. 核心控制器详解**

**Replication Controller / ReplicaSet Controller**

作用：确保指定数量的Pod副本始终运行

工作流程：
1. 监听ReplicaSet对象变化
2. 计算当前Pod数量与期望数量的差异
3. 创建或删除Pod以匹配期望数量
4. 更新ReplicaSet的status字段

**Deployment Controller**

作用：管理ReplicaSet的滚动更新和回滚

工作流程：
1. 监听Deployment对象变化
2. 创建对应的ReplicaSet
3. 根据更新策略执行滚动更新
4. 记录历史版本，支持回滚

**Node Controller**

作用：监控节点健康状态

工作流程：
1. 监听Node对象变化
2. 检查节点心跳（通过kubelet上报）
3. 标记NotReady节点
4. 驱逐NotReady节点上的Pod（可配置宽限期）

**Service Controller**

作用：管理Service的LoadBalancer类型

工作流程：
1. 监听Service对象变化
2. 调用云提供商API创建LoadBalancer
3. 更新Service的status.loadBalancer字段

**EndpointSlice Controller**

作用：管理Service的后端端点

工作流程：
1. 监听Service和Pod变化
2. 匹配Pod标签选择器
3. 创建/更新EndpointSlice对象
4. 将健康的Pod IP加入端点列表

**3. 控制器协作示例**

以创建Deployment为例，展示多个控制器的协作：

```
用户创建Deployment
    ↓
Deployment Controller监听到事件
    ↓
创建ReplicaSet
    ↓
ReplicaSet Controller监听到事件
    ↓
创建Pod
    ↓
Scheduler监听到未调度Pod
    ↓
绑定Pod到Node
    ↓
kubelet监听到Pod绑定
    ↓
创建容器
    ↓
Endpoint Controller监听到Pod就绪
    ↓
更新Service的Endpoints
```

#### 关键配置参数

```bash
--controllers=*  # 启用所有控制器
--controllers=deployment,replicaset  # 只启用指定控制器

# 控制器并发数
--concurrent-deployment-syncs=5
--concurrent-replicaset-syncs=5
--concurrent-service-syncs=1

# 节点控制器配置
--node-monitor-period=5s  # 节点检查周期
--node-monitor-grace-period=40s  # 节点标记NotReady的宽限期
--pod-eviction-timeout=5m  # Pod驱逐超时时间
```

#### 常见问题与排查

**问题1：控制器卡住不工作**

排查：
```bash
# 查看Controller Manager日志
kubectl logs -n kube-system kube-controller-manager-master -f

# 检查WorkQueue指标
kubectl get --raw /metrics | grep workqueue_depth

# 检查控制器是否启用
ps aux | grep kube-controller-manager | grep controllers
```

**问题2：Deployment更新卡住**

常见原因：
- 新Pod无法启动（镜像拉取失败、资源不足等）
- 健康检查失败
- PodDisruptionBudget限制

排查：
```bash
# 查看Deployment状态
kubectl describe deployment <deployment-name>

# 查看ReplicaSet状态
kubectl get rs -l app=<app-name>

# 查看Pod事件
kubectl describe pod <pod-name>
```

### 1.4 Scheduler - 集群的调度器

#### 核心作用

Scheduler负责将Pod调度到合适的Node上运行。它是K8s资源管理的核心，决定了工作负载的分布和资源利用率。

#### 工作原理深度解析

**1. 调度流程**

Scheduler的调度过程分为三个阶段：

```
┌──────────────────────────────────────────────┐
│         Scheduling Pipeline                  │
├──────────────────────────────────────────────┤
│                                              │
│  1. Scheduling Cycle (调度周期)               │
│     ├─ Filtering (过滤阶段)                   │
│     │  ├─ Predicates (预选)                  │
│     │  └─ 过滤不满足条件的节点                │
│     │                                        │
│     ├─ Scoring (打分阶段)                     │
│     │  ├─ Priorities (优选)                  │
│     │  └─ 对候选节点打分                      │
│     │                                        │
│     └─ Binding (绑定阶段)                     │
│        └─ 将Pod绑定到得分最高的节点           │
│                                              │
│  2. Binding Cycle (绑定周期)                  │
│     └─ 异步执行绑定操作                       │
│                                              │
└──────────────────────────────────────────────┘
```

**2. 过滤阶段（Predicates）**

过滤阶段会排除不满足条件的节点，常见的过滤策略：

- **PodFitsResources**：节点资源是否满足Pod需求
  - CPU、内存、存储等资源检查
  - 考虑已分配资源和可分配资源
  
- **PodFitsHostPorts**：节点端口是否冲突
  - 检查hostPort是否已被占用
  
- **PodMatchNodeSelector**：节点标签是否匹配
  - 检查nodeSelector和nodeAffinity
  
- **PodToleratesNodeTaints**：Pod是否容忍节点污点
  - 检查tolerations和taints
  
- **CheckNodeCondition**：节点状态是否正常
  - 检查Ready、MemoryPressure、DiskPressure等状态

**3. 打分阶段（Priorities）**

打分阶段对候选节点进行评分（0-10分），常见的打分策略：

- **LeastRequestedPriority**：优先选择资源使用率低的节点
  ```
  score = (capacity - requested) / capacity * 10
  ```
  
- **BalancedResourceAllocation**：资源分配均衡
  - CPU和内存使用率越接近，分数越高
  
- **NodeAffinityPriority**：节点亲和性
  - 满足亲和性规则的节点得分更高
  
- **InterPodAffinityPriority**：Pod亲和性
  - 与相关Pod在同一拓扑域的节点得分更高
  
- **TaintTolerationPriority**：污点容忍
  - 容忍更多污点的节点得分更高

**4. 调度器扩展**

K8s支持自定义调度器：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: custom-scheduled-pod
spec:
  schedulerName: my-custom-scheduler  # 指定自定义调度器
  containers:
  - name: nginx
    image: nginx
```

**5. 调度框架（Scheduling Framework）**

K8s 1.19+引入了调度框架，允许在调度流程的各个阶段插入扩展点：

```
QueueSort → PreFilter → Filter → PostFilter → 
PreScore → Score → NormalizeScore → Reserve → 
Permit → PreBind → Bind → PostBind
```

#### 关键配置参数

```bash
# 调度策略配置
--policy-config-file=/etc/kubernetes/scheduler-policy.json

# 性能调优
--algorithm-provider=DefaultProvider
--percentage-of-nodes-to-score=50  # 只对50%节点打分（大规模集群优化）

# 并发配置
--concurrent-deployment-syncs=10

# 绑定超时
--bind-timeout=60s
```

#### 常见问题与排查

**问题1：Pod一直处于Pending状态**

排查步骤：
```bash
# 查看Pod事件
kubectl describe pod <pod-name>

# 常见原因：
# - 资源不足：Insufficient cpu/memory
# - 节点选择器不匹配：node(s) didn't match node selector
# - 污点不容忍：node(s) had taints that the pod didn't tolerate
# - 持久卷挂载失败：persistentvolumeclaim "xxx" not found
```

**问题2：调度不均衡**

排查：
```bash
# 查看节点资源使用
kubectl top nodes

# 查看Pod分布
kubectl get pods -o wide --all-namespaces | awk '{print $7}' | sort | uniq -c

# 检查调度器日志
kubectl logs -n kube-system kube-scheduler-master -f
```

**问题3：调度器性能问题**

优化建议：
- 调整`--percentage-of-nodes-to-score`参数
- 使用节点标签缩小候选节点范围
- 启用调度器缓存
- 使用自定义调度器处理特定工作负载

## 二、Node节点组件详解

Node节点是K8s的工作节点，负责运行容器工作负载。每个Node节点运行三个核心组件：kubelet、kube-proxy和容器运行时。

### 2.1 kubelet - 节点的代理管家

#### 核心作用

kubelet是运行在每个Node上的主要代理，负责：
- 向Master节点汇报节点状态
- 管理Pod的生命周期
- 执行健康检查
- 挂载存储卷
- 与容器运行时交互

#### 工作原理深度解析

**1. Pod生命周期管理**

kubelet通过Pod Lifecycle Event Generator (PLEG)管理Pod：

```
┌────────────────────────────────────────────┐
│           kubelet Pod Management           │
├────────────────────────────────────────────┤
│                                            │
│  1. Pod配置来源                             │
│     ├─ API Server (主要)                   │
```

### 2.2 kube-proxy - 网络代理

#### 核心作用

kube-proxy负责维护节点上的网络规则，实现Service的负载均衡和服务发现。它是K8s网络模型的关键组件。

#### 工作原理深度解析

**1. 三种工作模式**

**userspace模式（已废弃）**

```
Client → kube-proxy (userspace) → iptables → Pod
```

- kube-proxy在用户空间监听请求
- 性能较差，已不推荐使用

**iptables模式（默认）**

```
Client → iptables (DNAT) → Pod
```

工作原理：
1. kube-proxy监听Service和EndpointSlice变化
2. 生成iptables规则
3. 客户端请求通过iptables直接转发到Pod
4. kube-proxy不参与数据转发路径

iptables规则示例：
```bash
# Service ClusterIP规则
-A KUBE-SERVICES -d 10.96.0.1/32 -p tcp -m tcp --dport 443 -j KUBE-SVC-NPX46M4PTMTKRN6Y

# Endpoint规则（随机选择）
-A KUBE-SVC-NPX46M4PTMTKRN6Y -m statistic --mode random --probability 0.3333 -j KUBE-SEP-W3
-A KUBE-SVC-NPX46M4PTMTKRN6Y -m statistic --mode random --probability 0.5 -j KUBE-SEP-W2
-A KUBE-SVC-NPX46M4PTMTKRN6Y -j KUBE-SEP-W1

# DNAT规则
-A KUBE-SEP-W1 -p tcp -m tcp -j DNAT --to-destination 10.244.1.2:8080
```

**IPVS模式（推荐）**

```
Client → IPVS (Load Balancer) → Pod
```

优势：
- 支持更多负载均衡算法：rr（轮询）、lc（最少连接）、dh（目标哈希）、sh（源哈希）等
- 性能更好：O(1)复杂度，iptables是O(n)
- 规则更新更快

IPVS规则示例：
```bash
# 查看IPVS规则
ipvsadm -Ln

# 输出示例
TCP  10.96.0.1:443 rr
  -> 10.244.1.2:6443          Masq    1      0          0
  -> 10.244.2.2:6443          Masq    1      0          0
```

**2. Service类型处理**

kube-proxy处理不同类型的Service：

- **ClusterIP**：虚拟IP，仅集群内部访问
- **NodePort**：在每个节点上开放端口
- **LoadBalancer**：调用云提供商创建负载均衡器
- **ExternalName**：CNAME映射到外部DNS

**3. 会话亲和性**

支持ClientIP会话亲和性：

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  sessionAffinity: ClientIP
  sessionAffinityConfig:
    clientIP:
      timeoutSeconds: 10800  # 3小时
```

#### 关键配置参数

```bash
# 代理模式
--proxy-mode=ipvs  # 或 iptables

# IPVS配置
--ipvs-scheduler=rr  # 负载均衡算法
--ipvs-min-sync-period=5s  # 最小同步间隔
--ipvs-sync-period=30s  # 同步周期

# iptables配置
--iptables-min-sync-period=5s
--iptables-sync-period=30s

# ClusterIP范围（需与API Server一致）
--cluster-cidr=10.96.0.0/12

# 性能调优
--conntrack-max-per-core=32768
--conntrack-tcp-timeout-established=24h
```

排查：

## 六、常见问题与最佳实践

### 6.1 常见问题FAQ

**Q1: Master节点可以运行工作负载吗？**

A: 可以，但不推荐。Master节点运行控制平面组件，资源消耗较大。建议通过taints/tolerations或nodeSelector将工作负载调度到Worker节点。

**Q2: etcd集群需要多少节点？**

A: 推荐3、5、7个奇数节点。Raft协议需要多数节点存活，3节点允许1个故障，5节点允许2个故障。节点数过多会降低写入性能。

**Q3: 如何选择CNI插件？**

A: 
- 小规模集群：Flannel（简单）
- 大规模集群：Calico（性能好，支持网络策略）
- 云原生环境：Cilium（eBPF，可观测性强）
- 混合云：Weave（跨云支持）

**Q4: kube-proxy应该用iptables还是IPVS模式？**

A: 
- 小规模集群（<1000个Service）：iptables足够
- 大规模集群（>1000个Service）：IPVS性能更好
- 需要高级负载均衡算法：IPVS

**Q5: 如何保证Master节点高可用？**

A: 
- API Server：多副本 + 负载均衡器
- etcd：奇数节点集群（3/5/7）
- Controller Manager/Scheduler：Leader Election自动选主
- 使用kubeadm等工具部署高可用集群

## 七、面试回答

在面试中回答"K8s有哪些组件及其作用"时，可以这样组织：

"Kubernetes采用Master-Worker架构，组件分为三类。**Master节点组件**包括：API Server作为集群统一入口，处理所有REST请求并负责认证授权；etcd作为分布式键值存储，保存集群所有状态数据；Controller Manager运行多种控制器，通过控制循环确保实际状态向期望状态收敛；Scheduler负责Pod调度，通过过滤和打分算法选择最优节点。**Node节点组件**包括：kubelet作为节点代理，管理Pod生命周期并与容器运行时交互；kube-proxy维护网络规则，实现Service负载均衡；容器运行时（如containerd）负责镜像管理和容器执行。**插件组件**包括CoreDNS提供服务发现、CNI插件配置Pod网络、Ingress Controller管理外部访问等。这些组件通过Watch机制协同工作，实现了声明式API和控制器模式，是K8s的核心设计理念。"
