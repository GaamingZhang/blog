# Kubernetes 数据持久化方式详解

## 引言

在容器化环境中，容器本身是临时的、易失的。当容器重启或删除时，容器内的数据会随之丢失。然而，许多应用场景需要数据持久化，如数据库、消息队列、日志存储等。Kubernetes 提供了多种数据持久化方案，满足不同场景的需求。

理解 Kubernetes 的数据持久化机制，选择合适的持久化方案，是构建生产级应用的关键。本文将深入剖析 Kubernetes 中的各种数据持久化方式及其应用场景。

## 数据持久化概述

### 持久化方式分类

| 类型 | 特点 | 适用场景 |
|-----|------|---------|
| **emptyDir** | 临时存储，Pod 生命周期 | 缓存、临时数据 |
| **hostPath** | 节点本地存储 | 节点级数据、调试 |
| **ConfigMap** | 配置数据存储 | 配置文件 |
| **Secret** | 敏感数据存储 | 密码、证书 |
| **PV/PVC** | 持久卷存储 | 数据库、持久化应用 |
| **CSI** | 容器存储接口 | 云存储、企业存储 |

## 一、emptyDir

### 概述

emptyDir 是最简单的存储类型，随 Pod 创建而创建，随 Pod 删除而删除。

### 配置示例

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: emptydir-demo
spec:
  containers:
  - name: app
    image: nginx:1.21
    volumeMounts:
    - name: cache
      mountPath: /var/cache
    - name: tmp
      mountPath: /tmp
  volumes:
  - name: cache
    emptyDir: {}
  - name: tmp
    emptyDir:
      sizeLimit: "100Mi"
      medium: Memory  # 使用内存作为存储介质
```

### emptyDir 特性

| 特性 | 说明 |
|-----|------|
| **生命周期** | 与 Pod 相同 |
| **存储介质** | 节点磁盘或内存 |
| **容量限制** | 可配置 sizeLimit |
| **共享** | Pod 内容器共享 |
| **持久性** | Pod 删除后数据丢失 |

### 使用场景

```yaml
# 场景1：缓存存储
volumes:
- name: cache
  emptyDir: {}

# 场景2：临时文件
volumes:
- name: tmp
  emptyDir:
    sizeLimit: "1Gi"

# 场景3：内存缓存
volumes:
- name: memory-cache
  emptyDir:
    medium: Memory
    sizeLimit: "256Mi"
```

## 二、hostPath

### 概述

hostPath 将节点文件系统上的文件或目录挂载到 Pod 中。

### 配置示例

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: hostpath-demo
spec:
  containers:
  - name: app
    image: nginx:1.21
    volumeMounts:
    - name: data
      mountPath: /var/data
  volumes:
  - name: data
    hostPath:
      path: /data/app
      type: DirectoryOrCreate
```

### hostPath 类型

| 类型 | 说明 |
|-----|------|
| **DirectoryOrCreate** | 目录不存在则创建 |
| **Directory** | 必须存在的目录 |
| **FileOrCreate** | 文件不存在则创建 |
| **File** | 必须存在的文件 |
| **Socket** | Unix socket |
| **CharDevice** | 字符设备 |
| **BlockDevice** | 块设备 |

### 使用场景

```yaml
# 场景1：访问节点日志
volumes:
- name: node-logs
  hostPath:
    path: /var/log
    type: Directory

# 场景2：访问 Docker
volumes:
- name: docker-sock
  hostPath:
    path: /var/run/docker.sock
    type: Socket

# 场景3：本地数据存储
volumes:
- name: local-data
  hostPath:
    path: /data/app
    type: DirectoryOrCreate
```

### 注意事项

- Pod 可能被调度到不同节点
- 需要考虑节点故障
- 存在安全风险
- 生产环境谨慎使用

## 三、ConfigMap

### 概述

ConfigMap 用于存储非敏感的配置数据。

### 配置示例

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  database_url: "mysql://db-service:3306/mydb"
  cache_size: "1024"
  config.properties: |
    server.port=8080
    logging.level=INFO
---
apiVersion: v1
kind: Pod
metadata:
  name: configmap-demo
spec:
  containers:
  - name: app
    image: nginx:1.21
    env:
    - name: DATABASE_URL
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: database_url
    volumeMounts:
    - name: config
      mountPath: /etc/config
  volumes:
  - name: config
    configMap:
      name: app-config
```

### 使用方式

```yaml
# 方式1：环境变量
env:
- name: DATABASE_URL
  valueFrom:
    configMapKeyRef:
      name: app-config
      key: database_url

# 方式2：命令行参数
command: ["./app", "--config=$(DATABASE_URL)"]

# 方式3：配置文件挂载
volumes:
- name: config
  configMap:
    name: app-config
    items:
    - key: config.properties
      path: application.properties
```

## 四、Secret

### 概述

Secret 用于存储敏感数据，如密码、Token、证书等。

### 配置示例

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
stringData:
  username: admin
  password: mypassword
  tls.crt: |
    -----BEGIN CERTIFICATE-----
    ...
    -----END CERTIFICATE-----
---
apiVersion: v1
kind: Pod
metadata:
  name: secret-demo
spec:
  containers:
  - name: app
    image: nginx:1.21
    env:
    - name: DB_USERNAME
      valueFrom:
        secretKeyRef:
          name: db-secret
          key: username
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-secret
          key: password
    volumeMounts:
    - name: tls
      mountPath: /etc/tls
      readOnly: true
  volumes:
  - name: tls
    secret:
      secretName: db-secret
```

### Secret 类型

| 类型 | 说明 |
|-----|------|
| **Opaque** | 用户定义的任意数据 |
| **kubernetes.io/service-account-token** | Service Account Token |
| **kubernetes.io/dockercfg** | Docker 配置 |
| **kubernetes.io/dockerconfigjson** | Docker 配置 JSON |
| **kubernetes.io/tls** | TLS 证书 |
| **kubernetes.io/basic-auth** | 基本认证 |
| **kubernetes.io/ssh-auth** | SSH 认证 |

## 五、PV 和 PVC

### 概述

PV（PersistentVolume）是集群级别的存储资源，PVC（PersistentVolumeClaim）是命名空间级别的存储请求。

### 工作原理

```
┌─────────────────────────────────────────────────────────────┐
│                    PV/PVC 工作原理                           │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              PersistentVolume (PV)                   │   │
│  │  - 集群级别资源                                      │   │
│  │  - 存储资源的抽象                                    │   │
│  │  - 独立于 Pod 生命周期                               │   │
│  └──────────────────────┬──────────────────────────────┘   │
│                         │                                   │
│                         │ 绑定                              │
│                         ▼                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              PersistentVolumeClaim (PVC)             │   │
│  │  - 命名空间级别资源                                  │   │
│  │  - 存储资源的请求                                    │   │
│  │  - 由 Pod 使用                                       │   │
│  └──────────────────────┬──────────────────────────────┘   │
│                         │                                   │
│                         │ 挂载                              │
│                         ▼                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                      Pod                             │   │
│  │  - 使用 PVC 挂载存储                                 │   │
│  │  - 数据持久化                                        │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### PV 配置示例

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-storage
spec:
  capacity:
    storage: 10Gi
  accessModes:
  - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: standard
  hostPath:
    path: /mnt/data
```

### PVC 配置示例

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: pvc-storage
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
  storageClassName: standard
```

### Pod 使用 PVC

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: pvc-demo
spec:
  containers:
  - name: app
    image: nginx:1.21
    volumeMounts:
    - name: data
      mountPath: /var/data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: pvc-storage
```

### 访问模式

| 访问模式 | 缩写 | 说明 |
|---------|------|------|
| **ReadWriteOnce** | RWO | 单节点读写 |
| **ReadOnlyMany** | ROX | 多节点只读 |
| **ReadWriteMany** | RWX | 多节点读写 |
| **ReadWriteOncePod** | RWOP | 单 Pod 读写 |

### 回收策略

| 策略 | 说明 |
|-----|------|
| **Retain** | 保留数据，需手动清理 |
| **Delete** | 自动删除存储资源 |
| **Recycle** | 已废弃，使用 Delete 替代 |

## 六、StorageClass 动态供给

### 概述

StorageClass 定义了存储的类型和供给方式，支持动态创建 PV。

### 配置示例

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: standard
provisioner: kubernetes.io/aws-ebs
parameters:
  type: gp2
  zone: us-east-1a
reclaimPolicy: Delete
volumeBindingMode: Immediate
allowVolumeExpansion: true
```

### 常见 StorageClass

```yaml
# AWS EBS
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: aws-ebs
provisioner: kubernetes.io/aws-ebs
parameters:
  type: gp3
  fsType: ext4

# GCE PD
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: gce-pd
provisioner: kubernetes.io/gce-pd
parameters:
  type: pd-ssd

# Azure Disk
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: azure-disk
provisioner: kubernetes.io/azure-disk
parameters:
  storageaccounttype: Standard_LRS

# NFS
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: nfs
provisioner: nfs-client
parameters:
  archiveOnDelete: "true"

# Local
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: local
provisioner: kubernetes.io/no-provisioner
volumeBindingMode: WaitForFirstConsumer
```

### 动态供给示例

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: dynamic-pvc
spec:
  accessModes:
  - ReadWriteOnce
  storageClassName: standard
  resources:
    requests:
      storage: 10Gi
```

## 持久化方式对比

| 方式 | 持久性 | 共享性 | 适用场景 |
|-----|--------|--------|---------|
| **emptyDir** | Pod 生命周期 | Pod 内共享 | 临时数据、缓存 |
| **hostPath** | 节点生命周期 | 节点级共享 | 节点数据、调试 |
| **ConfigMap** | 独立 | 只读共享 | 配置文件 |
| **Secret** | 独立 | 只读共享 | 敏感数据 |
| **PV/PVC** | 独立 | 可配置 | 持久化应用 |
| **CSI** | 独立 | 可配置 | 企业存储 |

## 面试回答

**问题**: Kubernetes 数据持久化的方式有哪些？

**回答**: Kubernetes 提供了多种数据持久化方式，适用于不同场景。

**临时存储**：`emptyDir` 随 Pod 创建而创建，Pod 删除后数据丢失，适用于缓存和临时数据。可以配置使用内存作为存储介质，提高访问速度。

**节点存储**：`hostPath` 将节点文件系统挂载到 Pod，数据存储在节点上，适用于节点级数据和调试场景，但存在节点故障风险。

**配置存储**：`ConfigMap` 存储非敏感配置数据，支持环境变量和文件挂载两种使用方式；`Secret` 存储敏感数据如密码、证书，数据经过 Base64 编码，支持加密存储。

**持久卷存储**：`PV`（PersistentVolume）是集群级别的存储资源，`PVC`（PersistentVolumeClaim）是命名空间级别的存储请求。PV 和 PVC 实现了存储供给和消费的解耦，支持多种访问模式（ReadWriteOnce、ReadOnlyMany、ReadWriteMany）和回收策略（Retain、Delete）。

**动态供给**：`StorageClass` 定义存储类型，支持动态创建 PV。通过配置不同的 provisioner（如 AWS EBS、GCE PD、Azure Disk、NFS、Ceph 等），可以自动创建和管理存储资源。

**CSI（Container Storage Interface）**：Kubernetes 的标准存储接口，支持第三方存储提供商实现自定义存储解决方案，如企业级存储、云存储等。

选择持久化方式时，需要考虑数据的生命周期、共享需求、性能要求和成本因素。生产环境推荐使用 PV/PVC 配合 StorageClass 实现动态供给，并配置适当的备份策略。
