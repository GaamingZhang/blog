# Kubernetes 资源管理和调度

## 引言

Kubernetes 资源管理和调度是集群运维的核心能力。通过合理配置资源请求和限制、资源配额、调度策略，可以确保应用获得所需资源，同时避免资源浪费和争抢。

## 资源管理概述

### 资源类型

```
┌─────────────────────────────────────────────────────────────┐
│              Kubernetes 资源类型                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  可压缩资源：                                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • CPU                                               │   │
│  │  • 超限会被限流                                      │   │
│  │  • 不会导致容器终止                                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  不可压缩资源：                                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • 内存                                              │   │
│  │  • 磁盘                                              │   │
│  │  • 超限会导致 OOMKilled                              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 资源请求和限制

### 基本配置

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: resource-pod
spec:
  containers:
  - name: app
    image: myapp:v1
    resources:
      requests:
        cpu: "100m"
        memory: "128Mi"
      limits:
        cpu: "500m"
        memory: "512Mi"
```

### Requests 和 Limits 说明

| 配置 | 说明 |
|-----|------|
| requests.cpu | 调度时的 CPU 最小保证 |
| requests.memory | 调度时的内存最小保证 |
| limits.cpu | 运行时的 CPU 最大限制 |
| limits.memory | 运行时的内存最大限制 |

## QoS 服务质量

### QoS 分类

```
┌─────────────────────────────────────────────────────────────┐
│                  QoS 服务质量等级                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Guaranteed（保证）：                                        │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • requests = limits                                │   │
│  │  • 最高优先级                                       │   │
│  │  • 最后被驱逐                                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  Burstable（可突发）：                                       │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • requests < limits                                │   │
│  │  • 中等优先级                                       │   │
│  │  • 中间被驱逐                                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  BestEffort（尽力而为）：                                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • 无 requests 和 limits                            │   │
│  │  • 最低优先级                                       │   │
│  │  • 最先被驱逐                                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### Guaranteed 配置

```yaml
resources:
  requests:
    cpu: "500m"
    memory: "512Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"
```

## ResourceQuota 资源配额

### 命名空间资源配额

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: compute-quota
  namespace: production
spec:
  hard:
    requests.cpu: "10"
    requests.memory: 20Gi
    limits.cpu: "20"
    limits.memory: 40Gi
    pods: "10"
    persistentvolumeclaims: "5"
```

### 资源配额说明

```yaml
spec:
  hard:
    requests.cpu: "10"        # CPU 请求总量
    requests.memory: 20Gi     # 内存请求总量
    limits.cpu: "20"          # CPU 限制总量
    limits.memory: 40Gi       # 内存限制总量
    pods: "10"                # Pod 数量限制
    services: "5"             # Service 数量限制
    secrets: "10"             # Secret 数量限制
```

## LimitRange 默认限制

### 配置默认资源限制

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: resource-limits
  namespace: production
spec:
  limits:
  - type: Container
    default:
      cpu: "500m"
      memory: "512Mi"
    defaultRequest:
      cpu: "100m"
      memory: "128Mi"
    min:
      cpu: "50m"
      memory: "64Mi"
    max:
      cpu: "2000m"
      memory: "2Gi"
```

## 调度策略

### nodeSelector

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: scheduled-pod
spec:
  nodeSelector:
    disktype: ssd
    zone: beijing
  containers:
  - name: app
    image: myapp:v1
```

### Node Affinity

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: affinity-pod
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: disktype
            operator: In
            values:
            - ssd
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 80
        preference:
          matchExpressions:
          - key: zone
            operator: In
            values:
            - beijing
  containers:
  - name: app
    image: myapp:v1
```

### Pod Affinity

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: pod-affinity
spec:
  affinity:
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchLabels:
            app: database
        topologyKey: kubernetes.io/hostname
    podAntiAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        podAffinityTerm:
          labelSelector:
            matchLabels:
              app: web
          topologyKey: kubernetes.io/hostname
  containers:
  - name: app
    image: myapp:v1
```

## 面试回答

**问题**: Kubernetes 如何进行资源管理和调度？

**回答**: Kubernetes 资源管理和调度包括资源配置、QoS 分类、资源配额、调度策略等方面：

**资源配置**：**requests** 用于调度决策，表示容器最小资源保证；**limits** 用于运行时约束，表示容器最大资源限制。CPU 是可压缩资源，超限会被限流；内存是不可压缩资源，超限会被 OOMKilled。

**QoS 服务质量**：**Guaranteed** 最高优先级，requests 等于 limits；**Burstable** 中等优先级，requests 小于 limits；**BestEffort** 最低优先级，无 requests 和 limits。节点资源不足时按优先级从低到高驱逐。

**ResourceQuota 资源配额**：限制命名空间的资源总量，包括 CPU、内存、Pod 数量等。防止单个命名空间占用过多资源。

**LimitRange 默认限制**：为未设置资源的 Pod 设置默认值；限制资源的最小和最大值。

**调度策略**：**nodeSelector** 简单的节点选择；**Node Affinity** 节点亲和性，支持软硬策略；**Pod Affinity** Pod 亲和性，将相关 Pod 调度到一起；**Taints/Tolerations** 污点和容忍，实现节点隔离。

**最佳实践**：为所有容器设置资源请求和限制；生产环境使用 Guaranteed QoS；配置 ResourceQuota 限制命名空间资源；使用 Pod 反亲和性分散副本。
