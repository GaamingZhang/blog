# Kubernetes 多环境部署

## 引言

在实际开发中，应用需要在多个环境（开发、测试、预发布、生产）中部署。每个环境的配置可能有所不同，如副本数、资源配置、环境变量等。Kubernetes 提供了多种方式实现多环境部署，本文介绍常用的方案和最佳实践。

## Kustomize 方案

### 目录结构

```
app/
├── base/
│   ├── deployment.yaml
│   ├── service.yaml
│   └── kustomization.yaml
└── overlays/
    ├── dev/
    │   ├── kustomization.yaml
    │   └── patch.yaml
    ├── staging/
    │   ├── kustomization.yaml
    │   └── patch.yaml
    └── production/
        ├── kustomization.yaml
        └── patch.yaml
```

### base/kustomization.yaml

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
- deployment.yaml
- service.yaml

commonLabels:
  app: myapp
```

### overlays/dev/kustomization.yaml

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

namespace: dev

resources:
- ../../base

patchesStrategicMerge:
- deployment-patch.yaml

configMapGenerator:
- name: app-config
  behavior: merge
  literals:
  - ENV=dev
  - LOG_LEVEL=debug

replicas:
- name: myapp
  count: 1

images:
- name: myapp
  newTag: dev
```

### overlays/production/kustomization.yaml

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

namespace: production

resources:
- ../../base

patchesStrategicMerge:
- deployment-patch.yaml

configMapGenerator:
- name: app-config
  behavior: merge
  literals:
  - ENV=production
  - LOG_LEVEL=info

replicas:
- name: myapp
  count: 3

images:
- name: myapp
  newTag: v1.0.0
```

### 使用 Kustomize

```bash
kubectl kustomize overlays/dev

kubectl apply -k overlays/dev

kubectl apply -k overlays/production
```

## Helm 方案

### values-dev.yaml

```yaml
namespace: dev

replicaCount: 1

image:
  repository: myapp
  tag: dev
  pullPolicy: Always

resources:
  requests:
    cpu: 100m
    memory: 128Mi
  limits:
    cpu: 500m
    memory: 512Mi

config:
  ENV: dev
  LOG_LEVEL: debug
```

### values-production.yaml

```yaml
namespace: production

replicaCount: 3

image:
  repository: myapp
  tag: v1.0.0
  pullPolicy: IfNotPresent

resources:
  requests:
    cpu: 200m
    memory: 256Mi
  limits:
    cpu: 1000m
    memory: 1Gi

config:
  ENV: production
  LOG_LEVEL: info

ingress:
  enabled: true
  hosts:
  - host: myapp.example.com
    paths:
    - path: /
      pathType: Prefix
```

### 使用 Helm

```bash
helm install myapp ./chart -f values-dev.yaml -n dev

helm install myapp ./chart -f values-production.yaml -n production

helm upgrade myapp ./chart -f values-production.yaml -n production
```

## GitOps 方案

### ArgoCD Application

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp-dev
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/org/app-config.git
    targetRevision: main
    path: overlays/dev
  destination:
    server: https://kubernetes.default.svc
    namespace: dev
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
---
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp-production
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/org/app-config.git
    targetRevision: main
    path: overlays/production
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

## 命名空间隔离

### 创建命名空间

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: dev
  labels:
    environment: dev
---
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    environment: production
```

### 资源配额

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: dev-quota
  namespace: dev
spec:
  hard:
    requests.cpu: "10"
    requests.memory: 20Gi
    limits.cpu: "20"
    limits.memory: 40Gi
    pods: "20"
```

## 最佳实践

### 1. 使用相同的配置结构

```yaml
resources:
- deployment.yaml
- service.yaml
- configmap.yaml
```

### 2. 环境差异化配置

```yaml
replicas:
- name: myapp
  count: 1  # dev
```

### 3. 使用 GitOps 管理

```yaml

  automated:
    prune: true
    selfHeal: true
```

### 4. 配置审计

```yaml

  annotations:
    environment: production
    version: v1.0.0
```

## 面试回答

**问题**: 如何使用 Kubernetes 进行多环境部署？

**回答**: Kubernetes 多环境部署有多种方案：

**Kustomize 方案**：使用 base + overlays 结构管理配置差异。base 目录存放通用配置，overlays 目录存放各环境的差异化配置。通过 kustomization.yaml 定义资源、补丁、配置等。使用 `kubectl apply -k overlays/<env>` 部署到指定环境。

**Helm 方案**：使用 values 文件管理不同环境配置。values-dev.yaml、values-staging.yaml、values-production.yaml 分别定义各环境配置。通过 `helm install -f values-<env>.yaml` 部署到指定环境。Helm 模板支持条件判断和循环，灵活性高。

**GitOps 方案**：使用 ArgoCD 或 Flux 实现 GitOps 工作流。配置存储在 Git 仓库，ArgoCD 自动同步到集群。支持多集群、多环境管理。变更通过 Git 提交触发，有完整的审计记录。

**命名空间隔离**：使用不同命名空间隔离环境。配置 ResourceQuota 限制资源使用。使用 NetworkPolicy 控制网络访问。

**最佳实践**：使用相同的配置结构；环境差异化配置最小化；使用 GitOps 管理配置变更；配置资源配额限制；使用标签和注解标识环境；自动化部署流程。
