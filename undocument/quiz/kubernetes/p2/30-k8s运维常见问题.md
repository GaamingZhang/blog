# Kubernetes运维常见问题与解决方案

## 引言

Kubernetes作为云原生时代的容器编排平台，以其强大的自动化能力、弹性伸缩和声明式配置赢得了广泛的应用。然而，在生产环境的实际运维过程中，K8s集群的复杂性带来了诸多挑战。从Pod生命周期管理到节点资源调度，从网络通信到存储持久化，每一个环节都可能成为故障的爆发点。

运维工程师不仅需要理解K8s的核心架构和工作原理，更需要掌握系统化的故障排查方法论。本文将深入剖析K8s运维过程中最常见的问题类型，从现象识别、根因分析到解决方案实施，为读者提供一套完整的故障处理指南。通过真实场景的案例分析和详细的排查步骤，帮助读者建立从问题发现到彻底解决的完整能力链路。

## 一、问题分类总览

在深入具体问题之前，我们先对K8s运维常见问题进行分类梳理：

| 问题类别 | 典型问题 | 影响范围 | 紧急程度 |
|---------|---------|---------|---------|
| Pod问题 | CrashLoopBackOff、ImagePullBackOff、OOMKilled | 应用服务 | 高 |
| 节点问题 | NotReady、资源不足、磁盘压力 | 集群调度 | 高 |
| 网络问题 | DNS解析失败、服务访问不通、网络策略冲突 | 服务通信 | 高 |
| 存储问题 | PV绑定失败、数据丢失、存储容量不足 | 数据持久化 | 高 |
| 证书问题 | API Server证书过期、组件证书过期 | 集群访问 | 严重 |
| 调度问题 | Pod Pending、节点亲和性冲突 | 资源分配 | 中 |
| 配置问题 | ConfigMap更新不生效、Secret挂载失败 | 应用配置 | 中 |

## 二、Pod常见问题

### 2.1 CrashLoopBackOff

#### 现象描述

Pod反复重启，状态显示为CrashLoopBackOff，通过`kubectl get pods`可以看到RESTARTS计数不断增加：

```bash
NAME                    READY   STATUS             RESTARTS   AGE
nginx-deployment-xxx    0/1     CrashLoopBackOff   5          10m
```

#### 原因分析

CrashLoopBackOff的本质是容器启动后立即退出，Kubelet根据重启策略不断尝试重启容器，但每次都失败。常见原因包括：

1. **应用程序错误**：应用程序代码存在致命错误，如空指针异常、配置文件缺失
2. **启动命令错误**：容器启动命令配置不当，或命令执行后立即退出
3. **健康检查失败**：livenessProbe配置不当，导致容器被误杀
4. **资源限制**：内存limit设置过低，触发OOMKilled
5. **依赖服务未就绪**：应用依赖的数据库、配置中心等服务不可用

#### 排查步骤

**步骤一：查看Pod详细信息**

```bash
kubectl describe pod <pod-name> -n <namespace>
```

重点关注Events部分和Last State字段，Last State会显示容器退出的原因和退出码。

**步骤二：查看容器日志**

```bash
# 查看当前容器日志
kubectl logs <pod-name> -n <namespace>

# 如果容器已重启，查看上一个容器的日志
kubectl logs <pod-name> -n <namespace> --previous
```

**步骤三：检查资源使用情况**

```bash
# 查看Pod资源使用
kubectl top pod <pod-name> -n <namespace>

# 查看Pod资源限制
kubectl get pod <pod-name> -n <namespace> -o yaml | grep -A 5 resources
```

**步骤四：进入容器调试**

```bash
# 如果容器能短暂运行，进入容器调试
kubectl exec -it <pod-name> -n <namespace> -- /bin/sh
```

#### 解决方案

根据排查结果采取针对性措施：

**场景一：应用启动命令问题**

对于需要持续运行的容器，确保启动命令不会立即退出：

```yaml
spec:
  containers:
  - name: nginx
    image: nginx:latest
    command: ["/bin/sh", "-c"]
    args: ["nginx -g 'daemon off;'"]  # 保持前台运行
```

**场景二：健康检查配置不当**

调整livenessProbe的初始延迟时间和检查频率：

```yaml
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 30  # 增加初始延迟
  periodSeconds: 10
  failureThreshold: 3      # 增加失败阈值
```

**场景三：资源不足**

调整资源限制，特别是内存limit：

```yaml
resources:
  requests:
    memory: "256Mi"
    cpu: "250m"
  limits:
    memory: "512Mi"  # 适当增加内存限制
    cpu: "500m"
```

### 2.2 ImagePullBackOff

#### 现象描述

Pod状态显示ImagePullBackOff或ErrImagePull，容器无法启动：

```bash
NAME                    READY   STATUS             RESTARTS   AGE
nginx-deployment-xxx    0/1     ImagePullBackOff   0          5m
```

#### 原因分析

镜像拉取失败的根本原因包括：

1. **镜像不存在**：镜像名称错误或镜像未推送到仓库
2. **认证失败**：私有镜像仓库未配置正确的Secret
3. **网络问题**：无法访问镜像仓库，特别是国外镜像源
4. **镜像仓库配额**：Docker Hub等仓库的拉取次数限制
5. **镜像标签错误**：使用了不存在的镜像标签

#### 排查步骤

**步骤一：查看详细错误信息**

```bash
kubectl describe pod <pod-name> -n <namespace>
```

在Events部分查看具体的错误信息，如"unauthorized"、"not found"等。

**步骤二：验证镜像是否存在**

```bash
# 在节点上手动拉取镜像测试
docker pull <image-name>:<tag>

# 或使用crictl（containerd环境）
crictl pull <image-name>:<tag>
```

**步骤三：检查镜像拉取Secret**

```bash
# 查看Pod配置的imagePullSecrets
kubectl get pod <pod-name> -n <namespace> -o yaml | grep -A 3 imagePullSecrets

# 查看Secret内容
kubectl get secret <secret-name> -n <namespace> -o yaml
```

#### 解决方案

**场景一：私有镜像仓库认证**

创建Docker Registry Secret：

```bash
kubectl create secret docker-registry regcred \
  --docker-server=<your-registry-server> \
  --docker-username=<your-username> \
  --docker-password=<your-password> \
  --docker-email=<your-email> \
  -n <namespace>
```

在Pod或Deployment中引用Secret：

```yaml
spec:
  imagePullSecrets:
  - name: regcred
  containers:
  - name: nginx
    image: your-registry/nginx:latest
```

**场景二：镜像仓库网络问题**

配置镜像加速器或使用国内镜像源：

```yaml
# containerd配置 /etc/containerd/config.toml
[plugins."io.containerd.grpc.v1.cri".registry.mirrors]
  [plugins."io.containerd.grpc.v1.cri".registry.mirrors."docker.io"]
    endpoint = ["https://mirror.ccs.tencentyun.com"]
```

**场景三：镜像标签问题**

确保使用正确的镜像标签，避免使用latest标签：

```yaml
spec:
  containers:
  - name: nginx
    image: nginx:1.21.6  # 使用明确的版本号
```

### 2.3 OOMKilled

#### 现象描述

Pod被系统OOM Killer杀死，状态显示OOMKilled：

```bash
NAME                    READY   STATUS      RESTARTS   AGE
java-app-xxx           0/1     OOMKilled   3          15m
```

#### 原因分析

OOMKilled表示容器内存使用超过了limit限制，触发Linux内核的OOM Killer机制。常见原因：

1. **内存limit设置过低**：未正确评估应用内存需求
2. **内存泄漏**：应用程序存在内存泄漏，内存持续增长
3. **JVM配置不当**：Java应用未正确设置堆内存参数
4. **并发请求过多**：突发流量导致内存激增

#### 排查步骤

**步骤一：确认OOMKilled状态**

```bash
kubectl describe pod <pod-name> -n <namespace>
```

在Last State中查看Exit Code，OOMKilled的退出码为137。

**步骤二：查看历史资源使用**

```bash
# 使用Prometheus查询历史内存使用
container_memory_usage_bytes{pod="<pod-name>"}

# 或使用kubectl top查看当前资源
kubectl top pod <pod-name> -n <namespace>
```

**步骤三：分析内存使用情况**

对于Java应用，启用JVM监控：

```bash
# 进入容器查看JVM内存使用
kubectl exec -it <pod-name> -n <namespace> -- jmap -heap <pid>
```

#### 解决方案

**场景一：调整内存限制**

根据实际内存需求调整limit：

```yaml
resources:
  requests:
    memory: "1Gi"
  limits:
    memory: "2Gi"  # 增加内存限制
```

**场景二：Java应用JVM调优**

设置合理的JVM堆内存参数：

```yaml
env:
- name: JAVA_OPTS
  value: "-Xms1g -Xmx1536m -XX:+UseG1GC"
```

注意：JVM堆内存应小于容器内存limit，预留内存给Metaspace、线程栈等。

**场景三：配置QoS等级**

对于关键应用，配置Guaranteed QoS：

```yaml
resources:
  requests:
    memory: "2Gi"
    cpu: "1"
  limits:
    memory: "2Gi"  # requests和limits相同
    cpu: "1"
```

## 三、节点常见问题

### 3.1 节点NotReady

#### 现象描述

节点状态显示NotReady，无法调度新的Pod：

```bash
NAME         STATUS     ROLES    AGE   VERSION
node-01      NotReady   master   30d   v1.24.0
node-02      Ready      worker   30d   v1.24.0
```

#### 原因分析

节点NotReady通常由以下原因导致：

1. **Kubelet服务异常**：Kubelet进程崩溃或停止
2. **容器运行时异常**：Docker或containerd服务异常
3. **网络插件问题**：CNI插件未正常运行
4. **证书过期**：Kubelet客户端证书过期
5. **资源耗尽**：节点磁盘、内存等资源耗尽
6. **时钟不同步**：节点时间与集群时间偏差过大

#### 排查步骤

**步骤一：查看节点详细信息**

```bash
kubectl describe node <node-name>
```

重点关注Conditions部分，查看具体的不健康状态。

**步骤二：在节点上检查服务状态**

```bash
# 检查Kubelet服务
systemctl status kubelet

# 检查容器运行时
systemctl status docker  # 或 containerd

# 查看Kubelet日志
journalctl -u kubelet -f
```

**步骤三：检查节点资源**

```bash
# 查看磁盘使用
df -h

# 查看内存使用
free -m

# 查看系统负载
top
```

**步骤四：检查网络插件**

```bash
# 查看CNI插件Pod状态
kubectl get pods -n kube-system | grep -E 'flannel|calico|weave'

# 查看CNI配置
ls -la /etc/cni/net.d/
```

#### 解决方案

**场景一：Kubelet服务异常**

```bash
# 重启Kubelet服务
systemctl restart kubelet

# 设置开机自启
systemctl enable kubelet
```

**场景二：容器运行时异常**

```bash
# Docker环境
systemctl restart docker

# Containerd环境
systemctl restart containerd
```

**场景三：磁盘压力**

清理无用资源：

```bash
# 清理无用容器
docker system prune -a

# 清理无用镜像
crictl rmi --prune

# 清理日志文件
find /var/log -type f -name "*.log" -mtime +7 -delete
```

**场景四：证书过期**

```bash
# 检查证书有效期
kubeadm certs check-expiration

# 更新证书
kubeadm certs renew all

# 重启Kubelet
systemctl restart kubelet
```

### 3.2 节点资源不足

#### 现象描述

节点资源使用率高，Pod无法调度或被驱逐：

```bash
# Pod调度失败
Warning  FailedScheduling  pod xxx  0/3 nodes are available: 3 Insufficient cpu.

# 节点压力驱逐
Warning  Evicted  pod xxx  The node was low on resource: ephemeral-storage.
```

#### 原因分析

资源不足的原因包括：

1. **过度调度**：节点上运行的Pod过多，资源requests总和超过节点容量
2. **资源泄漏**：僵尸进程、未清理的容器占用资源
3. **日志文件过大**：应用日志未做轮转，占用大量磁盘空间
4. **镜像文件过多**：历史镜像未清理，占用磁盘空间

#### 排查步骤

**步骤一：查看节点资源使用**

```bash
# 查看节点资源概览
kubectl describe node <node-name>

# 查看资源分配情况
kubectl get node <node-name> -o yaml | grep -A 10 allocatable

# 实时监控资源
kubectl top node
```

**步骤二：查看Pod资源使用**

```bash
# 查看所有Pod资源使用
kubectl top pods -A --sort-by=memory

# 查看特定命名空间的Pod
kubectl top pods -n <namespace> --sort-by=cpu
```

**步骤三：在节点上检查系统资源**

```bash
# 查看磁盘使用
df -h

# 查看大文件
du -sh /* | sort -hr | head -10

# 查看Docker资源使用
docker system df
```

#### 解决方案

**场景一：资源调度优化**

设置资源requests和limits，避免过度调度：

```yaml
resources:
  requests:
    cpu: "500m"
    memory: "512Mi"
  limits:
    cpu: "1000m"
    memory: "1Gi"
```

**场景二：配置资源配额**

使用ResourceQuota限制命名空间资源总量：

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: compute-quota
  namespace: production
spec:
  hard:
    requests.cpu: "10"
    requests.memory: 20Gi
    limits.cpu: "20"
    limits.memory: 40Gi
```

**场景三：清理磁盘空间**

```bash
# 清理Docker资源
docker system prune -a --volumes

# 清理containerd资源
crictl rmi --prune

# 清理日志
find /var/log -type f -name "*.log" -size +100M -delete
```

**场景四：配置日志轮转**

配置容器日志轮转策略：

```json
// /etc/docker/daemon.json
{
  "log-opts": {
    "max-size": "100m",
    "max-file": "3"
  }
}
```

## 四、网络常见问题

### 4.1 DNS解析失败

#### 现象描述

Pod内无法解析域名，应用访问外部服务失败：

```bash
# 在Pod内测试
kubectl exec -it <pod-name> -- nslookup kubernetes
;; connection timed out; no servers could be reached

# 应用日志
java.net.UnknownHostException: api.example.com
```

#### 原因分析

DNS解析失败的常见原因：

1. **CoreDNS服务异常**：CoreDNS Pod未正常运行
2. **CoreDNS配置错误**：CoreDNS ConfigMap配置不当
3. **Pod DNS配置错误**：Pod的dnsPolicy或dnsConfig配置不当
4. **网络策略限制**：NetworkPolicy阻止了DNS流量
5. **上游DNS问题**：外部DNS服务器不可用

#### 排查步骤

**步骤一：检查CoreDNS服务**

```bash
# 查看CoreDNS Pod状态
kubectl get pods -n kube-system -l k8s-app=kube-dns

# 查看CoreDNS Service
kubectl get svc -n kube-system kube-dns

# 查看CoreDNS日志
kubectl logs -n kube-system -l k8s-app=kube-dns
```

**步骤二：测试DNS解析**

```bash
# 创建测试Pod
kubectl run dns-test --image=busybox:1.28 --rm -it --restart=Never -- nslookup kubernetes

# 测试外部域名
kubectl run dns-test --image=busybox:1.28 --rm -it --restart=Never -- nslookup www.baidu.com
```

**步骤三：检查Pod DNS配置**

```bash
# 查看Pod的DNS配置
kubectl exec -it <pod-name> -- cat /etc/resolv.conf

# 查看Pod的DNS策略
kubectl get pod <pod-name> -o yaml | grep -A 5 dnsPolicy
```

**步骤四：检查CoreDNS ConfigMap**

```bash
kubectl get configmap coredns -n kube-system -o yaml
```

#### 解决方案

**场景一：CoreDNS服务异常**

重启CoreDNS Pod：

```bash
kubectl rollout restart deployment coredns -n kube-system
```

**场景二：CoreDNS配置优化**

修改CoreDNS ConfigMap，添加上游DNS服务器：

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: coredns
  namespace: kube-system
data:
  Corefile: |
    .:53 {
        errors
        health
        ready
        kubernetes cluster.local in-addr.arpa ip6.arpa {
           pods insecure
           fallthrough in-addr.arpa ip6.arpa
        }
        forward . /etc/resolv.conf 8.8.8.8 114.114.114.114
        cache 30
        loop
        reload
        loadbalance
    }
```

**场景三：Pod DNS配置**

使用正确的DNS策略：

```yaml
spec:
  dnsPolicy: ClusterFirst  # 优先使用集群DNS
  dnsConfig:
    nameservers:
    - 8.8.8.8
    searches:
    - default.svc.cluster.local
    - svc.cluster.local
    - cluster.local
    options:
    - name: ndots
      value: "2"
```

### 4.2 Service访问不通

#### 现象描述

Service无法访问后端Pod，或Service ClusterIP无法访问：

```bash
# 访问Service失败
curl http://<service-clusterip>:<port>
curl: (7) Failed to connect to <clusterip> port <port>: Connection refused

# Service Endpoints为空
kubectl get endpoints <service-name>
NAME         ENDPOINTS
my-service   <none>
```

#### 原因分析

Service访问不通的原因：

1. **Selector标签不匹配**：Service的selector与Pod标签不匹配
2. **Pod未就绪**：Pod的readinessProbe检查失败
3. **端口配置错误**：Service端口或targetPort配置错误
4. **kube-proxy异常**：kube-proxy未正常运行
5. **iptables/ipvs规则错误**：负载均衡规则未正确生成

#### 排查步骤

**步骤一：检查Service配置**

```bash
# 查看Service详细信息
kubectl describe svc <service-name>

# 检查Selector
kubectl get svc <service-name> -o yaml | grep -A 3 selector

# 检查匹配的Pod
kubectl get pods -l <label-selector>
```

**步骤二：检查Endpoints**

```bash
# 查看Endpoints
kubectl get endpoints <service-name>

# 查看Endpoints详细信息
kubectl describe endpoints <service-name>
```

**步骤三：检查Pod状态**

```bash
# 查看Pod标签
kubectl get pods --show-labels

# 查看Pod端口
kubectl get pod <pod-name> -o yaml | grep -A 5 ports
```

**步骤四：检查kube-proxy**

```bash
# 查看kube-proxy Pod
kubectl get pods -n kube-system -l k8s-app=kube-proxy

# 查看kube-proxy日志
kubectl logs -n kube-proxy-<node-name> -n kube-system
```

#### 解决方案

**场景一：Selector标签匹配**

确保Service selector与Pod标签一致：

```yaml
# Service配置
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  selector:
    app: nginx    # 必须与Pod标签匹配
  ports:
  - port: 80
    targetPort: 80

# Pod标签
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx    # 与Service selector匹配
```

**场景二：Pod就绪检查**

调整readinessProbe配置：

```yaml
readinessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 5
  failureThreshold: 3
```

**场景三：kube-proxy重启**

```bash
# 删除kube-proxy Pod，让其自动重建
kubectl delete pod -n kube-system -l k8s-app=kube-proxy
```

## 五、存储常见问题

### 5.1 PV绑定失败

#### 现象描述

PVC一直处于Pending状态，无法绑定PV：

```bash
kubectl get pvc
NAME         STATUS    VOLUME   CAPACITY   ACCESS MODES   STORAGECLASS   AGE
my-pvc       Pending                                       standard       5m
```

#### 原因分析

PV绑定失败的原因：

1. **没有可用的PV**：集群中没有满足PVC需求的PV
2. **StorageClass问题**：动态供给未配置或StorageClass不存在
3. **访问模式不匹配**：PV的访问模式与PVC需求不匹配
4. **容量不足**：PV容量小于PVC请求的容量
5. **PV被其他PVC占用**：PV已被绑定，无法重复使用

#### 排查步骤

**步骤一：查看PVC详细信息**

```bash
kubectl describe pvc <pvc-name>
```

查看Events中的错误信息。

**步骤二：检查可用PV**

```bash
# 查看所有PV
kubectl get pv

# 查看可用PV
kubectl get pv | grep Available
```

**步骤三：检查StorageClass**

```bash
# 查看StorageClass
kubectl get storageclass

# 查看StorageClass详细信息
kubectl describe storageclass <storageclass-name>
```

#### 解决方案

**场景一：创建静态PV**

手动创建PV：

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: my-pv
spec:
  capacity:
    storage: 10Gi
  accessModes:
  - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: standard
  nfs:
    server: 192.168.1.100
    path: /data/pv1
```

**场景二：配置动态供给**

配置StorageClass支持动态供给：

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: standard
provisioner: kubernetes.io/aws-ebs  # 或其他provisioner
parameters:
  type: gp2
reclaimPolicy: Delete
allowVolumeExpansion: true
```

**场景三：调整访问模式和容量**

确保PV和PVC的访问模式、容量匹配：

```yaml
# PVC配置
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes:
  - ReadWriteOnce    # 与PV访问模式匹配
  resources:
    requests:
      storage: 5Gi   # 小于等于PV容量
  storageClassName: standard
```

### 5.2 数据丢失风险

#### 现象描述

Pod重启或迁移后，数据丢失，持久化存储未生效。

#### 原因分析

数据丢失的原因：

1. **未使用持久化存储**：应用数据存储在容器内部
2. **PV回收策略错误**：PV的persistentVolumeReclaimPolicy设置为Delete
3. **存储后端故障**：NFS、Ceph等存储后端异常
4. **数据未同步**：应用未及时将数据写入持久化存储

#### 排查步骤

**步骤一：检查Pod挂载**

```bash
# 查看Pod挂载信息
kubectl describe pod <pod-name>

# 进入Pod查看挂载点
kubectl exec -it <pod-name> -- df -h
kubectl exec -it <pod-name> -- mount | grep <mount-path>
```

**步骤二：检查PV回收策略**

```bash
kubectl get pv <pv-name> -o yaml | grep persistentVolumeReclaimPolicy
```

**步骤三：检查存储后端**

```bash
# NFS存储检查
showmount -e <nfs-server>
mount -t nfs <nfs-server>:<path> /mnt/test

# Ceph存储检查
ceph status
```

#### 解决方案

**场景一：正确配置持久化存储**

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: data
      mountPath: /data    # 应用数据目录
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: my-pvc
```

**场景二：设置正确的回收策略**

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: my-pv
spec:
  persistentVolumeReclaimPolicy: Retain  # 保留数据
  # 其他配置...
```

**场景三：数据备份策略**

定期备份PV数据：

```bash
# 使用Velero备份
velero backup create my-backup --include-namespaces production

# 定时备份
velero schedule create daily-backup --schedule="0 2 * * *"
```

## 六、证书过期问题

### 6.1 API Server证书过期

#### 现象描述

集群突然无法访问，kubectl命令失败：

```bash
kubectl get nodes
Unable to connect to the server: x509: certificate has expired or is not yet valid
```

#### 原因分析

K8s证书默认有效期为1年，证书过期后会导致：

1. **API Server无法访问**：客户端证书过期
2. **组件通信失败**：组件间证书过期
3. **Kubelet无法注册**：Kubelet证书过期

#### 排查步骤

**步骤一：检查证书有效期**

```bash
# 查看所有证书有效期
kubeadm certs check-expiration

# 查看特定证书
openssl x509 -in /etc/kubernetes/pki/apiserver.crt -noout -text | grep -A 2 Validity
```

**步骤二：检查系统时间**

```bash
# 确认系统时间正确
date

# 检查时间同步
timedatectl status
```

#### 解决方案

**场景一：使用kubeadm更新证书**

```bash
# 更新所有证书
kubeadm certs renew all

# 重启控制平面组件
kubectl delete pod -n kube-system kube-apiserver-<node-name>
kubectl delete pod -n kube-system kube-controller-manager-<node-name>
kubectl delete pod -n kube-system kube-scheduler-<node-name>

# 更新kubeconfig
cp /etc/kubernetes/admin.conf ~/.kube/config
```

**场景二：更新Kubelet证书**

```bash
# 在节点上执行
kubeadm certs renew kubelet.conf
systemctl restart kubelet
```

**场景三：配置自动证书轮转**

Kubelet默认启用自动证书轮转，确保配置正确：

```yaml
# /var/lib/kubelet/config.yaml
rotateCertificates: true
```

## 七、常见问题快速参考

### 7.1 问题诊断流程图

```
Pod异常 → kubectl describe pod → 查看Events
    ↓
状态为Pending → 检查资源、节点、调度策略
    ↓
状态为Running但异常 → 查看日志、进入容器调试
    ↓
状态为Error/CrashLoopBackOff → 查看退出码、日志
```

### 7.2 常用排查命令速查表

| 场景 | 命令 | 说明 |
|-----|------|------|
| 查看Pod详情 | `kubectl describe pod <pod-name>` | 查看Pod事件和状态 |
| 查看Pod日志 | `kubectl logs <pod-name> --previous` | 查看上一个容器日志 |
| 进入容器 | `kubectl exec -it <pod-name> -- /bin/sh` | 进入容器调试 |
| 查看资源使用 | `kubectl top pods/nodes` | 实时资源监控 |
| 查看节点状态 | `kubectl describe node <node-name>` | 节点详细状态 |
| 查看Service | `kubectl get endpoints <svc-name>` | Service后端Pod |
| 查看DNS | `kubectl logs -n kube-system -l k8s-app=kube-dns` | CoreDNS日志 |
| 查看证书 | `kubeadm certs check-expiration` | 证书有效期 |
| 查看事件 | `kubectl get events --sort-by='.lastTimestamp'` | 集群事件 |

### 7.3 最佳实践建议

**资源管理**

1. 为所有Pod设置合理的requests和limits
2. 使用ResourceQuota限制命名空间资源总量
3. 使用LimitRange设置默认资源限制
4. 定期审计资源使用情况

**健康检查**

1. 配置livenessProbe和readinessProbe
2. 设置合理的初始延迟和检查间隔
3. 健康检查端点应轻量快速
4. 避免健康检查依赖外部服务

**日志和监控**

1. 配置日志轮转策略，避免磁盘占满
2. 使用集中式日志收集（ELK、Loki等）
3. 配置关键指标告警
4. 保留历史监控数据用于问题分析

**备份和恢复**

1. 定期备份etcd数据
2. 使用Velero备份PV和资源配置
3. 制定灾难恢复计划
4. 定期演练恢复流程

**证书管理**

1. 监控证书有效期，提前更新
2. 配置Kubelet自动证书轮转
3. 使用cert-manager管理证书
4. 建立证书更新SOP

## 八、面试回答

在面试中回答"K8s运维过程中遇到过哪些问题，如何解决的"这个问题时，可以这样组织回答：

"在K8s运维过程中，我遇到过多种类型的问题，主要包括Pod问题、节点问题、网络问题、存储问题和证书问题。以Pod问题为例，最常见的CrashLoopBackOff问题，我会先通过kubectl describe pod查看Pod事件和容器退出状态，然后使用kubectl logs --previous查看上一个容器的日志，分析是应用代码错误、资源不足还是健康检查配置不当导致。对于ImagePullBackOff，我会检查镜像是否存在、认证Secret是否正确配置、网络是否可达。节点NotReady问题通常需要登录节点检查kubelet和容器运行时服务状态、查看系统资源使用情况。网络问题中DNS解析失败比较常见，我会检查CoreDNS服务状态、ConfigMap配置以及Pod的DNS策略。存储问题主要关注PV绑定失败，需要检查StorageClass配置、PV容量和访问模式是否匹配。证书过期是严重问题，我会定期使用kubeadm certs check-expiration检查证书有效期，提前更新证书并重启相关组件。我的排查思路是：先看现象，再查日志，然后分析原因，最后针对性解决，同时建立监控告警机制，在问题影响业务前及时发现和处理。"
