# Kubernetes 标准输出日志收集：从原理到实践

## 引言：什么是标准输出日志？

在 Kubernetes 环境中，容器化应用的日志管理是一个核心问题。标准输出日志是指应用程序将日志信息输出到 stdout（标准输出）和 stderr（标准错误）的日志方式。这是容器化应用最推荐的日志输出方式，因为 Kubernetes 和容器运行时能够自动捕获、存储和管理这些日志。

与传统的文件日志不同，标准输出日志无需应用程序关心日志文件的创建、轮转和清理，这些工作由容器运行时和 Kubernetes 自动完成。这种设计理念遵循了"关注点分离"原则，让应用专注于业务逻辑，而将日志管理交给基础设施层处理。

标准输出日志的优势包括：

- **统一管理**：所有容器日志遵循统一的存储格式和位置
- **自动轮转**：容器运行时自动处理日志轮转，防止磁盘空间耗尽
- **易于收集**：日志采集器可以通过统一的接口读取所有容器日志
- **kubectl 集成**：可以通过 `kubectl logs` 命令直接查看容器日志

## 标准输出日志的存储位置

### 容器运行时的日志处理流程

当应用程序向 stdout 或 stderr 写入日志时，容器运行时（Docker、containerd 或 CRI-O）会捕获这些输出并进行处理。整个流程如下：

```
应用程序
    ↓
stdout/stderr
    ↓
容器运行时（Docker/containerd/CRI-O）
    ├── 捕获输出流
    ├── 添加元数据（时间戳、流类型）
    ├── 格式化为 JSON
    └── 写入日志文件
    ↓
节点文件系统
```

### 日志文件的具体位置

根据容器运行时的不同，日志文件的存储位置也有所差异：

**Docker 运行时**：
```
/var/lib/docker/containers/<container-id>/<container-id>-json.log
```

**containerd 运行时**：
```
/var/log/pods/<namespace>_<pod-name>_<pod-uid>/<container-name>/<restart-count>.log
```

**Kubernetes 符号链接**（统一接口）：
```
/var/log/containers/<pod-name>_<namespace>_<container-name>-<container-id>.log
```

这些符号链接指向实际的日志文件，为日志采集器提供统一的访问接口。

### 日志文件格式

容器运行时将标准输出日志以 JSON 格式存储，每行一条日志记录：

```json
{
  "log": "2024-01-15T10:30:00Z INFO User login successful\n",
  "stream": "stdout",
  "time": "2024-01-15T10:30:00.123456789Z"
}
```

字段说明：
- `log`：实际的日志内容
- `stream`：日志来源（stdout 或 stderr）
- `time`：容器运行时捕获日志的时间戳

### 查看日志文件

可以通过以下方式查看日志文件：

```bash
# 使用 kubectl 查看容器日志
kubectl logs <pod-name> -n <namespace>

# 查看前一个容器的日志（容器重启后）
kubectl logs <pod-name> -n <namespace> --previous

# 在节点上直接查看日志文件
ls -la /var/log/containers/
cat /var/log/containers/<pod-name>_<namespace>_<container-name>-<container-id>.log
```

## 日志收集架构

### 整体架构设计

Kubernetes 标准输出日志的收集架构通常采用 DaemonSet 模式，在每个节点上部署一个日志采集器 Pod。这种架构的优势在于：

- **资源效率高**：每个节点只需一个采集器实例
- **运维简单**：统一的配置和管理
- **性能开销小**：避免跨节点传输日志数据

```
┌─────────────────────────────────────────────────────────────┐
│                    Kubernetes 集群                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Node 1                                                     │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐                    │
│  │ Pod A    │ │ Pod B    │ │ Pod C    │                    │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘                    │
│       │            │            │                           │
│       └────────────┼────────────┘                           │
│                    ▼                                        │
│            ┌──────────────┐                                │
│            │  Log Agent   │                                │
│            │  (DaemonSet) │                                │
│            └──────┬───────┘                                │
│                   │                                         │
├───────────────────┼─────────────────────────────────────────┤
│  Node 2           │                                         │
│  ┌──────────┐     │                                         │
│  │ Pod D    │     │                                         │
│  └────┬─────┘     │                                         │
│       │           │                                         │
│       └───────────┼──► Log Agent (DaemonSet)                │
│                   │                                         │
└───────────────────┼─────────────────────────────────────────┘
                    │
                    ▼
          ┌──────────────────┐
          │  日志存储后端      │
          │  (ES/Loki/Kafka) │
          └──────────────────┘
```

### 日志采集器的工作原理

日志采集器通过以下步骤收集标准输出日志：

1. **监听日志文件**：通过 inotify 或轮询机制监听 `/var/log/containers/` 目录下的日志文件
2. **读取日志内容**：从日志文件的末尾开始读取新日志（类似 `tail -f`）
3. **解析日志格式**：解析 JSON 格式的日志记录
4. **添加 Kubernetes 元数据**：通过 Kubernetes API 获取 Pod、Namespace、Labels 等信息
5. **过滤和转换**：根据配置规则过滤、修改日志内容
6. **发送到后端**：将处理后的日志发送到 Elasticsearch、Loki 或 Kafka 等存储系统

## 常用收集方案对比

### Fluent Bit

Fluent Bit 是一个轻量级的日志采集器，专为云原生环境设计。它是 Fluentd 项目的子项目，具有更小的内存占用和更快的启动速度。

**核心特性**：
- 内存占用低（约 10-20MB）
- 支持多种输入、过滤和输出插件
- 原生支持 Kubernetes 元数据
- 配置简单，易于维护

**适用场景**：
- 资源受限的环境
- 边缘计算场景
- 大规模 Kubernetes 集群

### Fluentd

Fluentd 是一个成熟的日志收集器，拥有丰富的插件生态系统。相比 Fluent Bit，它提供了更强大的数据处理能力。

**核心特性**：
- 插件生态丰富（1000+ 插件）
- 支持复杂的日志处理逻辑
- Ruby 插件支持自定义处理
- 社区成熟，文档完善

**适用场景**：
- 需要复杂日志处理逻辑
- 多种数据源和目标
- 企业级日志系统

### Filebeat

Filebeat 是 Elastic 公司出品的轻量级日志采集器，与 Elasticsearch、Kibana 深度集成。

**核心特性**：
- 与 ELK 栈无缝集成
- 自动发现 Kubernetes Pod
- 内置模块支持常见日志格式
- 性能优异，资源占用低

**适用场景**：
- 使用 ELK 栈作为日志后端
- 需要快速部署和配置
- 对日志搜索和分析有高要求

### 方案对比表格

| 对比维度 | Fluent Bit | Fluentd | Filebeat |
|---------|-----------|---------|----------|
| 内存占用 | 10-20MB | 100-200MB | 20-50MB |
| CPU 占用 | 低 | 中 | 低 |
| 插件数量 | 100+ | 1000+ | 50+ |
| 配置复杂度 | 简单 | 中等 | 简单 |
| Kubernetes 支持 | 原生 | 原生 | 原生 |
| 日志处理能力 | 中 | 强 | 弱 |
| 社区活跃度 | 高 | 高 | 高 |
| 学习曲线 | 平缓 | 陡峭 | 平缓 |
| 适用规模 | 大型 | 中大型 | 中大型 |
| 推荐指数 | ★★★★★ | ★★★★ | ★★★★ |

**选型建议**：
- **资源受限环境**：选择 Fluent Bit
- **需要复杂处理**：选择 Fluentd
- **使用 ELK 栈**：选择 Filebeat
- **快速上手**：选择 Fluent Bit 或 Filebeat

```yaml

kind: DaemonSet
metadata:
  name: fluent-bit
  namespace: logging
  labels:
    app: fluent-bit
spec:
  selector:
    matchLabels:
      app: fluent-bit
  template:
    metadata:
      labels:
        app: fluent-bit
    spec:
      serviceAccountName: fluent-bit
      tolerations:
      - key: node-role.kubernetes.io/master
        effect: NoSchedule
      containers:
      - name: fluent-bit
        image: fluent/fluent-bit:2.1
        imagePullPolicy: IfNotPresent
        ports:
        - containerPort: 2020
          name: metrics
        volumeMounts:
        - name: varlog
          mountPath: /var/log
          readOnly: true
        - name: varlibdockercontainers
          mountPath: /var/lib/docker/containers
          readOnly: true
        - name: config
          mountPath: /fluent-bit/etc/
        resources:
          limits:
            cpu: 500m
            memory: 200Mi
          requests:
            cpu: 100m
            memory: 50Mi
      volumes:
      - name: varlog
        hostPath:
          path: /var/log
      - name: varlibdockercontainers
        hostPath:
          path: /var/lib/docker/containers
      - name: config
        configMap:
          name: fluent-bit-config
```

## 日志轮转机制

### 为什么需要日志轮转？

容器持续运行会产生大量日志，如果不进行轮转，会导致以下问题：

- **磁盘空间耗尽**：日志文件无限增长，占用节点磁盘空间
- **性能下降**：大文件读写性能下降
- **排查困难**：单个日志文件过大，难以快速定位问题

### Kubernetes 日志轮转机制

Kubernetes 通过 kubelet 和容器运行时共同实现日志轮转：

**kubelet 配置**（`/var/lib/kubelet/config.yaml`）：

```yaml
containerLogMaxSize: 10Mi      # 单个日志文件最大大小
containerLogMaxFiles: 5        # 保留的历史日志文件数量
```

**轮转流程**：

```
当前日志文件（<container-id>-json.log）
    ↓
达到最大大小（containerLogMaxSize）
    ↓
重命名为 <container-id>-json.log.1
    ↓
创建新的日志文件
    ↓
保留最新的 N 个文件（containerLogMaxFiles）
    ↓
删除最旧的日志文件
```

### 日志轮转示例

假设 `containerLogMaxSize=10Mi`，`containerLogMaxFiles=5`，日志轮转过程如下：

```
/var/lib/docker/containers/abc123/
├── abc123-json.log       # 当前活跃日志文件
├── abc123-json.log.1     # 第一次轮转的日志
├── abc123-json.log.2     # 第二次轮转的日志
├── abc123-json.log.3     # 第三次轮转的日志
└── abc123-json.log.4     # 第四次轮转的日志
```

当 `abc123-json.log` 达到 10Mi 时，会发生以下操作：

1. `abc123-json.log.4` 被删除
2. `abc123-json.log.3` 重命名为 `abc123-json.log.4`
3. `abc123-json.log.2` 重命名为 `abc123-json.log.3`
4. `abc123-json.log.1` 重命名为 `abc123-json.log.2`
5. `abc123-json.log` 重命名为 `abc123-json.log.1`
6. 创建新的 `abc123-json.log`

### 日志轮转最佳实践

**配置建议**：

| 场景 | containerLogMaxSize | containerLogMaxFiles | 总磁盘占用 |
|------|---------------------|----------------------|-----------|
| 开发环境 | 10Mi | 3 | 30Mi/容器 |
| 生产环境 | 50Mi | 5 | 250Mi/容器 |
| 高日志量应用 | 100Mi | 10 | 1Gi/容器 |

**注意事项**：

1. **磁盘空间规划**：确保节点有足够的磁盘空间存储日志
2. **日志采集延迟**：日志轮转时，采集器需要及时处理已轮转的文件
3. **监控告警**：监控节点磁盘使用率，及时清理过期日志
4. **应用日志级别**：生产环境使用合理的日志级别，避免产生过多日志

## 常见问题与最佳实践

### 常见问题

#### 1. 日志丢失怎么办？

**问题现象**：部分日志未能成功收集到后端存储。

**排查步骤**：

```bash
# 1. 检查日志采集器状态
kubectl get pods -n logging -l app=fluent-bit

# 2. 查看采集器日志
kubectl logs -n logging -l app=fluent-bit --tail=100

# 3. 检查节点日志文件
kubectl exec -n logging fluent-bit-xxxxx -- ls -la /var/log/containers/

# 4. 检查采集器指标
kubectl port-forward -n logging svc/fluent-bit 2020:2020
curl http://localhost:2020/api/v1/metrics
```

**常见原因及解决方案**：

| 原因 | 解决方案 |
|------|---------|
| Pod 已删除 | 使用集中日志系统，及时采集 |
| 磁盘空间满 | 配置日志轮转，清理旧日志 |
| 采集器崩溃 | 检查资源限制，增加资源 |
| 网络问题 | 检查输出端点连通性 |
| 日志轮转过快 | 调整轮转参数，增加采集速度 |

#### 2. 如何处理多行日志？

**问题场景**：Java 堆栈跟踪、Python 异常等多行日志被拆分成多条记录。

**解决方案**：使用 Fluent Bit 的多行解析器：

```ini
[MULTILINE_PARSER]
    Name          java-multiline
    Type          regex
    Flush_Timeout 1000
    Rule      "start_state"  "/^\d{4}-\d{1,2}-\d{1,2}/"  "cont"
    Rule      "cont"         "/^\s+at.*/"               "cont"
    Rule      "cont"         "/^\s+... \d+ more/"       "cont"
    Rule      "cont"         "/^\s*Caused by:.*/"       "cont"
```

#### 3. 如何过滤健康检查日志？

**问题场景**：Kubernetes 健康检查产生大量无用日志。

**解决方案**：使用 Fluent Bit 的 grep 过滤器：

```ini
[FILTER]
    Name    grep
    Match   kube.*
    Exclude log ^GET /healthz
    Exclude log ^GET /readiness
```

#### 4. 如何处理敏感信息？

**问题场景**：日志中包含密码、密钥等敏感信息。

**解决方案**：使用 modify 过滤器脱敏：

```ini
[FILTER]
    Name    modify
    Match   kube.*
    Regex   log password=\S+ password=***
    Regex   log token=\S+ token=***
```

#### 5. 如何实现日志采样？

**问题场景**：高频日志导致存储压力过大。

**解决方案**：使用 Lua 脚本实现采样：

```lua
-- sampling.lua
function sample(tag, timestamp, record)
    if math.random() < 0.1 then  -- 10% 采样率
        return 0, 0, record
    else
        return -1, 0, record
    end
end
```

配置：

```ini
[FILTER]
    Name    lua
    Match   kube.*
    script  sampling.lua
    call    sample
```

### 最佳实践

#### 1. 结构化日志输出

应用应输出结构化的 JSON 日志，便于解析和查询：

```json
{
  "timestamp": "2024-01-15T10:30:00.123Z",
  "level": "INFO",
  "service": "user-api",
  "trace_id": "abc123def456",
  "user_id": "12345",
  "message": "User login successful",
  "context": {
    "ip": "192.168.1.100",
    "duration_ms": 45
  }
}
```

#### 2. 合理设置资源限制

为日志采集器设置合理的资源请求和限制：

```yaml
resources:
  limits:
    cpu: 500m
    memory: 200Mi
  requests:
    cpu: 100m
    memory: 50Mi
```

#### 3. 监控采集器性能

监控关键指标：

- 日志处理速率（records/s）
- 输出延迟
- 错误率
- 内存使用

#### 4. 配置日志保留策略

根据业务需求配置日志保留时间：

```json
{
  "policy": {
    "phases": {
      "hot": {
        "min_age": "0ms",
        "actions": {
          "rollover": {
            "max_size": "50GB",
            "max_age": "1d"
          }
        }
      },
      "delete": {
        "min_age": "30d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}
```

#### 5. 使用标签和注解

利用 Kubernetes 标签和注解增强日志查询：

```yaml
metadata:
  labels:
    app: user-api
    version: v1.2.3
    environment: production
  annotations:
    logging/level: "info"
    logging/exclude: "false"
```
