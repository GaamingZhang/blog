# Kubernetes Scheduler 调度算法深度解析

## 引言：调度器的核心使命

在 Kubernetes 集群中，Scheduler（调度器）扮演着"资源分配决策者"的关键角色。当用户创建一个 Pod 时，Scheduler 需要从集群的所有节点中，选择一个最合适的节点来运行这个 Pod。这个看似简单的决策过程，实际上涉及复杂的资源评估、约束检查和多维度打分机制。

Scheduler 的工作直接影响集群的资源利用率、应用的高可用性和性能表现。一个优秀的调度决策需要在资源效率、负载均衡、亲和性约束、容灾能力等多个维度之间找到最佳平衡点。本文将深入剖析 Kubernetes Scheduler 使用的两种核心算法：**预选算法（Predicates/Filtering）** 和 **优选算法（Priorities/Scoring）**，揭示其背后的设计哲学和实现机制。

## 一、预选算法（Predicates/Filtering）：硬约束过滤

### 1.1 算法原理

预选算法是调度的第一阶段，其核心思想是**硬约束过滤**。Scheduler 会遍历集群中的所有节点，对每个节点执行一系列预定义的过滤规则，剔除不满足 Pod 运行条件的节点，最终得到一组"可行节点列表"（Feasible Nodes）。

这一阶段采用"一票否决"机制：只要节点在某一个过滤规则上不满足要求，就会被立即排除，不再参与后续的优选打分。这种设计确保了调度的正确性和安全性，避免将 Pod 调度到无法运行它的节点上。

### 1.2 核心过滤规则详解

#### 1.2.1 资源相关过滤

**PodFitsResources** 是最基础的资源过滤规则，检查节点是否拥有足够的 CPU、内存等资源来满足 Pod 的需求。

资源检查机制：
- **CPU 资源**：节点的可分配 CPU 减去已分配 CPU，必须大于等于 Pod 请求的 CPU
- **内存资源**：节点的可分配内存减去已分配内存，必须大于等于 Pod 请求的内存
- **扩展资源**：包括 GPU、FPGA 等特殊硬件资源的可用性检查

资源计算公式：
```
节点可用资源 = 节点总资源 - 系统预留资源 - Kubernetes 组件预留资源 - 已分配资源
```

**PodFitsHostPorts** 检查 Pod 所需的主机端口是否可用。如果 Pod 的 containerPort 配置了 hostPort，Scheduler 会确保该端口在目标节点上未被占用。

#### 1.2.2 存储相关过滤

**VolumeZonePredicate** 确保持久卷（PV）的可用区与节点所在区域匹配。对于使用云存储（如 AWS EBS、GCE PD）的场景，存储卷通常绑定到特定的可用区，Pod 必须调度到相同可用区的节点才能正常挂载。

**MaxEBSVolumeCount** 和 **MaxGCEPDVolumeCount** 分别限制节点上可挂载的 EBS 和 GCE PD 卷的最大数量，避免超过云厂商的限制。

**VolumeBindingPredicate** 检查 PVC 是否可以被绑定。对于动态供给的存储，Scheduler 需要确保存储类（StorageClass）支持延迟绑定，或者预先创建的 PV 满足 PVC 的需求。

#### 1.2.3 节点状态过滤

**CheckNodeCondition** 检查节点的健康状况，包括：
- 节点是否处于 Ready 状态
- 节点是否被标记为不可调度（Unschedulable）
- 节点是否处于网络不可达或磁盘压力状态

**CheckNodeUnschedulable** 过滤掉被手动标记为不可调度的节点（通过 `kubectl cordon` 命令）。

**NodeMemoryPressurePredicate** 和 **NodeDiskPressurePredicate** 分别检查节点是否存在内存压力和磁盘压力。当节点资源紧张时，Scheduler 会拒绝新的 Pod 调度，避免加剧资源竞争。

#### 1.2.4 亲和性约束过滤

**NodeAffinity** 实现节点亲和性调度，支持两种类型：
- **requiredDuringSchedulingIgnoredDuringExecution**：硬性要求，必须满足
- **preferredDuringSchedulingIgnoredDuringExecution**：软性偏好，在预选阶段不生效

节点亲和性通过节点标签（Label）进行匹配，支持 In、NotIn、Exists、DoesNotExist、Gt、Lt 等操作符。

**PodAffinity** 和 **PodAntiAffinity** 实现 Pod 间的亲和性和反亲和性调度：
- **PodAffinity**：将相关的 Pod 调度到同一拓扑域（如同一节点、同一可用区）
- **PodAntiAffinity**：将 Pod 分散到不同的拓扑域，提高容灾能力

#### 1.2.5 污点和容忍度过滤

**TaintTolerationPredicate** 检查 Pod 是否容忍节点上的污点（Taint）。污点用于标记节点的特殊属性或状态，如：
- `node-role.kubernetes.io/master:NoSchedule`：主节点不调度普通 Pod
- `node.kubernetes.io/not-ready:NoExecute`：不健康节点驱逐 Pod
- `node.kubernetes.io/disk-pressure:NoSchedule`：磁盘压力节点拒绝新 Pod

容忍度（Toleration）定义在 Pod 上，用于"抵消"污点的影响。只有当 Pod 容忍节点上的所有污点时，才能被调度到该节点。

## 二、优选算法（Priorities/Scoring）：软偏好打分

### 2.1 算法原理

优选算法是调度的第二阶段，其核心思想是**软偏好打分**。对于预选阶段筛选出的所有可行节点，Scheduler 会根据一系列优选规则对每个节点进行打分，分数范围为 0-10 分，最终选择得分最高的节点作为调度目标。

与预选的"一票否决"不同，优选阶段采用"加权评分"机制：每个优选规则独立打分，然后根据权重计算加权总分。这种设计允许在多个维度上进行权衡，选择综合表现最优的节点。

### 2.2 核心优选规则详解

#### 2.2.1 资源均衡优选

**LeastRequestedPriority** 优先选择资源使用率较低的节点，实现负载均衡。计算公式：

```
score = (capacity - requested) / capacity * 10
```

对于 CPU 和内存分别计算，然后取平均值。这个规则倾向于将 Pod 分散到不同的节点，避免单个节点负载过高。

**BalancedResourceAllocation** 优先选择 CPU 和内存使用率均衡的节点，避免资源碎片化。计算公式：

```
score = 10 - abs(cpuFraction - memoryFraction) * 10
```

其中 `cpuFraction` 和 `memoryFraction` 分别表示 CPU 和内存的使用比例。当两者接近时，得分较高。

**NodeResourcesFit** 是 Kubernetes 1.19 引入的新优选规则，综合考虑多种资源的拟合度，避免资源浪费。

#### 2.2.2 亲和性优选

**NodeAffinityPriority** 为节点亲和性打分。对于 `preferredDuringSchedulingIgnoredDuringExecution` 类型的亲和性规则，每个匹配的偏好项会贡献一个权重分，最终归一化到 0-10 分。

**InterPodAffinityPriority** 为 Pod 亲和性打分。计算目标节点所在拓扑域内已存在的匹配 Pod 数量，数量越多得分越高。这有助于将相关的 Pod 聚合在一起，减少网络延迟。

#### 2.2.3 污点和容忍度优选

**TaintTolerationPriority** 根据容忍度匹配程度打分。Pod 对节点污点的容忍度越高，得分越高。这个规则优先选择"更愿意接受"该 Pod 的节点。

#### 2.2.4 其他优选规则

**ImageLocalityPriority** 优先选择已存在 Pod 所需镜像的节点，减少镜像拉取时间。得分与节点上已存在的匹配镜像数量和大小相关。

**SelectorSpreadPriority** 将属于同一 Service、ReplicaSet 或 StatefulSet 的 Pod 分散到不同节点，提高容灾能力。这个规则会降低已运行相同控制器 Pod 的节点得分。

**EvenPodsSpreadPriority** 是 Kubernetes 1.18 引入的新特性，支持更灵活的 Pod 拓扑分布约束（TopologySpreadConstraints），实现跨拓扑域的均匀分布。

**DefaultPriorities** 是默认的优选规则集合，包括上述大部分规则，按默认权重组合使用。

## 三、调度流程完整解析

### 3.1 调度器架构

Kubernetes Scheduler 采用模块化架构设计，主要组件包括：

**Informer 缓存**：通过 Watch 机制监听 API Server，实时缓存节点、Pod、PVC 等资源信息，避免频繁访问 API Server。

**调度队列**：使用优先级队列（PriorityQueue）管理待调度的 Pod，支持优先级和抢占机制。

**调度框架**：提供插件化扩展点，允许自定义调度逻辑。主要扩展点包括：
- **PreFilter**：预选前预处理
- **Filter**：自定义过滤规则
- **PreScore**：优选前预处理
- **Score**：自定义打分规则
- **Reserve**：资源预留
- **Permit**：调度许可控制
- **PreBind/PostBind**：绑定前后钩子

**绑定器**：将调度决策提交到 API Server，更新 Pod 的 `spec.nodeName` 字段。

### 3.3 调度失败处理

当预选阶段没有找到可行节点时，Scheduler 会记录失败原因并触发以下机制：

**调度重试**：Pod 重新进入调度队列，等待下次调度周期。默认每秒尝试一次。

**抢占机制**：如果 Pod 配置了优先级（PriorityClass），Scheduler 可以驱逐低优先级的 Pod，为高优先级 Pod 腾出资源。抢占流程包括：
1. 识别可以抢占的节点
2. 选择最优抢占方案
3. 驱逐低优先级 Pod
4. 等待资源释放后重新调度

**调度失败事件**：通过 Event 记录详细的失败原因，包括：
- 哪些过滤规则失败
- 每个节点的具体失败原因
- 资源不足的详细信息

## 四、调度策略配置

### 4.1 调度器配置文件

Kubernetes 1.19+ 支持通过配置文件自定义调度策略。配置文件采用 YAML 格式，可以定义过滤规则、优选规则及其权重。

```yaml
apiVersion: kubescheduler.config.k8s.io/v1
kind: KubeSchedulerConfiguration
profiles:
  - schedulerName: default-scheduler
    plugins:
      queueSort:
        enabled:
          - name: PrioritySort
      preFilter:
        enabled:
          - name: NodeResourcesFit
          - name: NodePorts
          - name: VolumeRestrictions
      filter:
        enabled:
          - name: NodeUnschedulable
          - name: NodeName
          - name: NodePorts
          - name: NodeAffinity
          - name: TaintToleration
          - name: NodeResourcesFit
      preScore:
        enabled:
          - name: InterPodAffinity
          - name: PodAffinity
      score:
        enabled:
          - name: NodeResourcesFit
            weight: 1
          - name: NodeAffinity
            weight: 1
          - name: InterPodAffinity
            weight: 1
          - name: ImageLocality
            weight: 1
          - name: TaintToleration
            weight: 1
```

### 4.2 自定义调度器

除了配置默认调度器，还可以部署自定义调度器：

**多调度器部署**：在集群中运行多个调度器实例，Pod 通过 `spec.schedulerName` 字段指定使用的调度器。

**调度器扩展**：通过 Scheduler Extender 机制，将部分调度决策委托给外部服务。Extender 可以实现自定义的过滤和打分逻辑。

**调度框架插件**：开发自定义插件，编译到 Scheduler 二进制文件中，实现更深度的定制。

### 4.3 调度器性能调优

**并行度配置**：通过 `--parallelism` 参数控制并发调度的 Pod 数量，默认 16。在大型集群中可以适当提高。

**百分比配置**：通过 `--percentage-of-nodes-to-score` 参数控制参与打分的节点百分比。在节点数量超过 100 时，可以只对部分节点打分，提高调度速度。

**缓存同步**：通过 `--kube-api-qps` 和 `--kube-api-burst` 控制 Informer 与 API Server 的同步速率。

## 五、常见问题与最佳实践

### 5.1 常见问题解答

**问题 1：Pod 一直处于 Pending 状态，如何排查？**

**答**：首先使用 `kubectl describe pod <pod-name>` 查看事件，重点关注：
- Events 中是否有调度失败记录
- 失败原因是否为资源不足（Insufficient cpu/memory）
- 是否存在节点选择器或亲和性约束导致无匹配节点
- 节点是否有污点未被容忍
- PVC 是否处于 Pending 状态（存储未绑定）

**问题 2：如何实现 Pod 的跨可用区均匀分布？**

**答**：使用 TopologySpreadConstraints 配置拓扑分布约束：
```yaml
topologySpreadConstraints:
  - maxSkew: 1
    topologyKey: topology.kubernetes.io/zone
    whenUnsatisfiable: DoNotSchedule
    labelSelector:
      matchLabels:
        app: myapp
```
这确保每个可用区的 Pod 数量差异不超过 1。结合 PodAntiAffinity 可以进一步强化分散效果。

**问题 3：节点资源充足但调度失败，可能的原因？**

**答**：可能的原因包括：
- 节点被标记为 Unschedulable（通过 `kubectl cordon`）
- 节点存在污点，Pod 未配置容忍度
- Pod 配置了节点选择器或节点亲和性，但节点标签不匹配
- Pod 配置了 Pod 亲和性，但目标拓扑域内没有匹配的 Pod
- 存储卷可用区不匹配（VolumeZonePredicate 失败）
- 主机端口冲突（PodFitsHostPorts 失败）

**问题 4：如何避免资源碎片化？**

**答**：资源碎片化是指节点剩余资源不足以运行新 Pod，但总体资源充足。解决策略：
- 使用 `BalancedResourceAllocation` 优选规则，优先选择 CPU/内存均衡的节点
- 合理设置 Pod 的资源请求，避免过度分配
- 使用 `NodeResourcesFit` 插件，综合考虑多种资源
- 定期清理已完成和失败的 Pod
- 考虑使用 Descheduler 组件重新平衡集群

**问题 5：调度器性能下降，如何优化？**

**答**：调度器性能优化策略：
- 调整 `--percentage-of-nodes-to-score` 参数，减少打分节点数量
- 增加 `--parallelism` 参数，提高并发调度能力
- 优化 Informer 缓存配置，减少 API Server 访问
- 简化调度策略，移除不必要的过滤和优选规则
- 使用调度器性能配置文件（PerformanceProfile）
- 监控调度延迟指标，识别性能瓶颈

### 5.2 调度最佳实践

**资源管理最佳实践**：
- 为所有 Pod 设置合理的 resources.requests 和 resources.limits
- 使用 LimitRange 限制命名空间级别的资源使用
- 使用 ResourceQuota 控制命名空间资源配额
- 定期审查资源使用情况，调整资源请求

**高可用部署最佳实践**：
- 使用 PodAntiAffinity 将关键服务的 Pod 分散到不同节点
- 使用 TopologySpreadConstraints 实现跨可用区分布
- 为关键应用配置 PriorityClass，确保高优先级
- 使用 PodDisruptionBudget 保护应用可用性

**性能优化最佳实践**：
- 使用节点亲和性将 I/O 密集型应用调度到 SSD 节点
- 使用 ImageLocalityPriority 减少镜像拉取时间
- 使用 PodAffinity 将频繁通信的服务调度到同一节点
- 合理配置污点和容忍度，实现节点分级管理

**调度策略最佳实践**：
- 避免过度使用硬约束（requiredDuringScheduling），优先使用软约束
- 为调度策略设置合理的权重，平衡多个维度
- 使用命名空间隔离不同环境，避免调度冲突
- 定期审查调度失败日志，优化调度策略

## 六、调度算法演进与未来趋势

### 6.1 调度器版本演进

**Kubernetes 1.0-1.14**：传统调度器架构，硬编码的过滤和优选规则，扩展性有限。

**Kubernetes 1.15-1.17**：引入调度框架，支持插件化扩展，提高灵活性。

**Kubernetes 1.18-1.19**：引入 EvenPodsSpread 和 NodeResourcesFit，增强拓扑分布和资源管理能力。

**Kubernetes 1.20+**：支持配置文件和多 Profile，实现调度器实例化和策略定制。

### 6.2 新兴调度技术

**Volcano**：针对 AI/ML 和大数据场景的批处理调度器，支持 Gang Scheduling、Queue 管理、Fair Share 等高级特性。

**Karmada**：多云多集群调度器，支持跨集群的应用部署和资源调度。

**Koordinator**：阿里巴巴开源的混部调度器，支持在线离线混部、资源隔离、QoS 保障。

**YuniKorn**：统一的资源调度器，支持多种工作负载类型，提供细粒度的资源分配。

### 6.3 未来发展方向

**智能调度**：结合机器学习技术，预测资源需求，优化调度决策。

**实时调度**：支持更细粒度的资源分配和动态调整，适应无服务器场景。

**跨集群调度**：实现多云多集群的统一资源管理和调度。

**绿色调度**：考虑能耗和碳排放，优化数据中心的资源利用效率。

## 面试回答

**问题：简述 Kubernetes Scheduler 使用哪两种算法将 Pod 调度到节点？**

**回答**：Kubernetes Scheduler 使用**预选算法（Predicates/Filtering）**和**优选算法（Priorities/Scoring）**两个阶段完成 Pod 调度。预选算法是硬约束过滤阶段，遍历所有节点并执行一系列过滤规则（如资源检查、节点亲和性、污点容忍度等），剔除不满足 Pod 运行条件的节点，生成可行节点列表，采用"一票否决"机制确保调度正确性。优选算法是软偏好打分阶段，对可行节点根据优选规则（如资源均衡、亲和性偏好、镜像本地化等）进行 0-10 分打分，计算加权总分后选择最高分节点，如多个节点同分则随机选择。这种两阶段设计既保证了调度的正确性，又实现了多维度的优化权衡，是 Kubernetes 调度器的核心机制。
