# Kubernetes 容器资源限制和请求

## 引言

在 Kubernetes 中，合理配置容器的资源请求（Requests）和限制（Limits）是保证应用稳定运行和集群资源高效利用的关键。资源请求用于调度决策，资源限制用于运行时约束。理解两者的区别和配置方法，对于构建可靠、高效的容器化应用至关重要。

## 资源请求和限制概述

### Requests 和 Limits 区别

```
┌─────────────────────────────────────────────────────────────┐
│          Requests 与 Limits 对比                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Requests（请求）：                                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • 调度时使用：节点必须有足够的可用资源              │   │
│  │  • 保证容器获得的最小资源                            │   │
│  │  • 影响 Pod 调度决策                                 │   │
│  │  • 用于计算节点资源分配                              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  Limits（限制）：                                            │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • 运行时使用：容器最大可使用的资源                  │   │
│  │  • 防止容器占用过多资源                              │   │
│  │  • 超过限制会被限制或终止                            │   │
│  │  • 用于资源隔离和 QoS 分类                           │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  关系：Requests ≤ Limits                                     │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 资源类型

| 资源类型 | 说明 | 单位 |
|---------|------|------|
| cpu | CPU 资源 | millicores (m) 或 cores |
| memory | 内存资源 | bytes (Ki, Mi, Gi) |
| ephemeral-storage | 临时存储 | bytes (Ki, Mi, Gi) |
| hugepages-<size> | 大页内存 | bytes |

## CPU 资源管理

### CPU 单位说明

```
┌─────────────────────────────────────────────────────────────┐
│                  CPU 资源单位                                │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1 CPU = 1000m (millicores)                                 │
│                                                              │
│  示例：                                                      │
│  • 1 CPU    = 1000m = 1 个 CPU 核心                         │
│  • 0.5 CPU  = 500m  = 半个 CPU 核心                         │
│  • 0.1 CPU  = 100m  = 10% CPU 核心                          │
│  • 2 CPU    = 2000m = 2 个 CPU 核心                         │
│                                                              │
│  注意：                                                      │
│  • CPU 是可压缩资源，超限会被限流                            │
│  • 不会导致容器被终止                                        │
│  • 请求和限制可以是小数                                      │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### CPU 配置示例

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: cpu-demo
spec:
  containers:
  - name: app
    image: nginx:1.21
    resources:
      requests:
        cpu: "250m"
      limits:
        cpu: "500m"
```

### CPU 密集型应用配置

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: cpu-intensive-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: cpu-app
  template:
    metadata:
      labels:
        app: cpu-app
    spec:
      containers:
      - name: app
        image: cpu-intensive:v1
        resources:
          requests:
            cpu: "1000m"
            memory: "512Mi"
          limits:
            cpu: "2000m"
            memory: "1Gi"
```

## QoS 服务质量等级

### QoS 分类

```
┌─────────────────────────────────────────────────────────────┐
│                  QoS 服务质量等级                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Guaranteed（保证）：                                        │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  条件：                                              │   │
│  │  • 所有容器都设置了 requests 和 limits              │   │
│  │  • 所有容器的 requests.cpu = limits.cpu             │   │
│  │  • 所有容器的 requests.memory = limits.memory       │   │
│  │                                                      │   │
│  │  优先级：最高，最后被驱逐                            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  Burstable（可突发）：                                       │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  条件：                                              │   │
│  │  • 至少一个容器设置了 requests 或 limits            │   │
│  │  • 不满足 Guaranteed 条件                           │   │
│  │                                                      │   │
│  │  优先级：中等，中间被驱逐                            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  BestEffort（尽力而为）：                                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  条件：                                              │   │
│  │  • 所有容器都没有设置 requests 和 limits            │   │
│  │                                                      │   │
│  │  优先级：最低，最先被驱逐                            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### Guaranteed 配置

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: guaranteed-pod
spec:
  containers:
  - name: app
    image: nginx:1.21
    resources:
      requests:
        cpu: "500m"
        memory: "256Mi"
      limits:
        cpu: "500m"
        memory: "256Mi"
```

```yaml

kind: Pod

  name: burstable-pod
spec:
  containers:
  - name: app
    image: nginx:1.21
    resources:
      requests:
        cpu: "250m"
```

### BestEffort 配置

```yaml

kind: Pod

  name: besteffort-pod
spec:
  containers:
  - name: app
    image: nginx:1.21
```

## 资源配额和限制范围

### ResourceQuota

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: compute-quota
  namespace: production
spec:
  hard:
    requests.cpu: "10"
    requests.memory: 20Gi
    limits.cpu: "20"
    limits.memory: 40Gi
    pods: "10"
```

### LimitRange

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: resource-limits
  namespace: production
spec:
  limits:
  - type: Container
    default:
      cpu: "500m"
      memory: "512Mi"
    defaultRequest:
      cpu: "100m"
      memory: "128Mi"
    min:
      cpu: "50m"
      memory: "64Mi"
    max:
      cpu: "2000m"
      memory: "2Gi"
  - type: PersistentVolumeClaim
    max:
      storage: 50Gi
    min:
      storage: 1Gi
```

## 面试回答

**问题**: 什么是容器资源限制和容器资源请求？

**回答**: Kubernetes 容器资源请求和限制是资源管理的核心机制：

**资源请求**：用于调度决策，表示容器运行所需的最小资源。调度器根据 requests 判断节点是否有足够的可用资源来运行 Pod。requests 保证容器获得的最小资源量，影响 Pod 的调度结果。

**资源限制**：用于运行时约束，表示容器可以使用的最大资源。超过 CPU limits 会被限流（CPU 是可压缩资源），超过 memory limits 会被 OOMKilled（内存是不可压缩资源）。

**资源配置示例**：requests.cpu: "250m"、requests.memory: "256Mi"、limits.cpu: "500m"、limits.memory: "512Mi"。

**QoS 服务质量等级**：根据资源配置自动分为三类。**Guaranteed**（最高优先级）：所有容器的 requests 等于 limits，资源完全保证；**Burstable**（中等优先级）：设置了部分资源请求或限制，资源部分保证；**BestEffort**（最低优先级）：未设置任何资源请求和限制，资源不保证。节点资源不足时，按优先级从低到高驱逐 Pod。

**最佳实践**：生产环境建议使用 Guaranteed QoS；合理设置 requests 和 limits 的比例（通常 limits 是 requests 的 1.5-2 倍）；使用 LimitRange 设置命名空间默认值；使用 ResourceQuota 限制命名空间资源总量；监控实际资源使用并持续优化配置。
