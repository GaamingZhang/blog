# Kubernetes 配置文件安全管理

## 引言

在 Kubernetes 中，应用程序通常需要配置文件和敏感信息（如密码、密钥、证书等）。如何安全地管理这些配置信息是容器化应用安全的重要环节。Kubernetes 提供了 ConfigMap 和 Secret 两种资源对象来管理配置，同时结合 RBAC、加密存储等机制确保配置安全。

## ConfigMap 安全管理

### 创建 ConfigMap

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: production
data:
  app.properties: |
    server.port=8080
    log.level=info
    cache.enabled=true
  database.properties: |
    db.host=mysql-service
    db.port=3306
    db.name=myapp
```

### 使用 ConfigMap

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: my-app:v1
    envFrom:
    - configMapRef:
        name: app-config
    volumeMounts:
    - name: config
      mountPath: /etc/app
      readOnly: true
  volumes:
  - name: config
    configMap:
      name: app-config
```

### ConfigMap 安全最佳实践

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: secure-config
  namespace: production
  labels:
    app: myapp
    environment: production
  annotations:
    description: "Application configuration"
data:
  application.yml: |
    server:
      port: 8080
    logging:
      level: info
---
apiVersion: v1
kind: Pod
metadata:
  name: secure-app
spec:
  containers:
  - name: app
    image: my-app:v1
    volumeMounts:
    - name: config
      mountPath: /etc/app
      readOnly: true
    securityContext:
      runAsNonRoot: true
      runAsUser: 1000
  volumes:
  - name: config
    configMap:
      name: secure-config
      defaultMode: 0644
```

## Secret 安全管理

### Secret 类型

| 类型 | 用途 |
|-----|------|
| Opaque | 通用 Secret，存储任意数据 |
| kubernetes.io/service-account-token | Service Account Token |
| kubernetes.io/dockercfg | Docker 配置 |
| kubernetes.io/dockerconfigjson | Docker 配置（JSON 格式） |
| kubernetes.io/basic-auth | 基本认证 |
| kubernetes.io/ssh-auth | SSH 认证 |
| kubernetes.io/tls | TLS 证书 |
| bootstrap.kubernetes.io/token | Bootstrap Token |

### 创建 Secret

```bash
kubectl create secret generic db-secret \
  --from-literal=username=admin \
  --from-literal=password='S3cr3t!'

kubectl create secret tls tls-secret \
  --cert=path/to/tls.crt \
  --key=path/to/tls.key
```

### Secret YAML 定义

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
  namespace: production
type: Opaque
data:
  username: YWRtaW4=
  password: UzNjcjN0IQ==
stringData:
  username: admin
  password: S3cr3t!
---
apiVersion: v1
kind: Secret
metadata:
  name: tls-secret
  namespace: production
type: kubernetes.io/tls
data:
  tls.crt: LS0tLS1CRUdJTi...
  tls.key: LS0tLS1CRUdJTi...
```

### 使用 Secret

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-secret
spec:
  containers:
  - name: app
    image: my-app:v1
    env:
    - name: DB_USERNAME
      valueFrom:
        secretKeyRef:
          name: db-secret
          key: username
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-secret
          key: password
    volumeMounts:
    - name: tls
      mountPath: /etc/tls
      readOnly: true
  volumes:
  - name: tls
    secret:
      secretName: tls-secret
      defaultMode: 0600
```

## Secret 加密存储

### 启用 etcd 加密

```yaml
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
    - secrets
    providers:
    - aescbc:
        keys:
        - name: key1
          secret: <BASE64_ENCODED_SECRET>
    - identity: {}
```

### 配置 kube-apiserver

```bash
kube-apiserver \
  --encryption-provider-config=/etc/kubernetes/encryption-config.yaml \
  ...
```

### 加密现有 Secret

```bash
kubectl get secrets --all-namespaces -o json | kubectl replace -f -
```

## RBAC 权限控制

### 配置 RBAC 限制 Secret 访问

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: secret-reader
  namespace: production
rules:
- apiGroups: [""]
  resources: ["secrets"]
  resourceNames: ["db-secret"]
  verbs: ["get"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-secrets
  namespace: production
subjects:
- kind: ServiceAccount
  name: app-sa
  namespace: production
roleRef:
  kind: Role
  name: secret-reader
  apiGroup: rbac.authorization.k8s.io
```

### 使用 ServiceAccount

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-sa
  namespace: production
automountServiceAccountToken: false
---
apiVersion: v1
kind: Pod
metadata:
  name: app-with-sa
spec:
  serviceAccountName: app-sa
  automountServiceAccountToken: false
  containers:
  - name: app
    image: my-app:v1
```

## 外部密钥管理系统

### 使用 External Secrets Operator

```yaml
apiVersion: external-secrets.io/v1beta1
kind: SecretStore
metadata:
  name: vault-backend
spec:
  provider:
    vault:
      server: "https://vault.example.com"
      path: "secret"
      version: "v2"
      auth:
        kubernetes:
          mountPath: "kubernetes"
          role: "my-role"
---
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: vault-secret
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: SecretStore
  target:
    name: db-secret
    creationPolicy: Owner
  data:
  - secretKey: password
    remoteRef:
      key: database
      property: password
```

### 使用 HashiCorp Vault

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: vault-agent
spec:
  serviceAccountName: app-sa
  containers:
  - name: app
    image: my-app:v1
    volumeMounts:
    - name: secrets
      mountPath: /etc/secrets
      readOnly: true
  - name: vault-agent
    image: vault:1.13
    args:
    - agent
    - -config=/etc/vault/config.hcl
    volumeMounts:
    - name: vault-config
      mountPath: /etc/vault
    - name: secrets
      mountPath: /etc/secrets
  volumes:
  - name: vault-config
    configMap:
      name: vault-config
  - name: secrets
    emptyDir:
      medium: Memory
```

## 配置安全最佳实践

### 1. 最小权限原则

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: minimal-secret-access
rules:
- apiGroups: [""]
  resources: ["secrets"]
  resourceNames: ["specific-secret"]
  verbs: ["get"]
```

### 2. 使用命名空间隔离

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    environment: production
---
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
  namespace: production
```

### 3. 限制 Secret 挂载权限

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  containers:
  - name: app
    image: my-app:v1
    volumeMounts:
    - name: secrets
      mountPath: /etc/secrets
      readOnly: true
  volumes:
  - name: secrets
    secret:
      secretName: db-secret
      defaultMode: 0400
```

### 4. 使用 immutable Secret

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: immutable-secret
immutable: true
data:
  username: YWRtaW4=
  password: UzNjcjN0IQ==
```

### 5. 定期轮换密钥

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
  annotations:
    rotate/interval: "30d"
    rotate/last-rotated: "2024-01-01"
data:
  username: YWRtaW4=
  password: UzNjcjN0IQ==
```

## 常见安全问题与解决方案

### 问题一：Secret 泄露

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: unsafe-pod
spec:
  containers:
  - name: app
    image: my-app:v1
    env:
    - name: PASSWORD
      value: "plain-password"
```

**解决方案**：使用 Secret

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: safe-pod
spec:
  containers:
  - name: app
    image: my-app:v1
    env:
    - name: PASSWORD
      valueFrom:
        secretKeyRef:
          name: app-secret
          key: password
```

### 问题二：配置文件权限过大

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: unsafe-mount
spec:
  containers:
  - name: app
    image: my-app:v1
    volumeMounts:
    - name: config
      mountPath: /etc/app
  volumes:
  - name: config
    configMap:
      name: app-config
```

**解决方案**：设置只读权限

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: safe-mount
spec:
  containers:
  - name: app
    image: my-app:v1
    volumeMounts:
    - name: config
      mountPath: /etc/app
      readOnly: true
  volumes:
  - name: config
    configMap:
      name: app-config
      defaultMode: 0644
```

### 问题三：Secret 未加密存储

**解决方案**：启用 etcd 加密

```bash
kubectl get secrets -o jsonpath='{.items[*].data}' | jq
```

## 面试回答

**问题**: 如何在 Kubernetes 中进行配置文件的安全管理？

**回答**: Kubernetes 配置文件安全管理主要从以下几个方面进行：

**配置分类存储**：使用 ConfigMap 存储非敏感配置（如应用配置文件、环境变量），使用 Secret 存储敏感信息（如密码、密钥、证书）。Secret 支持 Base64 编码存储，并可配置 etcd 加密存储。

**访问控制**：通过 RBAC 限制对 Secret 的访问权限，遵循最小权限原则，只授予特定 ServiceAccount 对特定 Secret 的访问权限。使用命名空间隔离不同环境的配置。

**存储安全**：启用 etcd 加密存储 Secret，使用 EncryptionConfiguration 配置加密提供者。设置 Secret 挂载权限为只读，文件权限设置为最小（如 0400）。

**外部密钥管理**：对于生产环境，推荐使用外部密钥管理系统如 HashiCorp Vault、AWS Secrets Manager，通过 External Secrets Operator 同步到 Kubernetes。

**最佳实践**：使用 immutable Secret 防止意外修改；定期轮换密钥；审计 Secret 访问日志；避免在 Pod 定义中硬编码敏感信息；使用 ServiceAccount 并禁用自动挂载 Token。

通过以上措施，可以确保 Kubernetes 中配置文件的安全管理，防止敏感信息泄露。
