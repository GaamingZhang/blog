# Kubernetes 容灾和多集群管理

## 引言

容灾（Disaster Recovery）是保障业务连续性的重要措施。通过多集群部署和容灾方案，可以在发生故障时快速切换到备用集群，确保业务不中断。本文介绍 Kubernetes 容灾和多集群管理的方案。

## 多集群管理工具

### Kubernetes Federation

```yaml
apiVersion: core.kubefed.io/v1beta1
kind: KubeFedCluster
metadata:
  name: cluster-beijing
  namespace: kube-federation-system
spec:
  apiEndpoint: https://cluster-beijing.example.com
---
apiVersion: types.kubefed.io/v1beta1
kind: FederatedDeployment
metadata:
  name: myapp
  namespace: production
spec:
  placement:
    clusters:
    - name: cluster-beijing
    - name: cluster-shanghai
  overrides:
  - clusterName: cluster-beijing
    clusterOverrides:
    - path: spec.replicas
      value: 3
  - clusterName: cluster-shanghai
    clusterOverrides:
    - path: spec.replicas
      value: 2
  template:
    spec:
      replicas: 3
      selector:
        matchLabels:
          app: myapp
      template:
        spec:
          containers:
          - name: myapp
            image: myapp:v1
```

### ArgoCD 多集群

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp-beijing
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/org/app-config.git
    targetRevision: main
    path: overlays/beijing
  destination:
    server: https://cluster-beijing.example.com
    namespace: production
---
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp-shanghai
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/org/app-config.git
    targetRevision: main
    path: overlays/shanghai
  destination:
    server: https://cluster-shanghai.example.com
    namespace: production
```

  name: mysql-replica

  serviceName: mysql
  replicas: 1
  template:
    spec:
      containers:
      - name: mysql
        image: mysql:8.0
        env:
        - name: MYSQL_REPLICATION_MODE
          value: "slave"
        - name: MYSQL_MASTER_HOST

## 流量切换

### DNS 切换

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id $ZONE_ID \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "myapp.example.com",
        "Type": "CNAME",
        "TTL": 60,
        "ResourceRecords": [{"Value": "cluster-shanghai.example.com"}]
      }
    }]
  }'
```

### Ingress 切换

```yaml
apiVersion: networking.k8s.io/v1

    nginx.ingress.kubernetes.io/upstream-vhost: "cluster-shanghai.example.com"

    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: myapp
            port:
              number: 80
```

## 面试回答

**问题**: 如何进行 Kubernetes 容灾和多集群管理？

**回答**: Kubernetes 容灾和多集群管理需要从架构设计、工具选择、数据同步、流量切换等方面考虑：

**容灾架构**：**主备架构**主集群提供服务，备集群待命，故障时切换；**双活架构**两个集群同时提供服务，负载均衡，故障时自动切换。容灾指标包括 RPO（数据恢复点目标）和 RTO（恢复时间目标）。

**多集群管理工具**：**Kubernetes Federation** 官方多集群管理方案，支持资源分发和跨集群调度；**ArgoCD** 通过 Application 配置管理多集群部署；**Rancher** 提供多集群管理界面。

**数据同步**：数据库使用主从复制实现数据同步；对象存储使用工具（如 rclone）定期同步；应用状态使用外部存储（如 Redis、数据库）共享。

**流量切换**：**DNS 切换**修改 DNS 记录指向备用集群；**Ingress 切换**修改 Ingress 配置重定向流量；**Service Mesh** 使用 Istio 等实现流量管理。

**容灾演练**：定期进行容灾演练验证方案有效性。演练流程：准备 -> 执行 -> 恢复 -> 总结。记录演练结果，持续改进。

**最佳实践**：定期演练验证方案；配置监控告警；定期备份数据；文档化容灾流程；使用自动化工具减少人工操作。
