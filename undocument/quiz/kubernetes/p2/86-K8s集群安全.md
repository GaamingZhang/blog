# Kubernetes 集群安全机制详解

## 引言

安全性是 Kubernetes 集群生产环境部署的首要考虑因素。作为一个管理着关键业务应用的容器编排平台，Kubernetes 需要确保只有经过授权的用户和应用程序才能访问集群资源，同时保护敏感数据不被泄露。

Kubernetes 提供了一套多层次的安全机制，从 API 访问控制到容器运行时安全，从网络隔离到数据加密，构建了纵深防御体系。理解这些安全机制的工作原理和配置方法，是构建安全可靠的 Kubernetes 集群的基础。

## Kubernetes 安全模型概述

Kubernetes 的安全模型可以分为三个主要层次：

1. **API 访问安全**：控制谁可以访问 API Server 以及可以执行哪些操作
2. **工作负载安全**：确保运行在集群中的容器是安全的
3. **网络安全**：控制 Pod 之间以及 Pod 与外部之间的网络通信

## 一、API 访问安全

### 1.1 认证（Authentication）

认证是确定"谁在访问"的过程。Kubernetes 支持多种认证方式：

#### 认证方式类型

| 认证方式 | 说明 | 适用场景 |
|---------|------|---------|
| **客户端证书** | 使用 X.509 证书进行认证 | 组件间通信、管理员访问 |
| **静态 Token 文件** | 预定义的 Token 列表 | 简单场景、测试环境 |
| **Bootstrap Token** | 用于集群引导和节点加入 | 集群初始化、节点加入 |
| **Service Account** | Pod 内部访问 API | 应用程序访问 API |
| **OIDC** | OpenID Connect 集成 | 企业单点登录 |
| **Webhook Token** | 外部认证服务 | 自定义认证逻辑 |

#### 客户端证书认证配置

```yaml
# kube-apiserver 配置
apiServer:
  extraArgs:
    client-ca-file: /etc/kubernetes/pki/ca.crt
    tls-cert-file: /etc/kubernetes/pki/apiserver.crt
    tls-private-key-file: /etc/kubernetes/pki/apiserver.key
```

#### Service Account 认证

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-service-account
  namespace: default
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: read-pods-global
subjects:
- kind: ServiceAccount
  name: my-service-account
  namespace: default
roleRef:
  kind: ClusterRole
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

### 1.2 授权（Authorization）

授权是确定"可以做什么"的过程。Kubernetes 支持多种授权模块：

#### 授权模块类型

1. **Node 授权器**：专门处理 kubelet 的请求
2. **ABAC（Attribute-Based Access Control）**：基于属性的访问控制
3. **RBAC（Role-Based Access Control）**：基于角色的访问控制（推荐）
4. **Webhook**：外部授权服务

#### RBAC 核心概念

```yaml
# Role - 命名空间级别权限
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: default
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
- apiGroups: [""]
  resources: ["pods/log"]
  verbs: ["get"]
---
# ClusterRole - 集群级别权限
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: secret-reader
rules:
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get", "list", "watch"]
---
# RoleBinding - 绑定 Role 到用户/组
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: default
subjects:
- kind: User
  name: jane
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
---
# ClusterRoleBinding - 绑定 ClusterRole 到用户/组
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: read-secrets-global
subjects:
- kind: Group
  name: system:authenticated
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: secret-reader
  apiGroup: rbac.authorization.k8s.io
```

#### 常用 RBAC 配置示例

```yaml
# 只读用户
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: view-only
rules:
- apiGroups: [""]
  resources: ["pods", "services", "configmaps", "secrets"]
  verbs: ["get", "list", "watch"]
- apiGroups: ["apps"]
  resources: ["deployments", "replicasets", "statefulsets"]
  verbs: ["get", "list", "watch"]
---
# 开发者角色
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: developer
  namespace: development
rules:
- apiGroups: [""]
  resources: ["pods", "services", "configmaps", "secrets"]
  verbs: ["*"]
- apiGroups: ["apps"]
  resources: ["deployments", "replicasets"]
  verbs: ["*"]
- apiGroups: [""]
  resources: ["pods/portforward", "pods/exec"]
  verbs: ["create"]
```

### 1.3 准入控制（Admission Control）

准入控制是在对象持久化之前进行的验证和修改。Kubernetes 提供了多种准入控制器：

#### 内置准入控制器

| 准入控制器 | 功能 |
|-----------|------|
| **AlwaysPullImages** | 强制每次拉取镜像 |
| **DefaultStorageClass** | 自动添加默认 StorageClass |
| **LimitRanger** | 强制资源限制 |
| **NamespaceLifecycle** | 强制命名空间生命周期 |
| **NodeRestriction** | 限制 kubelet 修改节点信息 |
| **PodSecurityPolicy** | 强制 Pod 安全策略（已废弃，使用 Pod Security Standards） |
| **ResourceQuota** | 强制资源配额 |
| **ServiceAccount** | 自动添加 ServiceAccount |

#### ValidatingAdmissionWebhook 示例

```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingWebhookConfiguration
metadata:
  name: validation-webhook
webhooks:
- name: validate.example.com
  rules:
  - apiGroups: [""]
    apiVersions: ["v1"]
    operations: ["CREATE", "UPDATE"]
    resources: ["pods"]
  clientConfig:
    service:
      name: webhook-service
      namespace: default
      path: /validate
  admissionReviewVersions: ["v1"]
  sideEffects: None
  failurePolicy: Fail
```

#### MutatingAdmissionWebhook 示例

```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: MutatingWebhookConfiguration
metadata:
  name: mutation-webhook
webhooks:
- name: mutate.example.com
  rules:
  - apiGroups: [""]
    apiVersions: ["v1"]
    operations: ["CREATE"]
    resources: ["pods"]
  clientConfig:
    service:
      name: webhook-service
      namespace: default
      path: /mutate
  admissionReviewVersions: ["v1"]
  sideEffects: None
  failurePolicy: Ignore
```

## 二、Pod 安全

### 2.1 Pod Security Standards

Kubernetes 1.23 引入了 Pod Security Standards 替代 PodSecurityPolicy：

#### 安全级别

| 级别 | 说明 | 限制 |
|-----|------|------|
| **Privileged** | 不受限制 | 无限制，适用于系统和基础设施级 Pod |
| **Baseline** | 最小限制 | 禁止明显的提权，适用于大多数应用 |
| **Restricted** | 严格限制 | 严格限制，适用于安全敏感应用 |

#### 配置 Pod Security Standards

```yaml
# 命名空间级别配置
apiVersion: v1
kind: Namespace
metadata:
  name: my-namespace
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: latest
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/warn: restricted
```

### 2.2 Security Context

Security Context 定义了 Pod 或容器的安全配置：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: security-context-demo
spec:
  securityContext:
    runAsUser: 1000
    runAsGroup: 3000
    fsGroup: 2000
    runAsNonRoot: true
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: app
    image: nginx
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop:
        - ALL
    volumeMounts:
    - name: cache
      mountPath: /var/cache
  volumes:
  - name: cache
    emptyDir: {}
```

#### Security Context 配置项

| 配置项 | 说明 | 推荐值 |
|-------|------|--------|
| `runAsNonRoot` | 禁止以 root 运行 | true |
| `runAsUser` | 指定运行用户 ID | 非 0 值 |
| `runAsGroup` | 指定运行组 ID | 非 0 值 |
| `fsGroup` | 指定文件系统组 | 非 0 值 |
| `allowPrivilegeEscalation` | 禁止权限提升 | false |
| `readOnlyRootFilesystem` | 只读根文件系统 | true |
| `capabilities.drop` | 删除 Linux 能力 | ALL |
| `seccompProfile.type` | Seccomp 配置 | RuntimeDefault |

### 2.3 容器镜像安全

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: image-security
spec:
  containers:
  - name: app
    image: my-registry.example.com/my-app:v1.0@sha256:abc123...
    imagePullPolicy: Always
  imagePullSecrets:
  - name: registry-secret
```

镜像安全最佳实践：
- 使用私有镜像仓库
- 镜像签名验证
- 定期扫描镜像漏洞
- 使用最小化基础镜像
- 固定镜像版本（使用 digest）

## 三、网络安全

### 3.1 Network Policy

Network Policy 控制 Pod 之间的网络流量：

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-network-policy
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    - namespaceSelector:
        matchLabels:
          name: production
    ports:
    - protocol: TCP
      port: 8080
  egress:
  - to:
    - podSelector:
        matchLabels:
          app: database
    ports:
    - protocol: TCP
      port: 3306
  - to:
    - namespaceSelector: {}
      podSelector:
        matchLabels:
          k8s-app: kube-dns
    ports:
    - protocol: UDP
      port: 53
```

#### 默认拒绝所有流量

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: production
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress
```

### 3.2 Service Mesh 安全

使用 Istio 实现 mTLS：

```yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: istio-system
spec:
  mtls:
    mode: STRICT
---
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: api-authz
  namespace: production
spec:
  selector:
    matchLabels:
      app: api
  rules:
  - from:
    - source:
        principals: ["cluster.local/ns/production/sa/frontend"]
    to:
    - operation:
        methods: ["GET", "POST"]
        paths: ["/api/*"]
```

## 四、数据安全

### 4.1 Secret 管理

#### Secret 加密配置

```yaml
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
    - secrets
    providers:
    - aescbc:
        keys:
        - name: key1
          secret: <base64-encoded-secret>
    - identity: {}
```

#### 使用外部 Secret 管理

```yaml
# External Secrets Operator
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: my-secret
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secretsmanager
    kind: ClusterSecretStore
  target:
    name: my-secret
    creationPolicy: Owner
  data:
  - secretKey: password
    remoteRef:
      key: my-app/credentials
      property: password
```

## 五、审计日志

### 5.1 审计策略配置

```yaml
apiVersion: audit.k8s.io/v1
kind: Policy
rules:
# 记录敏感资源的修改
- level: RequestResponse
  resources:
  - group: ""
    resources: ["secrets", "configmaps"]
  - group: rbac.authorization.k8s.io
    resources: ["roles", "rolebindings", "clusterroles", "clusterrolebindings"]
  
# 记录 Pod 的创建和删除
- level: Request
  resources:
  - group: ""
    resources: ["pods"]
  verbs: ["create", "delete", "update"]

# 忽略系统命名空间的只读请求
- level: None
  namespaces: ["kube-system"]
  verbs: ["get", "list", "watch"]

# 默认记录元数据
- level: Metadata
```

### 5.2 审计日志配置

```yaml
# kube-apiserver 配置
apiServer:
  extraArgs:
    audit-log-path: /var/log/kubernetes/audit.log
    audit-log-maxage: "30"
    audit-log-maxbackup: "10"
    audit-log-maxsize: "100"
    audit-policy-file: /etc/kubernetes/audit-policy.yaml
```

## 六、安全最佳实践

### 6.1 集群安全清单

| 安全措施 | 说明 | 优先级 |
|---------|------|--------|
| 启用 RBAC | 使用 RBAC 控制访问权限 | 高 |
| 配置 Network Policy | 限制 Pod 间网络通信 | 高 |
| 启用 Pod Security Standards | 强制 Pod 安全配置 | 高 |
| Secret 加密 | 加密存储敏感数据 | 高 |
| 启用审计日志 | 记录 API 操作 | 中 |
| 镜像扫描 | 扫描镜像漏洞 | 中 |
| 使用私有镜像仓库 | 控制镜像来源 | 中 |
| 定期更新 | 更新 Kubernetes 版本 | 中 |
| 备份 etcd | 定期备份集群数据 | 中 |
| 监控告警 | 监控安全事件 | 低 |

```yaml
# Prometheus 安全告警规则

- name: kubernetes-security
  rules:
  - alert: PodRunningAsRoot
    expr: |
```

## 面试回答

**问题**: Kubernetes 如何保证集群的安全性？

**回答**: Kubernetes 通过多层次的安全机制来保证集群安全。首先是 **API 访问安全**，包括三个阶段：认证确定"谁在访问"，支持客户端证书、Service Account、OIDC 等方式；授权确定"可以做什么"，推荐使用 RBAC 基于角色的访问控制，通过 Role 和 RoleBinding 定义权限；准入控制在对象持久化前进行验证和修改，如 ResourceQuota、LimitRanger、ValidatingAdmissionWebhook 等。

其次是 **工作负载安全**，通过 Pod Security Standards 定义 Privileged、Baseline、Restricted 三个安全级别，在命名空间级别强制执行。Security Context 配置容器的安全设置，如禁止 root 运行、只读文件系统、删除 Linux 能力等。

第三是 **网络安全**，Network Policy 控制 Pod 之间的网络流量，可以实现命名空间隔离和微服务间的访问控制。Service Mesh（如 Istio）提供 mTLS 加密通信和细粒度的访问控制。

第四是 **数据安全**，Secret 支持静态加密，可以使用外部密钥管理系统（如 Vault、AWS Secrets Manager）管理敏感数据，etcd 数据也应该加密存储。

最后是 **审计日志**，记录 API 操作，便于安全审计和问题排查。生产环境应该启用 RBAC、Network Policy、Pod Security Standards、Secret 加密和审计日志，定期扫描镜像漏洞，及时更新 Kubernetes 版本。
