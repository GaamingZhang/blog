# Kubernetes CNI 模型深度解析

## 引言：容器网络的核心挑战

在 Kubernetes 集群中，Pod 的创建和销毁是频繁发生的日常操作。每个 Pod 都需要独立的 IP 地址，并且需要能够与集群内其他 Pod、Service 以及外部网络进行通信。这就带来了一个核心问题：**如何为动态创建的容器提供灵活、高效、可扩展的网络配置？**

传统的 Docker 网络模型（如 bridge 模式）无法满足 Kubernetes 的跨主机通信需求，而手动配置每个容器的网络接口既不现实也不可扩展。CNI（Container Network Interface）应运而生，它定义了一套标准的容器网络配置接口，让不同的网络实现能够以插件形式接入 Kubernetes，解决了容器网络管理的复杂性问题。

## 一、CNI 的定义和目标

### 1.1 什么是 CNI？

CNI（Container Network Interface）是由 Cloud Native Computing Foundation（CNCF）维护的一个开源项目，它定义了一套**标准的插件接口规范**，用于配置容器的网络接口。CNI 的核心思想是将容器运行时与网络实现解耦，让容器运行时专注于容器的生命周期管理，而网络配置则交给 CNI 插件来完成。

### 1.2 CNI 的设计目标

CNI 的设计遵循以下核心目标：

| 设计目标 | 说明 |
|---------|------|
| **简单性** | 接口规范简洁明了，易于理解和实现 |
| **可扩展性** | 支持多种网络实现方案，插件可插拔 |
| **通用性** | 不绑定特定的容器运行时，支持 Docker、containerd、CRI-O 等 |
| **可组合性** | 支持链式调用多个插件，实现复杂的网络配置 |
| **声明式配置** | 通过 JSON 配置文件描述网络需求 |

### 1.3 CNI 与其他网络模型的对比

| 网络模型 | 适用场景 | 特点 |
|---------|---------|------|
| **CNI** | Kubernetes、Mesos 等 | 轻量级、插件化、配置简单 |
| **CNM** | Docker 原生网络 | 功能丰富、与 Docker 深度集成 |
| **CNI vs CNM** | - | CNI 更轻量，CNM 功能更完善但复杂度更高 |

## 二、CNI 插件接口规范

### 2.1 核心接口定义

CNI 插件本质上是一个**可执行文件**，它需要实现以下四个核心操作：

```
ADD：    为容器添加网络接口
DEL：    删除容器的网络接口
CHECK：  检查容器网络配置是否正常（可选）
VERSION：查询插件支持的 CNI 版本
```

### 2.2 插件调用机制

CNI 插件的调用通过**环境变量**和**标准输入**传递参数：

**环境变量参数**：
- `CNI_COMMAND`：操作类型（ADD/DEL/CHECK/VERSION）
- `CNI_CONTAINERID`：容器 ID
- `CNI_NETNS`：容器网络命名空间路径
- `CNI_IFNAME`：需要配置的网络接口名称
- `CNI_ARGS`：额外参数
- `CNI_PATH`：插件可执行文件的搜索路径

**标准输入（JSON 配置）**：
```json
{
  "cniVersion": "0.4.0",
  "name": "my-network",
  "type": "bridge",
  "bridge": "cni0",
  "ipam": {
    "type": "host-local",
    "subnet": "10.244.0.0/16"
  }
}
```

### 2.3 插件配置文件结构

CNI 配置文件通常存放在 `/etc/cni/net.d/` 目录下，文件名格式为 `*.conflist` 或 `*.conf`。Kubernetes 会按照字母顺序选择第一个配置文件。

**单插件配置示例**：
```json
{
  "cniVersion": "0.4.0",
  "name": "bridge-network",
  "type": "bridge",
  "bridge": "cni0",
  "ipam": {
    "type": "host-local",
    "subnet": "10.244.0.0/16",
    "routes": [
      {"dst": "0.0.0.0/0"}
    ]
  }
}
```

**插件链配置示例**：
```json
{
  "cniVersion": "0.4.0",
  "name": "kube-network",
  "plugins": [
    {
      "type": "bridge",
      "bridge": "cni0",
      "ipam": {
        "type": "host-local",
        "subnet": "10.244.0.0/16"
      }
    },
    {
      "type": "portmap",
      "capabilities": {"portMappings": true}
    },
    {
      "type": "bandwidth",
      "capabilities": {"bandwidth": true}
    }
  ]
}
```

### 3.2 ADD 操作的详细步骤

当 Kubernetes 创建 Pod 时，CNI 插件的 ADD 操作会执行以下步骤：

**步骤 1：创建网络设备**
- 在宿主机上创建 `veth pair`（虚拟以太网对）
- 一端留在宿主机的网络命名空间（通常连接到网桥）
- 另一端移动到容器的网络命名空间

**步骤 2：配置网桥**
- 创建或使用现有的 Linux Bridge（如 `cni0`）
- 将 veth pair 的宿主机端连接到网桥

**步骤 3：IP 地址分配**
- 调用 IPAM 插件（如 host-local）
- 从配置的子网中分配一个可用的 IP 地址
- 配置容器的网络接口 IP 地址

**步骤 4：配置路由和规则**
- 配置容器内的默认路由
- 配置宿主机上的路由规则（用于跨主机通信）
- 设置 iptables 规则（如 NAT、端口映射等）

**步骤 5：返回结果**
- 插件通过标准输出返回 JSON 格式的配置结果
- 包含分配的 IP 地址、DNS 配置、路由信息等

## 四、常见 CNI 插件

### 4.1 主流 CNI 插件对比

| 插件名称 | 网络模式 | 特点 | 适用场景 |
|---------|---------|------|---------|
| **Flannel** | VXLAN/Host-GW/UDP | 简单易用、配置方便、支持多种后端 | 中小规模集群、快速部署 |
| **Calico** | BGP/IPIP | 性能优异、支持网络策略、可扩展性强 | 大规模集群、需要网络策略 |
| **Weave Net** | VXLAN | 自动拓扑、无需额外存储、支持加密 | 中小规模集群、快速体验 |
| **Cilium** | eBPF | 高性能、支持 L7 策略、可观测性强 | 云原生环境、需要高级网络功能 |
| **Canal** | Flannel + Calico | 结合两者优势、Flannel 网络 + Calico 策略 | 需要网络策略但不想用纯 Calico |

### 4.2 Flannel 工作原理

Flannel 是最简单易用的 CNI 插件之一，其核心原理是通过**覆盖网络（Overlay Network）**实现跨主机通信。

**Flannel 三种模式对比**：

| 模式 | 原理 | 性能 | 适用场景 |
|------|------|------|---------|
| **VXLAN** | 在 UDP 包中封装原始数据包 | 中等（封装开销） | 跨云环境、网络限制较多 |
| **Host-GW** | 通过宿主机路由表转发 | 高（无封装） | 同一子网、二层可达 |
| **UDP** | 用户态封装（已废弃） | 低（用户态开销大） | 仅用于测试 |

**Flannel VXLAN 模式工作流程**：

```
┌─────────────────────────────────────────────────────────────┐
│              Flannel VXLAN 跨主机通信流程                    │
└─────────────────────────────────────────────────────────────┘

节点 A (10.244.1.0/24)                    节点 B (10.244.2.0/24)
┌─────────────────────┐                  ┌─────────────────────┐
│ Pod1: 10.244.1.2    │                  │ Pod2: 10.244.2.3    │
│ ┌─────────────────┐ │                  │ ┌─────────────────┐ │
│ │ 发送数据包       │ │                  │ │ 接收数据包       │ │
│ │ 目标: 10.244.2.3│ │                  │ │                  │ │
│ └────────┬────────┘ │                  │ └────────▲────────┘ │
│          │          │                  │          │          │
│          ▼          │                  │          │          │
│ ┌─────────────────┐ │                  │ ┌────────┴────────┐ │
│ │ 容器 veth       │ │                  │ │ 容器 veth       │ │
│ └────────┬────────┘ │                  │ └────────▲────────┘ │
│          │          │                  │          │          │
│          ▼          │                  │          │          │
│ ┌─────────────────┐ │                  │ ┌────────┴────────┐ │
│ │ cni0 网桥       │ │                  │ │ cni0 网桥       │ │
│ └────────┬────────┘ │                  │ └────────▲────────┘ │
│          │          │                  │          │          │
│          ▼          │                  │          │          │
│ ┌─────────────────┐ │                  │ ┌────────┴────────┐ │
│ │ flannel.1       │ │                  │ │ flannel.1       │ │
│ │ (VXLAN VTEP)    │ │                  │ │ (VXLAN VTEP)    │ │
│ └────────┬────────┘ │                  │ └────────▲────────┘ │
│          │          │                  │          │          │
│          │  VXLAN封装                              │          │
│          │  UDP包 (外层IP: 节点A -> 节点B)         │          │
│          │          │                  │          │          │
│          ▼          │                  │          │          │
│ ┌─────────────────┐ │    网络传输      │ ┌────────┴────────┐ │
│ │ 物理网卡 eth0   │ ├─────────────────>│ │ 物理网卡 eth0   │ │
│ └─────────────────┘ │                  │ └─────────────────┘ │
└─────────────────────┘                  └─────────────────────┘
```

**Flannel 的关键组件**：

1. **flanneld**：运行在每个节点上的守护进程
   - 从 etcd 或 API Server 获取子网分配信息
   - 配置节点上的路由和 iptables 规则
   - 管理 flannel.1 接口

2. **子网管理**：
   - 每个节点分配一个子网（如 10.244.1.0/24）
   - 子网信息存储在 etcd 或 Kubernetes API 中
   - 避免跨节点的 IP 地址冲突

### 4.3 Calico 工作原理

Calico 是一个高性能、可扩展的 CNI 插件，使用**纯三层网络**实现，不依赖覆盖网络。

**Calico 核心组件**：

| 组件 | 功能 |
|------|------|
| **calico-node** | 运行在每个节点上的守护进程，负责路由和策略配置 |
| **calicoctl** | 命令行工具，用于管理 Calico 配置 |
| **Felix** | calico-node 的核心组件，负责配置路由、iptables 和网络策略 |
| **BIRD** | BGP 客户端，负责分发路由信息 |
| **confd** | 配置管理工具，监听 etcd 或 API Server 的变化 |
| **Typha** | 可选组件，用于减少 API Server 的压力 |

**Calico BGP 模式工作原理**：

```
┌─────────────────────────────────────────────────────────────┐
│                  Calico BGP 路由分发                         │
└─────────────────────────────────────────────────────────────┘

                    ┌──────────────────┐
                    │   BGP Peer       │
                    │  (路由反射器)     │
                    └────────┬─────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
              ▼              ▼              ▼
     ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
     │  节点 A     │  │  节点 B     │  │  节点 C     │
     │  BIRD       │  │  BIRD       │  │  BIRD       │
     │  Felix      │  │  Felix      │  │  Felix      │
     └─────────────┘  └─────────────┘  └─────────────┘
             │              │              │
             │              │              │
     ┌───────▼───────┐ ┌───▼────────┐ ┌───▼────────┐
     │ Pod: 10.0.1.2 │ │ Pod: 10.0.2.2│ │ Pod: 10.0.3.2│
     └───────────────┘ └────────────┘ └────────────┘

路由表示例（节点 A）：
10.0.1.0/24    dev cali1234    # 本节点 Pod
10.0.2.0/24    via 节点B      # 节点 B 的 Pod 子网
10.0.3.0/24    via 节点C      # 节点 C 的 Pod 子网
```

**Calico 的网络策略**：

Calico 支持丰富的网络策略，包括：

1. **Profile**：默认的安全策略，每个节点都有一个
2. **NetworkPolicy**：Kubernetes 原生的网络策略
3. **GlobalNetworkPolicy**：Calico 特有的全局网络策略
4. **NetworkSet**：IP 地址集合，用于策略规则

**网络策略示例**：

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend
  namespace: default
spec:
  podSelector:
    matchLabels:
      app: backend
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
```

### 4.4 Cilium 工作原理

Cilium 是基于 **eBPF（Extended Berkeley Packet Filter）** 的新一代 CNI 插件，提供了高性能和丰富的网络功能。

**Cilium 的核心优势**：

| 特性 | 说明 |
|------|------|
| **高性能** | eBPF 在内核态处理数据包，无需上下文切换 |
| **L7 策略** | 支持 HTTP、Kafka 等应用层协议的策略 |
| **可观测性** | 提供丰富的网络监控和追踪功能 |
| **负载均衡** | 内置高性能负载均衡器，可替代 kube-proxy |
| **安全** | 支持身份认证、加密通信等安全特性 |

## 五、CNI 与 Kubernetes 的集成

### 5.1 Kubernetes 网络模型要求

Kubernetes 对容器网络提出了以下核心要求：

| 要求 | 说明 |
|------|------|
| **所有 Pod 不使用 NAT 就能相互通信** | Pod 之间直接通信，无需地址转换 |
| **所有 Node 能与所有 Pod 通信** | 节点可以直接访问集群内所有 Pod |
| **Pod 看到的自己 IP 与其他人看到的一样** | Pod 的 IP 地址在集群内全局唯一且一致 |

### 5.3 kubelet 的 CNI 配置

kubelet 通过以下参数配置 CNI：

```bash
kubelet \
  --cni-conf-dir=/etc/cni/net.d \      # CNI 配置文件目录
  --cni-bin-dir=/opt/cni/bin \         # CNI 插件可执行文件目录
  --network-plugin=cni                 # 使用 CNI 网络插件
```

**kubelet 的 CNI 调用流程**：

1. kubelet 创建 Pod 时，调用 CRI Runtime 的 `RunPodSandbox` 接口
2. CRI Runtime 创建 Pause 容器和网络命名空间
3. CRI Runtime 调用 CNI 插件的 ADD 操作
4. CNI 插件配置网络接口、分配 IP 地址、设置路由
5. CRI Runtime 返回网络配置结果给 kubelet
6. kubelet 继续创建应用容器

### 5.4 CNI 配置文件的优先级

当 `/etc/cni/net.d/` 目录下存在多个配置文件时，kubelet 的选择规则：

1. 优先选择 `*.conflist` 文件（插件链配置）
2. 如果没有 `*.conflist`，选择 `*.conf` 文件
3. 按文件名字母顺序选择第一个

**推荐的文件命名方式**：
```
/etc/cni/net.d/
├── 10-cni.conflist      # 数字前缀控制优先级
├── 20-calico.conflist
└── 99-custom.conflist
```

## 六、CNI 插件的分类

### 6.1 按功能分类

CNI 插件可以分为以下几类：

| 类型 | 功能 | 示例插件 |
|------|------|---------|
| **主插件** | 创建网络接口、配置 IP 地址 | bridge, ipvlan, macvlan, ptp |
| **IPAM 插件** | IP 地址管理 | host-local, dhcp, static |
| **链式插件** | 扩展功能 | portmap, bandwidth, firewall, tuning |

### 6.2 主流插件详解

**bridge 插件**：
- 创建 Linux Bridge
- 连接容器和宿主机网络
- 最基础的 CNI 插件

**host-local IPAM**：
- 本地 IP 地址管理
- 从配置的子网中分配 IP
- IP 分配记录存储在本地文件

**portmap 插件**：
- 配置端口映射
- 实现 Pod 的端口转发
- 通常与主插件链式调用

**bandwidth 插件**：
- 限制 Pod 的网络带宽
- 使用 Linux TC（Traffic Control）
- 支持入站和出站带宽限制

## 七、常见问题与最佳实践

### 7.1 常见问题及解答

**问题 1：Pod 无法获取 IP 地址，一直处于 ContainerCreating 状态**

**原因分析**：
- CNI 插件未正确安装或配置
- IPAM 插件的 IP 地址池耗尽
- 网络配置文件格式错误
- CNI 插件可执行文件权限问题

**解决方案**：
```bash
# 检查 CNI 配置文件
ls -la /etc/cni/net.d/

# 检查 CNI 插件可执行文件
ls -la /opt/cni/bin/

# 查看 kubelet 日志
journalctl -u kubelet | grep -i cni

# 检查 IP 地址分配情况
cat /var/lib/cni/networks/<network-name>/*
```

**问题 2：跨节点 Pod 无法通信**

**原因分析**：
- 节点间网络不通（防火墙、安全组）
- 路由配置错误
- 覆盖网络封装问题（VXLAN 端口未开放）
- MTU 配置不当

**解决方案**：
```bash
# 检查节点间连通性
ping <node-ip>

# 检查 VXLAN 端口（默认 8472）
telnet <node-ip> 8472

# 检查路由表
ip route show

# 检查 MTU 设置
ip link show flannel.1
```

**问题 3：Pod 网络延迟高或丢包**

**原因分析**：
- 覆盖网络封装开销
- 网络策略规则过多
- 节点网络负载过高
- MTU 配置不当导致分片

**解决方案**：
- 使用 Host-GW 或 BGP 模式替代 VXLAN
- 优化网络策略规则
- 调整 MTU 值（通常设置为 1450 以适应 VXLAN 开销）
- 使用 Cilium 等高性能插件

**问题 4：CNI 插件升级后网络异常**

**原因分析**：
- 配置文件格式不兼容
- 旧版本插件残留
- 网络资源未正确清理

**解决方案**：
```bash
# 备份配置文件
cp -r /etc/cni/net.d /etc/cni/net.d.bak

# 清理旧的网络资源
ip link delete cni0
ip link delete flannel.1

# 重启 kubelet
systemctl restart kubelet
```

**问题 5：如何选择合适的 CNI 插件？**

**决策矩阵**：

| 场景 | 推荐插件 | 理由 |
|------|---------|------|
| 小型集群、快速体验 | Flannel | 简单易用、配置方便 |
| 生产环境、需要网络策略 | Calico | 性能好、功能完善 |
| 云原生环境、需要高级功能 | Cilium | 高性能、可观测性强 |
| 已有 Flannel 但需要策略 | Canal | 兼顾两者优势 |
| 多集群、混合云 | Calico/Cilium | 支持复杂的网络拓扑 |

### 7.2 最佳实践

**1. 配置文件管理**：
- 使用版本控制管理 CNI 配置文件
- 配置文件使用数字前缀控制优先级
- 保留默认配置文件作为备份

**2. 网络策略规划**：
- 默认拒绝所有流量，显式允许需要的流量
- 使用命名空间隔离不同环境
- 定期审计网络策略规则

**3. 性能优化**：
- 根据网络环境选择合适的网络模式
- 调整 MTU 值避免分片
- 使用高性能网络插件（如 Cilium）

**4. 监控和排错**：
- 监控网络延迟和丢包率
- 记录网络策略变更
- 建立网络故障排查流程

**5. 安全加固**：
- 限制 CNI 插件的权限
- 启用网络加密（如 WireGuard）
- 定期更新 CNI 插件版本

## 八、总结

### 核心要点回顾

CNI 作为 Kubernetes 网络的核心组件，通过标准化的插件接口实现了容器网络的灵活配置。其核心设计理念是将容器运行时与网络实现解耦，让不同的网络方案能够以插件形式无缝接入。

**关键要点**：
1. CNI 定义了四个核心操作：ADD、DEL、CHECK、VERSION
2. 插件通过环境变量和标准输入接收参数
3. 支持链式调用实现复杂网络配置
4. 主流插件各有特点，需根据场景选择
5. 与 Kubernetes 深度集成，由 kubelet 触发调用

### 面试回答

**面试官问：请简述 Kubernetes 的 CNI 模型？**

**回答**：

CNI（Container Network Interface）是 Kubernetes 用于配置容器网络的标准接口规范。它的核心思想是将容器运行时与网络实现解耦，通过插件化的方式让不同的网络方案能够接入 Kubernetes。

从技术实现上看，CNI 定义了四个核心操作：ADD 用于为容器添加网络接口，DEL 用于删除网络接口，CHECK 用于检查网络配置，VERSION 用于查询插件版本。CNI 插件本质上是一个可执行文件，通过环境变量和标准输入接收参数，执行网络配置后通过标准输出返回结果。

CNI 的工作流程是：当 kubelet 创建 Pod 时，会调用 CRI Runtime 创建网络命名空间，然后调用 CNI 插件的 ADD 操作，插件会创建 veth pair、配置网桥、分配 IP 地址、设置路由规则，最终让 Pod 具备网络通信能力。

常见的 CNI 插件包括 Flannel（简单易用，支持 VXLAN/Host-GW 模式）、Calico（高性能，支持 BGP 和网络策略）、Cilium（基于 eBPF，高性能且功能丰富）等。选择时需要根据集群规模、性能需求、是否需要网络策略等因素综合考虑。

CNI 的优势在于其简单性、可扩展性和通用性，让 Kubernetes 能够支持多种网络实现，满足不同场景的需求。同时，CNI 插件的链式调用机制也支持复杂的网络配置，如端口映射、带宽限制等扩展功能。
