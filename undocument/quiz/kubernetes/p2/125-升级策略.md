# Kubernetes 升级策略

## 引言

Kubernetes 集群升级是运维中的重要任务，需要确保升级过程中业务不受影响。合理的升级策略可以最小化风险，保证集群的稳定性和业务的连续性。本文介绍 Kubernetes 集群和应用的各种升级策略。

## 升级策略概述

### 升级类型

### 升级原则

| 原则 | 说明 |
|-----|------|
| 版本兼容性 | 只升级到相邻版本 |
| 备份数据 | 升级前备份 etcd |

| 验证测试 | 升级后验证功能 |

### 升级 kubeadm

```bash
apt-get update
apt-get install -y kubeadm=1.28.0-00

kubeadm version

kubeadm upgrade plan

kubeadm upgrade apply v1.28.0
```

### 升级控制平面

```bash
kubectl drain master-1 --ignore-daemonsets --delete-emptydir-data

kubeadm upgrade node

apt-get install -y kubelet=1.28.0-00 kubectl=1.28.0-00

systemctl daemon-reload
systemctl restart kubelet

kubectl uncordon master-1
```

### 升级工作节点


## 应用升级策略

### 滚动更新

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: rolling-update-app
spec:
  replicas: 5
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 1
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: app
        image: myapp:v2
        ports:
        - containerPort: 8080
```

### 使用 Argo Rollouts

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: myapp-rollout
spec:
  replicas: 5
  revisionHistoryLimit: 2
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: app
        image: myapp:v1
        ports:
        - containerPort: 8080
  strategy:
    canary:
      steps:
      - setWeight: 20
      - pause: {}
      - setWeight: 40
      - pause: {duration: 10}
      - setWeight: 60
      - pause: {duration: 10}
      - setWeight: 80
      - pause: {duration: 10}
```

```bash

```

```bash
kubectl get pods -l app=myapp

```

## 回滚操作

### 应用回滚

```bash
kubectl rollout undo deployment/myapp

kubectl rollout undo deployment/myapp --to-revision=2

kubectl rollout history deployment/myapp
```

### 集群回滚

```bash
kubeadm upgrade --rollback

systemctl restart kubelet
```

## 面试回答

**问题**: Kubernetes 的升级策略有哪些？

**回答**: Kubernetes 升级策略分为集群升级和应用升级两类：

**集群升级**：使用 kubeadm 升级，流程为：升级 kubeadm -> 升级第一个控制平面节点 -> 升级其他控制平面节点 -> 升级工作节点 -> 验证集群状态。升级原则：只升级到相邻版本；升级前备份 etcd；逐个节点升级；升级后验证功能；准备回滚方案。

**应用升级策略**：

**滚动更新（Rolling Update）**：默认策略，逐步替换旧版本 Pod。通过 maxSurge 和 maxUnavailable 控制更新速度。maxSurge 表示可以额外创建的 Pod 数量，maxUnavailable 表示不可用 Pod 的最大数量。优点是零停机、资源利用率高；缺点是回滚较慢。

**蓝绿部署（Blue-Green）**：同时部署新旧两个版本，通过切换 Service selector 实现流量切换。优点是快速切换、快速回滚；缺点是需要双倍资源。

**金丝雀发布（Canary）**：先部署少量新版本 Pod，逐步增加流量比例。通过调整副本数或使用 Istio 等工具控制流量比例。优点是风险可控、可以验证新版本；缺点是需要流量管理能力。

**最佳实践**：升级前备份数据；逐个节点升级；监控升级过程；准备回滚方案；使用 Argo Rollouts 等工具实现高级发布策略。
