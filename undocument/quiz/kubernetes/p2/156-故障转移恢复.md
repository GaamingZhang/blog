# Kubernetes 故障转移和恢复

## 引言

故障转移和恢复是保障 Kubernetes 集群高可用的关键能力。当节点、Pod 或组件发生故障时，Kubernetes 能够自动检测并进行故障转移，确保应用持续运行。本文介绍 Kubernetes 故障转移和恢复的机制和最佳实践。

## Pod 故障转移

### 自动重启

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: restart-pod
spec:
  restartPolicy: Always
  containers:
  - name: app
    image: myapp:v1
    livenessProbe:
      httpGet:
        path: /health
        port: 8080
      initialDelaySeconds: 30
      periodSeconds: 10
```

### restartPolicy 说明

| 策略 | 说明 |
|-----|------|
| Always | 总是重启（默认） |
| OnFailure | 失败时重启 |
| Never | 从不重启 |

### Deployment 自动恢复

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: app
        image: myapp:v1
        livenessProbe:
          httpGet:
            path: /health
            port: 8080
        readinessProbe:
          httpGet:
            path: /ready
            port: 8080
```

### 配置故障转移时间

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: fast-failover
spec:
  tolerations:
  - key: "node.kubernetes.io/not-ready"
    operator: "Exists"
    effect: "NoExecute"
    tolerationSeconds: 60
  - key: "node.kubernetes.io/unreachable"
    operator: "Exists"
    effect: "NoExecute"
    tolerationSeconds: 60
  containers:
  - name: app
    image: myapp:v1
```

### Pod Disruption Budget

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: myapp-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: myapp
```

### etcd 故障

```bash
# 检查 etcd 状态
ETCDCTL_API=3 etcdctl endpoint health

# 恢复 etcd
ETCDCTL_API=3 etcdctl snapshot restore snapshot.db
```

## 故障恢复流程

### Pod 故障恢复

```bash
# 查看 Pod 状态
kubectl get pods -n production

# 查看 Pod 事件
kubectl describe pod <pod-name> -n production

# 查看 Pod 日志
kubectl logs <pod-name> -n production --previous

# 强制删除 Pod
kubectl delete pod <pod-name> -n production --force --grace-period=0
```

### 节点故障恢复

```bash
# 查看节点状态
kubectl get nodes

# 查看节点详情
kubectl describe node <node-name>

# 驱逐节点上的 Pod
kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data

# 恢复节点
kubectl uncordon <node-name>

# 删除节点
kubectl delete node <node-name>
```

### 集群故障恢复

```bash
# 检查控制平面组件
kubectl get cs

# 检查 etcd
ETCDCTL_API=3 etcdctl endpoint health

# 重启控制平面组件
systemctl restart kubelet
systemctl restart docker
```

## 面试回答

**问题**: 如何进行故障转移和恢复？

**回答**: Kubernetes 故障转移和恢复分为 Pod 故障、节点故障、控制平面故障三种情况：

**Pod 故障转移**：**自动重启**通过 restartPolicy 配置（Always/OnFailure/Never）；**健康检查**livenessProbe 检测容器存活，失败时重启容器；**Deployment 自动恢复**控制器确保副本数符合预期，Pod 异常时自动重建。

**节点故障转移**：**故障检测**kube-controller-manager 监控节点心跳，节点失联后状态变为 NotReady；**Pod 迁移**等待 pod-eviction-timeout（默认 5 分钟）后，Pod 被驱逐并在其他节点重建；**tolerationSeconds** 配置 Pod 对节点故障的容忍时间。

**控制平面故障**：**API Server 故障**多副本部署 + 负载均衡实现高可用；**etcd 故障**定期备份，使用快照恢复；**Scheduler/Controller Manager** Leader Election 机制，主节点故障时自动切换。

**故障恢复流程**：**Pod 故障**查看状态、日志、事件，必要时强制删除重建；**节点故障**查看节点状态、驱逐 Pod、恢复或删除节点；**集群故障**检查控制平面组件、恢复 etcd、重启服务。

**最佳实践**：配置健康检查；设置资源限制；多副本部署；Pod 反亲和性分散副本；配置 PDB 保证最小可用副本；定期备份 etcd；制定故障恢复预案。
