# Kubernetes 高可用架构深度解析

## 引言：为什么K8s需要高可用？

在生产环境中，Kubernetes集群承载着关键业务应用，任何单点故障都可能导致服务中断，造成严重的业务损失。一个标准的单Master节点K8s集群存在多个潜在故障点：Master节点宕机会导致无法创建、更新或调度新的Pod；etcd数据存储损坏会造成集群状态丢失；单个Worker节点故障会影响该节点上运行的所有应用。

**高可用（High Availability, HA）的核心目标**是消除单点故障，确保集群在部分组件失效时仍能持续提供服务。Kubernetes的高可用架构涉及多个层面：控制平面高可用、数据存储高可用、工作节点高可用以及应用层高可用。本文将深入剖析每个层面的实现原理和最佳实践。

## 一、控制平面高可用

### 1.1 多Master架构

控制平面（Control Plane）是Kubernetes的大脑，包含四个核心组件：kube-apiserver、kube-controller-manager、kube-scheduler和cloud-controller-manager。在单Master架构中，这些组件仅运行在一个节点上，存在严重的单点故障风险。

**多Master架构的核心思想**是在多个节点上部署控制平面组件，通过负载均衡机制实现故障转移。标准的高可用配置通常采用3个或5个Master节点，遵循奇数原则以支持etcd的Raft一致性协议。

### 1.2 kube-apiserver高可用

kube-apiserver是控制平面的入口，所有组件和用户请求都通过它访问集群状态。实现apiserver高可用的关键在于**负载均衡层**。

**架构原理**：

```
                    负载均衡器 (LB)
                    /      |      \
                   /       |       \
          apiserver-1  apiserver-2  apiserver-3
          (Master-1)    (Master-2)   (Master-3)
```

负载均衡器可以是硬件设备（F5、A10）或软件实现（HAProxy、Nginx、keepalived）。每个apiserver实例都是无状态的，它们共享同一个etcd集群，因此可以水平扩展。

**关键配置要点**：

- **健康检查**：负载均衡器必须配置apiserver的健康检查端点（`/healthz`、`/livez`、`/readyz`），及时剔除故障节点
- **会话保持**：apiserver支持会话亲和性，但建议在负载均衡层禁用，让请求均匀分布
- **证书配置**：负载均衡器的证书必须包含apiserver的SAN（Subject Alternative Name），确保TLS握手成功

### 1.3 kube-controller-manager和kube-scheduler高可用

这两个组件通过**Leader Election（领导者选举）机制**实现高可用，而非负载均衡。

**实现原理**：

多个实例同时运行，但只有一个处于活跃状态（Leader），其他实例处于待命状态（Standby）。Leader通过etcd或Kubernetes Endpoints对象持有租约（Lease），定期续租以维持领导地位。如果Leader故障无法续租，其他实例通过竞争成为新的Leader。

**租约机制详解**：

```yaml
# Leader Election使用Kubernetes Lease对象
apiVersion: coordination.k8s.io/v1
kind: Lease
metadata:
  name: kube-controller-manager
  namespace: kube-system
spec:
  holderIdentity: master-1  # 当前Leader标识
  leaseDurationSeconds: 15   # 租约有效期
  acquireTime: "2026-03-12T10:00:00Z"
  renewTime: "2026-03-12T10:00:10Z"
```

关键参数解析：
- **leaseDurationSeconds**：租约有效期，Leader必须在此时间内续租，默认15秒
- **renewDeadlineSeconds**：续租超时时间，默认10秒
- **retryPeriodSeconds**：重试间隔，默认2秒

这种机制的**优势**在于无需外部协调服务，利用Kubernetes自身的资源对象实现选举，简化了架构复杂度。

## 二、etcd高可用

etcd是Kubernetes的存储后端，保存了集群的所有状态数据。etcd的高可用是整个K8s高可用架构的基石。

### 2.1 Raft一致性协议

etcd使用Raft协议实现分布式一致性。Raft协议的核心机制包括：

**领导者选举**：
- 集群启动时，所有节点处于Follower状态
- Follower在选举超时后转为Candidate，发起选举
- 获得多数节点投票的Candidate成为Leader
- Leader负责处理所有写请求，并将日志复制到Follower

**日志复制**：
- 客户端写请求发送到Leader
- Leader将操作追加到日志，并行发送到所有Follower
- 当多数节点确认后，Leader提交日志并应用到状态机
- 该机制确保了强一致性（Strong Consistency）

**安全性保证**：
- 所有已提交的日志在未来的Leader中必然存在
- 通过任期号（Term）和日志索引防止脑裂

### 2.2 集群规模与奇数原则

etcd集群规模必须是奇数（3、5、7），这源于Raft协议的多数派原则。集群的容错能力计算公式为：

```
容错节点数 = (N - 1) / 2
```

| 集群规模 | 容错节点数 | 推荐场景 |
|---------|-----------|---------|
| 1 | 0 | 开发测试环境 |
| 3 | 1 | 小规模生产环境 |
| 5 | 2 | 中大规模生产环境 |
| 7 | 3 | 超大规模关键业务 |

**为什么不用偶数节点？** 以4节点集群为例，容错能力仍为1（需要3节点存活），与3节点集群相同，但增加了网络开销和维护成本。

### 2.3 etcd部署模式

Kubernetes支持两种etcd部署模式，各有优劣：

**堆叠etcd（Stacked etcd）**：

etcd与控制平面组件部署在同一节点上，每个Master节点运行一个etcd实例。

```
┌─────────────────────────────────────┐
│         Master Node 1               │
│  ┌──────────┐  ┌─────────────────┐  │
│  │  etcd-1  │  │  apiserver-1    │  │
│  └──────────┘  │  controller-mgr │  │
│                │  scheduler       │  │
│                └─────────────────┘  │
└─────────────────────────────────────┘
```

**优势**：
- 架构简单，部署便捷
- 减少网络跳数，性能较好
- 节省硬件资源

**劣势**：
- etcd与Master耦合，节点故障影响双重组件
- 资源竞争可能影响性能
- 扩展性受限

**外部etcd（External etcd）**：

etcd集群独立部署，与控制平面节点分离。

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│   Master-1   │    │   Master-2   │    │   Master-3   │
│  apiserver   │    │  apiserver   │    │  apiserver   │
│  controller  │    │  controller  │    │  controller  │
│  scheduler   │    │  scheduler   │    │  scheduler   │
└──────┬───────┘    └──────┬───────┘    └──────┬───────┘
       │                   │                   │
       └───────────────────┼───────────────────┘
                           │
       ┌───────────────────┼───────────────────┐
       │                   │                   │
┌──────▼──────┐    ┌──────▼──────┐    ┌──────▼──────┐
│   etcd-1    │    │   etcd-2    │    │   etcd-3    │
└─────────────┘    └─────────────┘    └─────────────┘
```

**优势**：
- 控制平面与数据存储解耦，故障隔离性好
- etcd可独立扩展和优化
- 资源隔离，避免竞争

**劣势**：
- 架构复杂，运维成本高
- 增加网络延迟
- 需要更多硬件资源

### 2.4 etcd数据备份与恢复

高可用不等于数据安全，定期备份是灾难恢复的关键。etcd提供快照备份机制：

**备份原理**：
- etcd通过boltdb存储数据，支持在线快照
- 快照包含完整的数据状态，可用于恢复或迁移
- 建议定期备份（如每小时一次），并保留多个版本

**恢复流程**：
1. 停止所有etcd实例
2. 使用快照恢复单个节点的数据目录
3. 启动该节点，让其成为新集群的初始成员
4. 逐步添加其他节点，重建集群

## 三、工作节点高可用

工作节点（Worker Node）运行实际的应用负载，其高可用通过**冗余部署**和**故障转移机制**实现。

### 3.1 节点故障检测

kubelet定期向apiserver发送心跳，包括NodeStatus更新和Lease续租。节点控制器（Node Controller）监控节点状态，判断节点是否健康。

**节点状态判定逻辑**：

- **Ready**：节点健康，可调度Pod
- **NotReady**：节点失联超过40秒（默认node-monitor-grace-period）
- **Unknown**：节点状态未知

**Pod驱逐机制**：

当节点标记为NotReady后，节点控制器启动Pod驱逐流程：
- 等待pod-eviction-timeout（默认5分钟）
- 删除该节点上的所有Pod
- Deployment/ReplicaSet控制器在其他节点重建Pod

### 3.2 多Worker节点架构

生产环境应部署多个Worker节点，确保即使部分节点故障，集群仍有足够资源运行应用。

**容量规划原则**：

```
集群可用容量 = 总容量 - 单节点最大容量
```

假设集群有10个Worker节点，每个节点100个Pod，单节点最大容量为100个Pod，则：
- 集群总容量：1000个Pod
- 单节点故障影响：100个Pod
- 集群可用容量：900个Pod（考虑1个节点故障）

**资源预留**：
- 为系统守护进程预留资源（kubelet、kube-proxy、容器运行时）
- 为节点故障预留冗余容量
- 建议预留20%-30%的资源余量

## 四、应用层高可用

即使集群基础设施高可用，应用本身也必须设计为高可用架构。

### 4.1 多副本部署

通过Deployment控制Pod副本数，确保应用有多个实例运行。

**副本数设计原则**：
- 最少2个副本，支持滚动更新和故障转移
- 关键业务建议3个或更多副本
- 副本数应考虑节点数量和资源容量

### 4.2 Pod反亲和性

防止多个Pod副本调度到同一节点，避免节点故障导致应用完全不可用。

**反亲和性配置示例**：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3
  template:
    spec:
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 100
            podAffinityTerm:
              labelSelector:
                matchExpressions:
                - key: app
                  operator: In
                  values:
                  - web-app
              topologyKey: kubernetes.io/hostname
```

**topologyKey选择**：
- `kubernetes.io/hostname`：不同节点
- `topology.kubernetes.io/zone`：不同可用区
- `topology.kubernetes.io/region`：不同地域

### 4.3 PodDisruptionBudget（PDB）

PDB限制同时中断的Pod数量，确保应用在维护期间保持可用。

**工作原理**：
- 定义应用的最小可用副本数或最大不可用副本数
- 在节点排空（Drain）或集群升级时生效
- 阻止过多Pod同时被驱逐

**配置示例**：

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-app-pdb
spec:
  minAvailable: 2  # 至少保持2个Pod运行
  selector:
    matchLabels:
      app: web-app
```

### 4.4 健康检查与自愈

Kubernetes提供三种探针实现应用自愈：

**Liveness Probe（存活探针）**：
- 检测容器是否存活
- 失败时重启容器
- 适用于死锁、线程阻塞等场景

**Readiness Probe（就绪探针）**：
- 检测容器是否准备好接收流量
- 失败时从Service端点移除
- 适用于应用启动慢、依赖外部服务场景

**Startup Probe（启动探针）**：
- 检测容器是否启动成功
- 在启动期间禁用其他探针
- 适用于启动时间长的应用

## 六、高可用方案对比

| 对比维度 | 堆叠etcd | 外部etcd |
|---------|---------|---------|
| 架构复杂度 | 低 | 高 |
| 部署难度 | 简单 | 复杂 |
| 硬件成本 | 低 | 高 |
| 网络延迟 | 低 | 中等 |
| 故障隔离性 | 差 | 好 |
| 运维成本 | 低 | 高 |
| 性能表现 | 较好 | 好 |
| 扩展性 | 受限 | 灵活 |
| 推荐场景 | 中小规模集群 | 大规模关键业务 |

**选择建议**：
- 节点数少于50：堆叠etcd，架构简单
- 节点数50-200：堆叠etcd或外部etcd均可
- 节点数超过200：外部etcd，便于独立扩展和优化

## 七、实际部署配置示例

### 7.1 使用kubeadm部署高可用集群

kubeadm支持两种高可用部署方式：堆叠etcd和外部etcd。

**堆叠etcd模式初始化**：

```bash
# 在第一个Master节点初始化
kubeadm init \
  --control-plane-endpoint "lb.example.com:6443" \
  --upload-certs \
  --pod-network-cidr=10.244.0.0/16

# 在其他Master节点加入
kubeadm join lb.example.com:6443 \
  --token <token> \
  --discovery-token-ca-cert-hash sha256:<hash> \
  --control-plane \
  --certificate-key <cert-key>
```

**外部etcd模式配置**：

```yaml
# kubeadm-config.yaml
apiVersion: kubeadm.k8s.io/v1beta3
kind: ClusterConfiguration
kubernetesVersion: v1.28.0
controlPlaneEndpoint: "lb.example.com:6443"
etcd:
  external:
    endpoints:
    - https://etcd-1.example.com:2379
    - https://etcd-2.example.com:2379
    - https://etcd-3.example.com:2379
    caFile: /etc/kubernetes/pki/etcd/ca.crt
    certFile: /etc/kubernetes/pki/etcd/server.crt
    keyFile: /etc/kubernetes/pki/etcd/server.key
```

### 7.2 负载均衡器配置

使用HAProxy和keepalived构建高可用负载均衡器。

**HAProxy配置**：

```haproxy
frontend k8s-apiserver
    bind *:6443
    mode tcp
    option tcplog
    default_backend k8s-apiserver

backend k8s-apiserver
    mode tcp
    option tcp-check
    balance roundrobin
    server master-1 192.168.1.11:6443 check
    server master-2 192.168.1.12:6443 check
    server master-3 192.168.1.13:6443 check
```

**keepalived配置**：

```conf
vrrp_instance VI_1 {
    state MASTER
    interface eth0
    virtual_router_id 51
    priority 100
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass secret
    }
    virtual_ipaddress {
        192.168.1.100
    }
}
```

## 八、常见问题与最佳实践

### 常见问题解答

**Q1: Master节点全部宕机，集群还能运行吗？**

A: 已运行的Pod会继续工作，但无法创建新Pod、执行调度或更新资源。控制平面的高可用确保即使部分Master故障，集群仍可管理。建议至少3个Master节点，容许1个节点故障。

**Q2: etcd集群性能下降如何排查？**

A: 检查磁盘I/O性能（etcd对磁盘延迟敏感）、网络延迟、集群健康状态（`etcdctl endpoint health`）、数据库大小（`etcdctl endpoint status`）。建议使用SSD磁盘，保持网络延迟低于10ms。

**Q3: 如何实现跨机房高可用？**

A: 使用多集群架构（如KubeFed、Rancher多集群）或跨可用区部署。跨机房同步延迟较大，建议应用层实现数据同步，而非强依赖etcd跨机房集群。

**Q4: Pod反亲和性导致调度失败怎么办？**

A: 使用`preferredDuringSchedulingIgnoredDuringExecution`（软策略）而非`requiredDuringSchedulingIgnoredDuringExecution`（硬策略），或增加节点数量。检查节点资源是否充足。

**Q5: 如何验证高可用架构的有效性？**

A: 定期进行故障演练（Chaos Engineering），模拟Master节点宕机、etcd节点故障、Worker节点故障等场景，验证应用是否正常恢复。使用工具如Chaos Mesh、Litmus。

### 最佳实践

1. **控制平面冗余**：至少3个Master节点，使用负载均衡器分发apiserver流量
2. **etcd优化**：使用SSD磁盘、独立网络、定期备份、监控性能指标
3. **节点容量规划**：预留20%-30%资源余量，支持节点故障时的Pod迁移
4. **应用高可用**：多副本部署、Pod反亲和性、PDB保护、健康检查
5. **监控告警**：监控etcd性能、apiserver延迟、节点状态、Pod健康
6. **定期演练**：定期进行故障注入测试，验证高可用机制有效性
7. **备份策略**：定期备份etcd数据、集群配置、应用清单
8. **文档化**：记录架构设计、故障处理流程、恢复步骤

## 九、总结

Kubernetes高可用架构是一个多层次的系统工程，涉及控制平面、数据存储、工作节点和应用层四个维度。控制平面通过多Master和负载均衡实现apiserver高可用，通过Leader Election机制实现controller-manager和scheduler高可用。etcd作为数据存储核心，通过Raft协议确保强一致性，采用奇数节点集群提供容错能力。工作节点通过冗余部署和故障转移机制保障应用运行，应用层通过多副本、反亲和性、PDB和健康检查实现业务连续性。

选择堆叠etcd还是外部etcd取决于集群规模和业务需求，中小规模集群推荐堆叠etcd以简化架构，大规模关键业务推荐外部etcd以获得更好的隔离性和扩展性。无论采用哪种方案，定期备份、故障演练和监控告警都是保障高可用架构有效性的关键措施。

## 面试回答

**面试官：K8s集群中有没有高可用？高可用架构是什么？**

**回答**：Kubernetes集群有完善的高可用架构，主要从四个层面实现。第一是控制平面高可用，通过部署多个Master节点，apiserver前面加负载均衡器实现流量分发和故障转移，controller-manager和scheduler通过Leader Election机制实现主备切换，同一时间只有一个实例工作，其他待命。第二是etcd高可用，etcd使用Raft一致性协议，通常部署3或5个节点的集群，通过多数派机制保证数据一致性，允许少数节点故障。etcd可以与Master节点堆叠部署，也可以独立部署，各有优劣。第三是工作节点高可用，通过部署多个Worker节点，当某个节点故障时，节点控制器会驱逐该节点上的Pod，调度器会在其他节点重建。第四是应用层高可用，通过Deployment多副本部署、Pod反亲和性避免单节点风险、PDB限制同时中断数量、健康检查实现自愈。整体架构通过消除单点故障，确保集群在部分组件失效时仍能持续提供服务，生产环境建议至少3个Master节点、3个etcd节点、多个Worker节点，并定期进行故障演练验证高可用机制的有效性。
