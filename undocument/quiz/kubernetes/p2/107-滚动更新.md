# Kubernetes 滚动更新详解

## 引言

在现代应用部署中，如何在不中断服务的情况下更新应用是一个关键问题。Kubernetes 的滚动更新（Rolling Update）机制提供了一种平滑、渐进的应用更新方式，确保在更新过程中服务始终可用。

滚动更新是 Kubernetes Deployment 的核心功能之一，它通过逐步替换旧版本的 Pod 来实现应用的无缝升级。理解滚动更新的工作原理和配置方法，对于实现零停机部署至关重要。

## 滚动更新概述

### 什么是滚动更新

滚动更新是一种渐进式的应用更新策略，它逐步用新版本的 Pod 替换旧版本的 Pod，在整个更新过程中保持服务的可用性。

### 滚动更新的优势

| 优势 | 说明 |
|-----|------|
| **零停机** | 更新过程中服务始终可用 |
| **渐进式** | 可以逐步观察新版本状态 |
| **可回滚** | 发现问题可以快速回滚 |
| **可控性** | 可以控制更新速度 |

## 滚动更新配置

### 基本配置

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: app
        image: nginx:1.21
        ports:
        - containerPort: 80
```

### 更新策略参数

| 参数 | 说明 | 默认值 |
|-----|------|--------|
| **maxSurge** | 最多可以多创建的 Pod 数量或比例 | 25% |
| **maxUnavailable** | 最多不可用的 Pod 数量或比例 | 25% |

### 参数详解

#### maxSurge

```yaml
strategy:
  rollingUpdate:
    maxSurge: 1        # 绝对值：最多多创建 1 个 Pod
    # 或
    maxSurge: 25%      # 百分比：最多多创建 25% 的 Pod
```

#### maxUnavailable

```yaml
strategy:
  rollingUpdate:
    maxUnavailable: 0  # 绝对值：不允许任何 Pod 不可用
    # 或
    maxUnavailable: 25%  # 百分比：最多 25% 的 Pod 不可用
```

### 常见配置模式

#### 模式一：零停机更新

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1
    maxUnavailable: 0  # 不允许任何 Pod 不可用
```

特点：
- 先创建新 Pod，就绪后再删除旧 Pod
- 更新过程中副本数会暂时超过期望值
- 保证服务始终可用

#### 模式二：快速更新

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 0
    maxUnavailable: 1  # 允许 1 个 Pod 不可用
```

特点：
- 先删除旧 Pod，再创建新 Pod
- 更新过程中副本数不会超过期望值
- 更新速度更快，但有短暂不可用

#### 模式三：平衡模式

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 25%
    maxUnavailable: 25%
```

特点：
- 平衡更新速度和可用性
- 适合大多数场景

## 触发滚动更新

### 方式一：修改镜像

```bash
# 使用 kubectl set image
kubectl set image deployment/my-app app=nginx:1.22

# 查看更新状态
kubectl rollout status deployment/my-app
```

### 方式二：修改 YAML 文件

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  template:
    spec:
      containers:
      - name: app
        image: nginx:1.22  # 修改镜像版本
```

```bash
# 应用更新
kubectl apply -f deployment.yaml
```

### 方式三：使用 kubectl edit

```bash
# 直接编辑 Deployment
kubectl edit deployment my-app

# 在编辑器中修改镜像版本
# 保存后自动触发更新
```

### 方式四：使用 kubectl patch

```bash
# 使用 patch 修改
kubectl patch deployment my-app -p '{"spec":{"template":{"spec":{"containers":[{"name":"app","image":"nginx:1.22"}]}}}}'
```

### 触发更新的条件

以下变更会触发滚动更新：

| 变更类型 | 说明 |
|---------|------|
| **镜像变更** | 修改 container.image |
| **环境变量变更** | 修改 env 配置 |
| **ConfigMap/Secret 变更** | 挂载的配置变化 |
| **资源限制变更** | 修改 resources |
| **Pod 标签变更** | 修改 Pod 模板标签 |

## 暂停和恢复更新

### 暂停更新

```bash
# 暂停更新
kubectl rollout pause deployment/my-app

# 进行多次修改
kubectl set image deployment/my-app app=nginx:1.22
kubectl set resources deployment/my-app -c app --limits=cpu=200m,memory=256Mi
kubectl set env deployment/my-app ENV=production
```

### 恢复更新

```bash
# 恢复更新
kubectl rollout resume deployment/my-app

# 查看更新状态
kubectl rollout status deployment/my-app
```

### 使用场景

- 需要进行多项修改时
- 避免多次触发滚动更新
- 批量更新配置

## 回滚操作

### 查看历史版本

```bash
# 查看历史版本
kubectl rollout history deployment/my-app

# 输出示例
REVISION  CHANGE-CAUSE
1         <none>
2         kubectl set image deployment/my-app app=nginx:1.22
3         kubectl set image deployment/my-app app=nginx:1.23

# 查看特定版本详情
kubectl rollout history deployment/my-app --revision=2
```

### 回滚到上一版本

```bash
# 回滚到上一版本
kubectl rollout undo deployment/my-app

# 查看回滚状态
kubectl rollout status deployment/my-app
```

### 回滚到指定版本

```bash
# 回滚到指定版本
kubectl rollout undo deployment/my-app --to-revision=2
```

### 配置历史版本数量

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  revisionHistoryLimit: 10  # 保留 10 个历史版本
  template:
    spec:
      containers:
      - name: app
        image: nginx:1.21
```

## 滚动更新最佳实践

### 1. 配置健康检查

```yaml
livenessProbe:
  httpGet:
    path: /health
    port: 80
  initialDelaySeconds: 30
  periodSeconds: 10
readinessProbe:
  httpGet:
    path: /ready
    port: 80
  initialDelaySeconds: 5
  periodSeconds: 5
```

### 2. 配置优雅终止

```yaml
lifecycle:
  preStop:
    exec:
      command: ["/bin/sh", "-c", "sleep 15"]
terminationGracePeriodSeconds: 60
```

### 3. 设置合理的更新参数

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1
    maxUnavailable: 0
```

### 4. 使用 Pod Disruption Budget

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: my-app-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: my-app
```

### 5. 监控更新过程

```yaml
# Prometheus 告警规则
- alert: DeploymentRolloutStuck
  expr: |
    kube_deployment_status_replicas_updated != kube_deployment_spec_replicas
  for: 10m
  labels:
    severity: warning
  annotations:
    summary: "Deployment {{ $labels.deployment }} rollout is stuck"
```

## 金丝雀发布

### 概念

金丝雀发布是一种渐进式发布策略，先让小部分流量访问新版本，观察无误后再逐步扩大范围。

### 实现方式

```yaml
# 稳定版本
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-stable
spec:
  replicas: 9
  selector:
    matchLabels:
      app: my-app
      version: stable
  template:
    metadata:
      labels:
        app: my-app
        version: stable
    spec:
      containers:
      - name: app
        image: nginx:1.21
---
# 金丝雀版本
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-canary
spec:
  replicas: 1  # 10% 流量
  selector:
    matchLabels:
      app: my-app
      version: canary
  template:
    metadata:
      labels:
        app: my-app
        version: canary
    spec:
      containers:
      - name: app
        image: nginx:1.22
---
# Service
apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  selector:
    app: my-app
  ports:
  - port: 80
    targetPort: 80
```

## 蓝绿部署

### 概念

蓝绿部署是维护两套完全独立的环境，通过切换流量实现快速发布和回滚。

### 实现方式

```yaml
# 蓝色版本（当前生产）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-blue
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
      color: blue
  template:
    metadata:
      labels:
        app: my-app
        color: blue
    spec:
      containers:
      - name: app
        image: nginx:1.21
---
# 绿色版本（新版本）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-green
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
      color: green
  template:
    metadata:
      labels:
        app: my-app
        color: green
    spec:
      containers:
      - name: app
        image: nginx:1.22
---
# Service（切换 selector 切换流量）
apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  selector:
    app: my-app
    color: blue  # 切换为 green 即可切换流量
  ports:
  - port: 80
    targetPort: 80
```

## 面试回答

**问题**: 如何在 Kubernetes 中进行滚动更新？

**回答**: Kubernetes 滚动更新通过 Deployment 的 RollingUpdate 策略实现，它逐步用新版本的 Pod 替换旧版本的 Pod，在整个更新过程中保持服务可用。

**配置滚动更新**需要设置 `strategy.type` 为 `RollingUpdate`，并配置两个关键参数：`maxSurge` 定义最多可以多创建的 Pod 数量或比例，`maxUnavailable` 定义最多不可用的 Pod 数量或比例。例如 `maxSurge: 1, maxUnavailable: 0` 表示先创建新 Pod，就绪后再删除旧 Pod，实现零停机更新。

**触发滚动更新**的方式包括：`kubectl set image` 命令修改镜像；`kubectl apply` 应用修改后的 YAML；`kubectl edit` 直接编辑 Deployment；`kubectl patch` 使用 patch 命令。任何修改 Pod 模板的操作都会触发滚动更新。

**查看更新状态**使用 `kubectl rollout status` 命令，可以查看更新进度。更新过程中，Deployment 会创建新的 ReplicaSet，逐步调整新旧 ReplicaSet 的副本数。

**暂停和恢复更新**使用 `kubectl rollout pause` 和 `kubectl rollout resume` 命令，适合需要进行多项修改的场景。

**回滚操作**使用 `kubectl rollout undo` 命令，可以回滚到上一版本或指定版本。通过 `revisionHistoryLimit` 配置保留的历史版本数量。

生产环境最佳实践：配置 livenessProbe 和 readinessProbe 确保新 Pod 就绪后才接收流量；配置 preStop 钩子和 terminationGracePeriodSeconds 实现优雅终止；使用 Pod Disruption Budget 保证最小可用副本；监控更新过程，及时发现异常。
