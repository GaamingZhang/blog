# Kubernetes Pod IP 和 Service IP 访问超时处理完全指南

## 引言：当网络连接成为拦路虎

在 Kubernetes 集群的日常运维中，你是否遇到过这样的场景：通过 `kubectl get pods` 看到所有 Pod 都处于 Running 状态，服务看似一切正常，但当你尝试访问 Pod IP 或 Service ClusterIP 时，却遭遇连接超时？这种"表面风平浪静，暗地波涛汹涌"的网络问题，常常让运维人员陷入长时间的排查困境。

访问超时问题不仅影响服务的可用性，更可能隐藏着网络配置、安全策略或组件故障等深层次问题。本文将系统性地剖析 Pod IP 和 Service IP 访问超时的排查思路，帮助你快速定位问题根源，建立完整的网络故障处理知识体系。

## 核心概念：理解 Kubernetes 网络模型

在深入排查之前，我们需要理解 Kubernetes 网络的核心设计原则：

### Pod 网络模型

Kubernetes 要求所有 Pod 必须满足以下网络条件：

1. **所有 Pod 不使用 NAT 就能相互通信**：每个 Pod 拥有独立的 IP 地址，Pod 之间直接通过 IP 通信
2. **所有 Node 都能与所有 Pod 通信**：Node 可以直接访问任何 Pod 的 IP
3. **Pod 看到的自身 IP 与其他 Pod 看到的 IP 相同**：不存在 IP 转换或映射

### Service 网络模型

Service 为一组 Pod 提供稳定的访问入口，主要类型包括：

- **ClusterIP**：集群内部虚拟 IP，仅在集群内部可访问
- **NodePort**：在每个 Node 上开放端口，外部可通过 NodeIP:Port 访问
- **LoadBalancer**：对接云厂商负载均衡器
- **ExternalName**：将 Service 映射到外部 DNS 名称

## Pod IP 访问超时排查

### 排查思路

Pod IP 访问超时的排查遵循"由内而外、逐层递进"的原则：

```
应用层 → 容器网络 → Pod 网络 → 节点网络 → 集群网络
```

### 第一层：应用层检查

**检查应用是否正常监听端口**

```bash
# 进入 Pod 内部检查端口监听状态
kubectl exec -it <pod-name> -n <namespace> -- netstat -tlnp

# 或使用 ss 命令
kubectl exec -it <pod-name> -n <namespace> -- ss -tlnp

# 检查应用进程是否运行
kubectl exec -it <pod-name> -n <namespace> -- ps aux | grep <app-name>
```

**常见问题**：
- 应用未启动或启动失败
- 应用监听在 127.0.0.1 而非 0.0.0.0
- 应用监听端口与配置不一致

**解决方案**：
- 检查应用日志：`kubectl logs <pod-name> -n <namespace>`
- 确保应用监听在 `0.0.0.0:PORT` 而非 `127.0.0.1:PORT`
- 验证配置文件中的端口设置

### 第二层：容器网络检查

**检查容器网络接口**

```bash
# 查看 Pod 的网络接口
kubectl exec -it <pod-name> -n <namespace> -- ip addr

# 检查路由表
kubectl exec -it <pod-name> -n <namespace> -- ip route

# 测试 Pod 内部回环
kubectl exec -it <pod-name> -n <namespace> -- curl -v http://127.0.0.1:<port>
```

### 第三层：跨 Pod 网络连通性

**从其他 Pod 测试连接**

```bash
# 启动一个临时测试 Pod
kubectl run test-pod --rm -it --image=busybox -- sh

# 在测试 Pod 中尝试连接目标 Pod
wget -O- http://<pod-ip>:<port>
# 或使用 curl（如果镜像中有）
curl -v http://<pod-ip>:<port>

# 测试网络连通性
ping <pod-ip>
```

**检查同节点 vs 跨节点通信**

```bash
# 查看目标 Pod 所在节点
kubectl get pod <pod-name> -n <namespace> -o wide

# 在同节点的其他 Pod 测试
# 在不同节点的 Pod 测试
# 对比结果判断是否为跨节点网络问题
```

### 第四层：节点网络检查

**在目标节点上直接测试**

```bash
# SSH 到 Pod 所在节点
ssh <node-ip>

# 测试能否访问本节点上的 Pod IP
curl -v http://<pod-ip>:<port>

# 检查节点的网络接口
ip addr

# 检查节点路由表
ip route

# 检查 iptables 规则
iptables -L -n -v
iptables -t nat -L -n -v
```

**检查 CNI 插件状态**

```bash
# 查看节点上的 CNI 配置
ls -la /etc/cni/net.d/

# 检查 CNI 插件二进制文件
ls -la /opt/cni/bin/

# 查看 CNI 相关日志（以 Flannel 为例）
journalctl -u kubelet | grep -i cni
journalctl -u flanneld

# Calico 检查
calicoctl node status
```

### 第五层：网络策略检查

**检查 NetworkPolicy 是否阻止流量**

```bash
# 查看命名空间中的所有 NetworkPolicy
kubectl get networkpolicy -n <namespace>

# 查看具体策略详情
kubectl describe networkpolicy <policy-name> -n <namespace>

# 检查是否有默认拒绝策略
kubectl get networkpolicy -n <namespace> -o yaml | grep -A 5 "ingress:\|egress:"
```

**NetworkPolicy 排查要点**：

| 策略类型 | 影响范围 | 排查方法 |
|---------|---------|---------|
| Ingress 策略 | 控制进入 Pod 的流量 | 检查是否允许源 Pod 的访问 |
| Egress 策略 | 控制从 Pod 出去的流量 | 检查是否允许访问目标 Pod |
| 默认拒绝策略 | 拒绝所有未明确允许的流量 | 确认是否有 `policyTypes: Ingress` 但无规则 |

## Service ClusterIP 访问超时排查

Service ClusterIP 访问超时的排查比 Pod IP 更复杂，因为涉及 kube-proxy、iptables/IPVS 等组件。

### 排查思路

```
Service 配置 → Endpoints → kube-proxy → iptables/IPVS → 后端 Pod
```

### 第一层：Service 配置检查

**验证 Service 基本信息**

```bash
# 查看 Service 详情
kubectl get svc <service-name> -n <namespace> -o wide

# 检查 ClusterIP 和端口
kubectl describe svc <service-name> -n <namespace>

# 确认 Service 类型
kubectl get svc <service-name> -n <namespace> -o jsonpath='{.spec.type}'
```

**常见配置问题**：

| 问题现象 | 可能原因 | 解决方案 |
|---------|---------|---------|
| ClusterIP 为 None | Headless Service | 确认是否需要 ClusterIP |
| 端口映射错误 | targetPort 配置错误 | 检查 targetPort 是否匹配 Pod 容器端口 |
| 协议不匹配 | TCP/UDP 配置错误 | 确认应用协议类型 |

### 第二层：Endpoints 检查

**Endpoints 是 Service 与 Pod 的桥梁**

```bash
# 查看 Endpoints
kubectl get endpoints <service-name> -n <namespace>

# 查看 Endpoints 详细信息
kubectl describe endpoints <service-name> -n <namespace>

# 检查 Endpoints 是否有后端 Pod
kubectl get endpoints <service-name> -n <namespace> -o yaml
```

**Endpoints 为空的常见原因**：

1. **标签选择器不匹配**

```bash
# 查看 Service 的 selector
kubectl get svc <service-name> -n <namespace> -o jsonpath='{.spec.selector}'

# 查看 Pod 的标签
kubectl get pods -n <namespace> --show-labels

# 对比标签是否匹配
kubectl get pods -n <namespace> -l <label-selector>
```

2. **Pod 未就绪**

```bash
# 检查 Pod 的 Ready 状态
kubectl get pods -n <namespace>

# 查看Pod的就绪状态详情
kubectl describe pod <pod-name> -n <namespace> | grep -A 5 "Conditions:"

# 检查 readinessProbe 配置
kubectl get pod <pod-name> -n <namespace> -o jsonpath='{.spec.containers[*].readinessProbe}'
```

3. **Pod 端口不匹配**

```bash
# 检查 Pod 容器端口
kubectl get pod <pod-name> -n <namespace> -o jsonpath='{.spec.containers[*].ports}'

# 与 Service 的 targetPort 对比
kubectl get svc <service-name> -n <namespace> -o jsonpath='{.spec.ports[*].targetPort}'
```

### 第三层：kube-proxy 检查

kube-proxy 负责维护 Service 的负载均衡规则，是 Service 网络的核心组件。

**检查 kube-proxy 状态**

```bash
# 查看 kube-proxy Pod 状态
kubectl get pods -n kube-system -l k8s-app=kube-proxy

# 查看 kube-proxy 日志
kubectl logs -n kube-system <kube-proxy-pod-name>

# 检查 kube-proxy 配置
kubectl get configmap -n kube-system kube-proxy -o yaml
```

**kube-proxy 代理模式**

kube-proxy 支持三种代理模式，不同模式的排查方法不同：

| 代理模式 | 工作原理 | 排查重点 |
|---------|---------|---------|
| userspace | 用户空间代理 | kube-proxy 进程、性能瓶颈 |
| iptables | iptables 规则转发 | iptables 规则、规则数量 |
| IPVS | IPVS 负载均衡 | IPVS 规则、IPVS 内核模块 |

**查看当前代理模式**

```bash
# 查看 kube-proxy 配置中的模式
kubectl get configmap -n kube-system kube-proxy -o yaml | grep mode

# 或查看 kube-proxy 启动参数
kubectl get pods -n kube-system <kube-proxy-pod-name> -o yaml | grep -A 5 "args:"
```

### 第四层：iptables 模式排查

**查看 iptables 规则**

```bash
# 在节点上查看 Service 相关的 iptables 规则
iptables -t nat -L KUBE-SERVICES -n -v

# 查看特定 Service 的规则
iptables -t nat -L KUBE-SVC-<service-id> -n -v

# 查看完整的 NAT 链
iptables -t nat -L -n -v | grep <service-cluster-ip>

# 统计 iptables 规则数量
iptables -t nat -L -n | wc -l
```

**iptables 规则链结构**：

```
KUBE-SERVICES (入口)
  ├── KUBE-SVC-XXXX (Service 链)
  │   ├── KUBE-SEP-AAAA (Pod A 端点)
  │   ├── KUBE-SEP-BBBB (Pod B 端点)
  │   └── KUBE-SEP-CCCC (Pod C 端点)
  └── ...
```

**常见 iptables 问题**：

1. **规则缺失**：kube-proxy 未正确生成规则
2. **规则冲突**：自定义 iptables 规则干扰
3. **规则过多**：性能下降，连接超时

### 第五层：IPVS 模式排查

**查看 IPVS 规则**

```bash
# 在节点上安装 ipvsadm（如果没有）
apt-get install ipvsadm -y  # Debian/Ubuntu
yum install ipvsadm -y      # CentOS/RHEL

# 查看所有 IPVS 规则
ipvsadm -Ln

# 查看特定 Service
ipvsadm -Ln | grep <cluster-ip>

# 查看详细统计信息
ipvsadm -Ln --stats

# 查看连接信息
ipvsadm -Ln -c
```

**检查 IPVS 内核模块**

```bash
# 检查 IPVS 相关内核模块是否加载
lsmod | grep ip_vs

# 手动加载 IPVS 模块
modprobe ip_vs
modprobe ip_vs_rr
modprobe ip_vs_wrr
modprobe ip_vs_sh

# 检查内核参数
cat /proc/sys/net/ipv4/vs/conn_reuse_mode
cat /proc/sys/net/ipv4/vs/expire_nodest_conn
```

### 第六层：conntrack 连接跟踪

**conntrack 是连接跟踪的核心**

```bash
# 查看 conntrack 连接表
conntrack -L

# 查看特定 Service 的连接
conntrack -L | grep <cluster-ip>

# 统计连接数
conntrack -C

# 查看 conntrack 表大小限制
cat /proc/sys/net/netfilter/nf_conntrack_max

# 查看当前连接数
cat /proc/sys/net/netfilter/nf_conntrack_count

# 清空 conntrack 表（谨慎操作）
conntrack -F
```

**conntrack 相关问题**：

| 问题 | 现象 | 解决方案 |
|-----|------|---------|
| conntrack 表满 | 新连接超时 | 增加 nf_conntrack_max |
| 连接跟踪失效 | 请求无法到达后端 | 检查 conntrack 规则 |
| 连接复用问题 | 连接异常中断 | 调整 conntrack 参数 |

## 常见原因深度分析

### 1. 网络策略（NetworkPolicy）阻断

**原理分析**：

NetworkPolicy 基于 Label 选择器定义 Pod 的网络访问规则，通过 CNI 插件实现（如 Calico、Cilium）。策略生效后，CNI 插件会在节点上配置相应的 iptables 或 eBPF 规则。

**排查步骤**：

```bash
# 1. 列出所有 NetworkPolicy
kubectl get networkpolicy --all-namespaces

# 2. 检查策略详情
kubectl describe networkpolicy <policy-name> -n <namespace>

# 3. 验证 Pod 标签匹配
kubectl get pods -n <namespace> --show-labels

# 4. 临时删除策略测试
kubectl delete networkpolicy <policy-name> -n <namespace>
```

**典型案例**：

```yaml
# 默认拒绝所有入站流量
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: production
spec:
  podSelector: {}
  policyTypes:
  - Ingress
```

此策略会导致所有访问 production 命名空间中 Pod 的请求被拒绝。

### 2. kube-proxy 故障

**原理分析**：

kube-proxy 监听 API Server 中 Service 和 Endpoints 的变化，并在每个节点上维护负载均衡规则。如果 kube-proxy 故障，新的 Service 规则无法生成，导致访问超时。

**排查步骤**：

```bash
# 1. 检查 kube-proxy Pod 状态
kubectl get pods -n kube-system -l k8s-app=kube-proxy -o wide

# 2. 查看 kube-proxy 日志
kubectl logs -n kube-system <kube-proxy-pod-name> --tail=100

# 3. 检查 kube-proxy 与 API Server 的连接
kubectl exec -n kube-system <kube-proxy-pod-name> -- curl -k https://kubernetes.default.svc/api

# 4. 重启 kube-proxy
kubectl delete pod -n kube-system <kube-proxy-pod-name>
```

**常见错误日志**：

```
E0312 10:00:00.000000  12345 reflector.go:125] Failed to list *v1.Service: Get https://10.96.0.1:443/api/v1/services: dial tcp 10.96.0.1:443: connect: connection refused
```

### 3. iptables/IPVS 规则异常

**原理分析**：

在 iptables 模式下，kube-proxy 为每个 Service 创建一系列 iptables 规则，实现负载均衡和会话保持。规则异常会导致流量无法正确转发。

**iptables 规则生成流程**：

```
Service 创建 → kube-proxy 监听 → 生成 iptables 规则 → 规则下发到节点
```

**排查步骤**：

```bash
# 1. 检查 iptables 规则是否完整
iptables -t nat -S | grep KUBE

# 2. 查看规则统计（确认规则是否被匹配）
iptables -t nat -L KUBE-SERVICES -n -v

# 3. 追踪数据包路径
iptables -t nat -L KUBE-SVC-XXXX -n -v

# 4. 手动添加规则测试（仅用于排查）
iptables -t nat -A PREROUTING -d <cluster-ip> -p tcp --dport <port> -j DNAT --to-destination <pod-ip>:<port>
```

### 4. CNI 插件问题

**原理分析**：

CNI 插件负责 Pod 网络的配置和管理，包括 IP 地址分配、路由设置、网络隔离等。CNI 故障会导致 Pod 无法正常通信。

**常见 CNI 插件及排查**：

| CNI 插件 | 排查命令 | 常见问题 |
|---------|---------|---------|
| Flannel | `journalctl -u flanneld` | etcd 连接失败、网络配置错误 |
| Calico | `calicoctl node status` | BGP 会话异常、IPPool 耗尽 |
| Cilium | `cilium status` | eBPF 程序加载失败 |
| Weave | `weave status` | 快速发现失败、加密问题 |

**Flannel 排查示例**：

```bash
# 查看 Flannel Pod 状态
kubectl get pods -n kube-system -l app=flannel

# 查看 Flannel 日志
kubectl logs -n kube-system <flannel-pod-name>

# 检查 Flannel 网络配置
kubectl get configmap -n kube-system kube-flannel-cfg -o yaml

# 在节点上检查 Flannel 接口
ip addr show flannel.1
ip route | grep flannel
```

### 5. DNS 解析问题

**原理分析**：

Service 访问通常通过 DNS 名称（如 `service-name.namespace.svc.cluster.local`），如果 DNS 解析失败或解析到错误 IP，会导致连接超时。

**排查步骤**：

```bash
# 1. 测试 DNS 解析
kubectl exec -it <pod-name> -- nslookup kubernetes.default

# 2. 使用 dig 命令
kubectl exec -it <pod-name> -- dig kubernetes.default.svc.cluster.local

# 3. 检查 CoreDNS 状态
kubectl get pods -n kube-system -l k8s-app=kube-dns

# 4. 查看 CoreDNS 日志
kubectl logs -n kube-system <coredns-pod-name>

# 5. 检查 DNS 配置
kubectl exec -it <pod-name> -- cat /etc/resolv.conf

# 6. 直接使用 ClusterIP 测试（绕过 DNS）
kubectl exec -it <pod-name> -- curl http://<cluster-ip>:<port>
```

### 6. 节点网络配置问题

**原理分析**：

节点的网络配置（如路由、防火墙、内核参数）直接影响 Pod 和 Service 的网络通信。

**关键检查项**：

```bash
# 1. 检查节点路由表
ip route show

# 2. 检查防火墙状态
systemctl status firewalld  # CentOS/RHEL
ufw status                  # Ubuntu

# 3. 检查关键内核参数
sysctl net.ipv4.ip_forward
sysctl net.bridge.bridge-nf-call-iptables

# 4. 检查网络接口 MTU
ip link show

# 5. 检查 SELinux 状态（如果启用）
getenforce
```

**关键内核参数说明**：

| 参数 | 作用 | 推荐值 |
|-----|------|--------|
| net.ipv4.ip_forward | IP 转发 | 1（启用） |
| net.bridge.bridge-nf-call-iptables | 桥接流量经过 iptables | 1（启用） |
| net.bridge.bridge-nf-call-ip6tables | 桥接流量经过 ip6tables | 1（启用） |

## 系统化排查方法

### 排查流程图

```
┌─────────────────────────────────────────────────────────────────┐
│                    访问 Pod/Service IP 超时                      │
└───────────────────────────┬─────────────────────────────────────┘
                            │
                            ▼
                ┌───────────────────────┐
                │  1. 确认问题范围       │
                │  - 单个 Pod？         │
                │  - 单个 Service？     │
                │  - 整个命名空间？     │
                │  - 整个集群？         │
                └───────────┬───────────┘
                            │
                            ▼
                ┌───────────────────────┐
                │  2. 检查应用层        │
                │  - Pod 状态           │
                │  - 应用日志           │
                │  - 端口监听           │
                └───────────┬───────────┘
                            │
                            ▼
                ┌───────────────────────┐
                │  3. 检查网络连通性    │
                │  - Ping 测试          │
                │  - 同节点测试         │
                │  - 跨节点测试         │
                └───────────┬───────────┘
                            │
                            ▼
                ┌───────────────────────┐
                │  4. 检查 Service 层   │
                │  - Service 配置       │
                │  - Endpoints          │
                │  - kube-proxy         │
                └───────────┬───────────┘
                            │
                            ▼
                ┌───────────────────────┐
                │  5. 检查网络策略      │
                │  - NetworkPolicy      │
                │  - iptables 规则      │
                │  - IPVS 规则          │
                └───────────┬───────────┘
                            │
                            ▼
                ┌───────────────────────┐
                │  6. 检查 CNI 插件     │
                │  - CNI 状态           │
                │  - CNI 配置           │
                │  - CNI 日志           │
                └───────────┬───────────┘
                            │
                            ▼
                ┌───────────────────────┐
                │  7. 检查节点网络      │
```

## 常见问题与最佳实践

### 常见问题 FAQ

#### Q1: 为什么 Pod IP 可以 ping 通，但访问端口超时？

**A**: 这通常是因为应用层问题：
1. 应用未监听正确的端口或监听在 127.0.0.1
2. 应用启动失败或崩溃
3. 防火墙或 NetworkPolicy 阻止了特定端口

**排查方法**：
```bash
# 检查端口监听
kubectl exec -it <pod-name> -- netstat -tlnp

# 检查应用日志
kubectl logs <pod-name>
```

#### Q2: Service ClusterIP 可以访问，但通过 Service 名称访问超时？

**A**: 这是 DNS 解析问题：
1. CoreDNS 服务异常
2. Pod 的 DNS 配置错误
3. DNS 解析超时

**排查方法**：
```bash
# 测试 DNS 解析
kubectl exec -it <pod-name> -- nslookup <service-name>

# 检查 CoreDNS
kubectl get pods -n kube-system -l k8s-app=kube-dns
```

#### Q3: 同节点的 Pod 可以通信，跨节点 Pod 无法通信？

**A**: 这是跨节点网络问题：
1. CNI 插件配置错误（如 Flannel 的 VXLAN 配置）
2. 节点间网络不通
3. 节点防火墙阻止了 CNI 流量

**排查方法**：
```bash
# 检查节点间连通性
ping <node-ip>

# 检查 CNI 接口
ip addr show flannel.1  # Flannel
ip route | grep flannel

# 检查防火墙
iptables -L -n
```

#### Q4: Endpoints 为空，但 Pod 正常运行？

**A**: 这是 Service 与 Pod 关联问题：
1. Service 的 selector 与 Pod 标签不匹配
2. Pod 的 readinessProbe 失败
3. Pod 端口与 Service targetPort 不匹配

**排查方法**：
```bash
# 检查标签匹配
kubectl get pods --show-labels
kubectl get svc <service-name> -o jsonpath='{.spec.selector}'

# 检查 Pod 就绪状态
kubectl describe pod <pod-name> | grep -A 5 "Conditions:"
```

#### Q5: 访问 Service 偶尔超时，偶尔正常？

**A**: 这可能是负载均衡或后端 Pod 问题：
1. 部分后端 Pod 异常
2. kube-proxy 规则更新延迟
3. conntrack 表接近满载

**排查方法**：
```bash
# 检查所有后端 Pod 状态
kubectl get endpoints <service-name> -o yaml

# 检查 conntrack 使用率
cat /proc/sys/net/netfilter/nf_conntrack_count
cat /proc/sys/net/netfilter/nf_conntrack_max

# 逐个测试后端 Pod
curl http://<pod-ip>:<port>
```

### 最佳实践

#### 1. 网络策略管理

```yaml
# 推荐做法：明确允许需要的流量
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: backend
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
```

**建议**：
- 避免使用过于宽松的策略（如允许所有命名空间）
- 为每个命名空间设置默认策略
- 定期审计 NetworkPolicy

#### 2. Service 配置优化

```yaml
apiVersion: v1
kind: Service
metadata:
  name: backend
spec:
  selector:
    app: backend
  ports:
  - port: 80
    targetPort: 8080
    protocol: TCP
  # 会话保持（可选）
  sessionAffinity: ClientIP
  sessionAffinityConfig:
    clientIP:
      timeoutSeconds: 10800
```

**建议**：
- 明确指定 targetPort，避免使用名称
- 合理配置 readinessProbe，确保 Endpoints 准确
- 根据业务需求配置会话保持

#### 3. kube-proxy 配置优化

```yaml

  name: kube-proxy

      tcpEstablishedTimeout: 24h0m0s
```

**建议**：

- 合理配置 conntrack 参数

## 面试回答

**面试官问：访问 Pod 的 IP:端口或 Service 的 IP 显示超时，你会如何处理？**

**回答**：

访问超时问题需要系统化排查。首先，我会确认问题范围：是单个 Pod/Service 还是整个集群的问题。对于 Pod IP 超时，我会按照"应用层→网络层→节点层"的顺序排查：先检查 Pod 状态和应用是否正常监听端口，确保应用监听在 0.0.0.0 而非 127.0.0.1；然后通过测试 Pod 验证网络连通性，区分同节点和跨节点问题；最后检查节点的网络配置、CNI 插件状态和 NetworkPolicy 是否阻断流量。对于 Service ClusterIP 超时，排查重点在 Service 层：先检查 Service 配置和 Endpoints 是否正常，确认标签选择器匹配且 Pod 处于 Ready 状态；然后检查 kube-proxy 的运行状态和日志，查看 iptables 或 IPVS 规则是否正确生成；同时排查 conntrack 连接跟踪表是否满载。常见原因包括：应用监听地址错误、NetworkPolicy 阻断、kube-proxy 故障、CNI 插件异常、iptables/IPVS 规则缺失、DNS 解析失败等。我会使用分层排查策略，从应用层到网络层逐步定位，结合 kubectl、iptables、ipvsadm、conntrack 等工具，快速找到问题根源并解决。关键是要建立完整的排查思路，而不是盲目尝试。
