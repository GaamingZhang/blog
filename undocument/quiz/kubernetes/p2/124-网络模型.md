# Kubernetes 网络模型

## 引言

Kubernetes 网络是容器编排中最复杂的部分之一。Kubernetes 定义了一套网络模型规范，所有 Pod 都需要满足扁平网络的通信需求。理解 Kubernetes 网络模型，对于设计和排查容器网络问题至关重要。

## Kubernetes 网络模型概述

### 网络模型原则

```
┌─────────────────────────────────────────────────────────────┐
│              Kubernetes 网络模型原则                         │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 所有 Pod 不使用 NAT 就可以相互通信                       │
│     ┌─────────────────────────────────────────────────┐    │
│     │  Pod A (10.244.1.2) ─────────────────▶ Pod B   │    │
│     │                            (10.244.2.3)         │    │
│     │  直接通信，无需 NAT                            │    │
│     └─────────────────────────────────────────────────┘    │
│                                                              │
│  2. 所有 Node 都可以与所有 Pod 通信                          │
│     ┌─────────────────────────────────────────────────┐    │
│     │  Node (192.168.1.10) ───────────────▶ Pod      │    │
│     │                            (10.244.1.2)         │    │
│     └─────────────────────────────────────────────────┘    │
│                                                              │
│  3. Pod 看到的自身 IP 与其他 Pod 看到的 IP 相同              │
│     ┌─────────────────────────────────────────────────┐    │
│     │  Pod IP 在所有视角下一致                        │    │
│     │  无需进行地址转换                              │    │
│     └─────────────────────────────────────────────────┘    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 网络通信类型

| 通信类型 | 说明 |
|---------|------|
| Pod-to-Pod（同节点） | 同一节点上的 Pod 通过 veth pair 通信 |
| Pod-to-Pod（跨节点） | 跨节点的 Pod 通过网络插件通信 |
| Pod-to-Service | 通过 Service 实现服务发现和负载均衡 |
| External-to-Service | 外部流量通过 Service 访问 Pod |

## CNI 插件

### CNI 概述

CNI（Container Network Interface）是 Kubernetes 网络插件的标准接口，定义了容器运行时和网络插件之间的交互规范。

### 主流 CNI 插件

| 插件 | 类型 | 特点 |
|-----|------|------|
| Flannel | Overlay | 简单易用，支持多种后端 |
| Calico | 路由 | 高性能，支持网络策略 |
| Cilium | eBPF | 高性能，安全可观测 |
| Weave | Overlay | 简单，支持多主机 |
| Canal | 混合 | Flannel + Calico |

### Flannel 配置

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: kube-flannel
  namespace: kube-system
spec:
  containers:
  - name: kube-flannel
    image: flannelcni/flannel:v0.22.0
    command:
    - /opt/bin/flanneld
    args:
    - --ip-masq
    - --kube-subnet-mgr
    - --iface=eth0
    env:
    - name: POD_NAME
      valueFrom:
        fieldRef:
          fieldPath: metadata.name
    - name: POD_NAMESPACE
      valueFrom:
        fieldRef:
          fieldPath: metadata.namespace
    volumeMounts:
    - name: run
      mountPath: /run/flannel
  volumes:
  - name: run
    hostPath:
      path: /run/flannel
```

### Calico 配置

```yaml
apiVersion: operator.tigera.io/v1
kind: Installation
metadata:
  name: default
spec:
  calicoNetwork:
    ipPools:
    - blockSize: 26
      cidr: 10.244.0.0/16
      encapsulation: VXLANCrossSubnet
      natOutgoing: Enabled
      nodeSelector: all()
```

## Service 网络

### Service 类型

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  type: ClusterIP
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: nodeport-service
spec:
  type: NodePort
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 8080
    nodePort: 30080
---
apiVersion: v1
kind: Service
metadata:
  name: loadbalancer-service
spec:
  type: LoadBalancer
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 8080
```

## 网络策略

### NetworkPolicy 示例

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-network-policy
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
  egress:
  - to:
    - podSelector:
        matchLabels:
          app: database
    ports:
    - protocol: TCP
      port: 3306
```

## 面试回答

**问题**: 什么是 k8s 的网络模型？

**回答**: Kubernetes 网络模型定义了三个核心原则：所有 Pod 不使用 NAT 就可以相互通信；所有 Node 都可以与所有 Pod 通信；Pod 看到的自身 IP 与其他 Pod 看到的 IP 相同。

**网络通信类型**：**同节点 Pod 通信**通过 veth pair 和网桥实现；**跨节点 Pod 通信**通过 CNI 插件实现，支持 Overlay 网络（VXLAN、IPIP）或路由模式；**Pod-to-Service** 通过 kube-proxy 实现，支持 iptables 和 IPVS 两种模式；**External-to-Service** 通过 NodePort 或 LoadBalancer 类型 Service 实现。

**CNI 插件**：**Flannel** 简单易用，支持 VXLAN、host-gw 等后端；**Calico** 高性能，支持 BGP 路由和网络策略；**Cilium** 基于 eBPF，提供高性能和安全可观测性。

**Service 网络**：kube-proxy 负责将 Service VIP 转发到后端 Pod。iptables 模式使用 iptables 规则，默认模式；IPVS 模式使用 IPVS 负载均衡，性能更好，适合大规模集群。

**DNS 解析**：CoreDNS 提供 Service 域名解析，格式为 `<service-name>.<namespace>.svc.cluster.local`。

**网络策略**：NetworkPolicy 实现 Pod 级别的网络隔离，控制入站和出站流量。

**最佳实践**：根据场景选择合适的 CNI 插件；大规模集群使用 IPVS 模式；配置网络策略实现安全隔离。
