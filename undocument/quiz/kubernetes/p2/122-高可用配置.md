# Kubernetes 集群高可用性配置

## 引言

高可用性（High Availability，HA）是生产环境 Kubernetes 集群的基本要求。通过合理的架构设计和配置，可以确保集群在单点故障时仍能正常运行，保障业务的连续性。本文详细介绍 Kubernetes 集群高可用性的配置方法。

### etcd 集群配置

```yaml
apiVersion: kubeadm.k8s.io/v1beta3
kind: ClusterConfiguration
etcd:
  local:
    dataDir: /var/lib/etcd
    serverCertSANs:
    - "10.0.0.100"
    - "etcd-1"
    - "etcd-2"
    - "etcd-3"
    peerCertSANs:
    - "10.0.0.101"
    - "10.0.0.102"
    - "10.0.0.103"
    extraArgs:
      initial-cluster: etcd-1=https://10.0.0.101:2380,etcd-2=https://10.0.0.102:2380,etcd-3=https://10.0.0.103:2380
      initial-cluster-state: new
      initial-cluster-token: etcd-cluster
```

### etcd 健康检查

```bash
ETCDCTL_API=3 etcdctl \
  --endpoints=https://10.0.0.101:2379,https://10.0.0.102:2379,https://10.0.0.103:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/healthcheck-client.crt \
  --key=/etc/kubernetes/pki/etcd/healthcheck-client.key \
  endpoint health

ETCDCTL_API=3 etcdctl endpoint status --cluster -w table
```

## 控制平面高可用

### kube-apiserver 负载均衡

```yaml
apiVersion: kubeadm.k8s.io/v1beta3
kind: ClusterConfiguration
controlPlaneEndpoint: "10.0.0.100:6443"
apiServer:
  certSANs:
  - "10.0.0.100"
  - "master-1"
  - "master-2"
  - "master-3"
  - "k8s-api.example.com"
```

### HAProxy 配置

```
global
    log /dev/log local0
    chroot /var/lib/haproxy
    stats socket /run/haproxy/admin.sock mode 660
    stats timeout 30s

defaults
    log     global
    mode    tcp
    option  tcplog
    timeout connect 5000
    timeout client  50000
    timeout server  50000

frontend k8s-api
    bind *:6443
    mode tcp
    option tcplog
    default_backend k8s-api

backend k8s-api
    mode tcp
    option tcp-check
    balance roundrobin
    server master-1 10.0.0.101:6443 check
    server master-2 10.0.0.102:6443 check
    server master-3 10.0.0.103:6443 check
```

### Keepalived 配置

```
vrrp_script check_apiserver {
  script "curl -k https://localhost:6443/healthz"
  interval 3
  weight -2
  fall 10
  rise 2
}

vrrp_instance VI_1 {
    state MASTER
    interface eth0
    virtual_router_id 51
    priority 100
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass 1234
    }
    virtual_ipaddress {
        10.0.0.100
    }
    track_script {
        check_apiserver
    }
}
```

## 工作节点高可用

### 多工作节点部署

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ha-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: ha-app
  template:
    metadata:
      labels:
        app: ha-app
    spec:
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 100
            podAffinityTerm:
              labelSelector:
                matchLabels:
                  app: ha-app
              topologyKey: kubernetes.io/hostname
      containers:
      - name: app
        image: my-app:v1
        ports:
        - containerPort: 8080
```

### 节点故障转移

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ha-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: ha-app
  template:
    spec:
      containers:
      - name: app
        image: my-app:v1
      tolerations:
      - key: "node.kubernetes.io/not-ready"
        operator: "Exists"
        effect: "NoExecute"
        tolerationSeconds: 300
      - key: "node.kubernetes.io/unreachable"
        operator: "Exists"
        effect: "NoExecute"
        tolerationSeconds: 300
```

## 面试回答

**问题**: 如何进行 Kubernetes 集群的高可用性配置？

**回答**: Kubernetes 集群高可用配置需要从多个层面进行：

**etcd 高可用**：部署 3、5 或 7 个节点的 etcd 集群（奇数节点），使用 Raft 协议保证数据一致性。容错能力为 (N-1)/2 个节点故障。配置 initial-cluster 参数指定集群节点，定期备份 etcd 数据。

**控制平面高可用**：部署多个 Master 节点（通常 3 个），每个节点运行 kube-apiserver、kube-scheduler、kube-controller-manager。kube-apiserver 前端配置负载均衡（HAProxy/Nginx），使用 Keepalived 提供 VIP。kube-scheduler 和 kube-controller-manager 通过 Leader Election 机制实现高可用，同一时间只有一个实例工作。

**负载均衡配置**：使用 HAProxy 或 Nginx 对 kube-apiserver 进行负载均衡，使用 Keepalived 实现 VIP 漂移，确保负载均衡器本身的高可用。

**工作节点高可用**：部署多个工作节点，使用 Pod 反亲和性将副本分散到不同节点，配置 tolerationSeconds 控制节点故障时的 Pod 迁移时间。

**验证测试**：通过 kubectl get nodes、kubectl get cs 检查集群状态；模拟节点故障测试故障转移；验证服务在故障时仍可访问。

**最佳实践**：使用奇数节点；跨可用区部署实现异地多活；定期备份 etcd 数据；配置监控告警；制定故障恢复预案。
