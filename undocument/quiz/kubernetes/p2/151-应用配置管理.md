# Kubernetes 应用程序配置管理

## 引言

应用程序配置管理是 Kubernetes 运维的重要环节。通过 ConfigMap、环境变量、命令行参数等方式，可以将配置与代码分离，实现配置的灵活管理。本文介绍 Kubernetes 中应用程序配置管理的各种方式和最佳实践。

## ConfigMap 配置

### 创建 ConfigMap

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: production
data:
  APP_ENV: "production"
  LOG_LEVEL: "info"
  app.properties: |
    server.port=8080
    database.host=mysql-service
    database.port=3306
```

### 环境变量方式

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-env
spec:
  containers:
  - name: app
    image: myapp:v1
    env:
    - name: APP_ENV
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: APP_ENV
    - name: LOG_LEVEL
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: LOG_LEVEL
```

### 批量导入环境变量

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-envfrom
spec:
  containers:
  - name: app
    image: myapp:v1
    envFrom:
    - configMapRef:
        name: app-config
    - secretRef:
        name: app-secret
```

### 挂载配置文件

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-config-file
spec:
  containers:
  - name: app
    image: myapp:v1
    volumeMounts:
    - name: config
      mountPath: /etc/app
      readOnly: true
  volumes:
  - name: config
    configMap:
      name: app-config
```

### 挂载单个文件

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-single-file
spec:
  containers:
  - name: app
    image: myapp:v1
    volumeMounts:
    - name: config
      mountPath: /etc/app/app.properties
      subPath: app.properties
      readOnly: true
  volumes:
  - name: config
    configMap:
      name: app-config
```

## 命令行参数

### 使用环境变量作为参数

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-args
spec:
  containers:
  - name: app
    image: myapp:v1
    command: ["/app/server"]
    args:
    - "--port=$(APP_PORT)"
    - "--log-level=$(LOG_LEVEL)"
    env:
    - name: APP_PORT
      value: "8080"
    - name: LOG_LEVEL
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: LOG_LEVEL
```

### 直接传递参数

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-direct-args
spec:
  containers:
  - name: app
    image: myapp:v1
    command: ["/app/server"]
    args:
    - "--port=8080"
    - "--log-level=info"
```

## 配置热更新

### ConfigMap 更新机制

```
┌─────────────────────────────────────────────────────────────┐
│              ConfigMap 更新机制                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  更新 ConfigMap：                                            │
│  kubectl edit configmap app-config                          │
│       │                                                      │
│       ▼                                                      │
│  kubelet 检测到变更（约 1 分钟）                             │
│       │                                                      │
│       ▼                                                      │
│  更新 Pod 挂载的文件                                         │
│       │                                                      │
│       ▼                                                      │
│  应用检测文件变更并重新加载                                  │
│                                                              │
│  注意：                                                      │
│  • 环境变量不会自动更新                                      │
│  • subPath 挂载不会自动更新                                  │
│  • 需要应用支持热加载                                        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 使用 Reloader 自动重启

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  annotations:
    reloader.stakater.com/auto: "true"
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: app
        image: myapp:v1
        volumeMounts:
        - name: config
          mountPath: /etc/app
      volumes:
      - name: config
        configMap:
          name: app-config
```

## 配置管理最佳实践

### 1. 分离配置和代码

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  database.url: "mysql://mysql-service:3306/mydb"
```

### 2. 使用版本控制

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config-v1
  labels:
    version: "1"
```

### 3. 配置验证

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-validation
spec:
  containers:
  - name: app
    image: myapp:v1
    env:
    - name: APP_ENV
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: APP_ENV
          optional: false
```

### 4. 使用不可变 ConfigMap

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
immutable: true
data:
  APP_ENV: "production"
```

## 面试回答

**问题**: 如何在 Kubernetes 中实现应用程序的配置管理？

**回答**: Kubernetes 应用程序配置管理有多种方式：

**ConfigMap**：存储非敏感配置数据。可以通过环境变量或挂载文件方式传递给 Pod。支持键值对和文件格式配置。

**环境变量方式**：使用 env 从 ConfigMap 引用单个键值；使用 envFrom 批量导入所有键值。优点是简单直接；缺点是不会自动更新，需要重启 Pod。

**挂载文件方式**：将 ConfigMap 挂载为文件到 Pod 内。支持复杂配置结构（如 YAML、JSON、Properties 文件）。更新 ConfigMap 后文件会自动更新（约 1 分钟延迟），但需要应用支持热加载。

**命令行参数**：通过 command 和 args 传递启动参数。可以结合环境变量使用，如 `args: ["--port=$(PORT)"]`。

**配置热更新**：环境变量方式不会自动更新；volumeMounts 方式会自动更新文件（subPath 除外）；需要应用实现配置热加载；可以使用 Reloader 工具自动触发 Pod 重启。

**最佳实践**：分离配置和代码；使用版本控制管理配置；配置验证确保必需配置存在；使用不可变 ConfigMap 防止意外修改；敏感信息使用 Secret 存储。
