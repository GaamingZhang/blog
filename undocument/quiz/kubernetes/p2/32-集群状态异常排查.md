# 执行 kubectl get cs 查看集群状态不正常，显示 unhealthy 怎么排查?

## 引言

在日常 Kubernetes 集群运维中,我们经常需要检查集群的整体健康状态。`kubectl get cs` 是 Kubernetes 提供的一个非常实用的命令,用于查看集群核心组件的状态信息。CS 是 Component Status 的缩写,该命令会展示 etcd、controller-manager、scheduler 等核心组件的健康状况。

当你执行 `kubectl get cs` 命令时,如果发现某些组件显示为 `unhealthy` 状态,这通常意味着集群的核心组件出现了问题,需要立即进行排查和处理。本文将深入剖析 Component Status 的工作机制,并提供系统化的排查思路和解决方案。

## 一、kubectl get cs 命令详解

### 1.1 命令作用与输出格式

`kubectl get cs` 命令用于获取 Kubernetes 控制平面组件的状态信息。在较新版本的 Kubernetes 中,该命令已被弃用,推荐使用 `kubectl get componentstatuses`(简写为 `cs`)。

```bash
$ kubectl get cs
NAME                 STATUS    MESSAGE                         ERROR
scheduler            Healthy   ok
controller-manager   Healthy   ok
etcd-0               Healthy   {"health":"true"}
```

当组件状态正常时,STATUS 字段显示为 `Healthy`;当出现故障时,STATUS 字段显示为 `unhealthy`,并会在 ERROR 或 MESSAGE 字段中显示具体的错误信息。

### 1.2 涉及的组件

kubectl get cs 命令主要检查以下三类核心组件:

| 组件名称 | 作用描述 | 默认端口 |
|---------|---------|---------|
| scheduler | 负责 Pod 的调度,将 Pod 分配到合适的节点上 | 10251 |
| controller-manager | 负责各种控制器的运行,如 ReplicationController、Deployment 等 | 10252 |
| etcd | 分布式键值存储,保存集群的所有状态数据 | 2379 |

## 二、Component Status Controller 工作原理

### 2.1 健康检查机制

Kubernetes 的 Component Status Controller 定期对各个核心组件进行健康检查。检查机制基于 HTTP 协议的 GET 请求,每个组件都暴露了一个健康检查端点:

- **scheduler**: `http://localhost:10251/healthz`
- **controller-manager**: `http://localhost:10252/healthz`
- **etcd**: `http://localhost:2379/health`

当 Component Status Controller 向这些端点发送请求时,如果返回 HTTP 200 状态码且响应体为 "ok"(对于 scheduler 和 controller-manager)或 `{"health":"true"}`(对于 etcd),则认为该组件处于健康状态。

### 2.2 检查流程

```
┌─────────────────────────────────────────────────────────────┐
│                    kubectl get cs                           │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│              Component Status Controller                    │
│                  (kube-controller-manager                   │
│                   内置组件)                                   │
└─────────────────────────────────────────────────────────────┘
                           │
          ┌────────────────┼────────────────┐
          ▼                ▼                ▼
   ┌────────────┐   ┌────────────┐   ┌────────────┐
   │ scheduler  │   │  controller│   │   etcd     │
   │ :10251     │   │-manager   │   │  :2379     │
   │ /healthz   │   │ :10252    │   │  /health   │
   └────────────┘   │ /healthz   │   └────────────┘
                    └────────────┘
```

### 2.3 注意事项

需要特别注意的是,在 Kubernetes 1.19+ 版本中,`kubectl get cs` 命令已被标记为弃用。这是因为该命令存在一些局限性:

1. **不检查 API Server**: Component Status 检查并不包含 API Server 本身的状态
2. **本地健康检查**: 默认情况下只检查本地(localhost)组件,无法检查集群中其他节点的组件
3. **静态 Pod 无法检查**: 通过 kubelet 运行的静态 Pod 无法通过此方式检查

对于生产环境,建议结合使用 `kubectl get nodes`、`kubectl get pods -n kube-system` 等命令来综合判断集群状态。

## 三、etcd unhealthy 问题排查

### 3.1 健康检查机制

etcd 的健康检查通过向 `http://localhost:2379/health` 端点发送 GET 请求来完成。如果 etcd 正常运行且能够处理请求,该端点会返回 `{"health":"true"}` 的响应。

### 3.2 常见问题原因

etcd 出现 unhealthy 状态的原因主要包括:

| 问题类型 | 具体原因 | 影响程度 |
|---------|---------|---------|
| 存储问题 | 磁盘空间不足、磁盘 I/O 性能问题 | 高 |
| 网络问题 | etcd 节点间通信故障 | 高 |
| 配置问题 | 证书过期、配置错误 | 高 |
| 数据问题 | 数据损坏、快照加载失败 | 高 |
| 资源问题 | 内存不足、CPU 负载过高 | 中 |

### 3.3 排查步骤

**第一步:检查 etcd 服务状态**

```bash
# 查看 etcd 容器/进程状态
docker ps | grep etcd
# 或
systemctl status etcd

# 查看 etcd 日志
docker logs <etcd-container-id>
# 或
journalctl -u etcd -n 100
```

**第二步:检查 etcd 健康状态**

```bash
# 直接测试 etcd 健康检查端点
curl -s http://localhost:2379/health

# 检查 etcd 集群成员状态
ETCDCTL_API=3 etcdctl --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  endpoint health

# 查看集群状态
ETCDCTL_API=3 etcdctl --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  member list -w table
```

**第三步:检查磁盘和资源状态**

```bash
# 检查磁盘空间
df -h

# 检查磁盘 I/O 性能
iostat -x 1 5

# 检查内存使用情况
free -h

# 检查 CPU 负载
uptime
```

**第四步:检查证书状态**

```bash
# 检查 etcd 证书过期时间
openssl x509 -in /etc/kubernetes/pki/etcd/server.crt -noout -dates

# 检查证书链
openssl verify -CAfile /etc/kubernetes/pki/etcd/ca.crt \
  /etc/kubernetes/pki/etcd/server.crt
```

### 3.4 解决方案

**磁盘空间不足**

```bash
# 清理旧的 etcd 快照
rm -f /var/lib/etcd/snapshots/*

# 清理 Kubernetes 旧资源(谨慎操作)
kubectl delete pods --all-namespaces --field-selector=status.phase=Succeeded

# 扩容磁盘或清理数据
```

**证书过期**

```bash
# 重新生成 etcd 证书(需要重新初始化集群或使用 kubeadm 更新)
kubeadm init phase/etcds-only --skip-certificate-key-words

# 或者使用 kubeadm 更新证书
kubeadm certs renew all
```

**etcd 数据损坏**

```bash
# 如果是单节点 etcd,可以尝试恢复
# 1. 停止 etcd
# 2. 备份数据目录
# 3. 使用 etcdctl 进行数据修复
ETCDCTL_API=3 etcdctl --data-dir=/var/lib/etcd snapshot restore /path/to/snapshot
```

## 四、controller-manager unhealthy 问题排查

### 4.1 健康检查机制

kube-controller-manager 通过 `http://localhost:10252/healthz` 端点提供健康检查服务。该端点会检查控制器管理器的关键功能是否正常运行,包括:

- 云提供商控制器
- 路由控制器
- 服务控制器
- 副本控制器

### 4.2 常见问题原因

| 问题类型 | 具体原因 | 影响程度 |
|---------|---------|---------|
| 配置问题 | --kubeconfig 配置错误、--leader-elect 配置不当 | 高 |
| 证书问题 | API Server 通信证书过期或无效 | 高 |
| 资源问题 | 内存泄漏、CPU 不足导致超时 | 中 |
| 依赖问题 | 无法连接到 API Server | 高 |
| 并发问题 | 控制器并发配置不当 | 低 |

### 4.3 排查步骤

**第一步:检查 controller-manager 状态**

```bash
# 查看 controller-manager Pod 状态
kubectl get pods -n kube-system -l component=kube-controller-manager

# 查看 controller-manager 日志
kubectl logs -n kube-system kube-controller-manager-<pod-name>

# 查看详细日志
kubectl logs -n kube-system kube-controller-manager-<pod-name> --tail=200 -f
```

**第二步:检查健康检查端点**

```bash
# 本地健康检查
curl -s http://localhost:10252/healthz

# 检查返回结果
# 正常返回: ok
# 异常会返回具体错误信息
```

**第三步:检查进程和资源**

```bash
# 查看 controller-manager 进程
ps aux | grep kube-controller-manager

# 查看资源使用
top -p $(pgrep -f kube-controller-manager)

# 检查端口监听
netstat -tlnp | grep 10252
```

**第四步:检查配置和证书**

```bash
# 查看 controller-manager 配置
kubectl get cm -n kube-system kube-controller-manager -o yaml

# 检查与 API Server 的连接
curl -k https://localhost:6443/api/v1/namespaces/kube-system/endpoints/kubernetes

# 检查证书
openssl x509 -in /etc/kubernetes/pki/apiserver-kubelet-client.crt -noout -dates
```

### 4.4 解决方案

**API Server 连接问题**

```bash
# 检查 API Server 是否正常运行
kubectl get apiserver

# 如果 API Server 需要重启
systemctl restart kube-apiserver
# 或
docker restart kube-apiserver
```

**配置问题修复**

```bash
# 静态 Pod 配置文件位置
# /etc/kubernetes/manifests/kube-controller-manager.yaml

# 编辑配置后,kubelet 会自动重载
vi /etc/kubernetes/manifests/kube-controller-manager.yaml

# 确保以下参数正确
# --kubeconfig=/etc/kubernetes/controller-manager.conf
# --leader-elect=true
# --address=127.0.0.1
```

**资源不足问题**

```bash
# 增加资源限制
# 编辑 controller-manager 的静态 Pod 配置
vi /etc/kubernetes/manifests/kube-controller-manager.yaml

# 修改 resources 字段,增加 CPU 和内存限制
```

## 五、scheduler unhealthy 问题排查

### 5.1 健康检查机制

kube-scheduler 通过 `http://localhost:10251/healthz` 端点提供健康检查服务。该端点会验证调度器的核心功能是否正常工作,包括:

- 预选策略执行
- 优选策略计算
- 绑定逻辑

### 5.2 常见问题原因

| 问题类型 | 具体原因 | 影响程度 |
|---------|---------|---------|
| 配置问题 | 调度器配置文件错误 | 高 |
| 依赖问题 | 无法连接到 API Server | 高 |
| 资源问题 | 调度队列积压、性能问题 | 中 |
| 插件问题 | 自定义调度插件加载失败 | 中 |
| 权限问题 | RBAC 权限配置错误 | 高 |

### 5.3 排查步骤

**第一步:检查 scheduler 状态**

```bash
# 查看 scheduler Pod 状态

# 查看 scheduler 日志
```

**第二步:检查健康检查端点**

**第三步:检查进程和资源**

**第四步:检查调度器配置**

```bash
# 如果权限缺失,创建或更新 ClusterRoleBinding
kubectl create clusterrolebinding kube-scheduler-binding \
  --clusterrole=system:kube-scheduler \
  --serviceaccount=kube-system:kube-scheduler
```

## 七、常见问题与最佳实践

### 7.1 常见问题 FAQ

**Q1: kubectl get cs 命令在 Kubernetes 1.19+ 版本中显示 Warning,应该如何处理?**

A: 在 Kubernetes 1.19+ 版本中,`kubectl get cs` 命令已被弃用。建议使用以下替代方案:
- 使用 `kubectl get pods -n kube-system` 检查控制平面组件的 Pod 状态
- 使用 `kubectl get nodes` 检查节点状态
- 使用组件特定的健康检查端点进行验证

**Q2: 所有组件都显示 unhealthy,但集群功能正常,这是什么原因?**

A: 这种情况通常是因为健康检查端点的监听地址配置问题。检查组件的启动参数中的 `--address` 或 `--bind-address` 配置,确保监听在 `127.0.0.1` 或 `0.0.0.0`。

**Q3: etcd 显示 unhealthy,但 curl 测试健康端点正常,如何处理?**

A: 这可能是因为 Component Status Controller 检查的是不同的 etcd 实例或使用了不同的配置。检查:
- etcd 的监听地址配置
- 是否有多个 etcd 实例
- 证书配置是否正确

**Q4: controller-manager 和 scheduler 同时显示 unhealthy,可能的原因是什么?**

A: 这种情况通常是因为:
- API Server 出现故障,导致两个组件无法连接
- 网络配置问题
- 证书过期或配置错误

**Q5: 如何预防组件 unhealthy 问题?**

A: 建议采取以下预防措施:
- 定期检查和更新证书
- 监控磁盘空间和资源使用情况
- 配置告警规则,及时发现异常
- 定期备份 etcd 数据
- 使用高可用架构部署控制平面组件

### 7.2 最佳实践

**1. 监控和告警**

```yaml
# Prometheus 告警规则示例
groups:
- name: kubernetes-components
  rules:
  - alert: ComponentStatusUnhealthy
    expr: kube_componentcontroller_status_healthy == 0
    for: 5m
    labels:
      severity: critical
    annotations:
      summary: "Kubernetes component {{ $labels.component }} is unhealthy"
      description: "Component {{ $labels.component }} has been unhealthy for more than 5 minutes."
```

**2. 定期健康检查脚本**

```bash
#!/bin/bash
# 定期检查组件状态

# 检查 etcd
ETCD_STATUS=$(curl -s http://localhost:2379/health)
if [ "$ETCD_STATUS" != '{"health":"true"}' ]; then
  echo "ETCD is unhealthy: $ETCD_STATUS"
  # 发送告警
fi

# 检查 controller-manager
CM_STATUS=$(curl -s http://localhost:10252/healthz)
if [ "$CM_STATUS" != "ok" ]; then
  echo "Controller Manager is unhealthy: $CM_STATUS"
  # 发送告警
fi

# 检查 scheduler
SCHED_STATUS=$(curl -s http://localhost:10251/healthz)
if [ "$SCHED_STATUS" != "ok" ]; then
  echo "Scheduler is unhealthy: $SCHED_STATUS"
  # 发送告警
fi
```

**3. 证书自动更新**

```bash
# 使用 kubeadm 自动更新证书
# 添加到定时任务
0 0 1 * * kubeadm certs renew all && systemctl restart kubelet
```

**4. etcd 定期备份**

```bash
#!/bin/bash
# etcd 备份脚本
BACKUP_DIR="/backup/etcd"
DATE=$(date +%Y%m%d_%H%M%S)

ETCDCTL_API=3 etcdctl --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  snapshot save $BACKUP_DIR/etcd-snapshot-$DATE.db

# 保留最近7天的备份
find $BACKUP_DIR -name "*.db" -mtime +7 -delete
```

**5. 资源限制配置**

确保控制平面组件有足够的资源:

```yaml
# kube-controller-manager.yaml
resources:
  requests:
    cpu: 200m
    memory: 512Mi
  limits:
    cpu: 500m
    memory: 1Gi
```

## 八、总结

`kubectl get cs` 命令是 Kubernetes 集群健康状态检查的重要工具,通过 HTTP 健康检查端点监控 etcd、controller-manager 和 scheduler 三大核心组件的运行状态。当出现 unhealthy 状态时,需要系统化地进行排查:

1. **etcd 问题**: 重点检查磁盘空间、证书状态、网络连接和数据完整性
2. **controller-manager 问题**: 重点检查 API Server 连接、配置正确性和资源使用情况
3. **scheduler 问题**: 重点检查 API Server 连接、配置文件和 RBAC 权限

排查过程遵循"服务状态→健康端点→日志分析→配置检查→资源检查"的流程,逐步定位问题根源。同时,建立完善的监控告警机制、定期备份和证书更新策略,可以有效预防 unhealthy 问题的发生。

## 面试回答

**面试官**: 请问执行 kubectl get cs 查看集群状态显示 unhealthy,你会如何排查?

**回答**:

kubectl get cs 命令用于检查 Kubernetes 控制平面组件的健康状态,包括 etcd、controller-manager 和 scheduler。当出现 unhealthy 状态时,我会按照以下步骤进行排查:

首先,我需要理解 Component Status Controller 的工作原理。它通过 HTTP GET 请求访问各组件的健康检查端点:scheduler 的 10251 端口、controller-manager 的 10252 端口、etcd 的 2379 端口。如果返回 HTTP 200 且响应正确,则认为组件健康。

针对不同组件,排查思路有所不同:

**对于 etcd unhealthy**,我会先检查 etcd 服务状态和日志,然后使用 curl 测试健康端点,检查磁盘空间是否充足、证书是否过期、资源使用是否正常。常见原因是磁盘空间不足、证书过期或数据损坏,需要相应地进行清理、更新证书或恢复数据。

**对于 controller-manager unhealthy**,我会检查 Pod 状态和日志,测试健康端点,验证与 API Server 的连接,检查配置文件和证书。常见原因是无法连接 API Server、配置错误或资源不足,需要修复配置或重启服务。

**对于 scheduler unhealthy**,排查步骤与 controller-manager 类似,额外需要检查 RBAC 权限配置和调度器配置文件。

需要注意的是,在 Kubernetes 1.19+ 版本中,kubectl get cs 已被弃用,建议结合 kubectl get pods -n kube-system 和 kubectl get nodes 等命令综合判断集群状态。同时,应该建立监控告警机制、定期备份 etcd、及时更新证书,从预防角度减少 unhealthy 问题的发生。
