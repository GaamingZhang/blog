# 客户端访问K8s资源需要经过几关？分别是什么？

## 引言：为什么访问控制至关重要？

在Kubernetes集群中，API Server是整个集群的统一入口，所有对集群资源的访问和操作都必须经过API Server。作为一个多租户的容器编排平台，Kubernetes承载着企业的核心业务应用，涉及敏感数据、关键服务和基础设施资源。如果没有严格的访问控制机制，任何能够访问API Server的客户端都可以随意创建、修改或删除资源，这将带来巨大的安全风险。

想象一下，如果一名开发人员误删了生产环境的Namespace，或者恶意用户获取了集群管理员权限，后果将不堪设想。因此，Kubernetes设计了一套严密的访问控制体系，确保只有经过身份验证、拥有适当权限、符合安全策略的请求才能到达集群资源。

本文将深入剖析Kubernetes访问控制的三道关卡，揭示其工作原理和实现机制，帮助您构建更安全的Kubernetes集群。

## 访问控制三关概览

当客户端（无论是kubectl、用户应用还是系统组件）向Kubernetes API Server发起请求时，请求需要依次经过三道关卡的严格审查：

```
┌─────────────┐      ┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│   客户端     │─────▶│  第一关：    │─────▶│  第二关：    │─────▶│  第三关：    │─────▶ API Server
│  (kubectl)  │      │  认证(AuthN) │      │  授权(AuthZ) │      │ 准入控制     │      │  处理请求
└─────────────┘      └─────────────┘      └─────────────┘      └─────────────┘      └─────────────┘
                            │                     │                     │
                            ▼                     ▼                     ▼
                      你是谁？               你能做什么？            请求是否合规？
                    验证身份信息             验证操作权限            验证和修改请求
```

这三关分别是：

1. **认证（Authentication）**：验证客户端身份，回答"你是谁？"
2. **授权（Authorization）**：验证操作权限，回答"你能做什么？"
3. **准入控制（Admission Control）**：验证和修改请求，回答"请求是否合规？"

每一关都是一道安全屏障，任何一关验证失败，请求都会被拒绝。只有通过所有三关，请求才能到达API Server的核心逻辑进行处理。

## 第一关：认证（Authentication）——你是谁？

### 工作原理

认证是访问控制的第一道防线，其核心任务是验证客户端的身份真实性。当请求到达API Server时，认证模块会检查请求中携带的身份凭证，验证客户端是否为其声称的身份。

Kubernetes采用**插件化的认证架构**，支持多种认证方式同时启用。API Server启动时可以配置多个认证插件，这些插件会按顺序依次尝试认证请求：

```
请求到达
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│              认证插件链（Authentication Chain）           │
├─────────────────────────────────────────────────────────┤
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐│
│  │ X509证书  │─▶│ Bootstrap │─▶│  Token   │─▶│  OIDC    ││
│  │  认证    │  │   Token   │  │  认证    │  │  认证    ││
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘│
│       │              │             │             │      │
│       └──────────────┴─────────────┴─────────────┘      │
│                          │                              │
│                          ▼                              │
│                   任一插件认证成功                        │
│                    → 认证通过                            │
│                   所有插件认证失败                        │
│                    → 返回401                            │
└─────────────────────────────────────────────────────────┘
```

**关键机制**：
- 认证采用"或"逻辑：只要任一认证插件验证成功，请求就通过认证
- 认证失败返回401 Unauthorized错误
- 认证成功后，请求会被附加用户信息（Username、UID、Groups等），供后续授权使用

### 常见认证方式

| 认证方式 | 适用场景 | 工作机制 | 安全性 |
|---------|---------|---------|--------|
| **X509客户端证书** | kubelet、系统组件、管理员 | 使用证书中的CN作为用户名，O作为组名 | 高（基于PKI） |
| **Bootstrap Token** | 新节点加入集群 | 使用预共享的Token进行初始认证，用于获取正式证书 | 中（一次性） |
| **Service Account Token** | Pod内应用访问API | 自动挂载到Pod，使用JWT Token认证 | 中（需RBAC配合） |
| **Static Token File** | 测试环境、简单场景 | 静态文件存储Token和用户映射 | 低（需妥善保管） |
| **OIDC** | 企业环境、SSO集成 | 与企业身份提供商集成，支持OAuth2流程 | 高（企业级） |
| **Webhook Token Auth** | 自定义认证逻辑 | 将Token验证委托给外部服务 | 可定制 |

### 认证配置示例

API Server启动参数配置认证方式：

```bash
# 启用X509客户端证书认证
--client-ca-file=/etc/kubernetes/pki/ca.crt

# 启用Bootstrap Token认证
--enable-bootstrap-token-auth=true

# 配置OIDC认证
--oidc-issuer-url=https://accounts.google.com
--oidc-client-id=kubernetes
--oidc-username-claim=email

# 配置Service Account认证
--service-account-key-file=/etc/kubernetes/pki/sa.pub
--service-account-issuer=https://kubernetes.default.svc.cluster.local
```

### 认证信息传递

认证成功后，请求上下文中会包含以下用户信息：

```go
type UserInfo struct {
    Username string   // 用户名，如："system:node:node1"
    UID      string   // 用户唯一标识
    Groups   []string // 用户所属组，如：["system:nodes", "system:authenticated"]
    Extra    map[string][]string // 额外信息
}
```

这些信息将在后续的授权和准入控制阶段被使用。

## 第二关：授权（Authorization）——你能做什么？

### 工作原理

认证解决了"你是谁"的问题，授权则要回答"你能做什么"。授权模块根据认证阶段获取的用户信息，检查该用户是否有权限对目标资源执行请求的操作。

Kubernetes同样采用**插件化的授权架构**，支持多种授权模式。与认证不同的是，授权采用"与"逻辑：只有所有启用的授权模块都允许，请求才能通过授权。

```
认证通过的用户信息
        │
        ▼
┌───────────────────────────────────────────────────────┐
│            授权模块链（Authorization Chain）            │
├───────────────────────────────────────────────────────┤
│  ┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐  │
│  │  Node  │──▶│  RBAC  │──▶│  ABAC  │──▶│Webhook │  │
│  │ 授权   │   │  授权   │   │  授权   │   │  授权   │  │
│  └────────┘   └────────┘   └────────┘   └────────┘  │
│       │            │            │            │       │
│       └────────────┴────────────┴────────────┘       │
│                      │                               │
│                      ▼                               │
│              所有模块都允许 → 授权通过                 │
│              任一模块拒绝 → 返回403                   │
└───────────────────────────────────────────────────────┘
```

**授权决策依据**：
- **Who**：用户名、用户组
- **What**：请求的操作（get、list、create、update、delete等）
- **Which**：目标资源（Pod、Service、Secret等）
- **Where**：资源所在的Namespace或集群范围

### 授权模式详解

#### 1. RBAC（Role-Based Access Control）

RBAC是Kubernetes**最推荐、最常用**的授权模式，基于角色进行权限管理。

**核心概念**：

```
┌─────────────────────────────────────────────────────────┐
│                    RBAC 权限模型                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌──────────┐        ┌──────────┐       ┌──────────┐  │
│  │   User   │        │   Role   │       │  Rules   │  │
│  │  用户    │        │   角色   │       │  规则    │  │
│  └──────────┘        └──────────┘       └──────────┘  │
│       │                    │                  │        │
│       │  RoleBinding       │  定义            │        │
│       └────────────────────┤                  │        │
│                            │                  │        │
│                            └──────────────────┘        │
│                                                         │
│  ┌──────────┐        ┌──────────┐       ┌──────────┐  │
│  │  Cluster │        │ Cluster  │       │  Rules   │  │
│  │  Role    │        │  Role    │       │ (集群范围)│  │
│  └──────────┘        └──────────┘       └──────────┘  │
│       │                    │                  │        │
│       │ ClusterRoleBinding │  定义            │        │
│       └────────────────────┤                  │        │
│                            │                  │        │
│                            └──────────────────┘        │
└─────────────────────────────────────────────────────────┘
```

**四种核心资源**：

| 资源类型 | 作用范围 | 用途 |
|---------|---------|------|
| Role | Namespace级别 | 定义命名空间内的权限规则 |
| RoleBinding | Namespace级别 | 将Role绑定到用户/组/ServiceAccount |
| ClusterRole | 集群级别 | 定义集群范围的权限规则 |
| ClusterRoleBinding | 集群级别 | 将ClusterRole绑定到用户/组/ServiceAccount |

**权限规则示例**：

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: default
rules:
- apiGroups: [""]              # 核心API组
  resources: ["pods"]          # 资源类型
  verbs: ["get", "list", "watch"]  # 操作权限
- apiGroups: [""]              # 核心API组
  resources: ["pods/log"]      # 子资源
  verbs: ["get"]
```

**RBAC授权流程**：

```
请求到达授权模块
    │
    ▼
提取请求信息：user=alice, verb=get, resource=pod, namespace=default
    │
    ▼
查找用户绑定的所有RoleBinding和ClusterRoleBinding
    │
    ├─▶ 查找default命名空间的RoleBinding
    │       └─▶ 找到RoleBinding: alice-pod-reader
    │              └─▶ 关联Role: pod-reader
    │                     └─▶ 检查规则：允许get pods ✓
    │
    └─▶ 查找集群级别的ClusterRoleBinding
            └─▶ 无匹配
    │
    ▼
决策：允许访问
```

#### 2. Node授权

Node授权是专门为kubelet设计的授权模式，控制kubelet对节点相关资源的访问权限。

**工作原理**：
- 仅对用户名为`system:node:<nodeName>`的请求生效
- 限制kubelet只能访问本节点的Pod、Secret、ConfigMap等资源
- 防止kubelet访问其他节点的敏感信息

**授权逻辑**：

```
kubelet请求到达
    │
    ▼
验证用户名格式：system:node:<nodeName>
    │
    ├─▶ 用户名匹配 → 检查请求资源是否属于该节点
    │                  │
    │                  ├─▶ 属于本节点 → 允许
    │                  └─▶ 属于其他节点 → 拒绝
    │
    └─▶ 用户名不匹配 → 跳过Node授权，交由其他授权模块处理
```

#### 3. ABAC（Attribute-Based Access Control）

ABAC是基于属性的访问控制，通过策略文件定义授权规则。

**策略文件格式**：

```json
{
  "apiVersion": "abac.authorization.kubernetes.io/v1beta1",
  "kind": "Policy",
  "spec": {
    "user": "alice",
    "namespace": "default",
    "resource": "pods",
    "readonly": true
  }
}
```

**缺点**：
- 修改策略需要重启API Server
- 难以管理和维护
- 已逐渐被RBAC取代

#### 4. Webhook授权

Webhook授权将授权决策委托给外部服务，实现自定义授权逻辑。

**工作流程**：

```
请求到达授权模块
    │
    ▼
构造授权请求（SubjectAccessReview）
    │
    ▼
发送到外部授权服务
    │
    ▼
外部服务返回授权决策
    │
    ├─▶ allowed: true → 允许
    └─▶ allowed: false → 拒绝
```

**配置示例**：

```bash
--authorization-mode=Webhook
--authorization-webhook-config-file=/etc/kubernetes/authorization-webhook.yaml
```

### 授权模式对比

| 授权模式 | 优点 | 缺点 | 适用场景 |
|---------|------|------|---------|
| **RBAC** | 声明式配置、动态管理、细粒度控制 | 复杂场景规则较多 | 生产环境、企业应用 |
| **Node** | 专为kubelet设计、安全性高 | 仅适用于kubelet | 所有集群（默认启用） |
| **ABAC** | 灵活性高 | 需重启API Server、难维护 | 已淘汰 |
| **Webhook** | 完全自定义 | 依赖外部服务、增加延迟 | 特殊授权需求 |

**推荐配置**：`--authorization-mode=Node,RBAC`

## 第三关：准入控制（Admission Control）——请求是否合规？

### 工作原理

准入控制是访问控制的最后一道防线，位于授权之后、持久化存储之前。它不仅要验证请求是否符合集群策略，还可以修改请求内容。

准入控制分为两类：

1. **验证型准入控制（Validating Admission）**：只验证请求，不修改
2. **变更型准入控制（Mutating Admission）**：可以修改请求内容

**执行顺序**：

```
授权通过的请求
      │
      ▼
┌─────────────────────────────────────────────────────┐
│          准入控制链（Admission Chain）                │
├─────────────────────────────────────────────────────┤
│                                                     │
│  阶段1：变更型准入控制（Mutating Admission）          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐         │
│  │Namespace │─▶│  Limit   │─▶│ Service  │         │
│  │Auto-prov │  │  Ranger  │  │ Account  │         │
│  └──────────┘  └──────────┘  └──────────┘         │
│       │              │              │              │
│       └──────────────┴──────────────┘              │
│                      │                             │
│                      ▼ 可以修改请求内容              │
│                                                     │
│  阶段2：验证型准入控制（Validating Admission）        │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐         │
│  │  Limit   │─▶│   Pod    │─▶│  Node    │         │
│  │  Ranger  │  │Security  │  │Restrict  │         │
│  │(验证)    │  │ Context  │  │          │         │
│  └──────────┘  └──────────┘  └──────────┘         │
│       │              │              │              │
│       └──────────────┴──────────────┘              │
│                      │                             │
│                      ▼ 只验证，不修改               │
│                                                     │
│              所有准入控制器通过 → 请求通过           │
│              任一准入控制器拒绝 → 返回403           │
└─────────────────────────────────────────────────────┘
```

**关键机制**：
- 变更型准入控制先执行，可以修改请求
- 验证型准入控制后执行，验证修改后的请求
- 任一准入控制器拒绝，请求立即被拒绝
- 准入控制失败返回403 Forbidden

### 常见准入控制器

#### 1. 资源管理类

| 准入控制器 | 类型 | 功能 | 作用 |
|-----------|------|------|------|
| **NamespaceExists** | 验证型 | 检查Namespace是否存在 | 防止在不存在Namespace创建资源 |
| **NamespaceAutoProvision** | 变更型 | 自动创建不存在的Namespace | 简化管理（不推荐生产使用） |
| **LimitRanger** | 变更型+验证型 | 应用资源限制策略 | 为Pod设置默认资源限制 |
| **ResourceQuota** | 验证型 | 检查资源配额 | 限制Namespace资源使用总量 |

**LimitRanger工作原理**：

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: default
spec:
  limits:
  - default:          # 默认限制
      cpu: "500m"
      memory: "512Mi"
    defaultRequest:   # 默认请求
      cpu: "100m"
      memory: "128Mi"
    type: Container
```

当Pod未指定resources时，LimitRanger会自动注入：

```
创建Pod（无resources）
        │
        ▼
LimitRanger准入控制器拦截
        │
        ▼
检查LimitRange策略
        │
        ▼
自动注入resources：
  requests:
    cpu: 100m
    memory: 128Mi
  limits:
    cpu: 500m
    memory: 512Mi
        │
        ▼
修改后的Pod继续处理
```

#### 2. 安全策略类

| 准入控制器 | 类型 | 功能 | 作用 |
|-----------|------|------|------|
| **PodSecurityPolicy** | 验证型 | 验证Pod安全策略 | 限制Pod权限（已废弃，用PodSecurity替代） |
| **PodSecurity** | 验证型 | 强制Pod安全标准 | 替代PSP，实现三大安全级别 |
| **NodeRestriction** | 验证型 | 限制kubelet修改范围 | kubelet只能修改本节点Pod |
| **ServiceAccount** | 变更型 | 自动挂载ServiceAccount | 为Pod注入ServiceAccount Token |

**PodSecurity工作原理**：

```yaml
# Namespace标签定义安全策略
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    pod-security.kubernetes.io/enforce: restricted  # 强制执行restricted级别
    pod-security.kubernetes.io/enforce-version: latest
```

三大安全级别：

| 级别 | 限制程度 | 典型限制 |
|------|---------|---------|
| **Privileged** | 无限制 | 允许特权容器、主机网络、主机PID |
| **Baseline** | 最小限制 | 禁止特权容器、主机网络，允许capabilities |
| **Restricted** | 严格限制 | 禁止特权、必须runAsNonRoot、禁止capabilities |

#### 3. 业务逻辑类

| 准入控制器 | 类型 | 功能 | 作用 |
|-----------|------|------|------|
| **AlwaysPullImages** | 变更型 | 强制拉取镜像 | 确保使用最新镜像 |
| **ImagePolicyWebhook** | 验证型 | 外部镜像策略验证 | 集成镜像安全扫描 |
| **DefaultStorageClass** | 变更型 | 设置默认StorageClass | 自动为PVC设置StorageClass |
| **DefaultTolerationSeconds** | 变更型 | 设置默认容忍时间 | Pod默认容忍Node不可用时间 |

### 动态准入控制（Dynamic Admission Control）

除了内置准入控制器，Kubernetes还支持动态准入控制，通过Webhook实现自定义准入逻辑。

**架构**：

```
请求到达准入控制阶段
        │
        ▼
API Server构造AdmissionReview请求
        │
        ├─▶ 发送到ValidatingWebhook
        │       │
        │       └─▶ 外部服务验证请求
        │              └─▶ 返回allowed: true/false
        │
        └─▶ 发送到MutatingWebhook
                │
                └─▶ 外部服务修改请求
                       └─▶ 返回patch操作
```

**配置示例**：

```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingWebhookConfiguration
metadata:
  name: image-policy-webhook
webhooks:
- name: image-policy.example.com
  rules:
  - apiGroups: [""]
    apiVersions: ["v1"]
    resources: ["pods"]
    operations: ["CREATE", "UPDATE"]
  clientConfig:
    service:
      name: image-policy-service
      namespace: default
      path: "/validate"
  admissionReviewVersions: ["v1"]
```

### 准入控制器配置

API Server启动参数：

```bash
--enable-admission-plugins=NodeRestriction,LimitRanger,ResourceQuota,PodSecurity
--disable-admission-plugins=NamespaceAutoProvision
```

**推荐配置**（Kubernetes 1.25+）：


## 常见问题与最佳实践

### 常见问题

#### 问题1：为什么我的kubectl命令返回401 Unauthorized？

**原因分析**：
- kubeconfig文件中的证书或Token过期
- 客户端证书不被API Server信任
- Token格式错误或已被删除

**解决方案**：
```bash
# 检查kubeconfig配置
kubectl config view

# 检查证书有效期
openssl x509 -in ~/.kube/client-certificate.crt -text -noout

# 重新生成Token
kubectl create token default -n default
```

#### 问题2：为什么RBAC配置正确但仍然返回403 Forbidden？

**原因分析**：
- RoleBinding绑定到了错误的ServiceAccount
- 资源名称或API组配置错误
- 启用了多个授权模块，其中某个模块拒绝

**排查方法**：
```bash
# 检查用户权限
kubectl auth can-i list pods --as=alice -n default

# 查看RoleBinding详情
kubectl describe rolebinding alice-pod-reader -n default

# 检查授权日志
# API Server启动参数添加：
--v=5 --logtostderr=true
```

#### 问题3：如何实现Pod只能访问特定Namespace的资源？

**解决方案**：
使用RBAC限制ServiceAccount权限：

```yaml
# 创建ServiceAccount
apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-sa
  namespace: production

---
# 创建Role（仅限production命名空间）
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: production
rules:
- apiGroups: [""]
  resources: ["pods", "services"]
  verbs: ["get", "list"]

---
# 绑定ServiceAccount到Role
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: app-pod-reader
  namespace: production
subjects:
- kind: ServiceAccount
  name: app-sa
  namespace: production
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

#### 问题4：如何实现自定义准入控制逻辑？

**解决方案**：
使用ValidatingAdmissionWebhook或MutatingAdmissionWebhook：

```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingWebhookConfiguration
metadata:
  name: custom-validator
webhooks:
- name: validator.example.com
  rules:
  - apiGroups: [""]
    apiVersions: ["v1"]
    resources: ["pods"]
    operations: ["CREATE", "UPDATE"]
  clientConfig:
    service:
      name: webhook-service
      namespace: default
      path: "/validate"
  admissionReviewVersions: ["v1"]
  sideEffects: None
```

#### 问题5：如何调试准入控制器问题？

**排查方法**：
```bash
# 查看API Server启用的准入控制器
kubectl get pods -n kube-system kube-apiserver-master -o yaml | grep enable-admission-plugins

# 查看准入控制器日志
kubectl logs -n kube-system kube-apiserver-master | grep admission

# 检查Webhook配置
kubectl get validatingwebhookconfigurations
kubectl get mutatingwebhookconfigurations
```

## 面试回答

**面试官**：客户端访问Kubernetes资源需要经过几关？分别是什么？

**回答**：

客户端访问Kubernetes资源需要经过**三关**，分别是**认证（Authentication）、授权（Authorization）和准入控制（Admission Control）**。

**第一关是认证**，解决"你是谁"的问题。API Server通过多种认证插件（如X509证书、Bearer Token、OIDC等）验证客户端身份，认证成功后提取用户信息（Username、Groups、UID）。认证采用"或"逻辑，任一插件认证成功即通过，失败返回401。

**第二关是授权**，解决"你能做什么"的问题。根据认证阶段的用户信息，检查用户是否有权限执行请求的操作。Kubernetes支持多种授权模式，其中RBAC是最常用的，通过Role和RoleBinding实现细粒度的权限控制。授权采用"与"逻辑，所有启用的授权模块都必须允许，失败返回403。

**第三关是准入控制**，解决"请求是否合规"的问题。准入控制分为变更型和验证型两类，可以修改请求内容（如注入默认资源限制）或验证请求是否符合集群策略（如PodSecurity、ResourceQuota）。准入控制通过后，请求才会被持久化到ETCD。

这三关层层递进，构成了Kubernetes完整的安全防护体系，确保只有合法的、授权的、合规的请求才能访问集群资源。
