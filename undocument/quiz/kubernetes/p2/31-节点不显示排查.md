# kubectl get node 看不到某些节点的排查指南

## 引言

在 Kubernetes 集群运维过程中，`kubectl get node` 是最基础也是最常用的命令之一。然而，当我们执行该命令时，有时会发现某些节点"消失"了——明明物理服务器在运行，却无法在节点列表中看到。这种情况不仅影响集群的调度能力，还可能导致服务可用性下降。

节点不显示的问题看似简单，实则涉及 Kubernetes 节点注册机制、证书认证、网络通信、kubelet 运行状态等多个层面。本文将从节点注册的底层原理出发，系统性地分析导致节点不显示的各种原因，并提供详细的排查步骤和解决方案。

## 节点注册机制解析

在深入排查之前，我们需要理解 Kubernetes 节点是如何注册到集群中的：

1. **kubelet 启动**：kubelet 作为节点上的核心代理，负责与 API Server 通信
2. **证书认证**：kubelet 使用客户端证书向 API Server 进行身份认证
3. **节点对象创建**：认证通过后，kubelet 在 API Server 中创建或更新 Node 对象
4. **状态上报**：kubelet 定期向 API Server 上报节点状态（默认每 10 秒）

任何环节出现问题，都可能导致节点无法正常显示。

## 核心原因分析

### 原因一：kubelet 未运行或配置错误

#### 现象

- 执行 `kubectl get node` 看不到目标节点
- 节点上容器运行正常，但无法被调度新 Pod
- 节点状态长时间未更新

#### 排查步骤

**1. 检查 kubelet 服务状态**

```bash
# 检查 kubelet 是否运行
systemctl status kubelet

# 查看 kubelet 服务日志
journalctl -u kubelet -f

# 检查 kubelet 进程
ps aux | grep kubelet
```

**2. 检查 kubelet 配置文件**

```bash
# 查看 kubelet 配置
cat /var/lib/kubelet/config.yaml

# 检查启动参数
ps aux | grep kubelet | grep --color -- --config
```

**3. 验证 kubelet 是否连接到正确的 API Server**

```bash
# 检查 kubelet 配置中的 API Server 地址
grep -r "server:" /var/lib/kubelet/
grep -r "apiServer" /var/lib/kubelet/
```

#### 解决方案

| 问题类型 | 解决方法 |
|---------|---------|
| kubelet 未启动 | `systemctl start kubelet && systemctl enable kubelet` |
| kubelet 配置错误 | 检查并修正 `/var/lib/kubelet/config.yaml` |
| API Server 地址错误 | 更新配置文件中的 `apiServer` 地址 |
| kubelet 版本不兼容 | 升级或降级 kubelet 版本以匹配集群版本 |

### 原因二：网络连接问题

#### 现象

- kubelet 日志中出现 "connection refused" 或 "timeout" 错误
- 节点间歇性消失和出现
- 其他网络相关服务异常

#### 排查步骤

**1. 测试与 API Server 的网络连通性**

```bash
# 测试 API Server 端口连通性（默认 6443）
telnet <api-server-ip> 6443

# 或使用 nc
nc -zv <api-server-ip> 6443

# 测试 HTTPS 连接
curl -k https://<api-server-ip>:6443/healthz
```

**2. 检查防火墙规则**

```bash
# 查看 iptables 规则
iptables -L -n -v | grep 6443

# 查看防火墙状态（CentOS/RHEL）
firewall-cmd --list-all

# 查看防火墙状态（Ubuntu）
ufw status
```

**3. 检查网络接口和路由**

```bash
# 查看网络接口
ip addr show

# 查看路由表
ip route show

# 检查 DNS 解析
nslookup kubernetes.default.svc.cluster.local
```

**4. 检查 kubelet 日志中的网络错误**

```bash
# 过滤网络相关错误
journalctl -u kubelet | grep -i "connection\|timeout\|network\|refused"
```

#### 解决方案

| 问题类型 | 解决方法 |
|---------|---------|
| 防火墙阻止 | 开放 6443 端口：`firewall-cmd --add-port=6443/tcp --permanent` |
| 网络不通 | 检查网络配置、路由表、网关设置 |
| DNS 解析失败 | 配置正确的 DNS 服务器或使用 IP 地址 |
| 带宽或延迟过高 | 优化网络配置，确保节点与 master 网络质量 |

### 原因三：证书过期或无效

#### 现象

- kubelet 日志中出现 "certificate expired" 或 "unauthorized" 错误
- 节点突然从列表中消失（常见于集群运行一年后）
- 其他组件证书相关错误

#### 排查步骤

**1. 检查 kubelet 客户端证书有效期**

```bash
# 查看 kubelet 客户端证书有效期
openssl x509 -in /var/lib/kubelet/pki/kubelet-client-current.pem -noout -dates

# 查看 kubelet 服务端证书有效期
openssl x509 -in /var/lib/kubelet/pki/kubelet.crt -noout -dates

# 检查证书详细信息
openssl x509 -in /var/lib/kubelet/pki/kubelet-client-current.pem -noout -text
```

**2. 检查 CA 证书**

```bash
# 查看集群 CA 证书
openssl x509 -in /etc/kubernetes/pki/ca.crt -noout -dates

# 验证证书链
openssl verify -CAfile /etc/kubernetes/pki/ca.crt /var/lib/kubelet/pki/kubelet-client-current.pem
```

**3. 检查 kubelet 日志中的证书错误**

```bash
# 过滤证书相关错误
journalctl -u kubelet | grep -i "certificate\|expired\|unauthorized\|x509"
```

#### 解决方案

| 问题类型 | 解决方法 |
|---------|---------|
| 证书已过期 | 使用 kubeadm 更新证书：`kubeadm certs renew all` |
| kubelet 证书过期 | 重启 kubelet 自动续期，或手动生成证书 |
| CA 证书过期 | 更新 CA 证书并重新分发到所有节点 |
| 证书不匹配 | 重新生成证书并重启 kubelet |

**证书更新示例（kubeadm 集群）**：

```bash
# 1. 检查证书到期时间
kubeadm certs check-expiration

# 2. 更新所有证书
kubeadm certs renew all

# 3. 重启 kubelet
systemctl restart kubelet

# 4. 更新 kubeconfig（如果需要）
cp /etc/kubernetes/admin.conf ~/.kube/config
```

### 原因四：kubelet 未正确注册

#### 现象

- kubelet 进程正常运行
- 网络连接正常
- 但节点仍然不显示在列表中

#### 排查步骤

**1. 检查节点注册模式**

```bash
# 查看 kubelet 启动参数中的注册配置
ps aux | grep kubelet | grep -o "\--register[^ ]*"

# 检查配置文件
grep -i "register" /var/lib/kubelet/config.yaml
```

**2. 检查节点名称**

```bash
# 查看 kubelet 使用的主机名
hostname

# 查看 kubelet 配置中的节点名称
grep -i "hostname\|nodeName" /var/lib/kubelet/config.yaml

# 检查节点名称是否已存在
kubectl get nodes | grep <node-name>
```

**3. 检查节点标签和污点**

```bash
# 查看所有节点
kubectl get nodes --show-labels

# 检查节点是否有特殊污点
kubectl describe node <node-name> | grep -A5 Taints
```

**4. 查看 kubelet 注册日志**

```bash
# 查看节点注册相关日志
journalctl -u kubelet | grep -i "register\|node.*create\|node.*update"
```

#### 解决方案

| 问题类型 | 解决方法 |
|---------|---------|
| 自动注册被禁用 | 设置 `--register-node=true`（默认）或手动创建 Node 对象 |
| 节点名称冲突 | 修改 kubelet 配置中的 `hostname-override` 参数 |
| 节点标签问题 | 使用 `kubectl label node` 修正标签 |
| RBAC 权限不足 | 检查 kubelet 使用的 ServiceAccount 权限 |

**手动注册节点示例**：

```yaml
apiVersion: v1
kind: Node
metadata:
  name: <node-name>
  labels:
    kubernetes.io/hostname: <node-name>
spec:
  podCIDR: <pod-cidr>
```

### 原因五：节点被删除或隔离

#### 现象

- 节点之前可见，突然消失
- 可能伴随 Pod 驱逐事件
- 节点状态异常后消失

#### 排查步骤

**1. 检查节点删除事件**

```bash
# 查看节点相关事件
kubectl get events --field-selector involvedObject.kind=Node

# 查看所有事件中的节点信息
kubectl get events | grep -i node

# 查看集群级别事件
kubectl get events --all-namespaces | grep -i "node\|delete"
```

**2. 检查节点是否被标记为不可调度**

```bash
# 查看节点状态
kubectl get nodes

# 查看节点详细信息
kubectl describe node <node-name>

# 检查节点是否被 cordon
kubectl get node <node-name> -o jsonpath='{.spec.unschedulable}'
```

**3. 检查节点驱逐和隔离**

```bash
# 查看节点是否有 NoExecute 污点
kubectl describe node <node-name> | grep -A5 Taints

# 检查节点控制器日志（master 节点）
kubectl logs -n kube-system kube-controller-manager-<master-name> | grep -i "node.*delete\|node.*evict"
```

**4. 检查节点心跳和租约**

```bash
# 查看节点租约
kubectl get lease -n kube-node-lease

# 查看特定节点的租约
kubectl describe lease -n kube-node-lease <node-name>
```

#### 解决方案

| 问题类型 | 解决方法 |
|---------|---------|
| 节点被误删除 | 重新注册节点或重启 kubelet |
| 节点被隔离 | 移除 NoExecute 污点：`kubectl taint nodes <node-name> key:NoExecute-` |
| 节点心跳超时 | 检查 kubelet 状态和网络，确保心跳正常 |
| 节点控制器删除 | 调整 `--node-monitor-grace-period` 参数 |

## 常见问题与最佳实践

### 常见问题 FAQ

**Q1: 节点显示 NotReady 但仍然在列表中，如何处理？**

A: 这是节点状态异常但未完全失联的情况。检查 kubelet 是否正常运行，网络是否稳定，以及节点资源是否充足。如果节点长时间 NotReady，可能会被节点控制器驱逐。

**Q2: 如何预防证书过期导致的节点消失？**

A: 设置证书自动续期监控，使用 `kubeadm certs check-expiration` 定期检查证书有效期，建议在证书到期前 30 天进行更新。可以编写定时任务自动检查。

**Q3: 节点消失后，上面的 Pod 会怎样？**

A: 如果节点被删除或长时间失联（超过 `pod-eviction-timeout`，默认 5 分钟），Pod 会被控制器驱逐并在其他节点重建。如果使用 StatefulSet，需要特别注意数据持久化。

**Q4: 如何快速定位节点消失的根本原因？**

A: 按照排查流程图逐步检查，重点关注：
1. kubelet 日志中的错误信息
2. 系统日志中的异常
3. 网络连通性测试
4. 证书有效期检查

**Q5: 多个节点同时消失可能是什么原因？**

A: 可能原因包括：
- API Server 故障或负载过高
- 网络分区或网络设备故障
- 证书批量过期
- 集群配置错误影响多个节点

### 最佳实践

**1. 监控和告警**

```yaml
# Prometheus 告警规则示例
- alert: NodeUnavailable
  expr: kube_node_status_condition{condition="Ready",status="true"} == 0
  for: 5m
  labels:
    severity: critical
  annotations:
    summary: "节点 {{ $labels.node }} 不可用"
    description: "节点已超过 5 分钟未上报 Ready 状态"
```

**2. 证书管理自动化**

```bash
# 定期检查证书到期时间（可配置为 cron 任务）
#!/bin/bash
kubeadm certs check-expiration
if [ $? -ne 0 ]; then
  echo "证书即将过期，请及时更新" | mail -s "K8S 证书告警" admin@example.com
fi
```

**3. 节点健康检查脚本**

```bash
#!/bin/bash
# 节点健康检查脚本

NODE_NAME=$(hostname)
API_SERVER="https://kubernetes.default.svc"

# 检查 kubelet 状态
systemctl is-active kubelet || echo "kubelet 未运行"

# 检查网络连通性
curl -k -s ${API_SERVER}/healthz || echo "无法连接 API Server"

# 检查证书有效期
openssl x509 -in /var/lib/kubelet/pki/kubelet-client-current.pem -noout -checkend 2592000 || echo "证书将在 30 天内过期"

# 检查节点是否已注册
kubectl get node ${NODE_NAME} &>/dev/null || echo "节点未注册到集群"
```

**4. 日志聚合和分析**

使用 EFK 或 PLG（Promtail-Loki-Grafana）栈收集 kubelet 日志，便于快速定位问题。

**5. 配置管理**

- 使用配置管理工具（如 Ansible、SaltStack）统一管理 kubelet 配置
- 版本控制所有配置文件
- 配置变更前进行充分测试

## 面试回答

**面试官问：执行 kubectl get node 命令后看不到某些节点的原因有哪些？如何排查？**

**回答**：

节点不显示的问题主要涉及五个层面。第一是 kubelet 服务本身的问题，包括 kubelet 未启动、配置错误或版本不兼容，这会导致节点无法与 API Server 建立连接。第二是网络问题，防火墙阻止、网络不通或 DNS 解析失败都会导致 kubelet 无法连接到 API Server。第三是证书问题，这是生产环境中最常见的原因之一，kubelet 的客户端证书或 CA 证书过期会导致认证失败，节点无法注册或状态无法更新。第四是注册配置问题，kubelet 的自动注册功能被禁用，或者节点名称冲突、RBAC 权限不足。第五是节点被删除或隔离，可能是误操作删除了节点对象，或者节点被标记了 NoExecute 污点导致被驱逐。

排查时遵循从底层到上层的原则：首先检查 kubelet 服务状态和日志，确认服务正常运行；然后测试与 API Server 的网络连通性，检查防火墙规则；接着检查证书有效期，特别是运行时间较长的集群；再检查 kubelet 的注册配置和节点名称；最后查看集群事件和节点租约，确认节点是否被删除或隔离。通过这套系统化的排查流程，可以快速定位并解决节点不显示的问题。在生产环境中，建议建立证书到期监控、节点健康检查告警机制，并定期审查节点状态，预防此类问题的发生。

## 总结

节点不显示问题虽然表现形式简单，但背后涉及 Kubernetes 节点管理的多个核心机制。通过理解节点注册流程、掌握系统化的排查方法、建立完善的监控告警体系，我们可以快速定位并解决此类问题，保障集群的稳定运行。

记住排查的核心原则：**从服务状态到网络连通，从证书认证到注册配置，从日志分析到事件追踪**，层层递进，必能找到问题根源。
