# Kubernetes 服务发现

## 引言

服务发现是微服务架构的核心能力。在 Kubernetes 中，Pod 是动态创建和销毁的，IP 地址会变化。服务发现机制让应用能够找到并访问其他服务，而无需关心 Pod 的具体位置。本文介绍 Kubernetes 服务发现的原理和配置方法。

## 服务发现概述

### 服务发现方式

```

│              Kubernetes 服务发现方式                         │
├─────────────────────────────────────────────────────────────┤
│                                                              │

│  ┌─────────────────────────────────────────────────────┐   │
│  │  • Pod 创建时注入 Service 相关环境变量              │   │
│  │  • 格式：{SVCNAME}_SERVICE_HOST, {SVCNAME}_SERVICE_PORT│
│  │  • 只在 Pod 创建时设置，不动态更新                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  DNS 方式：                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • CoreDNS 提供域名解析                             │   │
│  │  • 格式：<service-name>.<namespace>.svc.cluster.local│   │

```

## DNS 服务发现

### Service DNS 格式

```
完整格式：
<service-name>.<namespace>.svc.cluster.local

简写格式：
<service-name>                    # 同命名空间
<service-name>.<namespace>        # 跨命名空间
```

### DNS 解析示例

```bash
nslookup my-service

nslookup my-service.production

nslookup my-service.production.svc.cluster.local

dig my-service.production.svc.cluster.local
```

### CoreDNS 配置

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: coredns
  namespace: kube-system
data:
  Corefile: |
    .:53 {
        errors
        health {
           lameduck 5s
        }
        ready
        kubernetes cluster.local in-addr.arpa ip6.arpa {
           pods insecure
           fallthrough in-addr.arpa ip6.arpa
           ttl 30
        }
        prometheus :9153
        forward . /etc/resolv.conf {
           max_concurrent 1000
        }
        cache 30
        loop
        reload
        loadbalance
    }
```

## Service 类型

### ClusterIP

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  type: ClusterIP
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 8080
```

### NodePort

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-nodeport
spec:
  type: NodePort
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 8080
    nodePort: 30080
```

### LoadBalancer

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-loadbalancer
spec:
  type: LoadBalancer
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 8080
```

### ExternalName

```yaml
apiVersion: v1
kind: Service
metadata:
  name: external-service
spec:
  type: ExternalName
  externalName: external.example.com
```

## 服务发现使用

### 同命名空间访问

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: client-pod
spec:
  containers:
  - name: client
    image: busybox
    command: ["sh", "-c", "while true; do wget -qO- my-service; sleep 1; done"]
```

### 跨命名空间访问

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: client-pod
  namespace: dev
spec:
  containers:
  - name: client
    image: busybox
    command: ["sh", "-c", "while true; do wget -qO- my-service.production; sleep 1; done"]
```

### 环境变量方式

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 8080
---
apiVersion: v1
kind: Pod
metadata:
  name: env-pod
spec:
  containers:
  - name: app
    image: myapp:v1
    env:
    - name: MY_SERVICE_HOST
      valueFrom:
        fieldRef:
          fieldPath: status.podIP
```

## Headless Service

### Headless Service 配置

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-headless
spec:
  clusterIP: None
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 8080
```

### Headless Service DNS 解析

```bash
nslookup my-headless

dig my-headless.production.svc.cluster.local
```

### StatefulSet 使用 Headless Service

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: web
spec:
  serviceName: my-headless
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: app
        image: nginx:1.21
        ports:
        - containerPort: 80
```

## 面试回答

**问题**: 怎样在 Kubernetes 中进行服务发现？

**回答**: Kubernetes 服务发现主要有两种方式：

**DNS 方式（推荐）**：CoreDNS 提供 Service 域名解析。域名格式为 `<service-name>.<namespace>.svc.cluster.local`。同命名空间可简写为 `<service-name>`，跨命名空间使用 `<service-name>.<namespace>`。DNS 方式动态更新，是推荐的服务发现方式。

**环境变量方式**：Pod 创建时自动注入 Service 相关环境变量，格式为 `{SVCNAME}_SERVICE_HOST` 和 `{SVCNAME}_SERVICE_PORT`。只在 Pod 创建时设置，不动态更新，已较少使用。

**Service 类型**：**ClusterIP** 默认类型，分配集群内部 IP，只能在集群内访问；**NodePort** 在每个节点上开放端口，可从外部访问；**LoadBalancer** 创建云负载均衡器，提供外部访问入口；**ExternalName** 映射到外部 DNS 名称。

**Headless Service**：设置 `clusterIP: None`，不分配 ClusterIP。DNS 查询返回所有 Pod IP，适用于 StatefulSet 和需要直接访问 Pod 的场景。

**服务发现使用**：同命名空间使用 Service 名称访问；跨命名空间使用 `<service-name>.<namespace>` 格式；通过 Service 实现负载均衡和故障转移。

**最佳实践**：使用 DNS 方式进行服务发现；配置有意义的 Service 名称；使用端口名称区分多端口服务；StatefulSet 使用 Headless Service。
