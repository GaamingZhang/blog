# Kubernetes 中的证书和私钥体系详解

## 引言：为什么 Kubernetes 需要如此复杂的 PKI 体系？

在 Kubernetes 集群中，各个组件之间的通信必须建立在安全可信的基础之上。想象一下，一个大型企业的办公大楼，每个房间（组件）之间都需要验证身份才能进出——这正是 Kubernetes PKI（Public Key Infrastructure，公钥基础设施）体系的作用。

Kubernetes 采用基于 TLS 的双向认证机制，确保：
- **身份验证**：确认通信双方的身份
- **数据加密**：保护传输数据不被窃听
- **完整性校验**：防止数据在传输过程中被篡改

一个典型的 Kubernetes 集群可能包含数十个证书和私钥，它们构成了一个复杂的信任网络。理解这些证书的作用和关系，对于集群的安全运维至关重要。

## 一、Kubernetes PKI 体系概览

Kubernetes 的 PKI 体系采用分层架构，主要由多个 CA（Certificate Authority，证书颁发机构）组成，每个 CA 负责签发特定用途的证书。

### 证书存储位置

Kubernetes 证书通常存储在以下两个位置：
- **/etc/kubernetes/pki/**：控制平面组件证书
- **/etc/kubernetes/pki/etcd/**：etcd 集群专用证书

## 二、核心 CA 证书详解

CA 证书是整个 PKI 体系的信任锚点，Kubernetes 集群中存在多个独立的 CA，各自负责不同的信任域。

### 2.1 Kubernetes CA（kubernetes-ca）

**用途**：Kubernetes 集群的核心 CA，用于签发大部分组件证书。

**存储位置**：
- 证书：`/etc/kubernetes/pki/ca.crt`
- 私钥：`/etc/kubernetes/pki/ca.key`

**签发对象**：
- API Server 服务器证书
- API Server 客户端证书（用于访问 etcd、kubelet）
- kubelet 证书
- Controller Manager 证书
- Scheduler 证书
- admin、system 等用户证书

**有效期**：10 年（kubeadm 默认配置）

**配置示例**：

```yaml
# /etc/kubernetes/pki/ca.crt 的典型配置
apiVersion: v1
kind: Config
clusters:
- cluster:
    certificate-authority: /etc/kubernetes/pki/ca.crt
    server: https://kubernetes-api:6443
  name: kubernetes
```

### 2.2 etcd CA（etcd-ca）

**用途**：专门为 etcd 集群提供独立的信任域，与主集群 CA 隔离。

**存储位置**：
- 证书：`/etc/kubernetes/pki/etcd/ca.crt`
- 私钥：`/etc/kubernetes/pki/etcd/ca.key`

**签发对象**：
- etcd 服务器证书
- etcd 客户端证书
- etcd peer 证书（节点间通信）

**设计原理**：etcd 作为独立组件，其 CA 与主集群分离，即使主 CA 泄露，etcd 的数据仍然受到保护。

### 2.3 Front Proxy CA（front-proxy-ca）

**用途**：用于扩展 API Server 的代理层认证，支持聚合 API（Aggregation Layer）。

**存储位置**：
- 证书：`/etc/kubernetes/pki/front-proxy-ca.crt`
- 私钥：`/etc/kubernetes/pki/front-proxy-ca.key`

**签发对象**：
- front-proxy-client 证书

**应用场景**：当使用 `kubectl proxy` 或自定义 API Server 时，front-proxy 证书用于验证代理请求的身份。

## 三、API Server 证书体系

API Server 作为集群的核心入口，拥有最复杂的证书配置。

### 3.1 API Server 服务器证书

**用途**：对外提供 HTTPS 服务，供客户端连接。

**存储位置**：
- 证书：`/etc/kubernetes/pki/apiserver.crt`
- 私钥：`/etc/kubernetes/pki/apiserver.key`

**签发者**：kubernetes-ca

**关键配置**：

```bash
# 查看 API Server 证书的 SAN（Subject Alternative Names）
openssl x509 -in /etc/kubernetes/pki/apiserver.crt -text -noout | grep -A 1 "Subject Alternative Name"
```

**SAN 包含**：
- Kubernetes 服务的 ClusterIP（通常是 10.96.0.1）
- kubernetes.default.svc
- kubernetes.default.svc.cluster.local
- 节点主机名
- 节点 IP 地址

### 3.2 API Server 访问 etcd 的客户端证书

**用途**：API Server 作为客户端访问 etcd 集群时的身份凭证。

**存储位置**：
- 证书：`/etc/kubernetes/pki/apiserver-etcd-client.crt`
- 私钥：`/etc/kubernetes/pki/apiserver-etcd-client.key`

**签发者**：etcd-ca

**配置位置**：API Server 启动参数

```bash
--etcd-certfile=/etc/kubernetes/pki/apiserver-etcd-client.crt
--etcd-keyfile=/etc/kubernetes/pki/apiserver-etcd-client.key
--etcd-cafile=/etc/kubernetes/pki/etcd/ca.crt
```

### 3.3 API Server 访问 kubelet 的客户端证书

**用途**：API Server 向 kubelet 发起请求（如日志查看、exec 命令）时的身份凭证。

**存储位置**：
- 证书：`/etc/kubernetes/pki/apiserver-kubelet-client.crt`
- 私钥：`/etc/kubernetes/pki/apiserver-kubelet-client.key`

**签发者**：kubernetes-ca

**组织信息**：证书中包含 `O=system:masters`，赋予 API Server 管理员权限。

## 四、kubelet 证书体系

kubelet 证书体系较为特殊，支持自动轮换机制。

### 4.1 kubelet 服务器证书

**用途**：kubelet 对外提供 HTTPS 服务（10250 端口）。

**存储位置**：
- 证书：`/var/lib/kubelet/pki/kubelet.crt`
- 私钥：`/var/lib/kubelet/pki/kubelet.key`

**签发方式**：
- **自签名模式**：kubelet 自己生成（已废弃）
- **自动轮换模式**：由 kubernetes-ca 签发（推荐）

### 4.2 kubelet 客户端证书

**用途**：kubelet 连接 API Server 时的身份凭证。

**存储位置**：
- 证书：`/var/lib/kubelet/pki/kubelet-client-current.pem`
- 私钥：同上文件

**签发者**：kubernetes-ca

**自动轮换机制**：

```yaml
# kubelet 配置
rotateCertificates: true
serverTLSBootstrap: true
```

kubelet 会在证书即将过期前自动申请新证书，通过 CSR（Certificate Signing Request）机制由 Controller Manager 自动批准。

## 五、etcd 证书体系

etcd 作为集群状态存储，其安全性至关重要。

### 5.1 etcd 服务器证书

**用途**：etcd 节点对外提供服务（2379 端口）。

**存储位置**：
- 证书：`/etc/kubernetes/pki/etcd/server.crt`
- 私钥：`/etc/kubernetes/pki/etcd/server.key`

**签发者**：etcd-ca

### 5.2 etcd Peer 证书

**用途**：etcd 集群节点间通信（2380 端口）。

**存储位置**：
- 证书：`/etc/kubernetes/pki/etcd/peer.crt`
- 私钥：`/etc/kubernetes/pki/etcd/peer.key`

**签发者**：etcd-ca

**配置示例**：

```bash
# etcd 启动参数
--peer-cert-file=/etc/kubernetes/pki/etcd/peer.crt
--peer-key-file=/etc/kubernetes/pki/etcd/peer.key
--peer-client-cert-auth=true
--peer-trusted-ca-file=/etc/kubernetes/pki/etcd/ca.crt
```

### 5.3 etcd 客户端证书

**用途**：供 etcdctl 等工具连接 etcd。

**存储位置**：
- 证书：`/etc/kubernetes/pki/etcd/healthcheck-client.crt`
- 私钥：`/etc/kubernetes/pki/etcd/healthcheck-client.key`

**签发者**：etcd-ca

## 六、Service Account 密钥对

Service Account 是 Kubernetes 的身份认证机制，使用非对称密钥对。

### 6.1 Service Account 公钥和私钥

**用途**：
- **私钥**：Controller Manager 使用私钥签发 Service Account Token
- **公钥**：API Server 使用公钥验证 Token 签名

**存储位置**：
- 私钥：`/etc/kubernetes/pki/sa.key`
- 公钥：`/etc/kubernetes/pki/sa.pub`

**工作原理**：

```
┌─────────────────────┐
│ Controller Manager  │
│                     │
│  sa.key (签名)      │
└─────────────────────┘
          │
          │ 签发 Token
          ▼
┌─────────────────────┐
│  Service Account    │
│       Token         │
└─────────────────────┘
          │
          │ 携带 Token 访问
          ▼
┌─────────────────────┐
│    API Server       │
│                     │
│  sa.pub (验证)      │
└─────────────────────┘
```

**配置位置**：

```bash
# Controller Manager 启动参数
--service-account-private-key-file=/etc/kubernetes/pki/sa.key

# API Server 启动参数
--service-account-key-file=/etc/kubernetes/pki/sa.pub
```

## 十、证书管理实战

### 10.1 查看证书信息

```bash
# 查看证书详情
openssl x509 -in /etc/kubernetes/pki/apiserver.crt -text -noout

# 查看证书有效期
openssl x509 -in /etc/kubernetes/pki/apiserver.crt -noout -dates

# 验证证书链
openssl verify -CAfile /etc/kubernetes/pki/ca.crt \
  /etc/kubernetes/pki/apiserver.crt

# 查看证书主题和颁发者
openssl x509 -in /etc/kubernetes/pki/apiserver.crt -noout -subject -issuer
```

### 10.2 kubeadm 证书管理命令

```bash
# 查看证书到期时间
kubeadm certs check-expiration

# 输出示例

apiserver                  Dec 22, 2026 08:54 UTC   284d            ca                      no
apiserver-etcd-client      Dec 22, 2026 08:54 UTC   284d            etcd-ca                 no
apiserver-kubelet-client   Dec 22, 2026 08:54 UTC   284d            ca                      no
controller-manager.conf    Dec 22, 2026 08:54 UTC   284d            ca                      no
etcd-healthcheck-client    Dec 22, 2026 08:54 UTC   284d            etcd-ca                 no
etcd-peer                  Dec 22, 2026 08:54 UTC   284d            etcd-ca                 no
etcd-server                Dec 22, 2026 08:54 UTC   284d            etcd-ca                 no

# 续订所有证书
kubeadm certs renew all

# 续订特定证书
kubeadm certs renew apiserver
kubeadm certs renew etcd-server

# 生成新的 CA 证书（谨慎操作）
kubeadm certs generate-csr
```

### 10.3 手动创建 CSR 并签发证书

```bash
# 1. 生成私钥
openssl genrsa -out myuser.key 2048

# 2. 创建 CSR
openssl req -new -key myuser.key -out myuser.csr -subj "/CN=myuser/O=mygroup"

# 3. 创建 Kubernetes CSR 对象
cat <<EOF | kubectl apply -f -
apiVersion: certificates.k8s.io/v1
kind: CertificateSigningRequest
metadata:
  name: myuser
spec:
  request: $(cat myuser.csr | base64 | tr -d '\n')
  signerName: kubernetes.io/kube-apiserver-client
  usages:
  - client auth
EOF

# 4. 批准 CSR
kubectl certificate approve myuser

# 5. 获取签发的证书
kubectl get csr myuser -o jsonpath='{.status.certificate}' | base64 -d > myuser.crt
```

## 十一、常见问题与最佳实践

### 11.1 常见问题

**Q1：证书过期后集群会怎样？**

A：证书过期后，组件间通信将失败。具体表现：
- API Server 无法启动
- kubectl 命令无法连接集群
- kubelet 无法与 API Server 通信
- etcd 集群通信中断

**Q2：如何提前发现证书即将过期？**

A：建议部署证书监控：

```yaml
# 使用 Prometheus 监控证书过期时间
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: cert-expiry-alert
spec:
  groups:
  - name: cert-expiry
    rules:
    - alert: CertificateExpiringSoon
      expr: x509_cert_not_after - time() < 86400 * 30
      for: 1h
      labels:
        severity: warning
      annotations:
        summary: "Certificate expiring in less than 30 days"
```

**Q3：续订证书后需要重启哪些组件？**

A：续订证书后需要重启控制平面组件：

```bash
# 重启控制平面 Pod
kubectl -n kube-system get pods -o name | grep -E 'kube-apiserver|kube-controller|kube-scheduler|etcd' | xargs kubectl -n kube-system delete

# 重启 kubelet
systemctl restart kubelet
```

**Q4：kubelet 证书自动轮换失败怎么办？**

A：检查以下配置：

```bash
# 查看 kubelet 配置
ps aux | grep kubelet | grep rotate

# 检查 CSR 状态
kubectl get csr

# 查看 kubelet 日志
journalctl -u kubelet -f | grep -i cert
```

**Q5：如何备份和恢复证书？**

A：备份整个 PKI 目录：

```bash
# 备份
tar -czf k8s-pki-backup-$(date +%Y%m%d).tar.gz /etc/kubernetes/pki

# 恢复
tar -xzf k8s-pki-backup-20260312.tar.gz -C /
```

### 11.2 最佳实践

1. **定期检查证书有效期**
   ```bash
   # 添加到 cron 定时任务
   0 0 * * * kubeadm certs check-expiration >> /var/log/cert-check.log 2>&1
   ```

2. **启用证书自动轮换**
   ```yaml
   # kubelet 配置
   featureGates:
     RotateKubeletServerCertificate: true
   ```

3. **使用外部 CA（生产环境推荐）**
   - 将 CA 私钥离线存储
   - 使用 Vault 等工具管理证书
   - 实现证书自动签发流程

4. **证书权限管理**
   ```bash
   # 确保证书文件权限正确
   chmod 600 /etc/kubernetes/pki/*.key
   chown root:root /etc/kubernetes/pki/*.key
   ```

5. **多集群证书管理**
   - 使用 cert-manager 自动化管理证书
   - 集成企业 PKI 系统
   - 统一证书策略

6. **审计证书使用**
   ```bash
   # 查看证书使用情况
   kubectl get clusterrolebinding -o yaml | grep -A 5 "system:masters"
   ```

## 十二、证书安全加固建议

### 12.1 密钥长度和算法

```bash
# 推荐配置
- RSA 密钥长度：至少 2048 位（推荐 4096 位）
- ECDSA：使用 P-256 或 P-384 曲线
- 签名算法：SHA-256 或更高
```

### 12.2 证书撤销机制

Kubernetes 不直接支持 CRL（证书撤销列表），但可以通过以下方式实现：

```bash
# 删除用户证书对应的权限
kubectl delete clusterrolebinding <username>-binding

# 或者使用 OPA/Gatekeeper 实现更细粒度的控制
```

### 12.3 网络策略限制

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-server-ingress
  namespace: kube-system
spec:
  podSelector:
    matchLabels:
      component: kube-apiserver
  policyTypes:
  - Ingress
  ingress:
  - from:
    - ipBlock:
        cidr: 10.0.0.0/8  # 限制访问来源
    ports:
    - protocol: TCP
      port: 6443
```

## 十三、面试回答

**面试官问：请介绍一下 Kubernetes 中的证书和私钥种类？**

**回答**：Kubernetes 采用分层 PKI 体系，主要包含三类 CA 证书。首先是 kubernetes-ca，这是集群的核心信任锚点，负责签发 API Server、kubelet、Controller Manager 等大部分组件证书。其次是 etcd-ca，专门为 etcd 集群提供独立的信任域，签发 etcd 的服务器、客户端和 peer 证书。最后是 front-proxy-ca，用于聚合 API 的代理层认证。

从组件角度看，API Server 拥有三套证书：服务端证书用于对外服务，etcd 客户端证书用于访问 etcd，kubelet 客户端证书用于访问节点。kubelet 证书支持自动轮换机制，通过 CSR 向 API Server 申请更新。Service Account 使用非对称密钥对，私钥由 Controller Manager 签发 Token，公钥由 API Server 验证签名。证书默认有效期 1 年，CA 证书 10 年，可通过 kubeadm certs 命令管理和续订。生产环境建议启用自动轮换、部署证书监控、使用外部 CA 加固安全性。
