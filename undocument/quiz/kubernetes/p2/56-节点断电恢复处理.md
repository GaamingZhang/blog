# Kubernetes 节点断电恢复处理：Pod 的命运与恢复机制

## 引言：节点断电的影响

在 Kubernetes 生产环境中，节点断电是一种常见但影响严重的故障场景。当某个 Node 节点突然断电时，该节点上运行的所有 Pod 将立即停止工作，这可能导致：

- **服务中断**：运行在该节点上的业务服务不可用
- **数据风险**：未持久化的数据可能丢失
- **集群状态异常**：集群调度器需要重新分配资源
- **应用恢复延迟**：有状态应用可能需要较长时间恢复

理解 Kubernetes 如何处理节点断电以及 Pod 的恢复机制，对于保障生产环境的高可用性至关重要。

## 一、节点断电后 Kubernetes 的响应机制

### 1.1 节点状态检测机制

Kubernetes 通过 kubelet 与 API Server 的心跳机制来监控节点状态：

| 检测机制 | 默认参数 | 说明 |
|---------|---------|------|
| Node Status Update | 每 10 秒 | kubelet 定期向 API Server 上报节点状态 |
| Node Lease | 每 10 秒 | kubelet 更新 Lease 对象以证明节点存活 |
| Node Monitor Grace Period | 40 秒 | 节点控制器等待节点状态更新的宽限期 |
| Pod Eviction Timeout | 5 分钟 | 节点 NotReady 后 Pod 被驱逐的超时时间 |

### 1.2 节点状态转换流程

当节点断电后，状态转换遵循以下流程：

```
正常状态 (Ready)
    ↓
心跳超时 (40秒内无响应)
    ↓
Unknown 状态
    ↓
达到 PodEvictionTimeout (默认5分钟)
    ↓
触发 Pod 驱逐流程
```

**关键参数配置**（kube-controller-manager）：

```bash
--node-monitor-grace-period=40s        # 节点状态监控宽限期
--pod-eviction-timeout=5m              # Pod 驱逐超时时间
--node-monitor-period=5s               # 节点控制器同步周期
```

### 1.3 节点控制器的角色

节点控制器（Node Controller）负责：

1. **监控节点健康状态**：定期检查节点心跳
2. **更新节点 Condition**：将节点标记为 Unknown
3. **触发 Pod 驱逐**：超时后创建驱逐请求
4. **维护节点列表**：同步节点信息到 etcd

## 二、Pod 的驱逐和重建过程

### 2.1 驱逐机制详解

当节点断电超过 `pod-eviction-timeout` 后，Kubernetes 会触发 Pod 驱逐流程：

**驱逐原则**：

| Pod 类型 | 驱逐行为 | 重建位置 |
|---------|---------|---------|
| DaemonSet Pod | 不驱逐 | 节点恢复后自动运行 |
| Static Pod | 不驱逐 | 由 kubelet 直接管理 |
| ReplicaSet/Deployment Pod | 驱逐并重建 | 调度到其他健康节点 |
| StatefulSet Pod | 按序驱逐 | 严格按序号重建 |
| 裸 Pod | 直接删除 | 不会重建 |

### 2.2 Pod 重建流程

以 Deployment 为例，Pod 重建流程如下：

```
节点断电
    ↓
节点状态变为 Unknown
    ↓
超过 pod-eviction-timeout (5分钟)
    ↓
节点控制器创建 Eviction 对象
    ↓
API Server 删除原 Pod (设置 DeletionTimestamp)
    ↓
Controller Manager 检测到 Pod 数量不足
    ↓
ReplicaSet Controller 创建新 Pod
    ↓
Scheduler 调度到健康节点
    ↓
kubelet 拉取镜像并启动容器
    ↓
就绪探针通过，Pod Ready
```

### 2.3 优雅终止与强制删除

**正常情况下的优雅终止**：

```yaml
spec:
  terminationGracePeriodSeconds: 30  # 默认30秒
```

**节点断电时的特殊情况**：

- 节点断电属于**硬故障**，kubelet 无法执行优雅终止
- Pod 的 `preStop` 钩子不会被调用
- 容器内的进程被强制终止
- 临时卷（emptyDir）数据丢失

**强制删除 Pod 的方法**（谨慎使用）：

```bash
kubectl delete pod <pod-name> --force --grace-period=0
```

## 三、节点恢复后的处理流程

### 3.1 节点重新上线流程

当断电节点恢复供电并重启后：

```
节点启动
    ↓
kubelet 服务启动
    ↓
向 API Server 发送心跳
    ↓
节点状态从 Unknown → Ready
    ↓
清理残留的 Pod 对象
    ↓
节点可接受新 Pod 调度
```

### 3.2 残留 Pod 的清理

节点恢复后，kubelet 会执行以下清理操作：

**清理机制**：

1. **对比实际状态与期望状态**：检查本地运行的容器与 API Server 中的 Pod 定义
2. **删除孤立容器**：清理已经不存在于 API Server 的容器
3. **清理残留卷**：移除不再挂载的卷
4. **更新节点状态**：重新上报节点资源信息

**相关参数**（kubelet）：

```bash
--maximum-dead-containers=0           # 保留的死容器数量
--minimum-container-ttl-duration=0s   # 死容器最小存活时间
--volume-stats-agg-period=1m          # 卷统计聚合周期
```

### 3.3 节点恢复后的调度行为

| 场景 | 行为 |
|-----|------|
| 原有 Pod 已重建到其他节点 | 恢复节点不接管，继续运行在新节点 |
| DaemonSet Pod | 自动在新节点上启动 |
| 节点资源充足 | 可接受新的调度请求 |
| 节点存在污点 | 需要容忍污点才能调度 |

## 四、数据持久化 Pod 的处理

### 4.1 PV/PVC 绑定关系

对于使用持久化存储的 Pod，处理机制如下：

**存储类型影响**：

| 存储类型 | 数据保留 | Pod 重建后行为 |
|---------|---------|--------------|
| emptyDir | 丢失 | 数据无法恢复 |
| hostPath | 保留 | 节点恢复后可访问 |
| NFS/云盘 | 保留 | 新 Pod 可重新挂载 |
| Local PV | 保留 | 必须调度回原节点 |

### 4.2 Local PV 的特殊处理

Local PV 绑定到特定节点，Pod 重建时必须调度回原节点：

**节点亲和性约束**：

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: local-pv
spec:
  capacity:
    storage: 10Gi
  accessModes:
  - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: local-storage
  local:
    path: /mnt/data
  nodeAffinity:
    required:
      nodeSelectorTerms:
      - matchExpressions:
        - key: kubernetes.io/hostname
          operator: In
          values:
          - node-1  # 绑定到特定节点
```

**节点断电时的影响**：

1. Pod 被驱逐，但 PVC 仍然绑定到 Local PV
2. 新 Pod 处于 Pending 状态，等待节点恢复
3. 节点恢复后，Pod 自动调度回原节点并挂载数据

### 4.3 数据一致性保障

**最佳实践**：

1. **使用 ReadWriteOnce 模式**：确保单节点独占访问
2. **配置适当的回收策略**：Retain 保留数据，Delete 自动清理
3. **定期备份**：使用 Velero 等工具备份 PV 数据
4. **应用层一致性**：数据库使用事务和 WAL 机制

## 五、有状态应用的恢复

### 5.1 StatefulSet 的恢复机制

StatefulSet 是有状态应用的典型控制器，其恢复机制具有以下特点：

**核心特性**：

- **稳定的网络标识**：Pod 名称为 `<statefulset-name>-<ordinal>`
- **稳定的持久化存储**：每个 Pod 绑定独立的 PVC
- **有序部署和扩展**：按序号顺序操作
- **有序删除和终止**：逆序删除

**恢复流程**：

```
节点断电
    ↓
StatefulSet Pod 标记为 Terminating
    ↓
超过 terminationGracePeriodSeconds
    ↓
强制删除 Pod
    ↓
StatefulSet Controller 检测到 Pod 缺失
    ↓
按序号顺序重建 Pod
    ↓
重新绑定原有 PVC
    ↓
应用启动并恢复数据
```

### 5.2 有状态应用的恢复策略

**数据库类应用**（如 MySQL、PostgreSQL）：

| 恢复阶段 | 操作 | 注意事项 |
|---------|------|---------|
| 启动前 | 检查数据完整性 | 使用 fsck 或数据库工具 |
| 启动时 | 恢复事务日志 | 重放 WAL 或 binlog |
| 启动后 | 验证数据一致性 | 执行一致性检查命令 |
| 加入集群 | 同步数据 | 与主节点或副本同步 |

**消息队列类应用**（如 Kafka、RabbitMQ）：

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: kafka
spec:
  serviceName: kafka-headless
  replicas: 3
  podManagementPolicy: OrderedReady  # 有序管理
  template:
    spec:
      containers:
      - name: kafka
        image: kafka:latest
        env:
        - name: KAFKA_BROKER_ID
          valueFrom:
            fieldRef:
              fieldPath: metadata.name  # 使用 Pod 序号作为 Broker ID
```

### 5.3 PodDisruptionBudget 的作用

PDB 可以限制同时中断的 Pod 数量，但在节点断电场景下作用有限：

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: web
```

**PDB 的局限性**：

- 仅对**主动驱逐**（如 kubectl drain）生效
- 对**节点故障**（如断电）无法阻止 Pod 被驱逐
- 主要用于维护期间的可用性保障

## 七、常见问题和最佳实践

### 7.1 常见问题

**Q1: 为什么节点断电后 Pod 不是立即迁移？**

A: Kubernetes 设计了 `pod-eviction-timeout`（默认 5 分钟）来避免网络抖动等临时故障导致的频繁迁移。这给节点恢复留出了时间窗口，减少不必要的 Pod 重建。

**Q2: 如何加快 Pod 的恢复速度？**

A: 可以通过以下方式优化：
- 调整 `pod-eviction-timeout` 为更短时间（如 1 分钟）
- 配置 Pod 优先级和抢占机制
- 使用预热镜像（image pre-pull）
- 优化应用启动时间

**Q3: Local PV 绑定的 Pod 如何处理？**

A: Local PV 具有节点亲和性，Pod 必须调度到原节点。节点断电期间，Pod 会一直处于 Pending 状态，直到节点恢复。建议为关键应用配置多副本。

**Q4: 节点恢复后，原节点上的 Pod 会迁移回来吗？**

A: 不会。已经重建到其他节点的 Pod 会继续在新节点运行，不会自动迁移回原节点。除非手动触发重新调度。

**Q5: 如何避免节点断电导致的服务中断？**

A: 采用以下高可用策略：
- 多副本部署（Replicas ≥ 3）
- 跨节点、跨可用区分布
- 配置 Pod 反亲和性
- 使用 PDB 保障最小可用副本数

### 7.2 最佳实践

#### 7.2.1 集群配置优化

**调整驱逐超时时间**：

```bash
# kube-controller-manager 配置
--pod-eviction-timeout=60s           # 降低为 1 分钟
--node-monitor-grace-period=20s      # 降低节点监控宽限期
--terminated-pod-gc-threshold=1000   # 清理已终止的 Pod
```

**配置节点污点自动添加**：

```bash
# kube-controller-manager 配置
--feature-gates=TaintBasedEvictions=true
```

节点 NotReady 时自动添加污点：

```yaml
# 节点污点示例
taints:
- key: node.kubernetes.io/unreachable
  operator: Exists
  effect: NoExecute
```

#### 7.2.2 应用配置优化

**配置合理的探针**：

```yaml
livenessProbe:
  httpGet:

```

**配置 Pod 反亲和性**：

```yaml

```

### 7.3 故障排查步骤

**节点断电后的排查流程**：

```bash
# 1. 检查节点状态
kubectl get nodes
kubectl describe node <node-name>

# 2. 检查节点事件
kubectl get events --field-selector involvedObject.kind=Node

# 3. 检查 Pod 状态
kubectl get pods --all-namespaces -o wide | grep <node-name>
kubectl get pods --field-selector spec.nodeName=<node-name>

# 4. 检查 Pod 事件
kubectl describe pod <pod-name>

# 5. 检查 PVC 绑定状态
kubectl get pvc -n <namespace>
kubectl describe pvc <pvc-name>

# 6. 检查存储卷状态
kubectl get pv
kubectl describe pv <pv-name>

# 7. 查看控制器日志
kubectl logs -n kube-system kube-controller-manager-<master-node>

# 8. 查看调度器日志
kubectl logs -n kube-system kube-scheduler-<master-node>
```

**强制删除卡在 Terminating 的 Pod**：

```bash
# 查看卡住的 Pod
kubectl get pods --field-selector metadata.deletionTimestamp!=null

# 强制删除
kubectl delete pod <pod-name> --force --grace-period=0 -n <namespace>

# 批量删除
kubectl get pods --field-selector metadata.deletionTimestamp!=null -n <namespace> -o name | xargs kubectl delete --force --grace-period=0 -n <namespace>
```

## 八、面试回答

**面试官问：Kubernetes 中一个节点突然断电，恢复后上面的 Pod 怎么办？**

**回答**：

节点断电后，Kubernetes 会通过节点控制器检测到节点心跳超时（默认 40 秒），将节点状态标记为 Unknown。经过 `pod-eviction-timeout`（默认 5 分钟）后，系统会触发 Pod 驱逐流程。

对于不同类型的 Pod，处理方式不同：
- **Deployment/ReplicaSet 管理的 Pod**：会被驱逐并在其他健康节点重建，emptyDir 数据丢失，持久化数据通过 PVC 保留
- **StatefulSet Pod**：按序号顺序重建，重新绑定原有 PVC，保证数据持久化和网络标识的稳定性
- **DaemonSet Pod**：不会被驱逐，节点恢复后自动启动
- **Local PV 绑定的 Pod**：必须等待原节点恢复，因为存储有节点亲和性

节点恢复后，kubelet 会重新上报心跳，节点状态从 Unknown 变为 Ready，并清理残留的 Pod 对象。已经迁移到其他节点的 Pod 不会自动迁回。

为了减少节点断电的影响，建议采用多副本部署、跨节点分布、配置合理的探针和优先级，并设置监控告警及时发现节点故障。关键应用应使用持久化存储并定期备份数据。
