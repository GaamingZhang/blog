# Kubernetes 热更新

## 引言

热更新是指在不重启应用的情况下更新配置或代码。在 Kubernetes 中，通过 ConfigMap、Secret 等机制可以实现配置的热更新，结合应用的热加载能力，实现配置变更的无缝生效。

## ConfigMap 热更新

### ConfigMap 配置

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: production
data:
  app.properties: |
    server.port=8080
    log.level=info
    cache.enabled=true
```

### 挂载 ConfigMap

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: myapp:v1
    volumeMounts:
    - name: config
      mountPath: /etc/app
      readOnly: true
  volumes:
  - name: config
    configMap:
      name: app-config
```

### 更新 ConfigMap

```bash
kubectl edit configmap app-config -n production

kubectl apply -f configmap.yaml

kubectl create configmap app-config --from-file=app.properties -n production --dry-run=client -o yaml | kubectl apply -f -
```

### ConfigMap 更新机制

```
┌─────────────────────────────────────────────────────────────┐
│              ConfigMap 更新机制                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  更新方式：                                                  │
│  • 原子更新：整个文件更新                                    │
│  • 符号链接切换：保证一致性                                  │
│                                                              │
│  更新延迟：                                                  │
│  • kubelet 同步周期：默认 1 分钟                            │
│  • 加上缓存时间：约 1-2 分钟                                │
│                                                              │
│  注意事项：                                                  │
│  • 环境变量不会自动更新                                      │
│  • 需要应用支持热加载                                        │
│  • subPath 挂载不会自动更新                                  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## Secret 热更新

### Secret 配置

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: app-secret
  namespace: production
type: Opaque
stringData:
  db_password: "new-password"
```

### 挂载 Secret

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
  - name: app
    image: myapp:v1
    volumeMounts:
    - name: secret
      mountPath: /etc/secrets
      readOnly: true
  volumes:
  - name: secret
    secret:
      secretName: app-secret
```

### 更新 Secret

```bash
kubectl edit secret app-secret -n production

kubectl create secret generic app-secret --from-literal=db_password=new-password -n production --dry-run=client -o yaml | kubectl apply -f -
```

## 应用热加载实现

### 文件监听方式

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: hot-reload-app
spec:
  containers:
  - name: app
    image: myapp:v1
    volumeMounts:
    - name: config
      mountPath: /etc/app
    env:
    - name: CONFIG_PATH
      value: /etc/app/app.properties
    lifecycle:
      postStart:
        exec:
          command:
          - /bin/sh
          - -c
          - |
            while true; do
              inotifywait -e modify /etc/app/app.properties
              kill -HUP 1
            done &
  volumes:
  - name: config
    configMap:
      name: app-config
```

### 使用 Reloader 工具

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  annotations:
    reloader.stakater.com/auto: "true"
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: app
        image: myapp:v1
        volumeMounts:
        - name: config
          mountPath: /etc/app
      volumes:
      - name: config
        configMap:
          name: app-config
```

### 使用 ConfigMap 版本控制

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config-v2
data:
  app.properties: |
    server.port=8080
    log.level=debug
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    spec:
      containers:
      - name: app
        image: myapp:v1
        volumeMounts:
        - name: config
          mountPath: /etc/app
      volumes:
      - name: config
        configMap:
          name: app-config-v2
```

## 环境变量热更新

### 环境变量限制

```
┌─────────────────────────────────────────────────────────────┐
│              环境变量热更新限制                               │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  环境变量不会自动更新：                                      │
│  • Pod 创建时注入环境变量                                    │
│  • 更新 ConfigMap 不会更新环境变量                          │
│  • 需要重启 Pod 才能生效                                    │
│                                                              │
│  解决方案：                                                  │
│  • 使用 volumeMounts 挂载配置文件                           │
│  • 使用 Downward API 注入版本信息                           │
│  • 使用外部配置中心                                          │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 使用 Downward API

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: downward-api-pod
spec:
  containers:
  - name: app
    image: myapp:v1
    env:
    - name: POD_NAME
      valueFrom:
        fieldRef:
          fieldPath: metadata.name
    - name: CONFIG_VERSION
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: version
```

## 最佳实践

### 1. 使用 volumeMounts 挂载配置

```yaml
volumeMounts:
- name: config
  mountPath: /etc/app
  readOnly: true
```

### 2. 应用支持热加载

```yaml
env:
- name: HOT_RELOAD_ENABLED
  value: "true"
```

### 3. 使用 Reloader 自动重启

```yaml
metadata:
  annotations:
    reloader.stakater.com/auto: "true"
```

### 4. 配置版本控制

```yaml
metadata:
  name: app-config-v1
  labels:
    version: "1"
```

## 面试回答

**问题**: 怎样进行热更新？

**回答**: Kubernetes 热更新主要针对配置变更，实现不重启应用的配置更新：

**ConfigMap/Secret 热更新**：将 ConfigMap 或 Secret 以 volumeMounts 方式挂载到 Pod，更新 ConfigMap/Secret 后，kubelet 会自动更新 Pod 内的文件。更新延迟约 1-2 分钟（kubelet 同步周期 + 缓存时间）。

**热更新限制**：环境变量方式挂载的配置不会自动更新，需要重启 Pod；使用 subPath 挂载的文件不会自动更新；需要应用支持热加载能力。

**应用热加载实现**：**文件监听**应用监听配置文件变化，自动重新加载配置；**信号通知**应用监听 SIGHUP 等信号，收到信号后重新加载；**Reloader 工具**自动检测 ConfigMap/Secret 变化，触发 Pod 滚动更新。

**最佳实践**：使用 volumeMounts 挂载配置文件而非环境变量；应用实现配置热加载能力；使用 Reloader 等工具自动触发更新；对配置进行版本控制；使用外部配置中心（如 Nacos、Apollo）实现更灵活的热更新。

**注意事项**：热更新依赖应用支持；更新有延迟；需要测试验证配置变更的正确性；关键配置变更建议使用滚动更新确保一致性。
