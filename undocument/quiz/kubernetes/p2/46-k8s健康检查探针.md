# Kubernetes 健康检查探针详解：三种探针机制深度解析

## 引言：为什么 Kubernetes 需要健康检查探针？

在传统的应用部署中，运维人员通常依赖进程监控工具（如 systemd、supervisord）来判断服务是否正常运行。当进程退出时，这些工具会自动重启服务。然而，在容器化环境中，这种简单的进程监控机制存在明显的局限性。

**进程存活不代表应用健康**。容器可能处于运行状态，但内部的应用程序可能已经陷入死锁、内存溢出、线程池耗尽或无响应状态。这种情况下，容器进程依然存活，但应用实际上已经无法正常提供服务。如果 Kubernetes 无法识别这种"假死"状态，就会导致流量被路由到异常的 Pod，引发服务故障。

**应用就绪不代表可以接收流量**。应用启动后，可能需要加载配置、预热缓存、建立数据库连接、等待依赖服务等初始化工作。在此期间，应用虽然进程正常，但还无法处理请求。如果过早地将流量路由到未完全就绪的 Pod，会导致请求失败或超时。

Kubernetes 通过健康检查探针（Probe）机制解决了这些问题。探针是 kubelet 对容器执行的定期诊断，通过主动探测应用的健康状态，实现自动故障恢复和流量管理。Kubernetes 提供了三种探针机制，每种探针承担不同的职责，协同保障应用的高可用性。

## 三种探针机制概览

Kubernetes 提供了三种健康检查探针，每种探针都有其特定的职责和应用场景：

| 探针类型 | 引入版本 | 核心职责 | 失败后果 | 适用场景 |
|---------|---------|---------|---------|---------|
| **Liveness Probe** | v1.0 | 检测容器是否存活 | 重启容器 | 应用死锁、资源耗尽、不可恢复故障 |
| **Readiness Probe** | v1.0 | 检测容器是否就绪 | 隔离流量 | 应用初始化、依赖服务未就绪、临时过载 |
| **Startup Probe** | v1.16 | 检测容器是否启动完成 | 重启容器 | 启动时间长的应用、启动时间不确定的应用 |

这三种探针协同工作，构成了完整的健康检查体系：
- **Startup Probe** 负责判断应用是否启动完成，为启动慢的应用提供保护
- **Liveness Probe** 负责判断应用是否存活，处理运行时的严重故障
- **Readiness Probe** 负责判断应用是否就绪，控制流量路由

## Liveness Probe（存活探针）

### 作用

Liveness Probe 用于检测容器是否存活。当存活探针检测失败时，Kubernetes 会认为容器已经处于不可恢复的故障状态，并根据重启策略（RestartPolicy）重启容器。这是一种"硬重启"机制，适用于处理应用死锁、资源耗尽、线程池阻塞等无法自愈的严重故障。

### 工作原理

存活探针的工作流程如下：

1. **周期性探测**：kubelet 按照配置的时间间隔周期性执行探针检查
2. **状态判断**：根据探针返回结果判断容器健康状态
3. **失败处理**：连续失败次数达到阈值（failureThreshold）时，触发容器重启
4. **重启执行**：kubelet 杀死当前容器，并根据 RestartPolicy 创建新容器

存活探针的核心价值在于处理那些进程依然运行但应用逻辑已经失效的场景。例如，Java 应用可能因为内存溢出而进入假死状态，进程依然存在但无法响应任何请求。此时，存活探针能够检测到异常并触发重启。

### 配置方式

存活探针支持三种检测方式，每种方式适用于不同的应用场景。

#### HTTP GET 探测

HTTP GET 是最常用的探测方式，适用于提供 HTTP 服务的应用。kubelet 向指定路径发送 HTTP GET 请求，根据响应状态码判断健康状态。

**工作原理**：
- kubelet 向容器的指定端口和路径发送 HTTP GET 请求
- 如果响应状态码在 200-399 范围内，认为探测成功
- 如果响应状态码不在该范围或请求超时，认为探测失败

**配置示例**：

```yaml
livenessProbe:
  httpGet:
    path: /health/live
    port: 8080
    scheme: HTTP
    httpHeaders:
    - name: X-Health-Check
      value: liveness
  initialDelaySeconds: 30
  periodSeconds: 10
  timeoutSeconds: 5
  failureThreshold: 3
  successThreshold: 1
```

**实现细节**：
- HTTP 请求由 kubelet 在节点本地发起，不经过集群网络
- 支持 HTTP 和 HTTPS 协议（通过 scheme 字段配置）
- 可以添加自定义 HTTP 头部，用于区分健康检查请求和正常业务请求
- 健康检查端点应该轻量、快速，避免执行复杂逻辑

#### TCP Socket 探测

TCP Socket 探测适用于非 HTTP 服务，如数据库、Redis、消息队列等。kubelet 尝试与指定端口建立 TCP 连接，根据连接结果判断健康状态。

**工作原理**：
- kubelet 尝试与容器的指定端口建立 TCP 连接
- 如果连接建立成功，立即断开连接并认为探测成功
- 如果连接建立失败或超时，认为探测失败

**配置示例**：

```yaml
livenessProbe:
  tcpSocket:
    port: 3306
  initialDelaySeconds: 30
  periodSeconds: 10
  timeoutSeconds: 5
  failureThreshold: 3
```

**实现细节**：
- TCP 探测只验证端口可达性，不验证应用逻辑
- 适用于那些不提供 HTTP 接口但监听 TCP 端口的服务
- 探测成功后会立即断开连接，不会保持长连接
- 对于需要认证的服务（如数据库），TCP 探测只能验证端口监听状态

#### Exec 探测

Exec 探测通过在容器内执行命令来判断健康状态，适用于需要执行复杂检查逻辑的场景。kubelet 在容器内执行指定命令，根据命令退出码判断健康状态。

**工作原理**：
- kubelet 在容器内执行指定的命令
- 如果命令退出码为 0，认为探测成功
- 如果命令退出码非 0，认为探测失败

**配置示例**：

```yaml
livenessProbe:
  exec:
    command:
    - /bin/sh
    - -c
    - pg_isready -h localhost -U postgres
  initialDelaySeconds: 30
  periodSeconds: 10
  timeoutSeconds: 5
  failureThreshold: 3
```

**实现细节**：
- 命令在容器内的命名空间中执行，具有容器的文件系统和环境变量
- 命令的超时时间由 timeoutSeconds 控制，超时视为失败
- 命令应该快速返回，避免长时间运行
- 适用于需要检查文件状态、进程状态或执行自定义健康检查脚本的场景

### 使用场景

存活探针适用于以下场景：

1. **应用死锁检测**：应用可能因为锁竞争、资源等待等原因进入死锁状态，进程存活但无法响应请求
2. **内存溢出处理**：Java 应用可能因为内存溢出而进入假死状态，需要重启恢复
3. **线程池耗尽**：应用线程池可能因为慢请求堆积而耗尽，导致无法处理新请求
4. **资源耗尽恢复**：文件句柄、数据库连接等资源耗尽导致应用异常
5. **不可恢复故障**：应用进入无法自愈的故障状态，需要重启恢复

## Readiness Probe（就绪探针）

### 作用

Readiness Probe 用于检测容器是否准备好接收流量。当就绪探针检测失败时，Kubernetes 会将 Pod 从 Service 的 Endpoints 中移除，停止向该 Pod 发送请求，但不会重启容器。这是一种"软隔离"机制，适用于处理应用临时不可用、依赖服务未就绪等可恢复的故障。

### 工作原理

就绪探针的工作流程如下：

1. **周期性探测**：kubelet 按照配置的时间间隔周期性执行探针检查
2. **状态判断**：根据探针返回结果判断容器是否就绪
3. **端点管理**：探测失败时，将 Pod IP 从 Service 的 Endpoints 中移除
4. **恢复处理**：探测成功后，将 Pod IP 重新加入 Service 的 Endpoints

就绪探针的核心价值在于流量控制。应用启动时可能需要加载配置、预热缓存、建立数据库连接等初始化工作。在此期间，应用虽然进程正常，但还无法处理请求。就绪探针确保只有真正准备好的 Pod 才会接收流量。

### 配置方式

就绪探针同样支持三种检测方式，配置方式与存活探针类似。

#### HTTP GET 探测

```yaml
readinessProbe:
  httpGet:
    path: /health/ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
  timeoutSeconds: 3
  failureThreshold: 3
  successThreshold: 1
```

#### TCP Socket 探测

```yaml
readinessProbe:
  tcpSocket:
    port: 6379
  initialDelaySeconds: 5
  periodSeconds: 5
  timeoutSeconds: 3
  failureThreshold: 3
```

#### Exec 探测

```yaml
readinessProbe:
  exec:
    command:
    - /bin/sh
    - -c
    - redis-cli ping | grep PONG
  initialDelaySeconds: 5
  periodSeconds: 5
  timeoutSeconds: 3
  failureThreshold: 3
```

### 使用场景

就绪探针适用于以下场景：

1. **应用初始化**：应用启动后需要加载配置、预热缓存、建立连接等初始化工作
2. **依赖服务检查**：应用依赖数据库、缓存、消息队列等服务，需要等待这些服务就绪
3. **临时过载处理**：应用可能因为请求量过大而临时过载，需要暂时隔离流量
4. **优雅降级**：应用检测到依赖服务不可用时，主动标记为未就绪，避免接收请求
5. **滚动更新保障**：确保新 Pod 完全就绪后，才终止旧 Pod

### 就绪探针的特殊行为

与存活探针不同，就绪探针有以下特殊行为：

1. **初始状态**：Pod 启动时，默认处于未就绪状态，直到就绪探针首次成功
2. **流量隔离**：探测失败不会重启容器，只是隔离流量
3. **恢复机制**：探测成功后会自动恢复流量，无需重启
4. **滚动更新**：新 Pod 必须通过就绪检查后，才会终止旧 Pod
5. **successThreshold 可配置**：可以要求连续成功多次才认为就绪

## Startup Probe（启动探针）

### 作用

Startup Probe 是 Kubernetes 1.16 引入的新特性，用于检测容器是否启动完成。当启动探针检测失败时，Kubernetes 会根据重启策略重启容器。Startup Probe 的核心价值在于为启动时间长的应用提供保护，避免因为存活探针过早检查导致应用在启动过程中被误判为不健康而重启。

### 工作原理

启动探针的工作流程如下：

1. **启动阶段优先**：Pod 启动后，首先执行启动探针检查
2. **禁用其他探针**：启动探针成功前，存活探针和就绪探针不会执行
3. **启动完成判断**：启动探针成功后，认为应用启动完成
4. **切换到常规探针**：启动探针成功后，开始执行存活探针和就绪探针

启动探针的核心价值在于处理启动时间不确定的应用。对于大型 Java 应用、机器学习模型加载、数据库初始化等场景，应用启动时间可能长达数分钟。如果只依赖存活探针的 initialDelaySeconds，需要设置很长的延迟时间，这会影响故障发现速度。Startup Probe 通过独立的启动检查机制，解决了这个矛盾。

### 配置方式

Startup Probe 同样支持三种检测方式，配置方式与其他探针类似。

#### HTTP GET 探测

```yaml
startupProbe:
  httpGet:
    path: /health/startup
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 10
  timeoutSeconds: 5
  failureThreshold: 30    # 最多等待 30 * 10 = 300 秒
  successThreshold: 1
```

#### TCP Socket 探测

```yaml
startupProbe:
  tcpSocket:
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 10
  timeoutSeconds: 5
  failureThreshold: 30
```

#### Exec 探测

```yaml
startupProbe:
  exec:
    command:
    - /bin/sh
    - -c
    - test -f /app/initialized
  initialDelaySeconds: 10
  periodSeconds: 10
  timeoutSeconds: 5
  failureThreshold: 30
```

### 使用场景

启动探针适用于以下场景：

1. **启动时间长的应用**：大型 Java 应用启动时间可能长达 2-3 分钟
2. **启动时间不确定的应用**：应用启动时间受数据量、网络状况等因素影响
3. **模型加载场景**：机器学习应用需要加载大型模型文件
4. **数据库初始化**：数据库启动后需要执行初始化脚本、数据迁移等操作
5. **避免启动期误判**：避免存活探针在应用启动过程中误判为不健康

### Startup Probe 与 Liveness Probe 的协同

Startup Probe 和 Liveness Probe 协同工作的机制如下：

1. **启动阶段**：Pod 启动后，首先执行 Startup Probe，Liveness Probe 和 Readiness Probe 暂不执行
2. **启动完成**：Startup Probe 成功后，认为应用启动完成，开始执行 Liveness Probe 和 Readiness Probe
3. **运行阶段**：Startup Probe 不再执行，Liveness Probe 和 Readiness Probe 持续运行

这种机制的优势在于：
- Startup Probe 可以配置较长的 failureThreshold，容忍应用启动时间长
- Liveness Probe 可以配置较短的 initialDelaySeconds 和 periodSeconds，快速发现运行时故障
- 避免了为 Liveness Probe 设置过长的 initialDelaySeconds 导致故障发现延迟

## 三种检测方式详解

### HTTP GET 探测

HTTP GET 探测是最常用的检测方式，适用于提供 HTTP/HTTPS 服务的应用。

**探测机制**：
- kubelet 向容器的指定端口和路径发送 HTTP GET 请求
- 请求由 kubelet 在节点本地发起，不经过集群网络
- 支持 HTTP 和 HTTPS 协议
- 可以添加自定义 HTTP 头部

**判断标准**：
- 响应状态码在 200-399 范围内：探测成功
- 响应状态码不在该范围：探测失败
- 请求超时：探测失败
- 连接失败：探测失败

**配置参数**：

```yaml
httpGet:
  path: /health        # 必填：请求路径
  port: 8080          # 必填：端口号（名称或数字）
  scheme: HTTP        # 可选：HTTP 或 HTTPS，默认 HTTP
  httpHeaders:        # 可选：自定义 HTTP 头部
  - name: X-Custom-Header
    value: health-check
```

**最佳实践**：
- 使用专门的轻量级健康检查端点
- 避免在健康检查中执行复杂逻辑或数据库查询
- 响应时间应控制在 1 秒以内
- 可以通过 HTTP 头部区分健康检查请求和业务请求

### TCP Socket 探测

TCP Socket 探测适用于非 HTTP 服务，如数据库、缓存、消息队列等。

**探测机制**：
- kubelet 尝试与容器的指定端口建立 TCP 连接
- 连接建立成功后立即断开，不发送任何数据
- 探测的是端口可达性，而非应用逻辑

**判断标准**：
- TCP 连接建立成功：探测成功
- TCP 连接建立失败：探测失败
- 连接超时：探测失败

**配置参数**：

```yaml
tcpSocket:
  port: 3306    # 必填：端口号（名称或数字）
```

**最佳实践**：
- 适用于不提供 HTTP 接口但监听 TCP 端口的服务
- 只能验证端口监听状态，无法验证应用逻辑
- 对于需要认证的服务，建议使用 Exec 探测
- 探测成功后会立即断开连接，不会保持长连接

### Exec 探测

Exec 探测通过在容器内执行命令来判断健康状态，适用于需要执行复杂检查逻辑的场景。

**探测机制**：
- kubelet 在容器内执行指定的命令
- 命令在容器的命名空间中执行，具有容器的文件系统和环境变量
- 命令执行超时由 timeoutSeconds 控制

**判断标准**：
- 命令退出码为 0：探测成功
- 命令退出码非 0：探测失败
- 命令执行超时：探测失败

**配置参数**：

```yaml
exec:
  command:        # 必填：要执行的命令
  - /bin/sh
  - -c
  - pg_isready -h localhost
```

**最佳实践**：
- 命令应该快速返回，避免长时间运行
- 可以检查文件状态、进程状态或执行自定义健康检查脚本
- 可以利用容器内的工具（如 mysqladmin、redis-cli）进行健康检查
- 命令输出不会影响探测结果，只有退出码有影响

## 探针参数详解

Kubernetes 为探针提供了丰富的参数配置，用于精细控制探测行为。

### 参数列表

| 参数 | 默认值 | 说明 |
|------|--------|------|
| **initialDelaySeconds** | 0 | 容器启动后等待多少秒才开始探测 |
| **periodSeconds** | 10 | 探测的间隔时间（秒） |
| **timeoutSeconds** | 1 | 探测的超时时间（秒） |
| **failureThreshold** | 3 | 连续探测失败多少次才认为探测失败 |
| **successThreshold** | 1 | 连续探测成功多少次才认为探测成功（仅对 Readiness Probe 和 Startup Probe 有效） |

### 参数配置策略

#### initialDelaySeconds（初始延迟）

**作用**：容器启动后等待多少秒才开始探测，避免在应用未完全启动时就开始检查。

**配置策略**：
- 根据应用的实际启动时间设置
- 对于 Java 等启动较慢的应用，通常需要 30-60 秒
- 对于 Go 等启动较快的应用，通常需要 5-15 秒
- 如果使用 Startup Probe，Liveness Probe 的 initialDelaySeconds 可以设置为 0 或较小值
- 可以通过观察应用启动日志，测量从容器启动到应用完全就绪的时间

**示例**：

```yaml
# 启动快的应用
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 10

# 启动慢的应用
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 60
  periodSeconds: 10
```

#### periodSeconds（探测间隔）

**作用**：探测的间隔时间，决定了健康检查的频率。

**配置策略**：
- 对于关键服务，可以设置较短的间隔（如 5-10 秒），快速发现故障
- 对于非关键服务，可以设置较长的间隔（如 30-60 秒），减少系统开销
- 需要在检测及时性和系统开销之间平衡
- 过短的间隔会增加 kubelet 和应用的负担

**示例**：

```yaml
# 关键服务：快速检测
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  periodSeconds: 5

# 非关键服务：减少开销
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  periodSeconds: 30
```

#### timeoutSeconds（超时时间）

**作用**：探测的超时时间，超时视为探测失败。

**配置策略**：
- 默认值为 1 秒，对于大多数应用足够
- 如果健康检查需要执行复杂逻辑，可以适当增加
- 对于网络不稳定的环境，可以适当增加
- 超时时间过长会延迟故障发现

**示例**：

```yaml
# 轻量级健康检查
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  timeoutSeconds: 1

# 需要执行复杂逻辑的健康检查
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  timeoutSeconds: 5
```

#### failureThreshold（失败阈值）

**作用**：连续探测失败多少次才认为探测失败，提供一定的容错空间。

**配置策略**：
- 默认值为 3 次，提供一定的容错空间
- 对于网络波动较大的环境，可以适当增加（如 5 次）
- 对于关键服务，可以适当减少（如 2 次），快速发现故障
- Startup Probe 通常需要较大的值（如 30 次），容忍应用启动时间长

**示例**：

```yaml
# 普通应用
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  failureThreshold: 3

# 网络不稳定环境
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  failureThreshold: 5

# 启动慢的应用（Startup Probe）
startupProbe:
  httpGet:
    path: /health
    port: 8080
  periodSeconds: 10
  failureThreshold: 30    # 最多等待 300 秒
```

#### successThreshold（成功阈值）

**作用**：连续探测成功多少次才认为探测成功。仅对 Readiness Probe 和 Startup Probe 有效，Liveness Probe 固定为 1。

**配置策略**：
- 默认值为 1 次
- 对于 Readiness Probe，可以设置为 2-3 次，确保应用真正稳定后再接收流量
- 对于 Startup Probe，通常设置为 1 次，启动成功即可
- 较大的值会增加应用就绪的延迟

**示例**：

```yaml
# 普通应用
readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  successThreshold: 1

# 需要确保稳定的应用
readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  successThreshold: 2    # 连续成功 2 次才认为就绪
```

## 三种探针对比

### 功能对比

| 对比维度 | Liveness Probe | Readiness Probe | Startup Probe |
|----------|----------------|-----------------|---------------|
| **核心作用** | 检测容器是否存活 | 检测容器是否就绪 | 检测容器是否启动完成 |
| **失败后果** | 重启容器 | 从 Service Endpoints 中移除 | 重启容器 |
| **适用场景** | 应用死锁、资源耗尽、不可恢复故障 | 应用初始化、依赖服务未就绪、临时过载 | 启动时间长的应用、启动时间不确定的应用 |
| **恢复方式** | 容器重启，应用重新初始化 | 自动恢复，无需重启 | 容器重启 |
| **初始状态** | 默认健康 | 默认未就绪 | 默认未启动完成 |
| **successThreshold** | 固定为 1 | 可配置，默认为 1 | 可配置，默认为 1 |
| **执行时机** | Startup Probe 成功后 | Startup Probe 成功后 | Pod 启动后立即执行 |
| **引入版本** | v1.0 | v1.0 | v1.16 |

### 协同工作机制

三种探针协同工作的完整流程如下：

1. **Pod 启动阶段**：
   - Pod 启动后，首先执行 Startup Probe
   - Liveness Probe 和 Readiness Probe 暂不执行
   - Startup Probe 持续探测，直到成功或达到 failureThreshold

2. **启动完成阶段**：
   - Startup Probe 成功后，认为应用启动完成
   - 开始执行 Liveness Probe 和 Readiness Probe
   - Startup Probe 不再执行

3. **初始化阶段**：
   - 应用进行初始化（加载配置、预热缓存、建立连接等）
   - Readiness Probe 持续失败，Pod 不接收流量
   - Liveness Probe 成功，容器不会被重启

4. **就绪阶段**：
   - 应用初始化完成，Readiness Probe 成功
   - Pod 开始接收流量
   - Liveness Probe 和 Readiness Probe 持续运行

5. **运行阶段**：
   - Liveness Probe 保障容器健康
   - Readiness Probe 保障服务可用
   - 两种探针协同工作，共同保障应用稳定性

6. **故障阶段**：
   - **临时故障**（如数据库连接超时）：Readiness Probe 失败，流量隔离，等待恢复
   - **严重故障**（如死锁）：Liveness Probe 失败，容器重启

**端点实现建议**：

```go
// /health/startup - 启动探针
// 检查应用是否完成启动初始化
func startupHandler(w http.ResponseWriter, r *http.Request) {
    if app.IsStartupComplete() {
        w.WriteHeader(http.StatusOK)
        fmt.Fprintf(w, "OK")
    } else {
        w.WriteHeader(http.StatusServiceUnavailable)
        fmt.Fprintf(w, "Starting")
    }
}

// /health/live - 存活探针
// 只检查进程存活状态，逻辑简单快速
func livenessHandler(w http.ResponseWriter, r *http.Request) {
    w.WriteHeader(http.StatusOK)
    fmt.Fprintf(w, "OK")
}

// /health/ready - 就绪探针
// 检查应用是否准备好接收流量，包含依赖检查
func readinessHandler(w http.ResponseWriter, r *http.Request) {
    if app.IsReady() && db.IsConnected() && cache.IsConnected() {
        w.WriteHeader(http.StatusOK)
        fmt.Fprintf(w, "OK")
    } else {
        w.WriteHeader(http.StatusServiceUnavailable)
        fmt.Fprintf(w, "Not Ready")
    }
}
```

## 常见问题

**Q1: 三种探针必须都配置吗？**

不是必须的。可以根据应用特点选择配置：
- 对于启动快的简单应用，可以只配置 Liveness Probe 和 Readiness Probe
- 对于启动慢的应用，建议配置 Startup Probe，避免启动期误判
- 对于不需要流量管理的批处理任务，可以只配置 Liveness Probe
- 对于需要精细控制的应用，建议配置全部三种探针

**Q2: Startup Probe 和 Liveness Probe 的 initialDelaySeconds 如何配置？**

如果配置了 Startup Probe，Liveness Probe 的 initialDelaySeconds 可以设置为 0 或较小值。因为 Startup Probe 成功后才会执行 Liveness Probe，应用已经启动完成。Startup Probe 的 initialDelaySeconds 应该根据应用的最小启动时间设置。

**Q3: 三种探针应该使用相同的端点吗？**

不建议使用相同端点。每种探针的职责不同，应该设计不同的检查逻辑：
- Startup Probe：检查应用是否启动完成，可以检查初始化标志
- Liveness Probe：检查进程存活状态，逻辑应该轻量、快速
- Readiness Probe：检查应用就绪状态，可以包含依赖服务检查

**Q4: 就绪探针失败会导致 Pod 重启吗？**

不会。就绪探针失败只会将 Pod 从 Service 的 Endpoints 中移除，停止接收流量，但不会重启容器。只有存活探针和启动探针失败才会触发容器重启。

**Q5: 如何处理探针检查导致的性能问题？**

探针检查应该轻量、快速，避免执行复杂逻辑。可以采取以下措施：
- 使用专门的轻量级健康检查端点
- 避免在健康检查中执行数据库查询或外部 API 调用
- 适当增加 periodSeconds 减少检查频率
- 使用缓存机制，避免重复计算
- 对于依赖服务检查，可以设置超时时间，避免长时间阻塞

## 面试回答

**面试官问：Kubernetes 常见健康检查的探针有几种？**

**回答**：Kubernetes 提供了三种健康检查探针：Liveness Probe（存活探针）、Readiness Probe（就绪探针）和 Startup Probe（启动探针，Kubernetes 1.16+ 引入）。

**Liveness Probe** 用于检测容器是否存活。当探测失败时，Kubernetes 会认为容器已经处于不可恢复的故障状态，根据重启策略重启容器。这适用于处理应用死锁、资源耗尽等严重故障。

**Readiness Probe** 用于检测容器是否准备好接收流量。当探测失败时，Kubernetes 会将 Pod 从 Service 的 Endpoints 中移除，停止向该 Pod 发送请求，但不会重启容器。这适用于处理应用初始化、依赖服务未就绪等临时性故障。

**Startup Probe** 用于检测容器是否启动完成。当探测失败时，Kubernetes 会重启容器。Startup Probe 的核心价值在于为启动时间长的应用提供保护，避免因为存活探针过早检查导致应用在启动过程中被误判为不健康而重启。Startup Probe 成功后，才会开始执行 Liveness Probe 和 Readiness Probe。

三种探针都支持三种检测方式：HTTP GET、TCP Socket 和 Exec。在实际应用中，三种探针协同工作：Startup Probe 负责判断应用是否启动完成，Liveness Probe 负责判断应用是否存活，Readiness Probe 负责判断应用是否就绪，共同保障应用的高可用性和稳定性。
