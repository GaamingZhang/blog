# Kubernetes 存储卷使用详解

## 引言

在容器化环境中，容器本身是临时的，容器内的数据会随着容器的删除而丢失。Kubernetes 通过存储卷（Volume）机制解决了这个问题，为 Pod 提供了持久化存储能力。

存储卷是 Kubernetes 中实现数据持久化和共享的核心机制。理解存储卷的类型、配置和使用方法，对于构建生产级应用至关重要。

## 存储卷概述

### 什么是存储卷

存储卷是 Pod 中可以被容器访问的目录，它的生命周期取决于存储卷的类型。存储卷将存储资源抽象化，使数据可以独立于容器生命周期存在。

### 存储卷的作用

```
┌─────────────────────────────────────────────────────────────┐
│                    存储卷的作用                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 数据持久化                                               │
│     - 容器重启后数据保留                                     │
│     - Pod 删除后数据保留                                     │
│                                                              │
│  2. 数据共享                                                 │
│     - Pod 内容器间共享数据                                   │
│     - Pod 间共享数据                                         │
│                                                              │
│  3. 配置管理                                                 │
│     - 挂载配置文件                                           │
│     - 挂载敏感信息                                           │
│                                                              │
│  4. 存储抽象                                                 │
│     - 屏蔽底层存储差异                                       │
│     - 统一访问接口                                           │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 存储卷类型

### 存储卷类型概览

| 类型 | 生命周期 | 特点 | 适用场景 |
|-----|---------|------|---------|
| **emptyDir** | Pod 生命周期 | 临时存储 | 缓存、临时数据 |
| **hostPath** | 节点生命周期 | 节点本地存储 | 节点数据、调试 |
| **ConfigMap** | 独立 | 配置数据 | 配置文件 |
| **Secret** | 独立 | 敏感数据 | 密码、证书 |
| **PVC** | 独立 | 持久化存储 | 数据库、持久化应用 |
| **nfs** | 独立 | 网络存储 | 共享存储 |
| **persistentVolumeClaim** | 独立 | 动态供给 | 生产环境 |

## emptyDir

### 概述

emptyDir 是最简单的存储卷类型，随 Pod 创建而创建，随 Pod 删除而删除。

### 配置示例

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: emptydir-demo
spec:
  containers:
  - name: app
    image: nginx:1.21
    volumeMounts:
    - name: cache
      mountPath: /var/cache
    - name: tmp
      mountPath: /tmp
  volumes:
  - name: cache
    emptyDir: {}
  - name: tmp
    emptyDir:
      sizeLimit: "100Mi"
      medium: Memory
```

### 配置选项

| 选项 | 说明 |
|-----|------|
| **medium** | 存储介质（默认磁盘或 Memory） |
| **sizeLimit** | 容量限制 |

### 使用场景

- 临时缓存
- 容器间数据共享
- 临时文件存储

## hostPath

### 概述

hostPath 将节点文件系统上的文件或目录挂载到 Pod 中。

### 配置示例

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: hostpath-demo
spec:
  containers:
  - name: app
    image: nginx:1.21
    volumeMounts:
    - name: data
      mountPath: /var/data
  volumes:
  - name: data
    hostPath:
      path: /data/app
      type: DirectoryOrCreate
```

### 类型选项

| 类型 | 说明 |
|-----|------|
| **DirectoryOrCreate** | 目录不存在则创建 |
| **Directory** | 必须存在的目录 |
| **FileOrCreate** | 文件不存在则创建 |
| **File** | 必须存在的文件 |
| **Socket** | Unix socket |
| **CharDevice** | 字符设备 |
| **BlockDevice** | 块设备 |

### 使用场景

- 访问节点日志
- 访问 Docker socket
- 本地数据存储

## ConfigMap

### 概述

ConfigMap 用于存储非敏感的配置数据。

### 配置示例

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  database_url: "mysql://db-service:3306/mydb"
  config.properties: |
    server.port=8080
    logging.level=INFO
---
apiVersion: v1
kind: Pod
metadata:
  name: configmap-demo
spec:
  containers:
  - name: app
    image: nginx:1.21
    volumeMounts:
    - name: config
      mountPath: /etc/config
      readOnly: true
  volumes:
  - name: config
    configMap:
      name: app-config
      items:
      - key: config.properties
        path: application.properties
```

### 使用方式

```yaml
# 方式1：环境变量
env:
- name: DATABASE_URL
  valueFrom:
    configMapKeyRef:
      name: app-config
      key: database_url

# 方式2：文件挂载
volumes:
- name: config
  configMap:
    name: app-config

# 方式3：特定文件挂载
volumes:
- name: config
  configMap:
    name: app-config
    items:
    - key: config.properties
      path: application.properties
```

## Secret

### 概述

Secret 用于存储敏感数据，如密码、Token、证书等。

### 配置示例

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
stringData:
  username: admin
  password: mypassword
---
apiVersion: v1
kind: Pod
metadata:
  name: secret-demo
spec:
  containers:
  - name: app
    image: nginx:1.21
    volumeMounts:
    - name: secret
      mountPath: /etc/secrets
      readOnly: true
  volumes:
  - name: secret
    secret:
      secretName: db-secret
```

### Secret 类型

| 类型 | 说明 |
|-----|------|
| **Opaque** | 用户定义的任意数据 |
| **kubernetes.io/tls** | TLS 证书 |
| **kubernetes.io/dockerconfigjson** | Docker 配置 |
| **kubernetes.io/basic-auth** | 基本认证 |
| **kubernetes.io/ssh-auth** | SSH 认证 |

## PVC（持久卷声明）

### 概述

PVC 是用户对存储资源的请求，通过 PV 或 StorageClass 获取实际的存储资源。

### 配置示例

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: data-pvc
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi
  storageClassName: standard
---
apiVersion: v1
kind: Pod
metadata:
  name: pvc-demo
spec:
  containers:
  - name: app
    image: nginx:1.21
    volumeMounts:
    - name: data
      mountPath: /var/data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: data-pvc
```

### 访问模式

| 访问模式 | 缩写 | 说明 |
|---------|------|------|
| **ReadWriteOnce** | RWO | 单节点读写 |
| **ReadOnlyMany** | ROX | 多节点只读 |
| **ReadWriteMany** | RWX | 多节点读写 |
| **ReadWriteOncePod** | RWOP | 单 Pod 读写 |

## NFS

### 概述

NFS 是网络文件系统，支持多节点共享存储。

### 配置示例

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nfs-demo
spec:
  containers:
  - name: app
    image: nginx:1.21
    volumeMounts:
    - name: nfs-data
      mountPath: /var/data
  volumes:
  - name: nfs-data
    nfs:
      server: 192.168.1.100
      path: /data/shared
      readOnly: false
```

### NFS 配置选项

| 选项 | 说明 |
|-----|------|
| **server** | NFS 服务器地址 |
| **path** | NFS 导出路径 |
| **readOnly** | 是否只读 |

## 多容器共享存储卷

### 配置示例

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: shared-volume-demo
spec:
  containers:
  - name: app
    image: nginx:1.21
    volumeMounts:
    - name: shared-data
      mountPath: /var/www/html
  - name: sidecar
    image: busybox
    command: ["/bin/sh", "-c", "while true; do echo $(date) >> /data/log.txt; sleep 10; done"]
    volumeMounts:
    - name: shared-data
      mountPath: /data
  volumes:
  - name: shared-data
    emptyDir: {}
```

## 存储卷最佳实践

### 1. 使用 PVC 管理持久化存储

```yaml
# 推荐：使用 PVC
volumes:
- name: data
  persistentVolumeClaim:
    claimName: data-pvc

# 不推荐：直接使用 hostPath
volumes:
- name: data
  hostPath:
    path: /data/app
```

### 2. 配置资源限制

```yaml
volumes:
- name: cache
  emptyDir:
    sizeLimit: "1Gi"
```

### 3. 使用 StorageClass 动态供给

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: data-pvc
spec:
  accessModes:
  - ReadWriteOnce
  storageClassName: standard
  resources:
    requests:
      storage: 10Gi
```

### 4. 配置安全选项

```yaml
volumes:
- name: secret
  secret:
    secretName: app-secret
    defaultMode: 0400  # 只读权限
```

### 5. 使用子路径

```yaml
volumeMounts:
- name: data
  mountPath: /var/www/html
  subPath: html
- name: data
  mountPath: /var/log/app
  subPath: logs
```

## 常见问题排查

### 存储卷无法挂载

```bash
# 检查 PVC 状态
kubectl get pvc

# 检查 PV 状态
kubectl get pv

# 检查 Pod 事件
kubectl describe pod <pod-name>

# 检查存储类
kubectl get storageclass
```

### PVC 处于 Pending 状态

```bash
# 检查 PVC 详情
kubectl describe pvc <pvc-name>

# 检查 StorageClass
kubectl get storageclass

# 检查是否有可用 PV
kubectl get pv
```

### 数据丢失

```bash
# 检查存储卷类型
kubectl get pod <pod-name> -o yaml | grep -A 10 volumes

# 检查 PVC 回收策略
kubectl get pv <pv-name> -o yaml | grep persistentVolumeReclaimPolicy
```

## 面试回答

**问题**: 如何进行存储卷的使用？

**回答**: Kubernetes 存储卷是 Pod 中可以被容器访问的目录，用于实现数据持久化和共享。存储卷的使用分为定义卷和挂载卷两个步骤。

**定义存储卷**在 Pod 的 `spec.volumes` 字段中配置，支持多种类型：`emptyDir` 是临时存储，随 Pod 生命周期；`hostPath` 挂载节点本地目录；`ConfigMap` 挂载配置数据；`Secret` 挂载敏感数据；`PVC` 挂载持久化存储；`nfs` 挂载网络存储。

**挂载存储卷**在容器的 `volumeMounts` 字段中配置，指定挂载路径。同一个存储卷可以被多个容器挂载，实现容器间数据共享。

**持久化存储**推荐使用 PVC。首先创建 PVC 请求存储资源，然后在 Pod 中通过 `persistentVolumeClaim` 引用。PVC 支持多种访问模式：ReadWriteOnce 单节点读写、ReadOnlyMany 多节点只读、ReadWriteMany 多节点读写。

**最佳实践**：使用 PVC 管理持久化存储，避免直接使用 hostPath；配置存储容量限制；使用 StorageClass 实现动态供给；配置适当的访问权限；使用子路径实现同一卷的多次挂载。
