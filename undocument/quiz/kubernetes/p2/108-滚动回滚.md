# Kubernetes 滚动回滚详解

## 引言

在应用部署过程中，新版本可能存在 bug 或性能问题，需要快速回退到稳定版本。Kubernetes 提供了强大的回滚机制，可以快速、安全地将应用恢复到之前的版本。

理解 Kubernetes 的回滚机制，包括版本历史管理、回滚操作和最佳实践，对于保障应用稳定性和快速响应生产问题至关重要。

## 回滚概述

### 什么是回滚

回滚是将应用从当前版本恢复到之前某个版本的操作。Kubernetes 通过维护 Deployment 的历史版本，支持快速回滚。

```
┌─────────────────────────────────────────────────────────────┐
│                    回滚过程                                  │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  版本历史：                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ REVISION  IMAGE          STATUS                     │   │
│  │ 1         nginx:1.20     已被替换                    │   │
│  │ 2         nginx:1.21     已被替换                    │   │
│  │ 3         nginx:1.22     当前版本（有问题）          │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  回滚操作：                                                  │
│  kubectl rollout undo deployment/my-app --to-revision=2    │
│                                                              │
│  回滚后：                                                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ REVISION  IMAGE          STATUS                     │   │
│  │ 1         nginx:1.20     已被替换                    │   │
│  │ 3         nginx:1.22     已被替换                    │   │
│  │ 4         nginx:1.21     当前版本（回滚后）          │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  注意：回滚会创建新的 REVISION                              │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 回滚的工作原理

Kubernetes 通过 ReplicaSet 实现版本管理和回滚：

1. 每次更新 Deployment 时，创建新的 ReplicaSet
2. 旧的 ReplicaSet 被保留，用于回滚
3. 回滚时，将旧的 ReplicaSet 扩容，新的 ReplicaSet 缩容

## 查看版本历史

### 查看历史版本列表

```bash
# 查看历史版本
kubectl rollout history deployment/my-app

# 输出示例
REVISION  CHANGE-CAUSE
1         <none>
2         kubectl set image deployment/my-app app=nginx:1.21
3         kubectl set image deployment/my-app app=nginx:1.22
```

### 查看特定版本详情

```bash
# 查看特定版本详情
kubectl rollout history deployment/my-app --revision=2

# 输出示例
REVISION  2
CONTAINER  IMAGE       ENVIRONMENTS
app        nginx:1.21

Pod Template:
  Labels:	app=my-app
  Containers:
   app:
    Image:	nginx:1.21
    Port:	80/TCP
```

### CHANGE-CAUSE 字段

CHANGE-CAUSE 记录了触发更新的原因：

```bash
# 记录变更原因
kubectl apply -f deployment.yaml --record

# 或使用 annotation
kubectl annotate deployment my-app kubernetes.io/change-cause="Update to v1.22"
```

## 执行回滚

### 回滚到上一版本

```bash
# 回滚到上一版本
kubectl rollout undo deployment/my-app

# 查看回滚状态
kubectl rollout status deployment/my-app

# 输出示例
deployment "my-app" successfully rolled out
```

### 回滚到指定版本

```bash
# 回滚到指定版本
kubectl rollout undo deployment/my-app --to-revision=2

# 查看回滚状态
kubectl rollout status deployment/my-app
```

### 查看回滚后的版本

```bash
# 查看历史版本
kubectl rollout history deployment/my-app

# 输出示例
REVISION  CHANGE-CAUSE
1         <none>
3         kubectl set image deployment/my-app app=nginx:1.22
4         kubectl rollout undo deployment/my-app --to-revision=2
```

## 配置历史版本

### revisionHistoryLimit

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  revisionHistoryLimit: 10  # 保留 10 个历史版本
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: app
        image: nginx:1.21
```

### 默认值

- 默认保留 10 个历史版本
- 设置为 0 将禁用回滚功能
- 建议根据更新频率设置合理值

## 回滚过程详解

### 回滚流程

```
┌─────────────────────────────────────────────────────────────┐
│                    回滚详细流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 获取目标版本 ReplicaSet                                 │
│     ┌─────────────────────────────────────────────────┐    │
│     │ kubectl rollout undo --to-revision=2            │    │
│     │ 查找 REVISION=2 对应的 ReplicaSet               │    │
│     └──────────────────────┬──────────────────────────┘    │
│                            │                                │
│                            ▼                                │
│  2. 更新 Deployment Pod 模板                                │
│     ┌─────────────────────────────────────────────────┐    │
│     │ 将 Pod 模板更新为目标版本                        │    │
│     │ 触发新的滚动更新                                │    │
│     └──────────────────────┬──────────────────────────┘    │
│                            │                                │
│                            ▼                                │
│  3. 执行滚动更新                                            │
│     ┌─────────────────────────────────────────────────┐    │
│     │ - 扩容目标 ReplicaSet                           │    │
│     │ - 缩容当前 ReplicaSet                           │    │
│     │ - 等待 Pod 就绪                                 │    │
│     └──────────────────────┬──────────────────────────┘    │
│                            │                                │
│                            ▼                                │
│  4. 创建新的 REVISION                                       │
│     ┌─────────────────────────────────────────────────┐    │
│     │ 回滚操作会创建新的 REVISION                      │    │
│     │ 不是复用旧的 REVISION                           │    │
│     └─────────────────────────────────────────────────┘    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 回滚时的 ReplicaSet 变化

```bash
# 回滚前
kubectl get rs -l app=my-app
NAME                DESIRED   CURRENT   READY   AGE
my-app-6b7f9c8d4    0         0         0       1h    # v1.20
my-app-7c8g9d9e5    0         0         0       30m   # v1.21
my-app-8d9h0e0f6    3         3         3       5m    # v1.22 (当前)

# 执行回滚
kubectl rollout undo deployment/my-app --to-revision=2

# 回滚后
kubectl get rs -l app=my-app
NAME                DESIRED   CURRENT   READY   AGE
my-app-6b7f9c8d4    0         0         0       1h    # v1.20
my-app-7c8g9d9e5    3         3         3       30m   # v1.21 (回滚后)
my-app-8d9h0e0f6    0         0         0       5m    # v1.22
```

## 回滚最佳实践

### 1. 记录变更原因

```bash
# 使用 --record 标志
kubectl apply -f deployment.yaml --record

# 或手动添加 annotation
kubectl annotate deployment my-app kubernetes.io/change-cause="Fix bug in login feature"
```

### 2. 保留足够的历史版本

```yaml
spec:
  revisionHistoryLimit: 10  # 根据更新频率设置
```

### 3. 回滚前验证

```bash
# 查看目标版本详情
kubectl rollout history deployment/my-app --revision=2

# 确认目标版本正确后再回滚
kubectl rollout undo deployment/my-app --to-revision=2
```

### 4. 监控回滚过程

```bash
# 实时查看回滚状态
kubectl rollout status deployment/my-app -w

# 查看 Pod 状态
kubectl get pods -l app=my-app -w
```

### 5. 回滚后验证

```bash
# 验证应用版本
kubectl describe deployment my-app | grep Image

# 验证服务可用性
kubectl exec -it <pod-name> -- curl localhost:80/health
```

## 回滚场景

### 场景一：新版本有 Bug

```bash
# 发现新版本有问题
# 1. 查看当前版本
kubectl rollout history deployment/my-app

# 2. 回滚到上一版本
kubectl rollout undo deployment/my-app

# 3. 验证回滚结果
kubectl rollout status deployment/my-app
```

### 场景二：性能问题

```bash
# 新版本性能下降
# 1. 监控指标确认问题
kubectl top pods -l app=my-app

# 2. 回滚到稳定版本
kubectl rollout undo deployment/my-app --to-revision=2

# 3. 验证性能恢复
kubectl top pods -l app=my-app
```

### 场景三：配置错误

```bash
# 配置变更导致问题
# 1. 查看历史版本
kubectl rollout history deployment/my-app

# 2. 回滚到正确配置的版本
kubectl rollout undo deployment/my-app --to-revision=1
```

## 回滚与滚动更新的关系

### 滚动更新创建新 ReplicaSet

```yaml
# 每次更新都会创建新的 ReplicaSet
REVISION 1: ReplicaSet my-app-6b7f9c8d4 (nginx:1.20)
REVISION 2: ReplicaSet my-app-7c8g9d9e5 (nginx:1.21)
REVISION 3: ReplicaSet my-app-8d9h0e0f6 (nginx:1.22)
```

### 回滚复用旧 ReplicaSet

```yaml
# 回滚不会创建新 ReplicaSet，而是复用旧的
REVISION 4: ReplicaSet my-app-7c8g9d9e5 (nginx:1.21)  # 复用 REVISION 2 的 ReplicaSet
```

## 常见问题排查

### 无法回滚

```bash
# 检查历史版本是否存在
kubectl rollout history deployment/my-app

# 检查 revisionHistoryLimit
kubectl get deployment my-app -o jsonpath='{.spec.revisionHistoryLimit}'

# 检查 ReplicaSet 是否存在
kubectl get rs -l app=my-app
```

### 回滚卡住

```bash
# 查看回滚状态
kubectl rollout status deployment/my-app

# 查看 Deployment 事件
kubectl describe deployment my-app

# 查看 Pod 状态
kubectl get pods -l app=my-app

# 查看 Pod 事件
kubectl describe pod <pod-name>
```

### 历史版本丢失

```bash
# 检查 revisionHistoryLimit 配置
kubectl get deployment my-app -o yaml | grep revisionHistoryLimit

# 如果设置为 0，历史版本不会保留
# 修改为合理值
kubectl patch deployment my-app -p '{"spec":{"revisionHistoryLimit":10}}'
```

## 回滚监控告警

### Prometheus 告警规则

```yaml
groups:
- name: deployment-alerts
  rules:
  - alert: DeploymentRollback
    expr: |
      increase(kube_deployment_status_replicas_updated[5m]) > 0
      and on(deployment)
      kube_deployment_status_replicas_updated != kube_deployment_spec_replicas
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "Deployment {{ $labels.deployment }} is rolling back"
      
  - alert: DeploymentRollbackStuck
    expr: |
      kube_deployment_status_replicas_updated != kube_deployment_spec_replicas
    for: 15m
    labels:
      severity: critical
    annotations:
      summary: "Deployment {{ $labels.deployment }} rollback is stuck"
```

## 面试回答

**问题**: 如何在 Kubernetes 中进行滚动回滚？

**回答**: Kubernetes 的回滚机制通过维护 Deployment 的历史版本来实现。每次更新 Deployment 时，Kubernetes 会创建新的 ReplicaSet 并保留旧的 ReplicaSet，用于可能的回滚操作。

**查看版本历史**使用 `kubectl rollout history deployment/<name>` 命令，可以看到所有历史版本的 REVISION 和 CHANGE-CAUSE。使用 `--revision=<n>` 可以查看特定版本的详细配置。

**执行回滚**有两种方式：`kubectl rollout undo deployment/<name>` 回滚到上一版本；`kubectl rollout undo deployment/<name> --to-revision=<n>` 回滚到指定版本。回滚操作实际上是触发一次滚动更新，将旧的 ReplicaSet 扩容，当前的 ReplicaSet 缩容。

**重要特性**：回滚会创建新的 REVISION，而不是复用旧的 REVISION 编号。例如，如果当前是 REVISION 3，回滚到 REVISION 2 后，会创建 REVISION 4，而不是恢复为 REVISION 2。回滚复用的是旧版本的 ReplicaSet，不会创建新的 ReplicaSet。

**配置管理**：通过 `spec.revisionHistoryLimit` 控制保留的历史版本数量，默认是 10 个。设置为 0 将禁用回滚功能。建议根据更新频率设置合理的保留数量。

**最佳实践**：使用 `--record` 标志或手动添加 `kubernetes.io/change-cause` annotation 记录变更原因，方便识别版本；回滚前先查看目标版本详情确认正确；回滚后验证应用功能正常；配置监控告警及时发现回滚异常。
