# 负载均衡原理

## 什么是负载均衡？

想象一家热门餐厅，门口排着长队。如果只有一个收银台，顾客等待时间会很长。聪明的餐厅经理开设了多个收银台，并安排服务员引导顾客到空闲的收银台——这就是负载均衡的基本思想。

在云计算中，负载均衡器将传入的流量分发到多个后端服务器，确保没有单台服务器过载，提高应用的可用性和响应速度。

## 负载均衡算法

### 常见算法

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  负载均衡算法详解：                                                       │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                                                                 │   │
│  │  1. 轮询（Round Robin）：                                        │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │                                                         │   │   │
│  │  │  原理：按顺序依次分发请求到每台服务器                     │   │   │
│  │  │                                                         │   │   │
│  │  │  请求1 → 服务器A                                         │   │   │
│  │  │  请求2 → 服务器B                                         │   │   │
│  │  │  请求3 → 服务器C                                         │   │   │
│  │  │  请求4 → 服务器A（循环）                                 │   │   │
│  │  │                                                         │   │   │
│  │  │  优点：简单、公平                                        │   │   │
│  │  │  缺点：不考虑服务器性能差异                              │   │   │
│  │  │ 适用：服务器性能相近的场景                                │   │   │
│  │  │                                                         │   │   │
│  │  └─────────────────────────────────────────────────────────┘   │   │
│  │                                                                 │   │
│  │  2. 加权轮询（Weighted Round Robin）：                           │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │                                                         │   │   │
│  │  │  原理：根据服务器权重分配请求                            │   │   │
│  │  │                                                         │   │   │
│  │  │  服务器A（权重3）：处理3个请求                           │   │   │
│  │  │  服务器B（权重2）：处理2个请求                           │   │   │
│  │  │  服务器C（权重1）：处理1个请求                           │   │   │
│  │  │                                                         │   │   │
│  │  │  请求分配：A A A B B C                                   │   │   │
│  │  │                                                         │   │   │
│  │  │  优点：考虑服务器性能差异                                │   │   │
│  │  │  缺点：需要手动配置权重                                  │   │   │
│  │  │ 适用：服务器性能差异较大的场景                            │   │   │
│  │  │                                                         │   │   │
│  │  └─────────────────────────────────────────────────────────┘   │   │
│  │                                                                 │   │
│  │  3. 最少连接（Least Connections）：                              │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │                                                         │   │   │
│  │  │  原理：将请求分发到当前连接数最少的服务器                 │   │   │
│  │  │                                                         │   │   │
│  │  │  当前连接数：                                            │   │   │
│  │  │  服务器A：10个连接                                       │   │   │
│  │  │  服务器B：5个连接  ← 新请求分发到这里                    │   │   │
│  │  │  服务器C：8个连接                                        │   │   │
│  │  │                                                         │   │   │
│  │  │  优点：动态负载均衡，更公平                              │   │   │
│  │  │  缺点：需要维护连接计数                                  │   │   │
│  │  │ 适用：长连接场景、请求处理时间差异大                      │   │   │
│  │  │                                                         │   │   │
│  │  └─────────────────────────────────────────────────────────┘   │   │
│  │                                                                 │   │
│  │  4. IP哈希（IP Hash）：                                          │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │                                                         │   │   │
│  │  │  原理：根据客户端IP地址哈希值选择服务器                   │   │   │
│  │  │                                                         │   │   │
│  │  │  hash(IP) % 服务器数量 = 服务器索引                      │   │   │
│  │  │                                                         │   │   │
│  │  │  客户端IP 192.168.1.1 → 哈希计算 → 服务器A               │   │   │
│  │  │  客户端IP 192.168.1.2 → 哈希计算 → 服务器B               │   │   │
│  │  │                                                         │   │   │
│  │  │  优点：会话保持，同一客户端始终访问同一服务器             │   │   │
│  │  │  缺点：可能导致负载不均                                  │   │   │
│  │  │ 适用：需要会话保持的场景                                  │   │   │
│  │  │                                                         │   │   │
│  │  └─────────────────────────────────────────────────────────┘   │   │
│  │                                                                 │   │
│  │  5. 一致性哈希（Consistent Hashing）：                           │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │                                                         │   │   │
│  │  │  原理：将服务器和请求映射到哈希环上                       │   │   │
│  │  │                                                         │   │   │
│  │  │            服务器A                                       │   │   │
│  │  │              ↑                                           │   │   │
│  │  │              │                                           │   │   │
│  │  │  请求1 ──→  ●  ←── 请求2                                │   │   │
│  │  │              │                                           │   │   │
│  │  │              ↓                                           │   │   │
│  │  │            服务器B                                       │   │   │
│  │  │                                                         │   │   │
│  │  │  优点：服务器增减时只影响相邻节点                        │   │   │
│  │  │  适用：缓存服务器集群                                    │   │   │
│  │  │                                                         │   │   │
│  │  └─────────────────────────────────────────────────────────┘   │   │
│  │                                                                 │   │
│  │  6. 加权最少连接（Weighted Least Connections）：                 │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │                                                         │   │   │
│  │  │  原理：结合权重和连接数选择服务器                        │   │   │
│  │  │                                                         │   │   │
│  │  │  选择标准：连接数/权重 最小的服务器                      │   │   │
│  │  │                                                         │   │   │
│  │  │  服务器A：连接10，权重3 → 10/3 = 3.33                    │   │   │
│  │  │  服务器B：连接5，权重2 → 5/2 = 2.5  ← 选择              │   │   │
│  │  │  服务器C：连接8，权重1 → 8/1 = 8                         │   │   │
│  │  │                                                         │   │   │
│  │  └─────────────────────────────────────────────────────────┘   │   │
│  │                                                                 │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### ALB路由配置示例

```yaml
# Terraform ALB配置示例

# 创建ALB
resource "aws_lb" "main" {
  name               = "my-application-lb"
  internal           = false
  load_balancer_type = "application"
  security_groups    = [aws_security_group.alb.id]
  subnets           = aws_subnet.public[*].id
  
  enable_deletion_protection = true
}

# 目标组
resource "aws_lb_target_group" "web" {
  name     = "web-tg"
  port     = 80
  protocol = "HTTP"
  vpc_id   = aws_vpc.main.id
  
  health_check {
    enabled             = true
    healthy_threshold   = 3
    interval            = 30
    matcher            = "200"
```

## 常见问题

### Q1: 如何选择L4还是L7负载均衡？

**选择建议**：
- L4：高性能场景、非HTTP协议、简单负载均衡
- L7：需要高级路由、SSL终止、内容缓存、基于URL路由

### Q2: 负载均衡器如何处理SSL？

**SSL终止**：
- 负载均衡器解密HTTPS请求
- 后端服务器使用HTTP通信
- 减轻后端服务器SSL计算负担

**SSL透传**：
- 负载均衡器不解密，直接转发
- 后端服务器处理SSL
- 端到端加密

### Q3: 如何实现跨区域负载均衡？

**方案**：
- DNS负载均衡（Route 53）
- 全局负载均衡器
- Anycast IP

### Q4: 负载均衡器本身如何高可用？

**方案**：
- 多可用区部署
- 自动扩展
- 健康检查和自动替换
- DNS故障转移

### Q5: 如何监控负载均衡器？

**监控指标**：
- 请求数量、错误率
- 响应时间（P50、P95、P99）
- 后端服务器健康状态
- 连接数、带宽

## 小结

负载均衡是分布式系统的核心组件，确保流量均匀分发到多个服务器，提高可用性和性能。

**核心原理**：
1. L4负载均衡：基于IP+端口，性能高
2. L7负载均衡：基于应用层内容，功能丰富

**负载均衡算法**：
- 轮询、加权轮询：简单公平
- 最少连接：动态负载均衡
- IP哈希：会话保持
- 一致性哈希：缓存场景

**关键特性**：
- 健康检查：自动排除故障服务器
- 会话保持：保证会话连续性
- 多可用区部署：高可用架构

## 面试回答总结

负载均衡是将传入流量分发到多个后端服务器的技术，提高应用可用性和性能。负载均衡分为L4（传输层，基于IP+端口）和L7（应用层，基于URL、Header等）。常见负载均衡算法包括：轮询（按顺序分发）、加权轮询（根据权重分发）、最少连接（分发到连接数最少的服务器）、IP哈希（根据客户端IP哈希分发，实现会话保持）、一致性哈希（服务器增减时只影响相邻节点）。健康检查机制通过TCP连接或HTTP请求检测服务器状态，连续N次失败标记为不健康并停止分发请求。云负载均衡服务如AWS ALB（L7，支持基于路径/主机路由、SSL终止）、NLB（L4，高性能、静态IP）。高可用架构通过多可用区部署实现，单可用区故障不影响整体服务。会话保持通过Cookie、IP Hash或Header实现，但最佳实践是无状态设计，会话数据存储在外部如Redis。
