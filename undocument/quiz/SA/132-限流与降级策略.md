# 限流与降级策略设计深度解析

## 为什么需要限流和降级?

2023年某票务系统在热门演唱会开售瞬间,每秒涌入100万请求,系统瞬间崩溃,导致真正想购票的用户无法访问,黄牛却通过脚本抢到了大量门票。这个案例揭示了流量控制的必要性:在有限的资源下,如何保护系统不被过载,同时保证核心业务的可用性。

限流和降级是系统高可用的两大基石。限流通过控制请求速率保护系统,降级通过牺牲非核心功能保证核心服务,两者结合构建起系统的自我保护机制。

**固定窗口算法实现**:

```java
public class FixedWindowRateLimiter {
    private final int limit;           // 窗口内最大请求数
    private final long windowSize;     // 窗口大小(毫秒)
    
    private AtomicInteger counter = new AtomicInteger(0);
    private AtomicLong windowStart = new AtomicLong(System.currentTimeMillis());
    
    public FixedWindowRateLimiter(int limit, long windowSize) {
        this.limit = limit;
        this.windowSize = windowSize;
    }
    
    public boolean tryAcquire() {
        long currentTime = System.currentTimeMillis();
        long currentWindowStart = windowStart.get();
        
        // 检查是否需要切换窗口
        if (currentTime - currentWindowStart >= windowSize) {
            // 尝试切换窗口
            if (windowStart.compareAndSet(currentWindowStart, currentTime)) {
                counter.set(0);
            }
        }
        
        // 检查是否超过限制
        return counter.incrementAndGet() <= limit;
    }
}
```

**令牌桶算法实现**:

```java
public class TokenBucketRateLimiter {
    private final long capacity;       // 桶容量
    private final long rate;           // 令牌生成速率(个/秒)
    
    private long tokens;               // 当前令牌数
    private long lastRefillTime;       // 上次填充时间
    
    public TokenBucketRateLimiter(long capacity, long rate) {
        this.capacity = capacity;
        this.rate = rate;
        this.tokens = capacity;
        this.lastRefillTime = System.currentTimeMillis();
    }
    
    public synchronized boolean tryAcquire() {
        // 计算需要补充的令牌数
        long currentTime = System.currentTimeMillis();
        long elapsedTime = currentTime - lastRefillTime;
        long tokensToAdd = (elapsedTime * rate) / 1000;
        
        // 补充令牌
        if (tokensToAdd > 0) {
            tokens = Math.min(capacity, tokens + tokensToAdd);
            lastRefillTime = currentTime;
        }
        
        // 尝试获取令牌
        if (tokens > 0) {
            tokens--;
            return true;
        }
        
        return false;
    }
}

// 使用Guava RateLimiter
public class GuavaRateLimiterExample {
    private final RateLimiter rateLimiter = RateLimiter.create(100); // 每秒100个
    
    public void processRequest() {
        if (rateLimiter.tryAcquire()) {
            // 处理请求
            doProcess();
        } else {
            // 限流
            throw new RateLimitException("请求过于频繁");
        }
    }
}
```

**Redis分布式限流实现**:

```java
@Component
public class RedisRateLimiter {
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    private static final String LUA_SCRIPT = 
        "local key = KEYS[1] " +
        "local limit = tonumber(ARGV[1]) " +
        "local window = tonumber(ARGV[2]) " +
        "local current = redis.call('INCR', key) " +
        "if current == 1 then " +
        "    redis.call('EXPIRE', key, window) " +
        "end " +
        "return current <= limit";
    
    public boolean tryAcquire(String key, int limit, int windowSeconds) {
        DefaultRedisScript<Boolean> script = new DefaultRedisScript<>(LUA_SCRIPT, Boolean.class);
        return redisTemplate.execute(script, 
            Collections.singletonList(key), 
            String.valueOf(limit), 
            String.valueOf(windowSeconds));
    }
}

// 使用示例
@Service
public class OrderService {
    @Autowired
    private RedisRateLimiter rateLimiter;
    
    public Order createOrder(OrderRequest request) {
        String userId = request.getUserId();
        String limitKey = "rate_limit:user:" + userId;
        
        // 用户级限流: 每秒最多10次
        if (!rateLimiter.tryAcquire(limitKey, 10, 1)) {
            throw new RateLimitException("请求过于频繁,请稍后再试");
        }
        
        // 创建订单逻辑
        return doCreateOrder(request);
    }
}
```

## 常见问题

### Q1: 如何选择限流算法?

**选择建议**:
- **固定窗口**: 实现简单,对精度要求不高的场景
- **滑动窗口**: 需要平滑限流的场景
- **令牌桶**: 需要处理突发流量的场景(推荐)
- **漏桶**: 需要严格恒定流出速率的场景

### Q2: 限流阈值如何设置?

**设置方法**:
- 压测确定系统容量
- 设置阈值为容量的70-80%
- 根据业务特点调整
- 动态调整,持续优化

### Q3: 降级后如何保证数据一致性?

**解决方案**:
- 写服务降级: 异步写入队列,后续补偿
- 读服务降级: 标注数据来源,提示用户
- 核心数据: 不降级或降级为同步等待

### Q4: 如何避免限流误杀正常用户?

**避免策略**:
- 多维度限流,避免单一维度过严
- 设置合理的阈值
- 提供友好提示和重试机制
- VIP用户白名单

### Q5: 限流降级如何测试?

**测试方法**:
- 压力测试验证限流效果
- 故障注入测试降级逻辑
- 混沌工程验证系统稳定性
- 定期演练,持续优化

## 小结

限流和降级是系统高可用的重要保障机制,需要在系统设计初期就充分考虑。

**限流算法**:
- **固定窗口**: 简单但有边界问题
- **滑动窗口**: 平滑但实现复杂
- **令牌桶**: 允许突发,应用广泛
- **漏桶**: 恒定流出,平滑流量

**限流策略**:
- 多维度限流: 用户/IP/接口/全局
- 分布式限流: Redis + Lua / 网关层
- 动态调整: 根据系统负载调整阈值

**降级策略**:
- 自动降级: 系统指标触发
- 手动降级: 配置中心开关
- 分级降级: P0-P3优先级
- 降级方式: 缓存/默认值/异步

**最佳实践**:
- 压测确定系统容量
- 设置合理阈值(70-80%容量)
- 建立完善的监控告警
- 定期演练和优化
- 提供友好的用户提示

限流和降级不是银弹,而是系统稳定性的最后一道防线。关键是要在系统设计时就考虑,而不是等到故障发生时才临时应对。

## 面试回答总结

限流和降级是系统高可用的两大基石。限流算法主要有四种:固定窗口算法实现简单但有边界问题,可能瞬间通过2倍请求;滑动窗口算法将窗口细分为多个小格子,解决边界问题但实现复杂;令牌桶算法以固定速率生成令牌,请求需要获取令牌才能通过,允许一定程度的突发流量,是应用最广泛的算法(Guava RateLimiter);漏桶算法以固定速率流出请求,强制恒定流出速率,适合需要平滑流量的场景。限流策略需要多维度设计:用户级限流(防止单个用户滥用)、IP级限流(防止恶意攻击)、接口级限流(保护核心接口)、全局限流(保护系统容量),多个维度组合使用。分布式限流可以采用Redis+Lua脚本实现原子性计数,或者在API网关层集中管理。降级策略分为自动降级(系统指标异常触发,如CPU>80%)和手动降级(配置中心开关控制)。降级方式包括:读服务降级返回缓存数据或默认值,写服务降级异步写入队列。降级要按优先级设计:P0核心交易服务尽量不降级,P1重要业务服务可降级为简化版本,P2辅助功能服务可直接关闭,P3后台管理服务可临时关闭。关键是要建立完善的监控体系,监控限流触发次数、拒绝率、降级触发次数、持续时间等指标,设置合理的告警规则,定期通过压力测试和故障注入验证限流降级的有效性。
