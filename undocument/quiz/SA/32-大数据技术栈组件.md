## 数据采集层组件

### 1. Flume

Flume是Cloudera提供的分布式日志收集系统，用于将大量日志数据从各种源收集、聚合并移动到集中存储。

```
Flume架构：

┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  数据源（Source）              Channel              Sink（目标）          │
│  ┌─────────────┐         ┌─────────────┐        ┌─────────────┐        │
│  │             │         │             │        │             │        │
│  │  Avro Source├────────→│ Memory      ├───────→│ HDFS Sink   │        │
│  │             │         │ Channel     │        │             │        │
│  │  Exec Source│         │             │        │ HBase Sink  │        │
│  │  (tail -f)  │         │ File        │        │             │        │
│  │             │         │ Channel     │        │ Kafka Sink  │        │
│  │  Spooling   │         │             │        │             │        │
│  │  Directory  │         │ JDBC        │        │ Logger Sink │        │
│  │             │         │ Channel     │        │             │        │
│  └─────────────┘         └─────────────┘        └─────────────┘        │
│                                                                         │
│  Agent = Source + Channel + Sink                                        │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘

典型配置示例：
# 定义agent
agent1.sources = source1
agent1.channels = channel1
agent1.sinks = sink1

# 配置source
agent1.sources.source1.type = exec
agent1.sources.source1.command = tail -F /var/log/app.log

# 配置channel
agent1.channels.channel1.type = memory
agent1.channels.channel1.capacity = 1000

# 配置sink
agent1.sinks.sink1.type = hdfs
agent1.sinks.sink1.hdfs.path = /flume/events/%Y-%m-%d
```

**适用场景**：
- 日志文件实时收集
- 多源日志聚合
- 数据传输到HDFS/HBase

### 2. Sqoop

Sqoop是Apache顶级项目，用于在Hadoop和关系型数据库之间高效传输批量数据。

```
Sqoop数据流向：

┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  导入（Import）：                                                        │
│  ┌─────────────┐                    ┌─────────────┐                    │
│  │   MySQL     │                    │    HDFS     │                    │
│  │   Oracle    │ ──────Sqoop──────→ │    Hive     │                    │
│  │   PostgreSQL│                    │    HBase    │                    │
│  └─────────────┘                    └─────────────┘                    │
│                                                                         │
│  导出（Export）：                                                        │
│  ┌─────────────┐                    ┌─────────────┐                    │
│  │    HDFS     │                    │   MySQL     │                    │
│  │    Hive     │ ──────Sqoop──────→ │   Oracle    │                    │
│  │             │                    │   PostgreSQL│                    │
│  └─────────────┘                    └─────────────┘                    │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘

常用命令示例：
# 从MySQL导入到HDFS
sqoop import \
  --connect jdbc:mysql://localhost:3306/db \
  --username root \
  --password pwd \
  --table users \
  --target-dir /user/hadoop/users \
  --m 4

# 从Hive导出到MySQL
sqoop export \
  --connect jdbc:mysql://localhost:3306/db \
  --username root \
  --password pwd \
  --table users \
  --export-dir /user/hive/warehouse/users
```

**核心特性**：
- 支持并行导入导出
- 自动类型映射
- 支持增量导入
- 压缩传输

### 3. Canal

Canal是阿里巴巴开源的MySQL数据库增量日志解析工具，提供增量数据订阅和消费。

```
Canal工作原理：

┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  MySQL主库                     Canal Server                 下游应用     │
│  ┌─────────────┐           ┌─────────────────┐         ┌─────────────┐ │
│  │             │           │                 │         │             │ │
│  │  业务数据   │           │  EventParser    │         │   Kafka     │ │
│  │     ↓       │           │  解析binlog     │         │             │ │
│  │  binlog日志 │ ──模拟──→ │       ↓         │ ──────→ │   Flink     │ │
│  │             │  Slave    │  EventSink      │         │             │ │
│  │             │           │  过滤/路由      │         │   Redis     │ │
│  │             │           │       ↓         │         │             │ │
│  │             │           │  EventStore     │         │   ES        │ │
│  │             │           │  存储/消费      │         │             │ │
│  └─────────────┘           └─────────────────┘         └─────────────┘ │
│                                                                         │
│  工作流程：                                                              │
│  1. Canal模拟MySQL Slave，发送dump协议                   │
│  2. MySQL收到dump请求，推送binlog给Canal                                │
│  3. Canal解析binlog，发送给下游                                         │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**适用场景**：
- 数据库实时同步
- 缓存更新
- 搜索索引构建
- 数据变更审计

## 计算层组件

### 1. MapReduce

MapReduce是Hadoop的分布式计算框架，采用分而治之的思想。

```
MapReduce执行流程：

┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  输入文件                                                                 │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │  HDFS: /input/data.txt (被切分为多个Split)                        │  │
│  └──────────────────────────────────────────────────────────────────┘  │
│                              ↓                                          │
│  Map阶段（并行执行）                                                      │
│  ┌────────────┐ ┌────────────┐ ┌────────────┐ ┌────────────┐          │
│  │   Mapper 1 │ │   Mapper 2 │ │   Mapper 3 │ │   Mapper 4 │          │
│  │            │ │            │ │            │ │            │          │
│  │ 读取Split  │ │ 读取Split  │ │ 读取Split  │ │ 读取Split  │          │
│  │ 解析数据   │ │ 解析数据   │ │ 解析数据   │ │ 解析数据   │          │
│  │ Map输出    │ │ Map输出    │ │ Map输出    │ │ Map输出    │          │
│  │ (K,V)     │ │ (K,V)     │ │ (K,V)     │ │ (K,V)     │          │
│  └─────┬──────┘ └─────┬──────┘ └─────┬──────┘ └─────┬──────┘          │
│        │              │              │              │                  │
│        └──────────────┴──────────────┴──────────────┘                  │
│                              ↓                                          │
│  Shuffle阶段（数据重组）                                                  │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │  Partition → Sort → Combiner → Group                            │  │
│  │  相同Key的数据发送到同一个Reducer                                  │  │
│  └──────────────────────────────────────────────────────────────────┘  │
│                              ↓                                          │
│  Reduce阶段（汇总计算）                                                   │
│  ┌────────────┐ ┌────────────┐ ┌────────────┐                         │
│  │  Reducer 1 │ │  Reducer 2 │ │  Reducer 3 │                         │
│  │            │ │            │ │            │                         │
│  │ 接收数据   │ │ 接收数据   │ │ 接收数据   │                         │
│  │ Reduce计算 │ │ Reduce计算 │ │ Reduce计算 │                         │
│  │ 输出结果   │ │ 输出结果   │ │ 输出结果   │                         │
│  └─────┬──────┘ └─────┬──────┘ └─────┬──────┘                         │
```

## 常见问题

### Q1: 如何选择合适的存储系统？

选择依据：
- **HDFS**：大文件顺序读写、离线批处理
- **HBase**：海量数据随机读写、实时查询
- **Kafka**：消息缓冲、流式数据
- **ClickHouse**：OLAP分析、实时查询

### Q2: Spark和MapReduce如何选择？

对比维度：
- **Spark**：内存计算、迭代计算、实时性要求高
- **MapReduce**：磁盘计算、一次性计算、资源受限

### Q3: Hive和Presto的区别是什么？

核心区别：
- **Hive**：基于MapReduce，适合离线批处理，延迟高
- **Presto**：纯内存计算，适合交互式查询，延迟低

### Q4: 如何选择流处理框架？

选择依据：
- **Flink**：低延迟、精确一次语义、状态管理
- **Spark Streaming**：微批处理、与Spark生态集成
- **Storm**：低延迟、简单场景

### Q5: 大数据平台如何做技术选型？

选型原则：
- 业务需求优先
- 团队技术能力
- 开源社区活跃度
- 运维复杂度
- 成本考量

## 小结

大数据技术栈涵盖了数据生命周期的各个环节：采集层负责数据收集，存储层负责数据持久化，计算层负责数据处理，查询分析层负责数据查询。各组件之间相互协作，共同构成完整的大数据平台。

作为解决方案架构师，需要深入理解各组件的原理和适用场景，根据业务需求、数据特点、性能要求等因素，选择合适的技术组件，构建高效、可靠、可扩展的大数据架构。

---

## 面试回答总结

大数据技术栈主要包括以下几个层次：数据采集层有Flume用于日志收集、Sqoop用于数据库同步、Canal用于增量数据捕获；存储层有HDFS分布式文件系统、HBase列式数据库、Kafka消息队列；计算层有MapReduce批处理框架、Spark内存计算引擎、Flink流处理引擎；查询分析层有Hive数据仓库、Presto交互式查询引擎、ClickHouse OLAP数据库；资源调度层有YARN负责集群资源管理。各组件的作用分别是：采集层负责从各种数据源收集数据，存储层负责海量数据的可靠存储，计算层负责分布式数据处理，查询分析层提供SQL查询能力，资源调度层实现资源的统一管理和调度。作为架构师，我认为技术选型需要根据业务场景、数据特点、性能要求和团队能力综合考量，避免过度设计或技术栈过于复杂。
