# 云数据备份恢复策略

## 为什么需要云数据备份恢复策略？

想象你的公司数据库突然被勒索病毒加密，或者误删除了关键业务数据。如果没有备份，这些数据可能永远丢失，导致业务停摆、客户流失、甚至公司倒闭。

云数据备份恢复策略就像"保险箱"——平时可能觉得多余，但关键时刻能救命。它确保在数据丢失、损坏、被攻击等情况下，能够快速恢复业务运行。

### 备份类型

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  备份类型对比：                                                           │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                                                                 │   │
│  │  全量备份（Full Backup）：                                       │   │
│  │  ├── 定义：备份所有数据                                         │   │
│  │  ├── 优点：恢复简单、完整                                       │   │
│  │  ├── 缺点：耗时长、占用空间大                                   │   │
│  │  └── 频率：每周一次                                             │   │
│  │                                                                 │   │
│  │  增量备份（Incremental Backup）：                                │   │
│  │  ├── 定义：备份自上次备份以来变化的数据                         │   │
│  │  ├── 优点：速度快、占用空间小                                   │   │
│  │  ├── 缺点：恢复复杂、依赖链                                     │   │
│  │  └── 频率：每天一次                                             │   │
│  │                                                                 │   │
│  │  差异备份（Differential Backup）：                               │   │
│  │  ├── 定义：备份自上次全量备份以来变化的数据                     │   │
│  │  ├── 优点：恢复比增量简单                                       │   │
│  │  ├── 缺点：占用空间比增量大                                     │   │
│  │  └── 频率：每天一次                                             │   │
│  │                                                                 │   │
│  │  备份策略组合：                                                  │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │                                                         │   │   │
│  │  │  周日：全量备份                                           │   │   │
│  │  │  周一：增量备份                                           │   │   │
│  │  │  周二：增量备份                                           │   │   │
│  │  │  周三：增量备份                                           │   │   │
│  │  │  周四：增量备份                                           │   │   │
│  │  │  周五：增量备份                                           │   │   │
│  │  │  周六：增量备份                                           │   │   │
│  │  │                                                         │   │   │
│  │  │  恢复时：周日全量 + 周一到目标日期的增量                   │   │   │
│  │  │                                                         │   │   │
│  │  └─────────────────────────────────────────────────────────┘   │   │
│  │                                                                 │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### AWS RDS备份配置示例

```yaml
# Terraform配置示例
resource "aws_db_instance" "production" {
  identifier             = "production-db"
  engine                 = "mysql"
  engine_version         = "8.0"
  instance_class         = "db.r5.xlarge"
  allocated_storage      = 500
  storage_encrypted      = true
  
  # 自动备份配置
  backup_retention_period = 30          # 保留30天
  backup_window          = "03:00-04:00" # 备份窗口
  
  # 时间点恢复
  skip_final_snapshot    = false
  final_snapshot_identifier = "production-db-final"
  
  # 跨区域复制
  replicate_source_db    = null
  
  # 删除保护
  deletion_protection    = true
  
  # 多可用区部署
  multi_az               = true
  
  # 参数组
  parameter_group_name   = aws_db_parameter_group.production.name
  
  tags = {
    Environment = "production"
    Backup      = "enabled"
  }
}

# 跨区域备份复制
resource "aws_db_instance_replica" "replica" {
  identifier             = "production-db-replica"
  replicate_source_db    = aws_db_instance.production.arn
  instance_class         = "db.r5.xlarge"
  
  # 备份配置
  backup_retention_period = 30
  
  tags = {
    Environment = "dr"
    Type        = "replica"
  }
}

# 手动快照
resource "aws_db_snapshot" "pre_upgrade" {
  db_instance_identifier = aws_db_instance.production.identifier
  db_snapshot_identifier = "production-db-pre-upgrade-${formatdate("YYYYMMDD", timestamp())}"
  
  tags = {
    Reason = "Pre-upgrade backup"
  }
}
```

### S3备份配置示例

```yaml
# Terraform配置示例

# 启用版本控制
resource "aws_s3_bucket_versioning" "backup" {
  bucket = aws_s3_bucket.backup.id
  versioning_configuration {
    status = "Enabled"
  }
}

# 跨区域复制
resource "aws_s3_bucket_replication_configuration" "backup" {
  bucket = aws_s3_bucket.backup.id
  role   = aws_iam_role.replication.arn
  
  rule {
    id     = "replicate-all"
    status = "Enabled"
    
    destination {
      bucket        = aws_s3_bucket.backup_dr.arn
      storage_class = "STANDARD"
    }
  }
}

# 生命周期策略
resource "aws_s3_bucket_lifecycle_configuration" "backup" {
  bucket = aws_s3_bucket.backup.id
  
  rule {
    id     = "transition-to-ia"
    status = "Enabled"
    
    transition {
      days          = 30
      storage_class = "STANDARD_IA"
    }
    
    transition {
      days          = 90
      storage_class = "GLACIER"
    }
    
    expiration {
      days = 365
    }
    
    noncurrent_version_transition {
      noncurrent_days = 30
      storage_class   = "STANDARD_IA"
    }
    
    noncurrent_version_expiration {
      noncurrent_days = 90
    }
  }
}

# 对象锁定（WORM）
resource "aws_s3_bucket_object_lock_configuration" "backup" {
  bucket = aws_s3_bucket.backup.id
  
  rule {
    default_retention {
      mode  = "COMPLIANCE"
      days  = 365
    }
  }
}
```

### Velero配置示例

```yaml
# Velero备份配置
apiVersion: velero.io/v1
kind: BackupStorageLocation
metadata:
  name: default
  namespace: velero
spec:
  provider: aws
  objectStorage:
    bucket: my-cluster-backup
  config:
    region: us-east-1
---
apiVersion: velero.io/v1
kind: Schedule
metadata:
  name: daily-backup
  namespace: velero
spec:
  schedule: "0 2 * * *"  # 每天凌晨2点
  template:
    includedNamespaces:
      - production
      - staging
    excludedResources:
      - events
      - pods
    storageLocation: default
    ttl: 720h  # 保留30天
    hooks:
      resources:
        - name: pre-backup-hook
          includedNamespaces:
            - production
          labelSelector:
            matchLabels:
              app: mysql
          pre:
            - exec:
                container: mysql
                command:
                  - /bin/bash
                  - -c
                  - "mysql -e 'FLUSH TABLES WITH READ LOCK;'"
                onError: Continue
---
apiVersion: velero.io/v1
kind: Backup
metadata:
  name: pre-upgrade-backup
  namespace: velero
spec:
  includedNamespaces:
    - production
  storageLocation: default
  ttl: 2160h  # 保留90天
  hooks:
    resources:
      - name: pre-backup-hook
        pre:
          - exec:
              container: app
              command:
                - /bin/bash
                - -c
                - "echo 'Starting backup'"
```

## 恢复策略设计

### 恢复时间目标（RTO）和恢复点目标（RPO）

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  RTO和RPO定义：                                                          │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                                                                 │   │
│  │  RTO（Recovery Time Objective）：                                │   │
│  │  ├── 定义：业务恢复所需的最长时间                               │   │
│  │  ├── 示例：4小时内恢复业务                                      │   │
│  │  └── 影响：决定备份恢复方案的选择                               │   │
│  │                                                                 │   │
│  │  RPO（Recovery Point Objective）：                               │   │
│  │  ├── 定义：可接受的数据丢失量                                   │   │
│  │  ├── 示例：最多丢失1小时数据                                    │   │
```

## 常见问题

### Q1: 如何选择备份保留期？

**建议**：
- 日备份：保留7-30天
- 周备份：保留4-12周
- 月备份：保留12-24个月
- 年备份：根据合规要求保留

### Q2: 如何处理大规模数据备份？

**策略**：
- 增量备份减少数据量
- 分片备份并行处理
- 使用专用备份网络
- 压缩和去重

### Q3: 如何验证备份数据完整性？

**验证方法**：
- 校验和验证
- 定期恢复测试
- 数据一致性检查
- 应用级验证

### Q4: 云备份如何控制成本？

**成本优化**：
- 生命周期策略自动分层
- 删除过期备份
- 压缩备份数据
- 选择合适的存储类型

### Q5: 如何应对勒索病毒？

**防护措施**：
- 不可变备份（WORM）
- 异地备份
- 离线备份
- 访问控制
- 定期恢复测试

## 小结

云数据备份恢复策略是保障数据安全的最后一道防线，需要从以下维度设计：

1. **备份策略**：遵循3-2-1原则，全量+增量备份组合
2. **备份类型**：自动备份、手动快照、跨区域备份
3. **恢复能力**：时间点恢复（PITR）、快速恢复
4. **RTO/RPO**：根据业务重要性设定合理目标
5. **最佳实践**：定期测试、加密、监控、文档化、自动化

## 面试回答总结

云数据备份恢复策略设计需要遵循3-2-1原则（3个副本、2种介质、1个异地）。备份类型包括全量备份（备份所有数据，恢复简单但耗时长）、增量备份（备份变化数据，速度快但恢复复杂）、差异备份（备份自上次全量以来的变化）。云数据库备份策略包括自动备份（每日全量+增量备份、事务日志备份）、手动快照（长期保留、跨区域复制）、时间点恢复PITR（恢复到任意时间点）。对象存储备份策略包括版本控制（保留所有版本）、跨区域复制CRR（异地容灾）、生命周期策略（自动分层存储）。Kubernetes备份使用Velero备份资源和PV数据。恢复策略需要定义RTO（恢复时间目标）和RPO（恢复点目标），关键业务系统RTO<1小时、RPO<15分钟，核心业务系统RTO<4小时、RPO<1小时。最佳实践包括定期测试恢复、加密备份数据、监控备份状态、文档化、自动化、满足合规性要求。
