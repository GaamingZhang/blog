# 分布式事务方案设计深度解析

## 为什么需要分布式事务?

想象你在电商平台购买了一件商品,系统需要完成三个操作:创建订单、扣减库存、扣减账户余额。在单体应用中,这三个操作在同一个数据库事务中完成,要么全部成功,要么全部失败。但在微服务架构中,订单服务、库存服务、账户服务分别使用不同的数据库,如何保证这三个操作的一致性?

这就是分布式事务要解决的核心问题:在分布式系统中,如何保证跨服务、跨数据库的操作满足ACID特性或最终一致性。

### 2. TCC(Try-Confirm-Cancel)

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  TCC事务模式:                                                            │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                                                                 │   │
│  │  核心思想: 将业务逻辑拆分为三个阶段                             │   │
│  │                                                                 │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │                                                         │   │   │
│  │  │  1. Try阶段: 尝试执行,预留资源                           │   │   │
│  │  │                                                         │   │   │
│  │  │  示例: 转账场景                                          │   │   │
│  │  │  ├── A账户: 冻结100元(余额不变,冻结金额+100)             │   │   │
│  │  │  └── B账户: 预增100元(余额不变,预增金额+100)             │   │   │
│  │  │                                                         │   │   │
│  │  └─────────────────────────────────────────────────────────┘   │   │
│  │                           │                                    │   │
│  │                           ↓                                    │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │                                                         │   │   │
│  │  │  2. Confirm阶段: 确认执行,提交事务                       │   │   │
│  │  │                                                         │   │   │
│  │  │  示例: 转账成功                                          │   │   │
│  │  │  ├── A账户: 扣减余额100元,清除冻结金额100元              │   │   │
│  │  │  └── B账户: 增加余额100元,清除预增金额100元              │   │   │
│  │  │                                                         │   │   │
│  │  └─────────────────────────────────────────────────────────┘   │   │
│  │                           │                                    │   │
│  │                           ↓                                    │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │                                                         │   │   │
│  │  │  3. Cancel阶段: 取消执行,回滚事务                        │   │   │
│  │  │                                                         │   │   │
│  │  │  示例: 转账失败                                          │   │   │
│  │  │  ├── A账户: 清除冻结金额100元                            │   │   │
│  │  │  └── B账户: 清除预增金额100元                            │   │   │
│  │  │                                                         │   │   │
│  │  └─────────────────────────────────────────────────────────┘   │   │
│  │                           │                                    │   │
│  │                           ↓                                    │   │
│  │  数据库设计:                                                    │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │                                                         │   │   │
│  │  │  账户表:                                                 │   │   │
│  │  │  ┌──────────────┬──────────┬──────────┬──────────┐     │   │   │
│  │  │  │ account_id   │ balance  │ frozen   │ pre_add  │     │   │   │
│  │  │  ├──────────────┼──────────┼──────────┼──────────┤     │   │   │
│  │  │  │ A            │ 1000     │ 100      │ 0        │     │   │   │
│  │  │  │ B            │ 500      │ 0        │ 100      │     │   │   │
│  │  │  └──────────────┴──────────┴──────────┴──────────┘     │   │   │
│  │  │                                                         │   │   │
│  │  │  Try: A.frozen=100, B.pre_add=100                       │   │   │
│  │  │  Confirm: A.balance=900, A.frozen=0                     │   │   │
│  │  │           B.balance=600, B.pre_add=0                    │   │   │
│  │  │  Cancel: A.frozen=0, B.pre_add=0                        │   │   │
│  │  │                                                         │   │   │
│  │  └─────────────────────────────────────────────────────────┘   │   │
│  │                           │                                    │   │
│  │                           ↓                                    │   │
│  │  优缺点:                                                        │   │
│  │  ├── 优点: 最终一致性,资源锁定粒度小                           │   │
│  │  ├── 缺点: 业务侵入性强,需要实现三个接口                       │   │
│  │  └── 适用: 对性能要求高,业务逻辑可控的场景                     │   │
│  │                                                                 │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**TCC实现示例**:

```java
// TCC事务协调器
@Service
public class TransferTccService {
    
    @Autowired
    private AccountTccService accountTccService;
    
    @GlobalTransactional
    public void transfer(String fromAccountId, String toAccountId, 
                        BigDecimal amount) {
        // Try阶段
        accountTccService.tryDeduct(fromAccountId, amount);
        accountTccService.tryAdd(toAccountId, amount);
        
        // Confirm/Cancel由TCC框架自动调用
    }
}

// 账户TCC服务
@Service
public class AccountTccService {
    
    @Autowired
    private AccountRepository accountRepository;
    
    // Try: 冻结金额
    public void tryDeduct(String accountId, BigDecimal amount) {
        Account account = accountRepository.findById(accountId);
        
        // 检查余额是否充足
        if (account.getBalance().compareTo(amount) < 0) {
            throw new InsufficientBalanceException("余额不足");
        }
        
        // 冻结金额
        account.setFrozen(account.getFrozen().add(amount));
        accountRepository.save(account);
    }
    
    // Confirm: 扣减余额,清除冻结
    public void confirmDeduct(String accountId, BigDecimal amount) {
        Account account = accountRepository.findById(accountId);
        
        // 扣减余额
        account.setBalance(account.getBalance().subtract(amount));
        // 清除冻结
        account.setFrozen(account.getFrozen().subtract(amount));
        
        accountRepository.save(account);
    }
    
    // Cancel: 清除冻结
    public void cancelDeduct(String accountId, BigDecimal amount) {
        Account account = accountRepository.findById(accountId);
        
        // 清除冻结
        account.setFrozen(account.getFrozen().subtract(amount));
        
        accountRepository.save(account);
    }
}
```

```
│  │  │     ├── 中心协调器统一调度                               │   │   │

│  │  └── 适用: 业务流程长,对实时性要求不高的场景                   │   │
```

**Saga实现示例(编排式)**:

```java
// 订单服务
@Service
public class OrderService {
    
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;
    
    public Order createOrder(OrderRequest request) {
        // 1. 创建订单
        Order order = new Order(request);
        orderRepository.save(order);
        
        // 2. 发布订单创建事件
        OrderCreatedEvent event = new OrderCreatedEvent(
            order.getId(),
            order.getItems()
        );
        kafkaTemplate.send("order-created", JSON.toJSONString(event));
        
        return order;
    }
    
    // 补偿操作: 取消订单
    @KafkaListener(topics = "order-cancel")
    public void cancelOrder(String message) {
        OrderCancelEvent event = JSON.parseObject(message, OrderCancelEvent.class);
        Order order = orderRepository.findById(event.getOrderId());
        order.setStatus(OrderStatus.CANCELLED);
        orderRepository.save(order);
    }
}

// 库存服务
@Service
public class InventoryService {
    
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;
    
    @KafkaListener(topics = "order-created")
    public void deductInventory(String message) {
        OrderCreatedEvent event = JSON.parseObject(message, OrderCreatedEvent.class);
        
        try {
            // 扣减库存
            inventoryRepository.deduct(event.getItems());
            
            // 发布库存扣减成功事件
            InventoryDeductedEvent deductedEvent = new InventoryDeductedEvent(
                event.getOrderId()
            );
            kafkaTemplate.send("inventory-deducted", JSON.toJSONString(deductedEvent));
            
        } catch (Exception e) {
            // 发布库存扣减失败事件
            InventoryDeductFailedEvent failedEvent = new InventoryDeductFailedEvent(
                event.getOrderId()
            );
            kafkaTemplate.send("inventory-deduct-failed", JSON.toJSONString(failedEvent));
        }
    }
    
    // 补偿操作: 恢复库存
    @KafkaListener(topics = "inventory-restore")
    public void restoreInventory(String message) {
        InventoryRestoreEvent event = JSON.parseObject(message, InventoryRestoreEvent.class);
        inventoryRepository.restore(event.getOrderId(), event.getItems());
    }
}

// 补偿流程编排
@Service
public class SagaOrchestrator {
    
    @KafkaListener(topics = "inventory-deduct-failed")
    public void handleInventoryDeductFailed(String message) {
        InventoryDeductFailedEvent event = JSON.parseObject(message, 
            InventoryDeductFailedEvent.class);
        
        // 发起补偿流程
        // 1. 取消订单
        kafkaTemplate.send("order-cancel", 
            JSON.toJSONString(new OrderCancelEvent(event.getOrderId())));
    }
}
```

**事务消息实现示例**:

```java
// RocketMQ事务消息生产者
@Service
public class OrderTransactionProducer {
    
    @Autowired
    private RocketMQTemplate rocketMQTemplate;
    
    public void createOrder(OrderRequest request) {
        // 构建消息
        Message<OrderCreatedEvent> message = MessageBuilder
            .withPayload(new OrderCreatedEvent(request))
            .build();
        
        // 发送事务消息
        rocketMQTemplate.sendMessageInTransaction(
            "order-group",           // 事务组名
            "order-created",         // Topic
            message,                 // 消息
            request                  // 本地事务参数
        );
    }
}

// 事务消息监听器
@RocketMQTransactionListener(rocketMQTemplateBeanName = "rocketMQTemplate")
public class OrderTransactionListener implements RocketMQLocalTransactionListener {
    
    @Autowired
    private OrderService orderService;
    
    @Autowired
    private OrderRepository orderRepository;
    
    // 执行本地事务
    @Override
    public RocketMQLocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        try {
            OrderRequest request = (OrderRequest) arg;
            
            // 执行本地事务: 创建订单
            orderService.createOrder(request);
            
            // 返回提交状态
            return RocketMQLocalTransactionState.COMMIT;
            
        } catch (Exception e) {
            // 返回回滚状态
            return RocketMQLocalTransactionState.ROLLBACK;
        }
    }
    
    // 事务回查
    @Override
    public RocketMQLocalTransactionState checkLocalTransaction(Message msg) {
        // 从消息中获取订单ID
        OrderCreatedEvent event = (OrderCreatedEvent) msg.getPayload();
        
        // 查询订单是否存在
        Order order = orderRepository.findById(event.getOrderId());
        
        if (order != null) {
            return RocketMQLocalTransactionState.COMMIT;
        } else {
            return RocketMQLocalTransactionState.ROLLBACK;
        }
    }
}
```

## 常见问题

### Q1: 如何保证幂等性?

**解决方案**:
- 唯一业务ID(订单号、流水号)
- 数据库唯一索引
- Redis分布式锁
- 状态机控制

### Q2: 如何处理补偿失败?

**解决方案**:
- 重试机制(指数退避)
- 人工介入
- 告警通知
- 定期对账

### Q3: 如何保证消息不丢失?

**解决方案**:
- 消息持久化
- ACK确认机制
- 消息重试
- 死信队列

### Q4: 如何避免分布式事务?

**设计建议**:
- 业务流程优化,减少跨服务事务
- 使用最终一致性替代强一致性
- 合理拆分服务边界
- 使用领域事件解耦

### Q5: Seata框架如何选择模式?

**模式选择**:
- AT模式: 无侵入,适合大多数场景
- TCC模式: 高性能,需要实现三个接口
- Saga模式: 长流程,适合编排式事务
- XA模式: 强一致性,性能较低

## 小结

分布式事务是微服务架构的核心挑战,需要根据业务场景选择合适的方案。

**核心方案**:
- **2PC**: 强一致性,性能低,适合金融场景
- **TCC**: 最终一致性,高性能,业务侵入性强
- **Saga**: 最终一致性,适合长流程业务
- **本地消息表**: 实现简单,可靠性高
- **事务消息**: 实时性好,依赖特定MQ

**设计原则**:
- 优先考虑业务流程优化
- 选择合适的一致性级别
- 保证幂等性和可补偿性
- 建立完善的监控和告警
- 设计合理的重试和补偿机制

**最佳实践**:
- 使用成熟框架(Seata)
- 建立完善的事务日志
- 定期对账和补偿
- 压测验证方案可行性
- 持续优化和演进

分布式事务没有银弹,关键是要在一致性、性能、复杂度之间找到平衡点。

## 面试回答总结

分布式事务方案设计需要根据业务场景选择合适的方案。主要方案有五种:2PC(两阶段提交)通过协调者统一管理参与者,Prepare阶段执行事务但不提交,Commit阶段统一提交,优点是强一致性,缺点是同步阻塞、单点故障、性能低,适合金融交易等对一致性要求极高的场景。TCC(Try-Confirm-Cancel)将业务逻辑拆分为三个阶段:Try阶段预留资源(如冻结金额)、Confirm阶段确认提交、Cancel阶段回滚补偿,优点是性能高、资源锁定粒度小,缺点是业务侵入性强需要实现三个接口,适合高并发场景。Saga模式将长事务拆分为多个本地事务,每个事务有对应的补偿操作,通过编排式(事件驱动)或协调式(中心协调器)实现,优点是适合长流程业务,缺点是补偿逻辑复杂、无隔离性。本地消息表在本地数据库中记录消息,保证本地事务和消息发送的原子性,通过定时任务扫描发送消息,优点是实现简单可靠性高,缺点是延迟较高。事务消息(如RocketMQ)通过半消息机制,先发送不可消费的半消息,执行本地事务后提交或回滚消息,优点是实时性好,缺点是依赖特定MQ。选型建议:强一致性要求用2PC或Seata AT模式,高并发场景用TCC,长流程业务用Saga,使用RocketMQ用事务消息,一般场景用本地消息表。关键是要保证幂等性(唯一业务ID、数据库唯一索引)、处理补偿失败(重试、人工介入、告警)、保证消息不丢失(持久化、ACK确认),并建立完善的监控告警和定期对账机制。
