# 云身份认证和访问管理（IAM）设计

## 什么是云IAM？

想象一家大型企业，有数千名员工、数百个系统和海量数据。传统方式下，每个系统都有独立的账号密码，员工需要记住多套凭证，管理员需要在多个系统中管理权限。这不仅效率低下，还存在安全隐患。

云IAM（Identity and Access Management）就像企业的"统一门禁系统"——员工只需一张门禁卡，就能根据权限访问不同区域，管理员可以集中管理所有人的访问权限，所有进出记录都被完整审计。

### 身份类型详解

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  身份类型详解：                                                           │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                                                                 │   │
│  │  用户（Users）：                                                 │   │
│  │  ├── 定义：代表具体人员                                         │   │
│  │  ├── 特点：长期身份、需要登录凭证                               │   │
│  │  ├── 示例：员工账号、外部合作伙伴账号                           │   │
│  │  └── 最佳实践：                                                 │   │
│  │      ├── 使用企业身份联合（SAML/OIDC）                          │   │
│  │      ├── 启用MFA                                                │   │
│  │      └── 避免使用根账号                                         │   │
│  │                                                                 │   │
│  │  组（Groups）：                                                  │   │
│  │  ├── 定义：用户的集合                                           │   │
│  │  ├── 特点：便于批量管理权限                                     │   │
│  │  ├── 示例：开发组、运维组、管理员组                             │   │
│  │  └── 最佳实践：                                                 │   │
│  │      ├── 按职能划分组                                           │   │
│  │      ├── 为组而非用户分配权限                                   │   │
│  │      └── 定期审查组成员                                         │   │
│  │                                                                 │   │
│  │  角色（Roles）：                                                 │   │
│  │  ├── 定义：临时身份                                             │   │
│  │  ├── 特点：无长期凭证、可被用户/服务扮演                        │   │
│  │  ├── 示例：EC2访问S3的角色、跨账号访问角色                      │   │
│  │  └── 最佳实践：                                                 │   │
│  │      ├── 使用角色而非长期凭证                                   │   │
│  │      ├── 设置信任策略                                           │   │
│  │      └── 定期轮换角色                                           │   │
│  │                                                                 │   │
│  │  服务账号（Service Accounts）：                                  │   │
│  │  ├── 定义：代表应用或服务                                       │   │
│  │  ├── 特点：用于服务间认证                                       │   │
│  │  ├── 示例：Kubernetes ServiceAccount、应用身份                  │   │
│  │  └── 最佳实践：                                                 │   │
│  │      ├── 使用工作负载身份                                       │   │
│  │      ├── 限制权限范围                                           │   │
│  │      └── 定期轮换凭证                                           │   │
│  │                                                                 │   │
```

### 最小权限原则实践

```yaml
# 最小权限策略示例

# 错误示例：过度授权
resource "aws_iam_policy" "bad_example" {
  name = "bad-policy"
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect = "Allow"
      Action = ["s3:*"]        # 所有S3操作，风险极高
      Resource = ["*"]         # 所有资源，风险极高
    }]
  })
}

# 正确示例：最小权限
resource "aws_iam_policy" "good_example" {
  name = "good-policy"
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect = "Allow"
      Action = [
        "s3:GetObject",
        "s3:ListBucket"
      ]
      Resource = [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ]
      Condition = {
        IpAddress = {
          "aws:SourceIp" = "10.0.0.0/8"
        }
        StringEquals = {
          "aws:PrincipalTag/Department" = "Finance"
        }
      }
    }]
  })
}

# 使用数据源限制权限
data "aws_iam_policy_document" "limited_access" {
  statement {
    effect = "Allow"
    
    actions = [
      "s3:GetObject",
      "s3:PutObject"
    ]
    
    resources = [
      "arn:aws:s3:::${var.bucket_name}/*"
    ]
    
    condition {
      test     = "StringEquals"
      variable = "s3:x-amz-acl"
      values   = ["private"]
    }
  }
}
```

## 常见问题

### Q1: 如何实现跨账号访问？

**方案**：
1. 在目标账号创建角色
2. 配置信任策略允许源账号扮演
3. 源账号用户扮演角色获取临时凭证

### Q2: 如何撤销紧急访问权限？

**步骤**：
1. 禁用用户账号
2. 删除访问密钥
3. 移除用户组
4. 吊销活动会话

### Q3: 如何处理权限过多的问题？

**解决方案**：
1. 权限审计
2. 使用Access Advisor查看使用情况
3. 删除未使用的权限
4. 实施权限边界

### Q4: 如何实现临时提权？

**方案**：
- 使用角色切换
- 设置会话时长限制
- 审计所有提权操作
- 自动过期

### Q5: 如何防止权限泄露？

**防护措施**：
- 不在代码中硬编码凭证
- 使用角色和环境变量
- 定期扫描泄露
- 使用Secrets Manager

## 小结

云IAM是云安全的基石，需要从以下维度设计：

1. **身份管理**：用户、组、角色、服务账号
2. **权限模型**：RBAC、ABAC
3. **策略设计**：最小权限原则、条件限制
4. **身份联合**：SAML、OIDC实现SSO
5. **安全实践**：MFA、凭证轮换、监控审计

## 面试回答总结

云IAM（身份认证和访问管理）设计需要从身份管理、认证、授权、审计四个维度构建。核心组件包括：用户（代表人员）、组（用户集合）、角色（临时身份）、服务账号（应用身份）。权限模型包括RBAC（基于角色的访问控制，用户→角色→权限）和ABAC（基于属性的访问控制，根据用户属性、资源属性、环境属性动态判断权限）。IAM策略包含Effect（Allow/Deny）、Action（操作）、Resource（资源）、Condition（条件）四个要素。设计原则遵循最小权限原则，仅授予完成任务所需的最小权限，使用条件限制（如IP地址、时间、标签）进一步约束。企业身份联合通过SAML 2.0或OIDC与企业AD集成，实现单点登录SSO，无需在云中创建用户。安全最佳实践包括：保护根账号（启用MFA、禁用访问密钥）、使用角色而非长期凭证、强制MFA、定期轮换凭证、启用CloudTrail审计、使用服务控制策略SCP限制权限。跨账号访问通过在目标账号创建角色并配置信任策略实现。
