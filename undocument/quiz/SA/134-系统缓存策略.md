# 系统缓存策略设计深度解析

## 为什么缓存如此重要?

想象你经营着一家图书馆,每天有成千上万的读者来借书。如果每个读者都要去书库找书,效率会非常低。聪明的做法是在前台设置一个热门书籍展示区,把借阅频率高的书放在这里,读者可以直接取走,无需进入书库。这个展示区就是"缓存"。

在计算机系统中,缓存是提升性能的利器。合理使用缓存可以将响应时间从秒级降低到毫秒级,将数据库压力降低90%以上。但缓存也是一把双刃剑,使用不当会导致数据不一致、缓存穿透、雪崩等问题。

## 缓存更新策略

### 1. 缓存更新模式

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  缓存更新策略:                                                           │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                                                                 │   │
│  │  1. Cache Aside(旁路缓存)                                       │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │                                                         │   │   │
│  │  │  读操作:                                                 │   │   │
│  │  │  ├── 1. 先读缓存                                        │   │   │
│  │  │  ├── 2. 缓存命中,直接返回                               │   │   │
│  │  │  ├── 3. 缓存未命中,读数据库                             │   │   │
│  │  │  └── 4. 写入缓存,返回结果                               │   │   │
│  │  │                                                         │   │   │
│  │  │  写操作:                                                 │   │   │
│  │  │  ├── 1. 先更新数据库                                    │   │   │
│  │  │  └── 2. 再删除缓存(推荐)或更新缓存                      │   │   │
│  │  │                                                         │   │   │
│  │  │  示例代码:                                               │   │   │
│  │  │  public Product getProduct(Long id) {                   │   │   │
│  │  │      Product product = cache.get(id);                   │   │   │
│  │  │      if (product == null) {                             │   │   │
│  │  │          product = db.query(id);                        │   │   │
│  │  │          cache.set(id, product, ttl);                   │   │   │
│  │  │      }                                                  │   │   │
│  │  │      return product;                                    │   │   │
│  │  │  }                                                      │   │   │
│  │  │                                                         │   │   │
│  │  │  public void updateProduct(Product product) {           │   │   │
│  │  │      db.update(product);                                │   │   │
│  │  │      cache.delete(product.getId());  // 删除缓存        │   │   │
│  │  │  }                                                      │   │   │
│  │  │                                                         │   │   │
│  │  └─────────────────────────────────────────────────────────┘   │   │
│  │                           │                                    │   │
│  │                           ↓                                    │   │
│  │  2. Read/Write Through(读写穿透)                               │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │                                                         │   │   │
│  │  │  读操作:                                                 │   │   │
│  │  │  ├── 缓存代理负责读取数据库                             │   │   │
│  │  │  └── 应用只与缓存交互                                   │   │   │
│  │  │                                                         │   │   │
│  │  │  写操作:                                                 │   │   │
│  │  │  ├── 缓存代理负责更新数据库                             │   │   │
│  │  │  └── 同步更新缓存                                       │   │   │
│  │  │                                                         │   │   │
│  │  │  优势: 应用代码简单,缓存与数据库一致性由代理保证         │   │   │
│  │  │  劣势: 需要缓存中间件支持                               │   │   │
│  │  │                                                         │   │   │
│  │  └─────────────────────────────────────────────────────────┘   │   │
│  │                           │                                    │   │
│  │                           ↓                                    │   │
│  │  3. Write Behind(异步写入)                                     │   │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │                                                         │   │   │
│  │  │  写操作:                                                 │   │   │
│  │  │  ├── 只更新缓存                                         │   │   │
│  │  │  └── 异步批量写入数据库                                 │   │   │
│  │  │                                                         │   │   │
│  │  │  优势: 写入性能极高                                     │   │   │
│  │  │  劣势: 数据可能丢失,一致性难以保证                      │   │   │
│  │  │                                                         │   │   │
│  │  └─────────────────────────────────────────────────────────┘   │   │
│  │                           │                                    │   │
│  │                           ↓                                    │   │
│  │  推荐策略: Cache Aside + 删除缓存                              │   │
│  │                                                                 │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2. 缓存更新时机

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  缓存更新时机选择:                                                       │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                                                                 │   │
│  │  ┌──────────┬────────────────────────────────────────────────┐ │   │
│  │  │  策略    │  说明                                         │ │   │
│  │  ├──────────┼────────────────────────────────────────────────┤ │   │
│  │  │  删除缓存│  更新数据库后删除缓存,下次读取时重建          │ │   │

│  │  │          │  缺点: 增加一次缓存未命中                     │ │   │
│  │  ├──────────┼────────────────────────────────────────────────┤ │   │
│  │  │  更新缓存│  更新数据库后更新缓存                         │ │   │
│  │  │          │  优点: 缓存始终可用                           │ │   │
│  │  │          │  缺点: 并发时可能产生脏数据                   │ │   │
│  │  ├──────────┼────────────────────────────────────────────────┤ │   │
│  │  │  延迟双删│  更新数据库前后各删除一次缓存                 │ │   │
│  │  │          │  优点: 解决并发时的脏数据问题                 │ │   │
│  │  │          │  缺点: 增加延迟,实现复杂                     │ │   │
│  │  └──────────┴────────────────────────────────────────────────┘ │   │
│  │                                                                 │   │
│  │  延迟双删实现:                                                  │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │                                                         │   │   │
│  │  │  public void updateProduct(Product product) {           │   │   │
│  │  │      // 1. 删除缓存                                     │   │   │
│  │  │      cache.delete(product.getId());                     │   │   │
│  │  │                                                         │   │   │
│  │  │      // 2. 更新数据库                                   │   │   │
│  │  │      db.update(product);                                │   │   │
│  │  │                                                         │   │   │
│  │  │      // 3. 延迟删除缓存(解决并发脏数据)                 │   │   │
│  │  │      CompletableFuture.runAsync(() -> {                 │   │   │
│  │  │          Thread.sleep(500);  // 延迟500ms               │   │   │
│  │  │          cache.delete(product.getId());                 │   │   │
│  │  │      });                                                │   │   │
│  │  │  }                                                      │   │   │
│  │  │                                                         │   │   │
│  │  └─────────────────────────────────────────────────────────┘   │   │
│  │                                                                 │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  缓存穿透问题:                                                           │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                                                                 │   │
│  │  问题: 查询不存在的数据,缓存和数据库都没有,每次都查询数据库    │   │
│  │                                                                 │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │                                                         │   │   │
│  │  │  恶意请求: 查询id=-1的数据                              │   │   │
│  │  │      │                                                   │   │   │
│  │  │      ↓                                                   │   │   │
│  │  │  缓存未命中                                              │   │   │
│  │  │      │                                                   │   │   │
│  │  │      ↓                                                   │   │   │
│  │  │  查询数据库(也没有)                                      │   │   │
│  │  │      │                                                   │   │   │
│  │  │      ↓                                                   │   │   │
│  │  │  返回空(不缓存)                                          │   │   │
│  │  │      │                                                   │   │   │
│  │  │      ↓                                                   │   │   │
│  │  │  下次请求继续查询数据库...                               │   │   │
│  │  │                                                         │   │   │
│  │  └─────────────────────────────────────────────────────────┘   │   │
│  │                           │                                    │   │
│  │                           ↓                                    │   │
│  │  解决方案:                                                      │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │                                                         │   │   │
│  │  │  方案1: 缓存空值                                        │   │   │
│  │  │  ├── 查询结果为空,也缓存(设置较短过期时间)              │   │   │
│  │  │  └── 示例: cache.set(id, NULL, 60s)                     │   │   │
│  │  │                                                         │   │   │
│  │  │  方案2: 布隆过滤器                                      │   │   │
│  │  │  ├── 在缓存前加布隆过滤器                               │   │   │
│  │  │  ├── 布隆过滤器判断数据是否存在                         │   │   │
│  │  │  └── 不存在直接返回,避免查询数据库                      │   │   │
│  │  │                                                         │   │   │
│  │  │  方案3: 参数校验                                        │   │   │
│  │  │  ├── 校验请求参数合法性                                 │   │   │

## 常见问题

### Q1: 如何选择缓存过期时间?

**设置建议**:
- 热点数据: 5-30分钟
- 一般数据: 1-24小时
- 配置数据: 1-7天
- 静态数据: 永不过期,手动更新

### Q2: 如何保证缓存与数据库一致性?

**一致性策略**:
- 强一致性: 使用分布式锁,性能低
- 最终一致性: 先更新数据库,再删除缓存
- 延迟双删: 解决并发时的脏数据问题
- 订阅Binlog: 异步更新缓存

### Q3: 如何选择缓存框架?

**框架选择**:
- 本地缓存: Caffeine(推荐)、Guava Cache
- 分布式缓存: Redis(推荐)、Memcached
- 多级缓存: Spring Cache + Caffeine + Redis

### Q4: 如何处理大Key问题?

**解决方案**:
- 拆分大Key为多个小Key
- 使用压缩算法
- 避免存储大对象
- 定期清理无用数据

### Q5: 如何设计缓存预热机制?

**预热策略**:
- 系统启动时预加载热点数据
- 定时任务刷新缓存
- 流量低峰期预热
- 监控缓存命中率,动态调整

## 小结

缓存是系统性能优化的核心手段,需要从架构、策略、监控多个维度进行设计。

**缓存架构**:
- **多级缓存**: 浏览器 → CDN → 本地 → 分布式 → 数据库
- **本地缓存**: 低延迟,容量有限,弱一致性
- **分布式缓存**: 可扩展,强一致性,延迟稍高

**缓存策略**:
- **更新策略**: Cache Aside(推荐)、Read/Write Through、Write Behind
- **更新时机**: 删除缓存(推荐)、更新缓存、延迟双删
- **过期策略**: 随机过期时间,避免雪崩

**常见问题**:
- **缓存穿透**: 布隆过滤器、缓存空值
- **缓存击穿**: 互斥锁、永不过期
- **缓存雪崩**: 随机过期、多级缓存、熔断降级

**最佳实践**:
- 合理设置过期时间
- 保证缓存与数据库一致性
- 建立完善的监控告警
- 定期优化缓存策略
- 预热热点数据

缓存设计需要在性能、一致性、复杂度之间找到平衡点,根据业务场景选择合适的策略。

## 面试回答总结

系统缓存策略设计需要从架构、更新策略、常见问题三个维度考虑。缓存架构采用多级设计:浏览器缓存(静态资源,1天-1年)→CDN缓存(静态资源,1小时-1天)→应用本地缓存(Caffeine/Guava,热点数据,5-30分钟,< 1ms延迟)→分布式缓存(Redis,业务数据,5分钟-1小时,1-5ms延迟)→数据库。本地缓存延迟低但容量有限、弱一致性,分布式缓存可扩展、强一致性但延迟稍高,通常组合使用。缓存更新策略推荐Cache Aside模式:读操作先读缓存,未命中则查数据库并写入缓存;写操作先更新数据库,再删除缓存(而不是更新缓存,避免并发脏数据)。对于热点数据可以使用延迟双删策略:更新数据库前后各删除一次缓存,解决并发时的脏数据问题。缓存常见问题有三种:缓存穿透(查询不存在的数据,每次都查数据库)解决方案是布隆过滤器或缓存空值;缓存击穿(热点数据过期,大量请求同时查数据库)解决方案是互斥锁(只让一个线程查数据库)或永不过期(后台异步更新);缓存雪崩(大量缓存同时过期)解决方案是随机过期时间(基础时间+随机时间)、多级缓存、熔断降级。关键是要建立完善的监控体系,监控命中率(目标>80%)、响应时间(目标<5ms)、内存使用率(目标<80%)等指标,并设置合理的告警规则。同时要合理设置过期时间(热点数据5-30分钟,一般数据1-24小时),保证缓存与数据库一致性(最终一致性),预热热点数据,定期优化缓存策略。
