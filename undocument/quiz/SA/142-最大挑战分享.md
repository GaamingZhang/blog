### 核心代码实现

```java
// 支付服务 - TCC接口定义
public interface PaymentService {
    
    /**
     * Try阶段：尝试冻结余额
     */
    @Transactional
    boolean tryFreezeBalance(String txId, Long userId, BigDecimal amount);
    
    /**
     * Confirm阶段：确认扣减余额
     */
    @Transactional
    boolean confirmDeductBalance(String txId, Long userId, BigDecimal amount);
    
    /**
     * Cancel阶段：释放冻结余额
     */
    @Transactional
    boolean cancelFreezeBalance(String txId, Long userId, BigDecimal amount);
}

// 支付服务实现
@Service
public class PaymentServiceImpl implements PaymentService {
    
    @Autowired
    private AccountMapper accountMapper;
    
    @Autowired
    private TransactionLogMapper transactionLogMapper;
    
    @Override
    public boolean tryFreezeBalance(String txId, Long userId, BigDecimal amount) {
        // 1. 检查事务是否已存在（幂等性）
        TransactionLog log = transactionLogMapper.selectByTxId(txId);
        if (log != null) {
            return true; // 已处理过
        }
        
        // 2. 查询账户余额
        Account account = accountMapper.selectByUserId(userId);
        if (account.getBalance().compareTo(amount) < 0) {
            throw new BusinessException("余额不足");
        }
        
        // 3. 冻结余额（乐观锁）
        int rows = accountMapper.freezeBalance(userId, amount, account.getVersion());
        if (rows == 0) {
            throw new BusinessException("冻结失败，请重试");
        }
        
        // 4. 记录事务日志
        TransactionLog txLog = new TransactionLog();
        txLog.setTxId(txId);
        txLog.setTxStatus("TRY");
        txLog.setTxType("PAYMENT");
        transactionLogMapper.insert(txLog);
        
        return true;
    }
    
    @Override
    public boolean confirmDeductBalance(String txId, Long userId, BigDecimal amount) {
        // 1. 检查事务状态
        TransactionLog log = transactionLogMapper.selectByTxId(txId);
        if (log == null || !"TRY".equals(log.getTxStatus())) {
            return false;
        }
        
        // 2. 确认扣减（冻结→扣减）
        int rows = accountMapper.confirmDeduct(userId, amount);
        if (rows == 0) {
            throw new BusinessException("确认扣减失败");
        }
        
        // 3. 更新事务状态
        transactionLogMapper.updateStatus(txId, "CONFIRM");
        
        return true;
    }
    
    @Override
    public boolean cancelFreezeBalance(String txId, Long userId, BigDecimal amount) {
        // 1. 检查事务状态
        TransactionLog log = transactionLogMapper.selectByTxId(txId);
        if (log == null || !"TRY".equals(log.getTxStatus())) {
            return false;
        }
        
        // 2. 释放冻结余额
        int rows = accountMapper.cancelFreeze(userId, amount);
        if (rows == 0) {
            throw new BusinessException("释放冻结失败");
        }
        
        // 3. 更新事务状态
        transactionLogMapper.updateStatus(txId, "CANCEL");
        
        return true;
    }
}

// TCC事务协调器
@Service
public class TccTransactionCoordinator {
    
    @Autowired
    private OrderService orderService;
    
    @Autowired
    private PaymentService paymentService;
    
    @Autowired
    private NotificationService notificationService;
    
    public void processPayment(PaymentRequest request) {
        String txId = generateTxId();
        
        try {
            // Try阶段
            orderService.tryCreateOrder(txId, request);
            paymentService.tryFreezeBalance(txId, request.getUserId(), request.getAmount());
            
            // Confirm阶段
            orderService.confirmOrder(txId);
            paymentService.confirmDeductBalance(txId, request.getUserId(), request.getAmount());
            
            // 异步发送通知（本地消息表）
            notificationService.sendNotificationAsync(txId, request);
            
        } catch (Exception e) {
            // Cancel阶段
            try {
                orderService.cancelOrder(txId);
                paymentService.cancelFreezeBalance(txId, request.getUserId(), request.getAmount());
            } catch (Exception cancelEx) {
                // 记录异常，人工处理
                log.error("Cancel failed: txId={}", txId, cancelEx);
            }
            throw e;
        }
    }
}
```

```java
// 本地消息表 - 通知服务
@Service
public class NotificationServiceImpl implements NotificationService {
    
    @Autowired
    private MessageMapper messageMapper;
    
    @Autowired
    private NotificationMapper notificationMapper;
    
    @Transactional
    public void sendNotificationAsync(String txId, PaymentRequest request) {
        // 1. 保存消息到本地消息表（与业务操作在同一事务）
        Message message = new Message();
        message.setTxId(txId);
        message.setStatus("PENDING");
        message.setContent(buildNotificationContent(request));
        messageMapper.insert(message);
    }
    
    // 定时任务扫描并发送消息
    @Scheduled(fixedDelay = 5000)
    public void sendPendingMessages() {
        // 1. 查询待发送消息
        List<Message> messages = messageMapper.selectPendingMessages(100);
        
        for (Message message : messages) {
            try {
                // 2. 发送通知
                notificationMapper.sendSms(message.getContent());
                
                // 3. 更新消息状态
                messageMapper.updateStatus(message.getId(), "SENT");
                
            } catch (Exception e) {
                // 4. 发送失败，增加重试次数
                int retryCount = message.getRetryCount() + 1;
                if (retryCount >= 3) {
                    messageMapper.updateStatus(message.getId(), "FAILED");
                } else {
                    messageMapper.updateRetryCount(message.getId(), retryCount);
                }
            }
        }
    }
}
```

## 常见问题

### Q1: TCC模式有什么缺点？

**主要缺点**：
1. **业务侵入性强**：需要编写Try、Confirm、Cancel三个接口
2. **开发成本高**：需要考虑各种异常场景
3. **调试困难**：分布式环境下问题定位复杂
4. **性能开销**：多次RPC调用，增加延迟

### Q2: 如何保证TCC的幂等性？

**幂等性保障**：
1. 每个操作都有唯一的事务ID
2. 执行前检查事务状态
3. 数据库唯一索引
4. 状态机模式，只允许特定状态转换

### Q3: 如果Confirm或Cancel失败怎么办？

**失败处理**：
1. **重试机制**：失败后自动重试
2. **定时任务**：扫描异常事务，自动补偿
3. **人工干预**：管理后台提供手动补偿功能
4. **告警通知**：异常事务发送告警

### Q4: 如何选择分布式事务方案？

**选择依据**：
1. **业务场景**：金融场景用TCC，普通场景用消息
2. **一致性要求**：强一致用TCC，最终一致用消息
3. **性能要求**：高并发用消息，低并发可用TCC
4. **团队能力**：考虑开发和运维成本

### Q5: 如何测试分布式事务？

**测试方法**：
1. **单元测试**：测试每个服务的TCC接口
2. **集成测试**：测试完整的事务流程
3. **异常测试**：模拟网络故障、服务宕机等
4. **压力测试**：测试高并发下的表现

## 面试回答总结

我遇到的最大挑战是金融支付系统的分布式事务数据一致性问题。项目背景是日均100万笔交易的支付系统,由于初期设计使用2PC方案性能太差,导致数据不一致率达到0.1%,每天有1000笔异常交易需要人工处理。我首先使用5 Whys方法深入分析根因,发现是技术选型不当、架构设计缺陷、团队能力不足等问题。然后对比了2PC、TCC、Saga、本地消息表等多种方案,最终选择TCC+本地消息表的组合方案:TCC模式处理核心交易流程(支付、充值、提现),保证资金安全;本地消息表处理通知类业务,保证最终一致性。实施过程中,我设计了Try-Confirm-Cancel三个阶段的详细流程,实现了幂等性保证、超时处理、补偿机制、监控告警等完整的异常处理体系。团队协作方面,建立了技术预研、架构设计、开发实施、测试验证、灰度上线的完整流程,通过每日站会、技术评审、知识分享等方式保证项目顺利推进。最终成果是数据不一致率从0.1%降低到0.001%,降低了99%;系统TPS从500提升到2000,提升了300%;人工处理时间从每天2小时降低到10分钟。这个挑战让我深刻理解了分布式事务的复杂性,掌握了多种分布式事务模式,提升了系统设计和团队协作能力。
