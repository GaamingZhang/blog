# HTTP常见状态码

## 概述

HTTP状态码（HTTP Status Code）是HTTP协议中用于表示服务器对客户端请求的响应状态的3位数字代码。当客户端向服务器发送请求时，服务器会返回一个包含状态码的响应头，用于指示请求的处理结果。

HTTP状态码的主要作用包括：
- 告知客户端请求是否成功处理
- 提供请求失败的原因（如果失败）
- 指导客户端进行下一步操作（如重定向）
- 帮助开发人员调试和诊断问题

## 状态码分类

HTTP状态码根据其第一个数字分为五大类，每类代表不同的响应类别：

| 分类 | 状态码范围 | 含义 |
|------|------------|------|
| 1xx  | 100-199    | 信息性状态码，表示请求已接收，继续处理 |
| 2xx  | 200-299    | 成功状态码，表示请求已成功被服务器接收、理解和处理 |
| 3xx  | 300-399    | 重定向状态码，表示需要客户端采取进一步操作才能完成请求 |
| 4xx  | 400-499    | 客户端错误状态码，表示客户端的请求有错误，服务器无法处理 |
| 5xx  | 500-599    | 服务器错误状态码，表示服务器在处理请求时发生了错误 |

接下来，我们将详细介绍各类中最常见和重要的HTTP状态码。

## 1xx 信息性状态码

信息性状态码表示服务器已接收请求，并正在继续处理。这类状态码很少直接显示给用户。

### 100 Continue
- **含义**：服务器已接收请求头，客户端应继续发送请求体（如果有）
- **使用场景**：主要用于客户端发送较大请求体前，测试服务器是否愿意接受该请求
- **工作流程**：客户端发送带`Expect: 100-continue`的请求头 → 服务器返回100状态码 → 客户端发送请求体

### 101 Switching Protocols
- **含义**：服务器同意客户端的请求，将切换到由`Upgrade`请求头指定的协议
- **使用场景**：WebSocket协议升级、HTTP/2协议切换等
- **工作流程**：客户端发送`Upgrade: websocket`请求头 → 服务器返回101状态码并切换协议

### 102 Processing
- **含义**：服务器已接收并正在处理请求，但尚未完成
- **使用场景**：处理耗时较长的请求时，向客户端表示请求正在进行中

## 2xx 成功状态码

成功状态码表示请求已成功被服务器接收、理解和处理。

### 200 OK
- **含义**：请求成功，服务器返回请求的内容
- **使用场景**：最常见的成功状态码，用于GET、POST等请求的成功响应
- **注意事项**：响应内容类型由`Content-Type`头指定

### 201 Created
- **含义**：请求成功并创建了新的资源
- **使用场景**：POST请求创建资源（如创建用户、文章等）
- **注意事项**：响应头应包含`Location`字段，指向新创建资源的URI

### 202 Accepted
- **含义**：服务器已接受请求，但尚未处理完成
- **使用场景**：异步处理请求时（如后台任务、批量处理等）
- **注意事项**：响应应包含请求处理状态的描述或轮询URI

### 204 No Content
- **含义**：请求成功，但服务器不需要返回响应体
- **使用场景**：DELETE请求、更新操作（只需要确认成功即可）
- **注意事项**：响应头不应包含`Content-Length`字段

### 206 Partial Content
- **含义**：服务器成功处理了部分GET请求
- **使用场景**：断点续传、视频流媒体等需要分块下载的场景
- **注意事项**：客户端需发送`Range`请求头指定要获取的内容范围

## 3xx 重定向状态码

重定向状态码表示客户端需要采取进一步操作才能完成请求。

### 301 Moved Permanently
- **含义**：资源已永久移动到新的URI
- **使用场景**：网站域名变更、资源路径重构等
- **注意事项**：浏览器会缓存此重定向，后续请求将直接访问新URI
- **SEO影响**：搜索引擎会更新索引，将旧URL指向新URL

### 302 Found
- **含义**：资源临时移动到新的URI
- **使用场景**：临时维护、A/B测试等临时重定向场景
- **注意事项**：浏览器不会缓存此重定向，每次请求都会先访问原URI
- **SEO影响**：搜索引擎不会更新索引，仍保留原URL

### 303 See Other
- **含义**：资源存在于另一个URI，客户端应使用GET方法获取
- **使用场景**：表单提交后重定向到结果页面（防止表单重复提交）
- **注意事项**：客户端必须使用GET方法访问新URI

### 304 Not Modified
- **含义**：资源自上次请求以来未被修改
- **使用场景**：浏览器缓存验证、减少不必要的数据传输
- **注意事项**：客户端需发送`If-Modified-Since`或`If-None-Match`请求头
- **性能影响**：大幅减少带宽使用和服务器负载

### 307 Temporary Redirect
- **含义**：资源临时移动到新的URI，客户端必须使用相同的HTTP方法
- **使用场景**：需要保持HTTP方法的临时重定向
- **注意事项**：与302不同，307严格要求保持原HTTP方法

### 308 Permanent Redirect
- **含义**：资源永久移动到新的URI，客户端必须使用相同的HTTP方法
- **使用场景**：需要保持HTTP方法的永久重定向
- **注意事项**：与301不同，308严格要求保持原HTTP方法

## 4xx 客户端错误状态码

客户端错误状态码表示客户端的请求有错误，服务器无法处理。

### 400 Bad Request
- **含义**：服务器无法理解客户端的请求
- **使用场景**：请求参数错误、请求体格式错误、URL格式错误等
- **注意事项**：响应应包含具体的错误信息，帮助客户端修正请求

### 401 Unauthorized
- **含义**：请求需要用户认证（身份验证失败）
- **使用场景**：访问需要登录的资源、API密钥无效等
- **注意事项**：响应头应包含`WWW-Authenticate`字段，指示认证方式

### 403 Forbidden
- **含义**：服务器理解请求，但拒绝执行（权限不足）
- **使用场景**：用户已认证，但没有足够权限访问资源
- **注意事项**：与401不同，403表示认证成功但授权失败

### 404 Not Found
- **含义**：服务器找不到请求的资源
- **使用场景**：请求不存在的URL、资源已删除等
- **注意事项**：是最常见的客户端错误状态码

### 405 Method Not Allowed
- **含义**：客户端使用的HTTP方法不被允许
- **使用场景**：尝试对只读资源使用POST方法、API端点不支持某些HTTP方法
- **注意事项**：响应头应包含`Allow`字段，列出支持的HTTP方法

### 406 Not Acceptable
- **含义**：服务器无法生成客户端请求的内容类型
- **使用场景**：客户端指定了不支持的`Accept`请求头
- **注意事项**：响应应包含服务器支持的内容类型

### 407 Proxy Authentication Required
- **含义**：客户端需要通过代理服务器的认证
- **使用场景**：访问需要代理认证的网络资源
- **注意事项**：响应头应包含`Proxy-Authenticate`字段

### 408 Request Timeout
- **含义**：服务器等待客户端请求超时
- **使用场景**：客户端请求太慢、网络连接不稳定等
- **注意事项**：客户端可以重试请求，但应优化请求速度

### 409 Conflict
- **含义**：请求与服务器当前状态冲突
- **使用场景**：并发编辑资源、资源版本冲突等
- **注意事项**：响应应包含冲突的详细信息

### 410 Gone
- **含义**：资源已永久删除，且不会再恢复
- **使用场景**：资源已被彻底移除，且没有替代URL
- **注意事项**：与404不同，410明确表示资源曾存在但现在已删除

### 413 Payload Too Large
- **含义**：请求体过大，服务器拒绝处理
- **使用场景**：上传大文件、请求参数过多等
- **注意事项**：响应头可包含`Retry-After`字段，指示何时可重试

### 414 URI Too Long
- **含义**：请求的URI过长，服务器拒绝处理
- **使用场景**：GET请求参数过多、恶意构造的长URL等
- **注意事项**：建议使用POST方法代替GET方法传递大量参数

### 415 Unsupported Media Type
- **含义**：服务器不支持请求体的媒体类型
- **使用场景**：上传不支持的文件格式、请求体格式错误等
- **注意事项**：响应应包含服务器支持的媒体类型

### 416 Range Not Satisfiable
- **含义**：请求的范围无效
- **使用场景**：断点续传时指定的范围超出资源实际大小
- **注意事项**：响应头应包含`Content-Range`字段，指示资源的有效范围

### 429 Too Many Requests
- **含义**：客户端在短时间内发送了太多请求（限流）
- **使用场景**：API接口限流、防止DDoS攻击等
- **注意事项**：响应头应包含`Retry-After`字段，指示何时可重试

## 5xx 服务器错误状态码

服务器错误状态码表示服务器在处理请求时发生了错误。

### 500 Internal Server Error
- **含义**：服务器内部发生错误，无法完成请求
- **使用场景**：服务器代码bug、配置错误等
- **注意事项**：这是最常见的服务器错误状态码，客户端无法直接修复

### 501 Not Implemented
- **含义**：服务器不支持请求的功能
- **使用场景**：请求使用了服务器尚未实现的HTTP方法或功能
- **注意事项**：与405不同，501表示功能完全未实现

### 502 Bad Gateway
- **含义**：作为网关或代理的服务器，从上游服务器收到无效响应
- **使用场景**：反向代理、负载均衡器等中间服务器场景
- **注意事项**：问题通常出在上游服务器，而非当前服务器

### 503 Service Unavailable
- **含义**：服务器暂时无法处理请求（过载或维护中）
- **使用场景**：服务器维护、流量过载、资源耗尽等
- **注意事项**：响应头应包含`Retry-After`字段，指示服务恢复时间

### 504 Gateway Timeout
- **含义**：作为网关或代理的服务器，从上游服务器获取响应超时
- **使用场景**：上游服务器处理缓慢、网络连接不稳定等
- **注意事项**：与502不同，504表示上游服务器超时而非返回无效响应

### 505 HTTP Version Not Supported
- **含义**：服务器不支持请求使用的HTTP版本
- **使用场景**：客户端使用了过时或不支持的HTTP版本
- **注意事项**：响应应包含服务器支持的HTTP版本

### 507 Insufficient Storage
- **含义**：服务器没有足够的存储空间完成请求
- **使用场景**：上传文件时服务器磁盘空间不足

### 508 Loop Detected
- **含义**：服务器在处理请求时检测到无限循环
- **使用场景**：重定向配置错误、资源之间的循环依赖等

### 511 Network Authentication Required
- **含义**：客户端需要通过网络认证才能访问资源
- **使用场景**：公共WiFi热点认证、企业网络访问控制等

## 相关常见问题及答案

### 1. HTTP状态码分为哪几类？每类的含义是什么？

**答案**：
HTTP状态码分为五大类：
- **1xx（信息性）**：100-199，表示服务器已接收请求，继续处理
- **2xx（成功）**：200-299，表示请求已成功被服务器接收、理解和处理
- **3xx（重定向）**：300-399，表示需要客户端采取进一步操作才能完成请求
- **4xx（客户端错误）**：400-499，表示客户端的请求有错误，服务器无法处理
- **5xx（服务器错误）**：500-599，表示服务器在处理请求时发生了错误

### 2. 请解释200、201、204和206状态码的区别

**答案**：
- **200 OK**：请求成功，服务器返回请求的内容，是最常见的成功状态码
- **201 Created**：请求成功并创建了新的资源，通常用于POST请求，响应头应包含Location字段指向新资源
- **204 No Content**：请求成功，但服务器不需要返回响应体，通常用于DELETE请求或只需要确认成功的更新操作
- **206 Partial Content**：服务器成功处理了部分GET请求，用于断点续传、视频流媒体等场景，客户端需发送Range请求头

### 3. 301、302、303、307和308状态码有什么区别？

**答案**：
- **301 Moved Permanently**：资源永久移动，浏览器缓存重定向，SEO友好
- **302 Found**：资源临时移动，浏览器不缓存，可能会改变HTTP方法（历史问题）
- **303 See Other**：资源存在于另一个URI，客户端必须使用GET方法获取，常用于表单提交后重定向
- **307 Temporary Redirect**：临时重定向，严格保持原HTTP方法，解决302的HTTP方法改变问题
- **308 Permanent Redirect**：永久重定向，严格保持原HTTP方法，解决301的HTTP方法改变问题

### 4. 401 Unauthorized和403 Forbidden有什么区别？

**答案**：
- **401 Unauthorized**：请求需要用户认证，通常是因为未提供认证信息或认证信息无效
- **403 Forbidden**：服务器理解请求，但拒绝执行，通常是因为用户已认证但没有足够的权限访问资源

### 5. 404 Not Found和410 Gone有什么区别？

**答案**：
- **404 Not Found**：服务器找不到请求的资源，可能是资源从未存在或已删除
- **410 Gone**：资源已永久删除，且不会再恢复，明确告诉客户端资源曾存在但现在已彻底移除

### 6. 502 Bad Gateway和504 Gateway Timeout有什么区别？

**答案**：
- **502 Bad Gateway**：作为网关或代理的服务器，从上游服务器收到了无效响应
- **504 Gateway Timeout**：作为网关或代理的服务器，等待上游服务器响应超时

### 7. 如何优化304 Not Modified状态码的使用？

**答案**：
304状态码用于缓存验证，优化方法包括：
- 正确设置`Last-Modified`和`ETag`响应头
- 客户端请求时使用`If-Modified-Since`和`If-None-Match`请求头
- 合理设置缓存过期时间
- 对静态资源启用强缓存和协商缓存结合的策略
- 避免频繁修改资源的文件名（使用版本号或哈希值）

### 8. 当遇到429 Too Many Requests状态码时，应该如何处理？

**答案**：
429表示客户端在短时间内发送了太多请求，处理方法包括：
- 检查响应头中的`Retry-After`字段，按照建议的时间重试
- 实现指数退避算法（exponential backoff）进行重试
- 优化请求频率，减少不必要的请求
- 如果是API请求，考虑申请更高的请求配额
- 实现请求队列，控制并发请求数量

### 9. HTTP状态码在RESTful API设计中有什么应用？

**答案**：
RESTful API广泛使用HTTP状态码表示操作结果：
- **200 OK**：GET请求成功获取资源
- **201 Created**：POST请求成功创建资源
- **204 No Content**：DELETE请求成功删除资源
- **400 Bad Request**：请求参数错误
- **401 Unauthorized**：认证失败
- **403 Forbidden**：授权失败
- **404 Not Found**：资源不存在
- **405 Method Not Allowed**：HTTP方法不被支持
- **409 Conflict**：资源版本冲突
- **500 Internal Server Error**：服务器内部错误

### 10. 如何设计良好的HTTP错误状态码返回机制？

**答案**：
设计良好的错误状态码返回机制应考虑：
- 使用标准的HTTP状态码，避免自定义状态码
- 在响应体中提供详细的错误信息（错误码、错误描述、解决方案等）
- 对不同类型的错误使用合适的状态码
- 为API提供统一的错误响应格式
- 包含足够的上下文信息，帮助客户端调试
- 对于4xx错误，明确指出客户端需要如何修正请求
- 对于5xx错误，避免暴露敏感的服务器内部信息
